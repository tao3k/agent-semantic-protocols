* Purpose

This report turns compact code into a parser-owned implementation track.  The
current optimized Rust example is better than whitespace-trimmed source, but it
is still only an initial projection.  The durable goal is not shorter text by
itself; it is a stable native syntax projection that lets an agent understand
control flow, effects, mutations, and exact expansion points with fewer
commands.

* Problem

The first Rust compact pass moved item query output from source slicing to
parser facts, but the output still exposed gaps:

- branch and loop ownership were visible, but terminal nodes such as
  =continue= and tail expressions were easy to lose
- mutation in compact arms such as =count += 1= was not represented as a
  mutation fact
- macro calls were reduced to the macro name, losing bounded argument evidence
- existing tests asserted that some lines existed, but did not snapshot the
  complete compact projection

Those gaps explain why agents keep falling back to extra owner/text/read
searches after an apparently successful query.  The compact output must carry
enough parser-owned evidence to make the next step obvious.

* Principles

1. Parser facts own compact structure.  Renderers may format, escape, or choose
   transport, but they must not invent branches, exits, calls, or mutations.
2. =--code= and line-protocol =|code text="..."= must serialize the same parser
   projection.  A drift between them is a protocol bug.
3. Compact text is for agent reasoning, not editable source replacement.  Exact
   edits still use read locators or expansion actions.
4. Snapshot tests are the regression boundary.  Every parser capability class
   gets a fixed fixture before broader real-project optimization.

* Capability Ladder

| Step | Capability | Required parser facts |
|------+------------+-----------------------|
| 1 | declaration | item kind, name, visibility, signature, exact read |
| 2 | control flow | if/else, match/case, loop boundaries, nesting depth |
| 3 | terminal/effect | return, break, continue, throw/raise, panic, await, try/? |
| 4 | mutation/binding | let/assignment target, ignored binding filtering, initializer summary |
| 5 | call/macro | callee name plus bounded high-signal argument evidence |
| 6 | expansion | node id, parent id, omitted fact, exact read, next query/read action |

Rust is the current proving ground with =syn=.  TypeScript should project from
the TypeScript compiler API.  Python should use =ast= nodes and preserve
indentation-sensitive block ownership in a parser outline.  Julia should use
JuliaSyntax and stay workspace-managed rather than a global-bin parity target.

* Snapshot Strategy

Each provider needs a small fixture suite for compact parser projection:

- pure compact code: =search owner <path> items --query <symbols> --code=
- line protocol: the same query without =--code=, proving =|code text="..."=
  is the same projection
- JSON packet: =projection.nodes[]=, =omitted[]=, and =expandActions[]= once the
  provider schema exposes them
- token-cost report: measured compact line output and query packet size against
  the raw fixture source, using the root =parser-compact-snapshots= runner

The first Rust fixture locks two functions:

- =branch_and_write= exercises if-let, ignored local macro call, explicit
  return, and tail =None=
- =match_and_loop= exercises loop ownership, match cases, =continue=, mutation,
  and tail =count=

Future fixtures should add async/effect handling, nested closures, unsafe
blocks, struct/enum projection, impl/member boundaries, and large-item outline
truncation.

* Acceptance Gates

Rust provider gate:

#+begin_src shell
direnv exec . cargo test --features cli cli::query::compact
direnv exec . cargo test --features cli cli::query
#+end_src

Cross-language gates should follow the same shape:

- provider-native parser fixture
- snapshot for pure compact projection
- snapshot or schema fixture for metadata-bearing packet output
- real-project replay showing fewer repeated owner/text/read searches

* Open Work

- Add stable projection node ids and parent ids to Rust compact facts.
- Promote projection nodes into =semantic-query-packet.v1= rather than only

- Align TypeScript =--code= output to compiler-owned projection nodes instead

- Add Python AST compact snapshots for function, class, branch, and raise
  flows.
- Use real public-library sandboxes to measure command count, packet size, and
  query retries before and after parser compact improvements.

* 2026-06-02 Update: Rust Pure Code Uses Parser Projection

The Rust provider no longer uses raw source slicing for pure =--code= owner item
queries.  =render_owner_item_code_lines= now serializes the same
parser-owned projection text used by line-protocol =|code text="..."=.

The locked fixture changed from formatted source:

#+begin_example
pub fn branch_and_write(flag: bool, block: &mut String) -> Option<String> {
    if let Some(line) = flag.then_some("ok") {
        let _ = writeln!(block, "{line}");
        return Some(line.to_string());
    }
}
#+end_example

to agent compact projection:

#+begin_example
pub fn branch_and_write(flag: bool, block: &mut String) -> Option<String>
if let Some(line) = flag.then_some("ok")
call writeln!(block, "{line}")
return Some(line.to_string())
tail None
#+end_example

This removes blank lines, indentation, and ignored-binding noise while retaining
the semantic facts the agent needs: declaration, branch, effect call, explicit
return, and tail value.  The next parser work is not renderer compaction; it is
projection metadata: stable node ids, parent ids, exact read locators per node,
and omitted/expand actions.

** 2026-06-02 follow-up: parser node metadata on =|code=

The Rust provider now emits parser-owned node tokens on line-protocol =|code=
rows:

#+begin_example
|code path=src/lib.rs lineRange=3:10 reason=item-query truncated=false nodes=n0:fn:declaration:0:3:10,n1:if:control-flow:1:4:7 text="..."
#+end_example

Pure =--code= output remains compact projection text only.  The metadata lives
on the default evidence-bearing line protocol, where agents need reverse
navigation and hot-block expansion.  The semantic query JSON renderer now
prefers =nodes= to build =projection.nodes[]=, derives =parentId= from depth,
and emits hot-block =expandActions[]= for parser roles such as
=control-flow=, =mutation=, =effect=, and =terminal=.  Compact text
classification remains only as a legacy fallback.

** 2026-06-02 follow-up: root snapshot and token-cost contract

The root repository now owns a portable parser compact fixture contract under
=tests/fixtures/parser-compact/=.  Each case manifest validates against
=parser-compact-case.v1= and points to:

- compact =--code= output
- =semantic-query-packet.v1= JSON from provider-native =query --json=
- =parser-compact-token-cost.v1= JSON generated by the Python runner

The first root case is the feature-level =control-flow-basic= case with Rust,
Python, and TypeScript variants.  Each variant exercises the same semantic
shape: loop, guard return, assignment or augmented assignment, branch, and tail
return.

With the byte tokenizer fallback, the current baseline is:

| Variant | Raw source | Compact line | Ratio | Query packet |
|---------+------------+--------------+-------+--------------|
| Rust | 283 | 189 | 0.6678 | 5716 |
| Python | 286 | 201 | 0.7028 | 8793 |
| TypeScript | 256 | 221 | 0.8633 | 8983 |

The full query packets are intentionally recorded as optimization evidence:
JSON carries node projection and expansion metadata for validators and
artifacts, while agent-facing compact line output remains the low-token
reasoning surface.  The TypeScript variant is now semantic-outline output
rather than a one-line brace-block source compaction, but its ratio remains the
next optimization target.

The root entry point is:

#+begin_src shell
uv run parser-compact-snapshots --case control-flow-basic --tokenizer byte
#+end_src

Use =--refresh= only for intentional fixture updates and =--check-provider= to
replay the real provider command against the expected output.  Future variants
should be added under the same feature-level case directory when they test the
same compact capability.  The root runner remains a command wrapper and
snapshot/token comparator; language providers own all parser compact
implementation.  Provider commands may use manifest placeholders such as
={repoRoot}= to avoid PATH drift, as the Python variant does with
=uv run --project {repoRoot}/languages/python-lang-project-harness --frozen
py-harness=.
