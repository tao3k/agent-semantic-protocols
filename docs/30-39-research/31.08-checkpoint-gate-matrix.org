* Purpose

This checkpoint prevents the hook/query/search work from spreading into one
unreviewable change set.  The current repository has root protocol changes,
root hook runtime changes, sandtable changes, and nested provider changes in
flight at the same time.  Future work should close one checkpoint group at a
time and keep validation evidence attached to that group.

* Checkpoint Groups

| Group | Scope | Status | Required proof |
|-------+-------+--------+----------------|
| hook-query-route | exact direct reads, wrapper line locators, decision route =kind=query= | ready for root gate | root hook tests, decision schema tests, real hook replay |
| query-window-set | =semantic-query-packet=, owner-local query repair, =windowSet= line protocol | partial | root schema tests, sandtable real provider replay, provider query tests |

* Execution Order

1. Close =hook-query-route= first because it is already root-validated and
   protects every provider from direct source-read escapes.
2. Close =query-window-set= next because it turns hook denials into useful,
   bounded provider evidence instead of repeated owner/text searches.
3. Close =handle-policy= after provider gates prove that policy ids, schema
   fixtures, and test cases are first-class facts.
4. Keep =evidence-layer= schemas separate until their provider adapters and
   tests are intentionally selected.

* Worktree Rule

Do not mix root protocol commits with nested provider commits.  If a provider
repo under =languages/= is dirty, validate and checkpoint that provider inside
its own repository before advancing the root pointer or root schema copies.
