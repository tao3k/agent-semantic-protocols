* Purpose

This plan turns the StakGraph comparative study and parser-compact research
into ASP-owned code work.

The target is not to copy StakGraph's tool surfaces.  The target is to use
ASP's strengths:

#+begin_example
native provider parser facts
  -> schema-backed search/read packets
  -> hook-deny guide routes that replay
  -> compact graph/code frontier
  -> sandtable command-count and token-cost evidence
#+end_example

The first implementation slice must reduce real command inflation after
=direct-source-read= denial.  Relation-frontier and causal-path research stays
behind RFC/schema work until provider-owned relation facts exist.

* Implementation Slice 2: Rust Read-Frontier Packet

Goal: wide Rust =direct-source-read= ranges should return a symbol/window/test
frontier, not a single =local-window= range that encourages manual scanning.

Code owners:

- =languages/rust-lang-project-harness/src/cli/query.rs=
- =languages/rust-lang-project-harness/src/cli/semantic_read_json.rs=
- =languages/rust-lang-project-harness/tests/unit/cli/query/hook/range.rs=

Tasks:

1. Replace =range-outline= for wide line selectors with =range-frontier= when
   the parser can identify overlapping Rust items.
2. Use native parser facts to collect symbols overlapping the requested range.
   Do not infer items from string slicing.
3. Rank frontier entries:
   - selected item containing the range first;
   - owner-local named items next;
   - covering tests when available;
   - bounded fallback windows last.
4. Emit compact text:

#+begin_example
[read-plan] mode=range-frontier code=false reason=wide-selector maxWindow=32 alg=symbol-frontier frontier=S.code,T.tests,W.code avoid=repeat-wide-read,manual-window-scan,raw-read
|range path=... requested=1:120 selected=...
|symbol item=run_install kind=fn lineRange=... read=... reason=parser-item
|window id=W lineRange=... read=... reason=bounded-remainder
#+end_example

5. Emit JSON that validates against =semantic-read-packet.v1.schema.json=:
   =readPlan.symbols=, =readPlan.windows=, =readPlan.frontier=,
   =readPlan.avoid=, =readPlan.omit=, =readPlan.maxWindowLines=,
   =readPlan.algorithm=, and =readPlan.nextActions=.
6. For no parser symbols, return =search-repair= or =range-split= with a
   =query --from-hook ... --term ... --surface owners,tests --view seeds=
   next action.

Acceptance:

#+begin_src shell
asp rust query --from-hook direct-source-read \
  --selector src/search/guide.rs:1:80 .
#+end_src

The output must contain =mode=range-frontier= or a schema-valid search repair
plan.  It must not return only =item=local-window kind=range= for parser-visible
Rust source.

* Implementation Slice 3: Cross-Language Read-Frontier Alignment

Goal: Rust, TypeScript, and Python use one read-plan contract.  Language
differences stay in parser-owned facts, not command behavior.

Tasks:

1. Python: preserve the current symbol-frontier behavior and add schema fixture
   coverage for =nextActions= and token-cost.
2. TypeScript: implement equivalent range-frontier using the TypeScript
   compiler API.  Wide =*.ts= reads should produce source-file symbols,
   class/function/method handles, and covering tests where known.
3. Rust: use =syn= item ranges and existing owner/test ranking.
4. Shared: update package-local schema copies only if root schema changes.
   Current v1 fields should be enough for the first slice.

Acceptance:

#+begin_src shell
asp rust query --from-hook direct-source-read --selector <wide-rust-range> .
asp typescript query --from-hook direct-source-read --selector <wide-ts-range> .
asp python query --from-hook direct-source-read --selector <wide-py-range> .
#+end_src

All three must return the same high-level contract:
=readPlan.code=false=, =frontier=, =avoid=manual-window-scan=, and at least one
executable next action or read locator.

* Implementation Slice 4: Parser Compact Expansion

Goal: read-frontier should lead into compact parser output, not raw source.

Tasks:

1. Every =frontier= entry with action =code= must point to a bounded provider
   locator that =query --code= can expand.
2. =query --code= must keep using native compact projection, not raw source
   slicing.
3. The parser-compact snapshot runner should include read-frontier expansion
   cases from the same real-library fixture pool.
4. Token-cost should be recorded for:
   - read-plan packet;
   - selected =query --code= output;
   - combined two-step flow;
   - equivalent raw source range baseline.

Acceptance:

#+begin_example
wide read denied
  -> read-frontier packet
  -> one selected query --code
  -> compact parser projection
#+end_example

This flow should be measured as a command-count and token-count improvement
over manual source-window scanning.

* Implementation Slice 5: Sandtable And Black-Box Gates

Goal: prevent the same escape and command-inflation failures from returning.

Tasks:

1. Add black-box hook tests for:
   - =Read *.rs=, =Read *.py=, =Read *.ts=;
   - =sed -n ... *.rs|*.py|*.ts=;
   - =rtk read -n *.rs= and language variants;
   - =git diff= / file-list patterns that become source dumps.
2. For each deny, assert:
   - guide routes use =asp <language>= in agent-facing output;
   - no =--json= is suggested;
   - no stale provider command shape is suggested;
   - replay exits successfully.
3. Extend real-trigger receipts with:
   - command id;
   - tokenCost per command;
   - readPlanCount;
   - manualWindowScanCount;
   - repeatedAxisCount;
   - firstUsefulLocatorCommandIndex.

Acceptance:

The sandtable should fail if a denied broad source read leaves the agent with a
range-only plan and no executable frontier.

- =search reasoning map= command.

routes are present.

* Success Criteria

The first complete implementation is successful when a real denied wide source
read follows this shape:

#+begin_example
deny direct-source-read
  -> ASP guide route
  -> read-frontier packet with ranked parser-owned handles
  -> one selected compact query/code expansion
  -> no manual overlapping line-window scan
#+end_example

Measured success:

- command count drops for the 31.15 trigger;
- prompt token cost drops versus raw source windows;
- guide replay failure count is zero;
- stale route count is zero;
- each language has at least one schema-valid wide-read frontier fixture.
