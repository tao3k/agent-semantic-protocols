* Purpose

This report records a real Rust workflow where the agent obeyed the project
skill by using =rs-harness= query/search instead of raw source reads, but still
fell into many repeated owner-local symbol probes.

The trace matters because graph synthesis has already improved cross-owner
planning.  The remaining waste is inside one selected owner: the agent does not
have enough parser-owned item-discovery evidence to stop guessing function
names.

* Replayed Trace

* Observed Problems

** Owner selection succeeded, item discovery did not

The selected owner was stable: =src/command/raw_search.rs=.  Repeated commands
were not discovering new project areas; they were guessing top-level item names
inside an already selected owner.

** Symbol seeds are too shallow for large owners

=owner items --view seeds= reported =item=30= but only emitted five symbol
seeds before truncation.  That leaves the agent with incomplete evidence and
encourages guessed queries such as =parse_ripgrep_scope=.

** Miss packets do not repair the query

=status=miss match=none next=revise-query= is correct but underspecified.  The
agent needs parser-owned nearby candidates, prefix candidates, or a suggested
broader query.  Without that, it repeats narrow symbol probes.

** Multi-term owner-local query is missing

The sequence was one logical owner-local question:

#+begin_src text
RawSearchScope | terms_for_raw_search | matching_providers | parse_* helpers
#+end_src

The protocol should support one owner-local query-set that returns per-term
coverage.  It should not force one process startup and parse pass per guessed
symbol.

** Shell syntax leaks into protocol usage

=--query RawSearchScope|terms_for_raw_search= is dangerous unless quoted; the
shell treats =|= as a pipe.  A first-class repeated =--term= flag is safer for
agent-generated commands and easier to explain in hook guidance.

* Root Cause

The search contract has matured at the owner graph level, but query still lacks
an owner-local planning layer.

#+begin_src text
selected owner
  -> incomplete item seeds
  -> guessed exact query
  -> miss without candidates
  -> another guessed exact query
  -> repeated direnv/provider startup
#+end_src

The latency is not mainly Rust parser cost.  It is repeated process startup and
repeated parsing caused by a contract that makes the agent ask the same owner
question many times.

* Improvement Direction

** 31.04-A Add owner-local query-set terms

Provider query commands should accept repeated terms:

#+begin_src shell
rs-harness query src/command/raw_search.rs \
  --term RawSearchScope \
  --term terms_for_raw_search \
  --term matching_providers \
  crates/semantic-agent-hook
#+end_src

Each term should return coverage:

#+begin_src text
|query term=RawSearchScope status=hit match=exact item=1
|query term=parse_ripgrep_scope status=miss match=none candidate=parse_ripgrep_like_scope next=query:parse_
#+end_src

** 31.04-B Add miss candidates

When an exact/contains owner item query misses, the provider should return a
small, parser-owned candidate list.  Candidate ranking can stay conservative:

1. exact case-insensitive item name
2. common prefix or suffix token overlap
3. normalized separator/camel/snake similarity
4. high-ranked owner-local top-level declarations

Candidate output must not invent names from raw text.  It should only name
parser-owned items.

** 31.04-C Add names-only/outline mode

For broad terms such as =parse_=, returning code for six functions is useful
but expensive.  Agents first need an item index, then exact code for the chosen
items.

Expected compact shape:

#+begin_src text
[query] owner=src/command/raw_search.rs terms=1 mode=names-only match=fallback-contains item=6
|query term=parse_ status=hit match=fallback-contains item=6 next=code
|item parse_ripgrep_like_scope kind=fn lineRange=463:544
|item parse_grep_like_scope kind=fn lineRange=546:590
#+end_src

** 31.04-D Canonicalize owner paths

If the command is run with project root =crates/semantic-agent-hook= and selector
=crates/semantic-agent-hook/src/command/raw_search.rs=, the provider should
surface the canonical owner =src/command/raw_search.rs= in a structured field.
This prevents the agent from treating =parserOwner=false= as a reason to
restart discovery.

* Protocol Impact

This belongs in the query contract, not in hook-private logic:

- RFC: document owner-local query-set, names-only mode, and miss candidates.
- Schema: extend =semantic-query-packet.v1= with =queryCoverage=,
  =candidateItems=, and =renderMode=.
- Rust provider: implement repeated =--term=, =--names-only=, miss candidates,
  and canonical owner fields.
- Tests: validate compact output, JSON packet shape, and raw_search real trigger
  reduction.

* Success Criteria

A raw_search implementation turn should not need a dozen single-symbol probes.
The expected bounded flow is:

#+begin_src shell
rs-harness search owner crates/semantic-agent-hook/src/command/raw_search.rs items --view seeds crates/semantic-agent-hook
rs-harness query src/command/raw_search.rs --term parse_ripgrep_scope --term parse_ --names-only crates/semantic-agent-hook
rs-harness query src/command/raw_search.rs --term parse_ripgrep_like_scope --term parse_fd_scope crates/semantic-agent-hook
#+end_src

The first query packet should explain the miss and name the candidate.  The
second should render compact code only for selected exact items.

* 2026-06-01 Landing Notes

The first implementation pass moved the raw_search loop from repeated exact
symbol probes toward an owner-local discovery phase.

** Implemented contract changes

- RFC 005 now documents owner-local repeated =--term= queries, names-only item
  discovery, and parser-owned miss candidates.
- =semantic-query-packet.v1= now allows =outputMode= =code|names|outline=,
  =queryCoverage= entries, bounded =candidateItems=, and matches without
  =code= when the packet is a names/outline discovery response.
- Rust =rs-harness search owner ... items --query ... --names-only= and
  =rs-harness query <owner> --term ... --names-only= now keep item discovery
  inside the selected owner.

** Real raw_search replay

The stale guessed symbol no longer returns only =next=revise-query=:

#+begin_src text

** 2026-06-01 Registry visibility follow-up

The query packet is now advertised through the provider registry instead of
remaining an implementation detail.  =semantic-language-registry.v1= accepts
=query/*= methods and =command="query"=, and the Rust provider registers
=query/owner-items= with:

#+begin_src text

outputSchemaIds=agent.semantic-protocols.semantic-query-packet

#+end_src

This matters for the real raw_search loop because the parent agent can discover
the owner-local query surface from =rs-harness agent doctor --json= before it
starts guessing source reads or single-symbol commands.

** 2026-06-01 TypeScript and Python provider alignment

The root cause of the repeated real-project trigger was not only a missing Rust
query primitive.  The protocol also needed the other maintained providers to
advertise the same owner-local shape so agents can learn one query workflow
instead of language-specific fallbacks.

TypeScript and Python now both expose =query/owner-items= through
=agent doctor --json= and implement a top-level =query= command with repeated
=--term=, =--names-only=, =--code=, and =--json= modes.  The JSON form emits
=semantic-query-packet.v1=, and names-only mode suppresses =|code= lines while
retaining parser-owned item locations:

#+begin_src shell
ts-harness query src/cli/semantic-language.ts \
  --term semanticLanguageRegistryDocument \
  --names-only .

py-harness query src/python_lang_project_harness/_semantic_language.py \
  --term semantic_language_registry_document \
  --names-only .
#+end_src

This is the cross-language contract that prevents agents from falling back to a
chain of broad =search lexical= probes followed by whole-file reads.  A provider
can now answer the narrow question:

#+begin_src text
inside this owner, which parser-owned items match these terms, and what exact
read locators should the agent use next?
#+end_src
