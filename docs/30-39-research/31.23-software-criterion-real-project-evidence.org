* Scope

This records the first ready-provider real-project evidence pass for the agent
software criterion contract.  The run exercises the TypeScript, Python, and Rust
language harness repositories through their public policy check commands.  It
does not claim Julia readiness, blocking severity promotion, compact-output
parity for every signal, or repair success.

The TypeScript pass includes the false-positive audit for
=native-idiom.manual-transform-loop=.  The final run requires a loop to push a
value that depends on the loop variable and requires the enclosing function to
directly return the built collection.  Argument collection and command argument
accumulation are therefore excluded from the final evidence.

* Commands

The measured commands were run twice for each provider.  The second run is
reported as the warm elapsed value; packet sizes and finding counts were
unchanged between the two runs.

#+begin_example
cd languages/typescript-lang-project-harness
node dist/src/cli/main.js check --full --json .

cd languages/python-lang-project-harness
uv run py-harness check --full --json .

cd languages/rust-lang-project-harness
target/debug/rs-harness check --full --json .
#+end_example

* Observations

- TypeScript now has real-project signal coverage without the broad
  =positionals.push(arg)= false positive.  The remaining manual transform loop
  findings are functions that build and return collections, including
  =ownersForHits=, =uniqueOwners=, =sourceTextHits=,
  =collectPublicDataFields=, and =collectSemanticGraphFieldFacts=.
- Python has a narrow criterion finding in the harness CLI, but the JSON packet is
  28.58 MB for one criterion finding.  This reinforces the RFC rule that compact
  text is the default agent surface and JSON remains a validation/artifact
  surface.
- Rust returned parseable JSON with four criterion findings and exit code 1.  The
  non-zero status reflects policy findings from the check command rather than a
  JSON evidence failure, but any CI use of this evidence needs to document that
  status convention.
- Repeated two-run measurements did not change finding counts, signal counts, or
  packet sizes.  This evidence does not prove provider cache replay; it only
  shows stable repeated command output for these projects.

* Promotion blockers

- Do not promote any software criterion to blocking severity from this record alone.
  The record proves localization, not repair usefulness.
- Add compact-output parity checks before agents rely on JSON-only fields for
  the signal vocabulary.
- Track Python packet size separately from signal correctness.  A correct
  finding with a 28.58 MB JSON packet is still too expensive for default agent
  loops.
- Clarify Rust check exit semantics before wiring this evidence shape into a
  pass/fail CI gate.

* Next

- Use this record as the Phase 4 evidence anchor in the quality-policy RFC.
- Add one repair-oriented real-project pass after compact-output parity is
