* Purpose

This report records the live Claude SDK sandtable findings from the Tokio deep
question flow.  The goal is not to make live prompts run longer.  The goal is to
make structure-grounded AST search/query strong enough that an agent reaches
useful hot selectors inside the existing 120 second per-prompt limit.

The live run exposed a useful distinction:

#+begin_example
agent can trigger ASP search/query       -> baseline success
agent still spends too many commands     -> flow design failure
agent times out on meta optimization     -> prompt/pipe contract is too loose
#+end_example

Future work should reduce command count, token cost, repeated search stages,
and broad selector uncertainty by improving the search/query pipe itself.

The active scenario is:

* Exposed Problems

** 31.22-A Token cost is too high for live search calibration

Two completed prompts consumed roughly:

#+begin_example
330k-400k total tokens per prompt
$0.43-$0.53 per prompt
100k-162k stdout bytes per prompt
#+end_example

This is too expensive for iterative sandtable tuning.  The cost is dominated by
large cache-read context plus verbose stream output.  The next research slice
must treat token cost as a first-class budget, not only elapsed time.

** 31.22-B The agent can trigger ASP but still wastes search stages

The first prompt proves that "uses ASP" is not sufficient.  It ran a complex
flow but still repeated =search prime= and ran seven =search fzf= commands.

The desired flow is not:

#+begin_example
prime -> fzf -> fzf -> fzf -> reasoning -> owner -> query -> more query
#+end_example

The desired flow is:

#+begin_example
prime once -> one typed frontier -> one owner/reasoning refinement ->
one exact selector query -> answer
#+end_example

The provider should make the next step obvious enough that the agent does not
explore the same semantic axis repeatedly.

** 31.22-C Field/type semantics are not structural enough

The deep question is about =Vec<scalar>= collection fields.  The current flow
pushes the agent toward text terms such as =Vec=, =scalar=, =collection=, and
=field=.  That produces broad search behavior.

The parser-owned frontier should expose typed AST facts such as:

- field name;
- field type;
- generic constructor, such as =Vec<T>=, =Option<T>=, =HashMap<K,V>=;
- collection-vs-scalar classification;
- enclosing struct/enum/impl owner;
- first selector for the field declaration and enclosing item.

Without that element model, the agent has to rediscover structural facts
through repeated text searches and tree-sitter probes.

** 31.22-D The live prompt can read the guide but still miss the intended pipe

The prompt asks the agent to use =SKILL.md= and ASP search/query.  That works at
a coarse level, but the agent still has to choose among =search prime=,
=search fzf=, =search reasoning=, =query guide treesitter=, and selector reads.

This suggests the live prompt should not carry the whole burden.  The provider
needs an agent-facing pipe command or graph frontier that encodes the intended
next actions, budgets, and avoid rules as data.

The 2026-06-07 live rerun after removing the private =ASP_LIVE_CLAUDE_CLI=
gate proved the runtime environment is no longer the blocker: Claude was found
from the inherited global =PATH= and the SDK produced token/cost evidence.
The flow still failed because =search pipe= rendered reasoning commands before
selector commands, so the agent spent its limited turns on =search reasoning=
and never crossed into =asp rust query --selector ... --code .=.  The durable
fix is selector-first pipe projection: when graph-turbo has an exact selector,
=recommendedNext=, =nextCommand=, =frontierActions=, and =pipeCommands= should
make that selector read the first executable branch action.

After applying selector-first projection in both =asp-graph-turbo= and ASP
=pipePlan=, the single-prompt Tokio live run passed within the existing budget:

#+begin_example
elapsedMs=58813 asp=4 search=3 querySelector=1 reasoning=1 repeated=0 fzf=0
totalTokens=334897 costUsd=0.399453 stdoutBytes=59831
#+end_example

The observed ASP sequence was:

#+begin_example

asp rust search pipe 'Vec scalar collection fields' --view seeds .
asp rust query --selector stress-test/examples/simple_echo_tcp.rs:39:39 --code .
asp rust search reasoning owner-query --owner stress-test/examples/simple_echo_tcp.rs --query 'Vec scalar collection fields' --view seeds .
#+end_example

This validates the ordering fix: the agent crossed into the exact selector
before branch reasoning, did not repeat prime or pipe, and did not fall back to
=search fzf=.  The remaining research problem is evidence quality for Tokio
itself: the selected branch is executable but still not the ideal
field/type/collection owner for the question.

The next optimization moved Rust =semantic-facts= from lexical-owner gating to
a provider-owned =syn= project scan.  Search-pipe candidates now seed priority
only; they do not decide whether field/type/collection facts can exist.  The
Tokio no-token smoke now emits parser-owned facts:

This changes the remaining problem again: the pipe now reaches actual

valid field facts, not recovering from missing parser facts.

The graph-turbo query weighting then started reading provider nested
=fields=, so =elementShape=scalar= participates in =query_node_match_bonus=.
For the same Tokio smoke, field nodes now rank ahead of the collection family
node:

#+begin_example

#+end_example

Field nodes also carry a parser-owned =contextLocator=.  The compact node still
prints the exact declaration locator, but =nextCommand= / =pipeCommands= should
use the bounded context selector when available.  This keeps the next read
structural and avoids a follow-up manual window widening step.

** 31.22-E Non-zero Claude SDK exits can contain valid evidence

Both completed prompts returned exit code 1 but still produced usable stream
summaries, token cost, and command-flow observations.  The scenario now allows
non-zero exit for this live evidence class, but that should not be confused
with success.

The report should distinguish:

- transport/process success;
- model reached max turns or returned non-zero;
- enough command-flow evidence was captured;
- the final answer quality met the prompt goal.

** 31.22-F The third prompt is too broad for the 120 second budget

"Audit the ASP search/query flow" is a meta-task.  It asks the model to both
perform a search and critique the search pipeline.  With the current command
surface, that is too open-ended.

The next version should split this into bounded sub-questions or provide a
native pipe-flow diagnostic that emits the audit facts directly.

* Research Directions

** 31.22-0 Expected ASP flow as benchmark contract

Long-term optimization cannot rely only on the commands Claude happened to run
in one live stream.  The scenario JSON must preserve the expected command flow
as a contract:

#+begin_example
deepQuestionCases[].expectedAspFlow.canonicalCommands
deepQuestionCases[].expectedAspFlow.requiredStages
deepQuestionCases[].expectedAspFlow.forbiddenStages
steps[].expect.pipeFlow
#+end_example

The canonical command list is not a brittle exact transcript.  It is the target
pipe shape: run =search prime= once to establish the project graph context,
start =search pipe= for the concrete question, follow the emitted exact
=query-selector= command when one exists, then use reasoning profiles only for
branch expansion or selector-less fallback.
Concrete owner paths and selectors can vary by fixture, but the stage budget
and forbidden behavior should not vary.

This turns live optimization into a measurable delta:

#+begin_example
expected: asp<=8 search<=4 query<=4 repeated=0 prime=1 pipe=1 reasoning>=1 selector>=1
actual:   extracted from Claude SDK stream observations
result:   pass/fail plus concrete pipeFlow drift messages
#+end_example

Without this JSON-level contract, every run becomes anecdotal and cannot drive
regression testing.

*** Reflection: what the run actually proves

The run proves that =pipePlan= is directionally useful, but not yet a complete
agent contract.

The first prompt passed because the agent could build the graph context with
=search prime=, enter the concrete question with =search pipe=, follow two
=search reasoning owner-query= calls, and finally issue one selector query.
This is the desired shape.  The important signal is not just lower command
count; it is that the search stages became typed and non-repeated:

#+begin_example
prime=1 pipe=1 reasoning=2 selector=1 repeated=0 directRead=0
#+end_example

The second prompt failed in a more useful way.  It reached =prime=, =pipe=, and
=reasoning= with only five ASP commands, but never crossed the boundary into a
selector query.  That means the agent understood the high-level pipe but did
not receive a concrete enough next action to select code.  The pipeline is now
strong enough to reduce exploration, but too weak to force a selector.

The third prompt is not a good optimization smoke test.  It is a meta-prompt
about the search flow itself, so the agent stops after observing =prime= and
=pipe= and explains the surface instead of continuing through reasoning and
selector reads.  It should not be part of the early live loop.

The fundamental problem is therefore not "agent did not use ASP" and not
"disable reasoning to save turns".  The problem is that =search pipe= has to be
a composable query/search pipeline, not merely a graph renderer or a command
template printer.  The pipe should express stages such as =prime=, =search=,
=rank=, =filter=, =expand=, and =project=, then expose selected projections for
the agent.  =frontierActions= and =pipeCommands= are only projections of the
pipeline result.  They are not the definition of pipe itself.

The live timeout after the first =pipeExpr= implementation shows a second
constraint: bounded branch exploration is useful, but branch repetition is not.
The pipe may expose several ranked branches for reasoning; the agent should run
each selected branch at most once, then fan in to a selector decision instead of
replaying the same owner-query commands.

The default branch cap should be top 3.  Those branches should come from
graph-turbo ranking, including sparse graph/path signals, not from raw lexical
walk order.  More than three choices adds token and turn pressure before the
agent has enough evidence to discriminate, and should be reserved for explicit
debug views or later fan-out experiments.

only the =vec scalar meaning= prompt.  The three-prompt matrix is useful after
the pipe contract stabilizes, but during early optimization it triples token
cost without adding proportionate signal.  Keep the single prompt as the smoke
test for =prime -> pipe -> reasoning -> selector=, then restore additional
prompts only when the concrete =pipeCommands= action frontier is strong enough.

** 31.22-A Structure-grounded AST query for fields

Add or expose a field-level query model that can answer questions like:

#+begin_example
fields where type.constructor=Vec
fields where type.classification=collection
fields under owner=<struct-or-module>
first selectors for matching field and enclosing item
#+end_example

For Rust this can start with parser-owned facts from the Rust harness:

| fact | example |
|------+---------|
| =element.kind= | =field= |
| =type.constructor= | =Vec= |
| =type.arguments= | =T= |
| =type.classification= | =collection= |
| =owner.kind= | =struct= |
| =selector.field= | =path:start-end= |
| =selector.owner= | =path:start-end= |

This should reduce the need for repeated =fzf= and broad tree-sitter query
attempts.

** 31.22-B Search pipe as a provider-owned frontier

The agent needs a single compact surface for deep code questions:

#+begin_example
asp rust search pipe '<question>' --view seeds .
#+end_example

or an equivalent reasoning profile.  The output should contain:

- interpreted axes, such as =field-type=, =owner=, =tests=, =deps=;
- ranked next commands;
- query profiles to run, with complete argv;
- avoid rules, especially =repeat-prime=, =broad-fzf=, and =raw-read=;
- a stop condition for when enough selectors have been found.

This is not a replacement for =search fzf= or =query=.  It is a typed planner
frontier that should call those surfaces in a bounded order.

** 31.22-C Token-cost budgets in sandtable reports

The live runner now extracts =tokenCost= from Claude SDK stream JSON.  The next
step is to make those numbers actionable:

- per-step =maxTotalTokensWarn=;
- per-step =maxCostUsdWarn= when SDK cost is available;
- per-scenario aggregate token/cost budgets;
- top-cost command/profile attribution when command-level usage exists;
- fail-on-cost mode for repeated benchmark runs.

Token cost should be audited alongside command count and elapsed time.

** 31.22-D Partial evidence on timeout

Timeout should not erase evidence.  The runner should keep:

- partial stdout/stderr byte and line counts;
- partial ASP command extraction;
- partial =pipeFlow= classification;
- partial token usage if the SDK has already emitted usage;
- timeout reason and elapsed time.

This allows a failed live run to still guide optimization without rerunning the
same expensive prompt.

** 31.22-E Prompt design should become smaller, not longer

The third prompt should be replaced by one or more bounded probes:

#+begin_example
prompt A: run the canonical search pipe and report commands only
prompt B: answer the domain question using only pipe-selected selectors
prompt C: critique missing pipe stages from the recorded command list
#+end_example

Each prompt must stay under 120 seconds.  If a prompt needs more time, the
protocol should be improved, not the budget.

* 2026-06-07 Typed field fact optimization

The latest root cause is that the graph-turbo request was still mostly lexical:
it could rank lines that mention =Vec=, =collection=, or =fields=, but it did
not give the agent an explicit =field -> type -> collection -> hot= graph path.
That left the model to recover structure with extra owner search, fzf, or
tree-sitter guide probes after =search pipe= had already selected branches.

The optimization landed in three layers:

- RFC contract: =owner-query= now allows =field=, =type=, and =collection=
  nodes plus =has_type= and =collection_of= relations.
- Provider fact path: ASP core only forwards bounded candidates to the active
  provider.  Rust field/type/collection facts are produced by =rs-harness search
  semantic-facts= through the Rust harness parser path, not by ASP-owned Rust
  syntax heuristics.
- Python graph-turbo: typed nodes are weighted and projected into top-three
  branch selectors, with query coverage aware of =field/fields= and
  =collection/collections= aliases.

The next live Tokio run should inspect whether =asp rust search pipe 'Vec scalar
collection fields' --view seeds= emits a =frontierActions= selector for a
field-level hot range before launching Claude.  If the pipe output still points
only at comments or modules, the remaining fix is to strengthen the Rust

The local smoke after the provider-owned fact path showed the next branch
projection issue: graph rank still surfaced lexical =collection= and =vec= items
from the same owner even after the =field= node ranked first.  The graph-turbo
projection now stops branch selection on provider-owned =field= nodes when they
exist, so =frontierActions= and =pipeCommands= expose the =scalars= field
selector without spending extra branches on lexical collection/Vec hot ranges.
