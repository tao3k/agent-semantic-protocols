#+TITLE: RFC 10.31 - EvidenceGraph flow against codebase-memory-mcp
#+STARTUP: overview

* Purpose

This RFC makes the comparison target explicit. "CodeBase" in this line of
work means =codebase-memory-mcp=, not generic repository search. The goal is to
prove that ASP's EvidenceGraph flow can reach authoritative selector evidence
with fewer turns when possible, and with stronger receipts than the
=codebase-memory-mcp= reference baseline when the task needs language-native
precision, freshness, or recovery evidence.

=codebase-memory-mcp= is the reference benchmark, not a mechanical pass/fail
floor. ASP must record the reference baseline and the ASP result, then explain
the delta:

#+begin_src text
roundDelta = aspSearchQueryRounds - codebaseMemoryMcpQueryRounds
#+end_src

Negative delta is the preferred result. Zero or positive delta is acceptable
only when the EvidenceGraph packet proves an advantage that the reference
baseline does not provide, such as native parser selector identity, dirty-tree
overlay freshness, recovery receipts, or legal edit/action framing.

The comparison is not won by raw latency or recall alone. It is won by reducing
turns from intent to a legal next action:

#+begin_src text
repository question -> authoritative selector -> proven edit boundary
#+end_src

* Reference baseline

=codebase-memory-mcp= demonstrates a product lesson ASP should absorb, not a
playbook ASP should copy:

#+begin_src text
search_graph / semantic_query / query_graph / search_code
-> trace_path when relationship evidence is needed
-> get_code_snippet when bounded source evidence is needed
#+end_src

Its advantage is that callers are guided through a small graph-oriented tool
surface instead of drifting through broad file search. ASP keeps that
constraint but follows its own route: ReasoningTree for the run trace,
EvidenceGraph for shared proof structure, GraphRoute for navigation, and
NextAction for the exact parent command. The reference baseline is a comparison
lens, while the ASP playbook is EvidenceGraph-native and parser-owned.

* EvidenceGraph flow

The ASP flow is:

#+begin_src text
intent
-> Search Frame seeds
-> native selector expansion
-> dependency / test / artifact topology expansion
-> dynamic overlay reconciliation
-> ranked EvidenceGraph bundle
-> nextCommand / nextClasses / sourceTrace / avoid receipt
-> WhereFrame
-> HowFrame
-> scenario and benchmark receipt
#+end_src

Each stage has a required artifact:

| Stage | Required artifact | Failure if missing |
| intent -> seeds | Search Frame query receipt with =sourceTrace= | broad tool use remains possible |
| seeds -> selectors | parser-owned selector candidates | line ranges or file paths masquerade as identity |
| selectors -> topology | ownership, dependency, test, and artifact edges | no proof of why this owner is the edit boundary |
| topology -> overlay | source-index or dynamic-overlay receipt | stale durable facts can outrank current code |
| overlay -> bundle | ranked EvidenceGraph evidence packet | the agent has facts but no decision surface |
| bundle -> frame | replayable =nextCommand= and =avoid= list | fallback behavior remains model-invented |
| frame -> action | =WhereFrame= and =HowFrame= receipts | the agent can locate code but not choose a legal edit |
| action -> gate | scenario and benchmark receipt | the comparison remains qualitative |

* Comparison criterion

The comparison criterion is turns-to-authoritative-selector.

| Criterion | =codebase-memory-mcp= baseline | ASP EvidenceGraph target |
| turns | graph/search command, optional trace, snippet | Search Frame, selector/topology expansion, exact selector |
| identity | graph node or snippet span | native parser selector plus owner and artifact identity |
| next action | tool playbook guidance | replayable =nextCommand= plus =avoid= and frame receipts |
| freshness | indexed graph plus snippet lookup | durable DB facts plus dynamic overlay reconciliation |
| proof | MCP result shape | Search Frame, EvidenceGraph, WhereFrame, HowFrame, and gate receipts |

ASP only beats =codebase-memory-mcp= when it reaches exact selector evidence
without falling back to path-only, display-line-only, or raw-source search
behavior. Owner candidates and package clusters are useful orientation
evidence, but they are not a winning result unless the frame also emits exact
selector candidates or a selector-bound next command.

* Reference benchmark gate

The benchmark counts ASP search/query commands and reference playbook rounds,
not model messages. Every scenario that compares against =codebase-memory-mcp=
must store both numbers:

| Field | Meaning |
| codebaseMemoryMcpQueryRounds | measured reference playbook rounds for the same question |
| aspSearchQueryRounds | ASP search/query rounds needed to reach authoritative selector evidence |
| selectorEvidence | parser-owned selector or selector-bound next command |
| roundDelta | =aspSearchQueryRounds - codebaseMemoryMcpQueryRounds= |
| evidenceAdvantage | native selector, overlay freshness, recovery, or action-frame advantage when roundDelta is not negative |

Static ASP ceilings still apply:

| Step | Budget | Passing evidence |
| ambiguous intent -> owner cluster | <= 2 commands | owner candidates with =sourceTrace= and =avoid= |
| owner cluster -> exact selector | <= 2 commands | parser-owned selector candidates or selector query |
| exact selector -> edit boundary | <= 1 command | WhereFrame-ready evidence and validation route |

A scenario fails if the preferred next action is raw source access, path-only
inspection, display-line targeting, or generic =asp rg= / =asp fd= fallback.
Those commands may exist for diagnostics, but they cannot be the Search Frame
success path.

The static ceilings are guardrails, not the comparison result. A negative
roundDelta is the preferred compression win. A zero or positive roundDelta must
carry an explicit evidenceAdvantage; otherwise the scenario is a search-design
regression because ASP spent more turns without proving a stronger result.

These facts refine the comparison rule:
- An ASP search result is not valid benchmark evidence unless the active child can run =status= and =search pipe= without same-turn re-registration.
- =sourceTrace= and elapsed timing are part of the intent-quality record; a correct owner with multi-second elapsed time is still a performance follow-up.
- Ranking success is owner-specific: broad implementation facades, config files, and model-mapping files can be evidence, but they are drift when a more precise routing, scenario, or RFC owner exists.
- Ranking correction belongs to Graph Turbo query scoring and receipt projection. Rust source-index code may emit neutral candidates, exact selectors, source traces, and parser-owned evidence fields, but it must not encode intent-domain ranking heuristics that Graph Turbo cannot explain or reproduce for other providers.
- Exact-selector metadata belongs to the main-agent Direct Inventory Lane, not the final evidence lane. When the main agent already has a concrete selector and asks for =--view metadata=, the ASP receipt proves structure, headings, item candidates, and next selectors. It does not prove the bounded content or code. Delegating that command to a subagent is redundant unless the metadata receipt itself asks for fan-out, compression, or cross-document synthesis; if the task needs proof, the next step is a direct =--content= or =--code= fetch by the main agent.
- The main-session binary gate must allow exact-selector direct inventory/fetch commands and continue to deny broad search. Allowed shapes are selector-bound =query= commands with =--code=, =--content=, or document-provider =--view metadata=. Denied shapes include =search=, =--term= query discovery, and hook-owned =--from-hook direct-source-read=. This keeps final proof cheap for the main agent without moving search calibration out of =asp-explore=.
- Language direct fetch requires an item or bounded range selector, such as =path:start:end --code=. A bare file selector can legally return an owner/item frontier and should be classified as Direct Inventory, not final proof, even when =--code= is present.
- Provider surfaces are not interchangeable. Document providers expose selector metadata through =--view metadata=, while Rust source discovery is owned by the dynamic lexical overlay, for example =asp rust search lexical TERM --workspace .=.
- Subagent output should return one compact =GraphRoute= receipt per independent parent intent branch: ranked selector evidence, typed route relations, route state, and one safe parent =next= action. It must not return snippets, source bodies, line ranges, confidence labels, missing-coverage claims, not-found inventories, or cheap-model recommendations.
The main-agent lanes split bounded work by evidence strength:
| Lane | Command shape | Main-agent action | Delegation rule |
| Direct Fetch | exact language selector + =--code= | Run directly and consume bounded code evidence | Do not delegate |
| Direct Fetch | exact document selector + =--content= | Run directly and consume bounded document evidence | Do not delegate |
| Direct Inventory | exact document selector + =--view metadata= | Run directly to identify headings, item candidates, and next selectors | Do not treat as final proof |
| Direct Inventory | exact owner + =items= / =tests= | Run directly and consume selectors or coverage receipts | Do not delegate |
| Direct Receipt | selector-bound =nextCommand= | Run directly unless it fans out across independent axes | Delegate only for fan-out |
| Direct Receipt | session/status/rollout receipt | Run directly to decide lifecycle state | Do not delegate |

The SubAgent lane is reserved for work that changes the evidence shape:
| Search shape | SubAgent responsibility | Return contract |
| ambiguous intent | refine clauses, owners, and next commands | Evidence Bundle |
| independent fan-out axes | search each axis without editing | nodes, edges, selectors |
| large content compression | extract bounded snippets and metadata summaries | snippets plus selectors |
| cross-provider topology | connect deps, tests, artifacts, or overlays | graph edges and receipts |
| long-running search | report heartbeat/progress receipts | sourceTrace and elapsed |
The intended interactive loop is:
#+begin_src text
main intent understanding
-> direct inventory when selectors are already known
-> subagent reasoning only when the next selectors require search calibration
-> main direct fetch of final =--code= / =--content= evidence
-> main edit or answer
#+end_src
The subagent should normally finish the search-calibration pass within a bounded 30-50 second window. After that pass, the main agent should know which exact files, owners, or selectors to fetch directly.

* Implementation sequence

1. Extend Search Frame receipts so owner-oriented search can emit exact selector
   candidates or a selector-bound =nextCommand=.
2. Promote source-index owner-items evidence into the EvidenceGraph packet and
   let Graph Turbo own query-clause, topology, local-evidence, and drift ranking
   before rendering the final action surface.
3. Keep provider-local rankers neutral. Any scenario-driven ranking correction
   must appear as a Graph Turbo score adjustment, trace metric, or receipt
   reason.
4. Add the round-budget scenarios above under the search scenario suite.
5. Treat a path-only or display-line-only answer as an incomplete frame even
   when the owner cluster is correct.
6. Use WhereFrame and HowFrame receipts to prove the final edit boundary and
   legal action, not just the search hit.

* Closure criteria

This RFC is closed when:

- Search Frame can turn the source-index owner-items path into exact selector
  evidence within the round budget;
- the preferred next action is never generic =asp rg= / =asp fd= for a passing
  EvidenceGraph flow;
- every passing scenario records Search Frame, EvidenceGraph, WhereFrame, and
  HowFrame receipts or an explicit structured reason for stopping earlier;
- the comparison with =codebase-memory-mcp= records reference rounds, ASP
  rounds, roundDelta, and evidenceAdvantage instead of relying on a qualitative
  RFC claim.
