#+title: RFC 10.17: Rust SQL Source Index Boundary
#+status: draft
#+date: 2026-06-16

* One-sentence summary

ASP source indexing is Rust-owned DB Engine infrastructure: language providers
emit bounded semantic facts, while Rust owns workspace source discovery, owner
identity, selector ranges, cache invalidation, query acceleration,
Turso-backed stable lookup, and optional external seed import.

* Problem

The early Gerbil Scheme harness proved useful search packet shapes, but it also
let Scheme collect project files, assemble broad workspace search packets, and
recreate source-index behavior. That makes broad search slow, duplicates ranking
and owner discovery in a provider runtime, and produces behavior that other
language providers cannot reproduce.

Provider-local source indexing also creates unclear agent output. When a query
times out or returns a fallback packet, the agent cannot tell whether the
failure came from missing package installation, missing Rust index state, stale
provider facts, or a provider-local recursive scan.

* Contract

Rust owns the workspace source index.

The Rust source index is DB Engine-backed and owns:

- workspace source discovery
- file identity
- owner identity
- selector ranges
- query acceleration
- cache invalidation
- graph topology inputs
- rank inputs
- optional external index import

Language providers may emit language-native owner facts and semantic facts, but
they must not rebuild a workspace source index in Scheme, Python, Julia,
TypeScript, or any other provider runtime.

For Gerbil Scheme, bounded provider facts include imports, definitions, package
metadata, POO forms, structural owner facts, and policy findings. They do not
include walking the whole workspace for every broad search, maintaining
provider-local source caches, ranking owners, or implementing provider-local
owner discovery.

Broad search resolves candidate owners through the Rust DB Engine source index first.
Only after Rust chooses an owner set may the CLI invoke language providers for
bounded semantic fact extraction or owner-local validation.

* Project Scope

The Rust source index covers language project scopes derived from package-management anchor files. Examples include =Cargo.toml= for Rust packages and workspaces, =pyproject.toml= for Python projects, =Project.toml= for Julia packages, =package.json= for TypeScript/JavaScript projects, and =gerbil.pkg= for Gerbil packages.

The cover set is the source and configuration surface owned by those anchored projects. Hidden cache roots such as =.data=, =.codex=, =.devenv=, =.direnv=, =.cache=, VCS metadata, package dependency directories, and build outputs are excluded while discovering anchors and while walking anchored package source directories.

For Cargo workspaces, =workspace.exclude= is part of the project-scope contract. Excluded package roots and nested VCS roots are not indexed as part of the parent workspace by default. Agents that need those sources must activate or index that package root explicitly.

This boundary is part of search quality, not just performance. Agent-facing broad search must not rank vendored Codex source, plugin cache files, or runtime-source mirrors ahead of the active language project unless a future command explicitly requests an external-source scope.

* Runtime-source checkout indexing
Runtime-source checkouts are explicit external-source scopes. They remain
outside the active project source-index scope, because broad project search must
not rank versioned runtime mirrors ahead of the workspace being edited.

=asp cache runtime-source acquire= is the materialization boundary for those
mirrors. After clone/fetch/checkout succeeds, the command must write DB Engine
source-index rows for the checkout root in the active project client DB with
=rawSourceStored=false=. The source-index generation project root is the
checkout directory, not the active workspace root. Rows use the acquired
language id and the supplied =--index-owner= as the owner/provider identity,
retain file hashes with byte length and mtime, skip VCS metadata directories,
and index only source-like files for the acquired language.

The acquisition receipt must report =sourceIndex=refreshed=, the active DB
Engine path, generation id, checkout project root, file count, owner count,
selector count, reused-generation state, and =rawSourceStored=false=. This does
not replace provider structural artifacts: =asp cache import= remains the next command for replaying
provider-emitted structural-index artifacts into the same client DB.

Refresh is incremental. If the latest matching project-root/schema generation
has identical file hash evidence, refresh must reuse that generation and skip
rewriting owner and selector rows. Warm refresh should reuse unchanged file
hashes from the prior generation when path, byte length, and mtime are
unchanged, so unchanged files do not need to be read again just to recompute
SHA-256. It should also retain scope evidence for the provider registry,
provider-declared package roots, source roots, config parents, and indexed
source directories. When that evidence is unchanged, refresh may rebuild
the candidate file list from the latest owner rows and skip provider
=workspace-scope= calls and recursive source-root discovery entirely. Agent-
facing receipts must expose whether the source-index generation was reused.
=asp cache source-index lookup= is the read-side debug and agent-routing
surface for this index. It must read from the active project client DB while
accepting an explicit =--index-root= for external/runtime scopes. The root
command is unfiltered; language-scoped lookup must use the existing facade form
=asp <language> cache source-index lookup= rather than an inner
=--language-id= cache flag. A hit prints
compact candidate rows with path, language, provider, source kind, line count,
and query keys. A miss prints a compact =noOutput= line with the miss state,
query, and index root instead of emitting explanatory prose.
* Cold rebuild transaction and bounded recovery

=asp cache source-index refresh= remains a warm metadata/reuse check. The
explicit =asp cache source-index rebuild= route may assemble and persist a new
generation, but its complete cold path has one fixed end-to-end budget. The
budget covers file reads, owner/query-key and selector assembly, and the Turso
write transaction. A cold rebuild MUST either commit the complete generation
within that budget or explicitly roll back every uncommitted generation, owner,
and selector row. It MUST NOT retain a writer transaction after a timeout or
write failure. An assembly timeout MUST identify its stage and processed/total
file counts; a write timeout MUST identify intended owner/selector row counts.

Stable owner recall is served directly from the committed source-index tables.
Cold rebuild MUST NOT duplicate every source-index owner into a generic Turso
FTS document table as a completion side effect: that projection reopens the FTS
reader per owner and turns one source-index transaction into an unbounded
second indexing pass. Generic search documents remain an overlay/fallback
surface with their own materialization contract; source-index rebuild receipts
describe only the source-index generation they committed.

A cold source-index import holds the DB Engine's source-index write access
guard before it opens a writable Turso handle or bootstraps schema. Concurrent
source-index lookups use a nonblocking read guard: when a rebuild is waiting
or active, they return the existing =Busy= lookup state instead of opening a
competing read-only handle. This is a concurrency boundary, not a fallback to
provider search or a permission to mutate the read path.

The completion boundary is atomic: readers and reuse checks observe either the
previous complete generation or the newly committed complete generation, never
a partially written generation. After a successful cold commit, the writer
MUST checkpoint and truncate the source-index WAL before returning its receipt.
The bounded failure text MUST identify the cold-write phase, elapsed budget,
and intended owner/selector row counts so an agent can distinguish a slow
rebuild from a missing or stale index. Regression gates cover the warm check,
successful rebuild-to-reuse handoff, cold-write rollback, WAL checkpoint, and
concurrent lookup behavior under rebuild pressure.

* V1 source-index performance stabilization

** Hot DDL, dirty refresh, and bounded projection

Source-index schema DDL is a migration concern, not a lookup or write concern.
Every operation MAY probe the canonical layout with one read-only statement,
but it MUST run =CREATE TABLE= or =CREATE INDEX= only when that probe proves the
layout is absent or incomplete. Trace receipts distinguish =schema-migrated=
from =schema-verified= so a hot write cannot silently regress to DDL.

=refresh= is semantically complete for tracked dirty worktrees. A clean
worktree reuses the latest generation without source scanning. A dirty
worktree must content-hash its scoped source and publish a new generation
through the changed-owner batch transaction; it must not return
=cold-required= merely because Git reports tracked edits. A future collection
optimization may reduce the dirty file set, but it may not weaken hash or
owner-membership correctness.

The token projection is the database-owned candidate ranker. Lookup applies the
caller limit in SQL and decodes selector/query-key JSON only for the final
candidate rows. Rust-side JSON projection must never expand a small lookup into
an implicit 64-512 owner reranking scan.

The routing projection retains bounded selector-source tokens when they are
needed for V1 multi-word semantic recall. It stores the deduplicated token list
with each owner and rebuilds a source-index-owned token dictionary in Turso with
one row per token and a compact owner-path set. Candidate selection joins
requested tokens to that dictionary and expands owner paths in SQL; it MUST NOT
materialize one B-tree row per =(token, owner)= pair. The dictionary is replaced
atomically with its owner scope and is not duplicated into generic search
documents. Final selector facts remain available only for already-ranked
owners, preserving V1 lookup packets and owner-local follow-up.

Composite identifier tokens preserve the complete normalized spelling and add
their underscore and hyphen components to the same owner token list. This keeps
SQL candidate recall compatible with V1's final substring scoring: a query for
=report= may nominate =report_chain= before the final ranker decides the exact
owner ordering.

Cold snapshot publication uses parameterized set-based writes: one owner upsert
over the canonical changed-owner JSON payload and one term insert over a
flattened ={ownerPath, token}= JSON payload inside the same transaction. The
write path MUST NOT generate SQL text by interpolating owner, selector, token,
or project values and pass that text to =execute_batch=.
Those value batches create parse and prepare work proportional to payload size,
duplicate the parameterized incremental path, and make a cold import slower as
the repository grows. V1 keeps the same canonical owner/term tables and
generation receipt; only the internal statement shape converges.

Before inserting term rows, the transaction deletes every affected owner path.
The canonical owner builder emits each owner once and term tokens are a set, so
the subsequent term batch has no possible primary-key conflict. It MUST use a
plain parameterized insert rather than performing a redundant per-row conflict
probe. This ordering is part of the transaction's atomicity invariant, not an
optimization that may be skipped by a future incremental path.

** Active-fact storage convergence

V1 generation receipts remain generation-based, but physical source facts converge on an active owner table and an active selector table for each =(project_root, schema_id, schema_version)= scope. The active owner key is =(project_root, schema_id, schema_version, owner_path)=; the active selector key additionally includes =selector_id=. Both retain =file_hash= and every public candidate field.

One rebuild transaction replaces the complete active owner and selector scope, writes the generation receipt, and publishes the active pointer last. A failed replacement MUST roll back and leave the previously published active facts and pointer intact. Pointer-present readers query only active facts; pointer-absent databases retain the existing legacy generation-table fallback until their next rebuild. The public lookup packet, language filtering, scoring, deduplication, limits, and generation receipts do not change.

This eliminates one active-membership write per owner and bounds storage by the current scope instead of historical file hashes. Cached prepared statements are the baseline import API. Parameterized multi-VALUES may be used only when the locked Turso version and the registered large-repository gate show a net improvement; raw SQL data interpolation is prohibited.

*** Registered large-repository evidence

The registered DB gate uses a fresh Git-initialized checkout containing the tracked and unignored worktree files, with the active =.bin/asp= provider bridge. On the 1193-file checkout, =env ASP_SOURCE_INDEX_TRACE=1 direnv exec . asp cache source-index rebuild --workspace <fixture>= recorded =rows-written= at 1029 milliseconds and 5.41 seconds process wall time. This is a 38 percent DB-write improvement over the prior 1663 millisecond v1 baseline, but remains 29 milliseconds above the one-second cold target and therefore is not a cold-gate pass.

The same fixture with 12 added Rust files (1.0 percent dirty) recorded 54 milliseconds for =rows-written= and 3.53 seconds process wall time. A subsequent =direnv exec . asp cache source-index refresh --workspace <fixture>= reached =scope-reused= in 23 milliseconds. The gate records all three command receipts and fails the cold phase until the one-second budget is met; it must not replace the DB phase timing with process wall time, which includes provider startup and scope discovery.

*** Current token-dictionary evidence

The active 499-owner Rust harness workspace produced 21,152 normalized semantic
tokens. A fresh-state =direnv exec . env ASP_SOURCE_INDEX_TRACE=1
target/release/asp cache source-index rebuild --workspace
languages/rust-lang-project-harness= wrote 2,486 token dictionary rows in 75
milliseconds and completed the Turso import phase in 120 milliseconds. The
preceding one-token-per-owner-pair projection wrote the same scope in about 502
milliseconds. A fresh rebuild followed by a dirty refresh of the unchanged
tracked worktree returned =dirty-scope-reused= in 273 milliseconds without
source collection. These are DB-phase and refresh receipts, not process-wall
substitutes for the registered 1,193-file gate above.

The release-only 1,193-owner DB gate produced 20,281 semantic tokens and 4,785
token dictionary rows. Its =rows-written= receipt was 211 milliseconds and the
complete transaction was 220 milliseconds, satisfying the one-second V1 cold
DB-write target. The scheduled large-library runtime workflow executes this
release gate before the 14-corpus command matrix; unoptimized debug builds are
not used as a proxy for the release DB latency contract.

This work stabilizes the existing source-index v1 contract. It MUST NOT create
a v2 schema id, command, packet, generation meaning, or compatibility route.
The existing =refresh=, =rebuild=, lookup result states, and generation
receipts remain the public surface. Internal Turso tables and indexes may be
migrated additively when they preserve those v1 observations.

The source-index write path MUST bootstrap only the base client DB state and
the source-index v1 tables it needs. It MUST NOT initialize graph, overlay,
route-receipt, or generic FTS tables as a side effect. Source-index rows are
the stable recall store; generic FTS remains owned by its own explicit search
materialization path.

V1 lookup MUST push filtering, term matching, rank aggregation, and =LIMIT=
into Turso. It MUST NOT load every owner and selector row for the latest
generation into Rust merely to construct haystacks and score candidates. An
additive source-index term index may support the existing query-key semantics;
source-specific FTS is permitted only when it replaces rather than duplicates
a bounded indexed query path.

V1 rebuild remains generation-based, but unchanged files are reusable facts.
When a prior compatible generation exists, the importer MUST copy unchanged
owner, selector, and term rows with set-based SQL and assemble/write only
changed files. Fresh imports use prepared or batched statements; a new
generation id makes conflict clearing and unconditional per-row delete work
invalid unless a retry is actually resuming that generation.

The v1 performance gate uses a representative real repository and records
scope discovery, file hashing, fact assembly, DB bootstrap, row import,
commit/checkpoint, SQL statement count, WAL bytes, and final process wall time.
For the registered large-repository fixture, the target budgets are: clean
cold rebuild <= 1 second, 1 percent dirty rebuild <= 200 milliseconds, warm
refresh <= 50 milliseconds, and indexed lookup p95 <= 10 milliseconds. A
phase exceeding its share must fail with its phase receipt; aggregate wall time
alone is not sufficient evidence of a pass.

* Missing Index Behavior

A provider command that cannot answer from bounded facts must return an
explicit missing-index or missing-capability response. That response must
include the Rust index command to run next. It must not silently fall back to a
slow recursive source scan.

This is an agent-facing requirement. The agent must be able to tell whether the
next step is:

- build or refresh the Rust DB Engine source index
- install package dependencies
- ask a provider for bounded owner facts
- narrow the query
- inspect a concrete selector

* Dependency Incremental Index

Dependency search is not a prompt-output cache and not a raw source cache. It is
a Rust-owned incremental DB Engine index derived from durable project facts.

The dependency index input keys are:

- package manifest hashes such as =Cargo.toml=, =package.json=,
  =pyproject.toml=, =Project.toml=, and =gerbil.pkg=
- lockfile or resolver hashes such as =Cargo.lock=, =package-lock.json=,
  =pnpm-lock.yaml=, =uv.lock=, =Manifest.toml=, and provider-specific
  resolved-version packets
- provider registry coverage hashes, including package roots, source roots,
  config files, source extensions, and ignored path prefixes
- provider-emitted import and dependency-usage fact hashes
- optional public API or docs-use fact hashes when a provider can emit them
  without fetching or storing dependency source text

The dependency index output is a compact frontier:

- dependency package nodes with current/external/unknown version scope
- manifest and lockfile evidence rows
- import/API usage owner rows
- related test owner rows
- next actions for owner, tests, import, docs-use, crate-source, or
  language-equivalent dependency evidence

The index must store structural facts, query keys, hashes, and locators only. It
must not store raw source text from the active workspace or dependencies.

Incremental refresh must be file-hash driven. A change to a normal source file
may invalidate only import/API usage facts for that file. A change to a
manifest, lockfile, provider registry descriptor, or package root boundary
invalidates the affected dependency package rows and version-scope rows.
Unchanged dependency rows must remain reusable across agent edits.

This is the performance boundary for dependency search. A warm =asp <language>
search dependency <package> --workspace . --view seeds= command should answer
from the Rust dependency index and replay/render path without invoking a full
language provider scan. Cold refresh may invoke bounded provider extractors, but
it must report provider command count, changed input hashes, row counts, and
elapsed time.

* ASP Hot Path v1

** Provider Contract v1

Provider Contract v1 is the only Rust ASP hot-path contract for language
harnesses and freezes the no-compatibility boundary. A provider is the authority
for native parser/typechecker/project manager facts and bounded owner-local
projection only. It must not own broad workspace search, source indexing, fact
indexing, dependency indexing, route selection, graph ranking, prompt graph
rendering, cache invalidation, source content cache, or dependency cache.

The shared schema is [[../../schemas/semantic-provider-contract.v1.schema.json][semantic-provider-contract.v1]]. Provider registrations and scenario receipts should validate against this schema before advertising a method to agents.
Required provider commands:

- =asp <language> agent doctor --workspace . --json= reports provider-local
  capability and contract metadata.
- =asp <language> index export --workspace . --scope changed|owner|workspace --json=
  exports bounded native facts for Rust import.
- =asp <language> search owner <owner> items --query '<terms>' --workspace . --json=
  returns owner-local parser facts only.
- =asp <language> query --selector '<structuralSelector|nativeFactId>' --workspace . --view skeleton|outline|signature|comments|names|unsafe-blocks|diagnostic-capsule|flow-capsule --json=
  performs exact provider projection after Rust has selected identity.
- =asp <language> query --selector '<structuralSelector|nativeFactId>' --workspace . --code=
  returns exact code only for parser-owned identity.

Forbidden provider commands and advertised methods:

- no =query/direct-source-read=
- no =read-packet= output mode for provider projection
- no provider =cacheReplay=true=
- no executable =path:start:end= or =displayLineRange= selector
- no wildcard or workspace broad search through provider query

Executable identity is limited to =structuralSelector= and =nativeFactId=.
=sourceLocatorHint= and =displayLineRange= remain display/transport evidence
only. If an agent has only =path:start:end=, it must return to Rust ASP
SourceIndex/FactIndex materialization instead of invoking provider code.
The source-index boundary extends to the whole agent-facing search/query hot path. The durable shape is:
#+begin_src text
language harness = bounded native parser/typechecker/project-manager facts
Rust ASP         = source/fact/dependency index, route, materialize, render, receipt
Python graph    = graph algorithms, ranking, ablation, and sparse graph reuse
#+end_src

The root performance problem this RFC prevents is a split control plane: some searches using Rust cache replay, some using provider-local workspace scans, some using finder candidates, some using optional semantic-facts side channels, and some using graph-turbo request builders that no longer share one normalized fact model. Optimizations must land in the Rust client/index/materializer path unless they are a provider-local bounded projection optimization.

Provider CLIs are fact authorities, not performance control planes. A language provider may expose:
- =agent doctor --json= for capabilities, source roots, ignored prefixes, and project-manager evidence
- =index export --scope changed|owner|workspace --json= for bounded native fact packets
- =search owner <owner> items --json= for owner-local facts only
- =query --selector <structural-selector|native-fact-id> --view ... --json= for bounded projections
- =query --selector <structural-selector|native-fact-id> --code= for exact code transport

Provider CLIs must not own broad recursive source search, workspace ranking, graph-turbo orchestration, source-content caches, dependency caches, or prompt graph rendering. If the Rust index state needed for a command is missing, the command must return an explicit missing-index or missing-capability response with the next Rust index command. It must not silently perform a slow provider-local workspace scan.
* Native Fact Index Import
Provider-emitted =semantic-native-syntax-fact-index.v1= packets are imported into a Rust-owned fact index. The fact index is the bridge from native parser/typechecker facts to language-neutral search/rank/materialize rows.

The normalized row model must preserve:
- language id and provider id
- owner id and owner path
- native fact id
- structural selector
- source locator hint and display line range as non-executable hints
- query keys, roles, relations, visibility, test/export flags, and fact hash

Agent-facing candidate selection must use this normalized fact model instead of reducing parser facts to only =path + line + text=. Line ranges may be rendered as display evidence, but they are not executable selector identity. Exact code materialization requires a structural selector or native fact id, except for explicitly marked degraded fallback routes.
* Projection Materializer
Provider execution is allowed at the final projection boundary. The Rust materializer chooses a typed action from indexed evidence and calls the provider only for selected exact projections.

Materialization rules:
- owner and item exploration should prefer skeleton, names, outline, diagnostic, unsafe, or flow capsule projections before code
- =query-code= requires exact parser identity or native fact identity
- =sourceLocatorHint= may be passed as transport evidence, but must not be treated as an executable selector by default renderers
- multiple exact projections should be batched into one provider process when the provider supports batch query
- prompt stdout and machine packets should be produced in one provider invocation through =--packet-out= when supported

If a provider lacks batch projection or packet-out support, Rust may use a compatibility fallback, but the receipt must report the fallback mode so sandtable and CI can keep the debt visible.
* Cold Warm Hot Gates
Search/query performance is verified in three states.

Cold path may build or refresh source, fact, or dependency indexes. Its receipt must report provider process count, provider elapsed time, indexed files, indexed owners, indexed facts, changed input hashes, Turso write counts, graph-turbo time, render time, packet-out mode, and the explicit refresh command that produced the state.

Warm broad search must answer from Rust-owned source/fact/dependency indexes before exact projection. It should not start a language provider for workspace owner discovery, dependency topology, broad search ranking, or graph-turbo input assembly. Warm receipts must expose =sourceIndexHit=, =factIndexHit= or =dependencyIndexHit= as applicable, =providerProcessCount=, first route, graph-turbo time, and render time.

Hot path reuses the same Rust index generation plus graph-turbo structural backend fingerprint. It should only re-rank or render the selected frontier and should be millisecond-level for broad routing.

The shared performance gate schema is =schemas/semantic-hot-path-performance-gate.v1.schema.json=. Scenario and sandtable tests should use it to prove:
- =warmBroadProviderProcessCount == 0=
- relevant index hit flags are true on warm paths
- first route is not =prime= when owner, selector, dependency, failure, or test evidence already exists
- exact code routes require parser-owned identity
- line ranges are display hints, not executable selectors
- projection provider process count is bounded, preferably one batch process
- packet-out compatibility fallback is explicit
* gtags Integration

GNU Global/gtags can participate only as an optional Rust-side source-index
seed.

When available, Rust may import =GTAGS=, =GRTAGS=, and =GPATH= facts into the
SQL index as candidate definitions, references, and path ownership hints.
=gtags= is not a required provider dependency and is not an alternative owner
for ASP search semantics.

If =gtags= is absent, stale, or incomplete, Rust still owns the SQL source index
and builds it from filesystem manifests, parser packets, and bounded language
fact extractors.

* Implementation Sequence

1. Define the shared SQL source-index packet and registry descriptor in
   =schemas/=.
2. Implement the Rust SQL source-index owner and cache invalidation path.
3. Add optional =gtags= import behind capability detection.
4. Add the Rust dependency incremental index schema, DB tables, refresh command,
   and warm dependency-search replay path.
5. Change language providers so broad search asks Rust for owner sets before
   provider-local semantic extraction.
6. Remove or downgrade provider-local workspace indexing to bounded owner fact
   extraction.
7. Add CI and real-project evidence that broad search does not call provider
   workspace scans and that missing index state yields explicit next commands.

* State layout materialization
State paths must be derived from the shared state layout, not rebuilt in each
cache, activation, source-index, or history module. Runtime materialization
must resolve the config-owned layout once and create directories from that
layout instance instead of repeating git/config discovery per directory.
* Non-goals

This RFC does not require =gtags= in developer environments or CI.

This RFC does not make Rust parse every language's semantic forms. Rust owns
source indexing and graph/rank infrastructure; language providers still own
language-specific semantic facts.

This RFC does not make provider binaries resident services. Provider calls stay
bounded and explicit.
* Source-content cache removal
The Rust SQL source index must not store full source text as a cache product.
Source contents change too quickly under agent edits, and replaying cached source
text creates stale-code risk. The durable cache surface is structural:
- project/workspace scope
- provider identity
- language id
- file identity
- owner identity
- selector ranges
- symbol and item names
- graph/rank inputs

Broad search can use this structural index to find owners quickly, then read or
query the current workspace file through the provider-owned selector route when
code content is needed. This keeps cache invalidation cheap and makes the agent
output clear: cached structure may accelerate discovery, but current code still
comes from the active workspace.
* Provider activation boundaries
Source discovery is driven by the active provider registry snapshot, not by
hard-coded language heuristics inside the SQL indexer. A provider descriptor may
declare source roots, config files, source extensions, and ignored path prefixes.
The Rust indexer uses those fields to collect candidate files and to reject
provider-ignored paths before inserting rows.

Manual activation snapshots supplied by tests or downstream projects are
authoritative inputs. Loading such a snapshot must not silently resync it from
the current workspace and erase the downstream provider boundary. This makes the
same source-index code usable for fixture calibration, local projects, and
larger dependency trees.
* Project context and state layout
Rust-owned indexing and cache mutation commands must resolve project identity
through a single project context before opening state. The context owns:
- current working directory
- git toplevel
- project home
- project environment status
- client state layout

The config crate owns the underlying runtime layout. Client, runtime,
activation, history, and source-index code may wrap that layout for local API
ergonomics, but they must not reimplement git toplevel discovery, project home
selection, =.envrc= detection, or protocol state paths independently.

The git toplevel is the workspace boundary for source-index refresh and other
workspace-owned mutation paths. Paths outside that boundary are invalid unless a
future command explicitly activates another workspace root.

Project environment variables such as =PRJHOME= and related =PRJ*= values are
only trustworthy when the git toplevel contains =.envrc=. A nested =.envrc=
inside a package directory is not enough to make the workspace project
environment available. This keeps package-local development shells from
silently changing ASP state ownership.

State paths must be derived from the shared state layout, not rebuilt in each
cache, activation, source-index, or history module. The layout owns the state
root, client cache directory, cache manifest path, artifact directory, and
Turso database parent. The state root is the protocol directory, normally
=<git-toplevel>/.cache/agent-semantic-protocol=. The client cache directory is
=<state-root>/client= and the artifact directory is =<state-root>/artifacts=.
Mutation and audit paths should consume the layout before opening, rewriting,
or backfilling state.
* State layout materialization
State paths must be derived from the shared state layout, not rebuilt in each
cache, activation, source-index, or history module. The layout owns the state
root, client cache directory, cache manifest path, artifact directory, and
Turso database parent.

The state root is the protocol directory, normally
=<git-toplevel>/.cache/agent-semantic-protocol=. The client cache directory is
=<state-root>/client= and the artifact directory is =<state-root>/artifacts=.
Mutation and audit paths should consume the layout before opening, rewriting,
or backfilling state.

Runtime materialization should resolve the config-owned layout once and create
directories from that layout instance. It should not call per-directory helpers
that each repeat git/config discovery.
* V1 immutable file facts and active membership
** Import transaction and WAL maintenance

The import transaction commits immutable facts, active path membership, generation
metadata, and the active pointer atomically. A source-index import MUST NOT run a
synchronous =wal_checkpoint(TRUNCATE)= after that commit: checkpoint truncation is
unbounded on a long-lived client DB and is cache maintenance rather than import
work. Turso WAL readers observe the committed pointer directly; normal WAL
auto-checkpoint policy owns maintenance outside the source-index hot path.

The import trace MUST expose transaction and deferred-checkpoint boundaries so a
real checkout benchmark can distinguish fact materialization from background WAL
maintenance.

The source-index public protocol remains v1.  This is an internal Turso
layout migration, not a new schema id, command, receipt, or compatibility
route.

The existing generation-keyed owner and selector rows make a one-file rebuild
copy every unchanged row into a new generation.  Turso measurements show that
this copy is slower than the prepared full write, so v1 must not implement
incremental rebuilds by copying a generation.

The replacement layout has three internal responsibilities:

- an immutable file-fact identity keyed by project, source-index schema,
  owner path, and file hash;
- owner and selector facts owned by that file fact rather than by a generation;
- one active membership per project/schema/owner path that points at the
  current file fact, together with an active-generation pointer used for
  existing generation receipts and reuse checks.

On a dirty rebuild, Turso inserts facts only for changed or new files, upserts
only their active memberships, removes memberships for deleted files, records
the new generation receipt, and advances the active pointer in the same
transaction.  It does not copy unchanged owner or selector rows.  Initial
bootstrap may dual-write the existing v1 owner/selector tables until the
reader migration is complete; reads then resolve the active pointer and active
memberships while preserving the current candidate, receipt, and JSON shapes.

The migration must be additive and restart-safe.  A read may use the legacy
generation rows only until a complete active membership is recorded for the
project/schema.  Write failure must leave both the active pointer and active
membership unchanged.  Rebuild, refresh, lookup, and their public receipts
remain v1 throughout.
