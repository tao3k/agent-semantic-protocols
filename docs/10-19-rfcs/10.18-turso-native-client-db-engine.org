#+TITLE: RFC 10.18 - ASP DB Engine v1 and Turso Backend
#+AUTHOR: agent-semantic-protocols
#+DATE: 2026-06-28
#+OPTIONS: toc:2

* One-sentence summary
ASP owns a DB Engine v1 contract for durable search, state, topology,
artifact evidence, and future writing workflows; Turso is the production
backend and the public backend token is =turso=.

* Contract
The DB Engine is the product boundary. Providers and command facades emit
schema-valid facts; search, query, recall, artifact review, and writing
workflows consume typed DB Engine read models. Public contracts must not expose
old DB handles, old backend tokens, or project-local cache fallback routes.

The stable public contract is:
- backend token: =turso=
- schema version: =1=
- state root: State Core under =ASP_STATE_HOME= or =$HOME/.agent-semantic-protocols=
- active DB file: =live/client/client.turso=
- active manifest: =live/client/manifest.json=
- durable artifact root: the paired State Core workspace artifact directory

=futureBackend=turso= remains only as a v1 report field during the cutover
window. It is not a second backend and not a version suffix.

* Turso rationale
Turso is useful because it matches the agent loop better than a synchronous
single-writer storage shape:
- async local access aligned with Tokio provider transport;
- concurrent source and structural index imports;
- observable cache invalidation through durable DB state;
- Turso-owned FTS/BM25 for stable generation search;
- overlay-search capability through the DB Engine boundary;
- future sync, encryption, CDC, and vector experiments without changing State
  Core paths.

ASP should consume Turso-owned search semantics through the DB Engine. It
should not introduce a long-lived direct =tantivy= backend, and it should not
let dynamic dirty-code overlay writes pollute stable project generations.

* Implementation checkpoint
The implementation target is now executable in =agent-semantic-client-db=:
- =ClientDbEngine= resolves State Core paths and exposes backend, layout
  version, repo/workspace/scope identity, DB path, manifest path, client dir,
  artifact path, and backend capability reports.
- =ClientDbEngineBackend= is the adapter seam; the Turso adapter owns active
  =client.turso= traffic in production builds.
- =schemas/semantic-db-engine-report.v1.schema.json= and
  =schemas/agent-semantic-client-receipt.v1.schema.json= are the analyzer-facing
  report contracts.
- =asp cache status= renders a compact =|dbEngine= row only as CLI shorthand;
  receipt JSON is the durable analyzer surface.
- =ClientReceipt.dbEngine= embeds the DB Engine report for cache status, import,
  flush, invalidate, writeback, and replay receipts.

The Turso backend currently owns these traffic families behind the DB Engine
facade:
- cache-generation read model;
- syntax-query replay object;
- provider-command selection;
- artifact events;
- agent session registry storage;
- graph entity and edge projections;
- stable search documents;
- overlay documents;
- source-index generation, owner, selector, and search projection;
- structural-index read model;
- route receipts.

* Source-index and structural-index boundary
Source-index stable data is written through
=ClientDbEngine::refresh_source_index_import_from_client_dir=. The method
bootstraps =client.turso=, writes generation/owner/selector rows, persists the
source-index EvidenceGraph into =asp_graph_entity= / =asp_graph_edge=, and
writes =source-index= namespace rows into =asp_search_document= for stable FTS
lookup.

Structural-index data follows the same DB Engine rule: packet parsing and raw
source rejection stay in the owning crate, while the DB Engine owns durable
generation rows, graph projection, search documents, and lookup facades.

* Dynamic overlay boundary
Dynamic dirty-code search is not a stable-generation write path. The router
should merge evidence in this order:
1. receipt/context anchors;
2. active overlay facts;
3. dirty-file provider parse facts;
4. stable Turso FTS/BM25 over current generations;
5. semantic/vector fallback when available.

Overlay state is keyed by repo/workspace/session/base generation and should
store bounded replayable evidence only: dirty hashes, patch receipts,
changed/deleted selector ids, parse-delta summaries, active failures, and
temporary overlay-local entity ids. Accepted overlays compact into a new stable
generation; rejected overlays remain session-scoped audit evidence.

* Command and crate ownership
=crates/agent-semantic-protocol/src/command/*= must stay a thin public facade:
parse CLI arguments, dispatch to package-owned services, and render typed
reports. It must not construct DB paths, open the DB, rank search candidates, or
hard-code language-specific database behavior.

Ownership is:
- =agent-semantic-runtime::state_core= resolves State Core identity and paths;
- =agent-semantic-client-core= exposes client-facing State Core services;
- =agent-semantic-client-db= owns DB Engine schema, open/bootstrap, read/write
  sessions, and Turso adapters;
- =agent-semantic-search= owns search routing, overlay merge, candidate
  projection, and source-index lookup orchestration;
- language harnesses own native parse facts and shared schema packets.

* Validation gates
The Turso cutover is accepted only when these gates pass:
- public docs/schemas contain no old backend token or version-suffixed Turso
  backend name;
- =cargo test -p agent-semantic-client-db= passes;
- =cargo test -p agent-semantic-client-core= passes;
- client receipt and DB Engine report schema tests pass;
- =asp state locate --json= and =asp cache status= expose State Core paths and
  =backend=turso=;
- source-index cold functional gate refreshes a clean workspace and then search
  returns =source=source-index= instead of silently falling back to finder;
- overlay search uses DB Engine/search-adapter semantics and keeps stable
  generations clean;
- per-language cold functional gates record elapsed time, provider spawn count,
  stdout bytes, route source, and fallback reason.
* Turso Best Practices Reference

The downstream implementation guide for Turso-specific concurrency, retries,
multi-process CLI access, and future MVCC writer runtime is maintained in:

- [[file:10.29-turso-db-engine-best-practices.org][10.29 Turso DB Engine Best Practices]]

Any DB Engine code change that touches Turso =Builder= flags, connection
pragmas, transaction mode, retry classification, or FTS schema must first
refresh the 10.29 upstream-doc audit against the official Turso docs and the
workspace-resolved =turso= crate.

This RFC remains the architecture target: Turso replaces the retired RustSQL /
SQLite implementation as ASP's DB Engine. The best-practices guide is the
normative downstream implementation policy for Turso opens, transaction retry,
multi-process CLI access, pressure gates, and the future high-throughput AI
writer runtime.
