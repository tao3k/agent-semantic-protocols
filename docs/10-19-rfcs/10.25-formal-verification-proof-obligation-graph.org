* Summary

This RFC defines how formal verification enters ASP.

Formal verification is not a replacement for search, policy, scenarios, or
runtime tests. It is a high-trust evidence source for branch legality. It helps
ASP stop defensive engineering by converting stable architecture claims into
explicit proof obligations, verified facts, and compact receipts that can be
used by the Code Reasoning Tree before an agent writes code.

The practical model is:

- EvidenceGraph answers what evidence exists.
- DynamicTopology answers what the target means inside the current architecture.
- GraphRoute answers how to jump through that topology with minimal tokens.
- Policy/Scenarios answer how to change code with high engineering quality.
- Formal verification adds proof-backed facts that can make a branch legal,
  illegal, or unproven.

The key design rule is selective formalization. ASP must not try to formalize
all code. It should formalize stable contracts that sit behind fast-changing
code:

- schema invariants
- state-machine transitions
- parser and renderer round-trip laws
- selector identity rules
- cache/index invalidation rules
- dependency and ownership constraints
- high-value safety and correctness properties

The checkout is a Python client and CLI for AXLE, the Axiom Lean Engine API. It
does not vendor a Lean project directly. Its value for ASP is the interface
shape around proof verification and Lean metaprogramming utilities:

- =verify_proof=: validates candidate Lean content against a trusted formal
  statement.
- =check=: compiles Lean code and reports compiler/tool findings, but this is
  weaker than strict proof verification.
- =extract_decls=: splits Lean code into declarations and reports declaration
  metadata, dependencies, type hashes, tactic counts, heartbeats, timings, and
  messages.
- =sorry2lemma= and =have2lemma=: turn incomplete proof state or local proof
  steps into standalone lemmas.
- =repair_proofs=: attempts deterministic proof repair, but should be treated
  as suggestion/evidence rather than a hard policy source.

The AXLE technical report frames AXLE as cloud infrastructure for Lean 4 proof
manipulation, extraction, and verification. The useful idea for ASP is not the
cloud service itself. The useful idea is the proof-tool interface:

- separate a trusted statement from a candidate implementation
- verify the candidate against that statement
- expose proof failures as structured messages
- expose declaration graphs and proof metadata as reusable data
- preserve a receipt that can be referenced without dumping full proof text

* Why This Matters For ASP

ASP is trying to solve two core problems for agents:

1. Where to change code.
2. How to change code well.

Search, Query, EvidenceGraph, DynamicTopology, and GraphRoute mainly solve
"where". Policy and Scenarios mainly solve "how". Formal verification improves
both, but it must enter through facts, not through broad hooks.

The failure mode we want to avoid is defensive engineering:

#+begin_quote
The agent sees an uncertain consumer boundary and adds more optional checks,
fallbacks, duplicated validation, or lossy compatibility code. The code may
pass tests, but the architecture gets weaker because the real invariant was not
made explicit.
#+end_quote

Formal verification changes the decision surface:

#+begin_quote
If the invariant is supposed to hold at the producer boundary, ASP should create
or locate a proof obligation for that invariant. If the proof receipt exists,
HowFrame can tell the agent not to add a consumer fallback. If the proof receipt
does not exist, HowFrame can route the agent to the owner boundary where the
invariant must be established.
#+end_quote

This is a different level of guidance than a generic policy warning. It says:

- this branch is illegal because it hides the invariant at the wrong boundary
- this branch is legal because the invariant is proven or scenario-backed
- this branch is unproven and needs a receipt before the edit should proceed

* Compilation Is Not Verification

AXLE exposes the distinction that ASP must preserve:

- =check= answers whether Lean code compiles and reports findings.
- =verify_proof= answers whether candidate content satisfies a formal statement.

For ASP this maps to:

- tests/scenarios can say a behavior was observed
- policy can say an edit shape is recommended or forbidden
- formal proof can say a property follows from a model or statement

These are not interchangeable.

An agent should not treat "it compiled" as "the architecture invariant holds".
It should also not treat "policy said this is preferred" as "the proof exists".
The receipt must preserve evidence class and trust level.

* Trust Levels

Formal proof receipts must carry a trust level. Lean proof validation has
different threat models:

- =compile=: code elaborates and builds.
- =axiom-audited=: proof does not rely on unwanted axioms or =sorry=.
- =verify-proof=: candidate satisfies a trusted formal statement.
- =external-checker=: proof was rechecked by a stricter external checker path
  such as lean4checker/comparator style validation.

ASP should start with honest-proof assumptions for internal engineering
invariants. For untrusted or externally supplied proofs, the receipt must be
marked as requiring external checker validation before it can prune branches.

* Core Objects

** ProofObligation

A ProofObligation is a stable question that should be answerable by a proof,
scenario, benchmark, or human review.

It is not a line-number target. It must be selector-owned.

Minimal fields:

#+begin_src json
{
  "schema": "semantic-proof-obligation.v1",
  "id": "proof-obligation:<stable-id>",
  "project": "<project-id>",
  "topologySlice": "<topology-slice-id>",
  "ownerSelector": "<selector>",
  "intent": "<why this obligation exists>",
  "claim": "<plain-language claim>",
  "formalStatementRef": "<artifact-ref-or-inline-hash>",
  "evidenceClass": "formal-proof",
  "trustRequired": "verify-proof",
  "createdFrom": ["policy-fact:<id>", "scenario:<id>", "topology:<id>"],
  "status": "open|verified|disproved|stale"
}

Examples:

- Every search packet identity is selector-based and does not require line
  numbers for stable identity.
- Rendering a compact graph preserves node identity.
- A Turso cache invalidation transition cannot leave an active stale manifest
  for the same selector.
- A provider command result either contains a schema-valid error receipt or a
  schema-valid evidence packet.

* Progressive Agent Disclosure

Formal verification is valuable only if it stays compact.

Agent-facing disclosure should have three levels:

** L0: Decision Fact

#+begin_src text
FormalFact: selector identity is verified at producer boundary.
Do not add line-number fallback in consumers.
Route missing selector to provider contract or schema migration.
#+end_src

** L1: Receipt Summary

#+begin_src text
ProofReceipt search-packet-selector-identity:
  checker: axle.verify_proof
  trust: verify-proof
  appliesTo: semantic-search-packet.v1
  failedDeclarations: []
  branchEffect: consumer fallback illegal
#+end_src

** L2: Full Artifact

The full Lean statement, candidate proof, AXLE request, and tool messages are
available only when the agent explicitly needs to inspect or revise the proof.

This keeps token cost low while preserving traceability.

* Mental Simulation: Avoiding Defensive Engineering

Intent:

#+begin_quote
Fix search packet rendering after a provider starts omitting selector metadata
in one path.
#+end_quote

Naive agent plan:

1. Add =Option<String>= handling to every renderer.
2. Fall back to path plus line number when selector is absent.
3. Add tests for missing selector.

This "works", but it damages ASP because selector identity is the core rule
that prevents line-number-native search.

ASP with formal verification:

1. WhereFrame locates provider result construction, renderer, schema, and
   search packet tests.
2. DynamicTopology identifies the producer boundary and renderer consumer.
3. EvidenceGraph finds a FormalFact:

   #+begin_src text
   selector identity is required and verified for semantic-search-packet.v1
   #+end_src

4. HowFrame marks the consumer fallback branch as illegal.
5. Policy says the fix must happen at the producer/schema boundary.
6. The agent edits the provider result builder or schema migration, not every
   downstream renderer.
7. Observation runs schema tests and updates EvidenceGraph. If the invariant
   changed, a new ProofObligation is opened rather than weakening all
   consumers.

The result is not "more checks". It is fewer, better-placed checks.

* How This Relates To Policy And Scenarios

Policy and Scenarios are still required.

Policy answers:

- Which edit shapes are high quality?
- Which edit shapes are architecture damage?
- Which dependency or runtime patterns are preferred?
- Which local recipes have benchmark or snapshot support?

Scenarios answer:

- Did this recipe work in a representative benchmark?
- What latency, memory, packet size, and failure cases were observed?
- Which dependency or implementation path was validated?

Formal verification answers:

- Is this stable claim proven against a trusted statement?
- Does the candidate implementation preserve the stated invariant?
- Can this branch be pruned because it violates a proven property?

They should be combined inside HowFrame:

#+begin_src text
HowFrame:
  target: provider result construction
  policyFacts:
    - do not create consumer fallbacks for producer-owned invariants
  scenarioFacts:
    - selector-only search reduced repeated query turns in scenario X
  formalFacts:
    - selector identity invariant verified for packet schema v1
  illegalBranches:
    - path+line fallback in renderer
  legalBranches:
    - fix provider construction
    - bump schema and migrate receipts if invariant changed
#+end_src

* How This Relates To Topology And GraphRoute

Topology is the map of the current architecture slice. It says:

- what the target is
- which owners and dependencies are connected
- which edges are hot paths
- which historical failures matter
- which policy/scenario/proof facts apply
- which branches are legal

GraphRoute is the navigation algorithm. It decides:

- where to jump next
- which evidence node is enough
- whether to expand or stay compact
- whether a proof receipt makes a search branch unnecessary

Formal verification belongs to Topology as branch legality evidence. It does
not replace GraphRoute. A proof receipt can make GraphRoute skip a branch, but
GraphRoute is still the algorithm that chooses the skip.

* Python And Julia Roles

Python is the runtime graph implementation layer:

- ingest ProofObligation and ProofReceipt artifacts
- connect receipts to selectors, schemas, policy facts, and scenarios
- update EvidenceGraph and DynamicTopology
- implement graph algorithms that route the agent to the smallest sufficient
  evidence slice
- expose compact HowFrame projections

Julia is the research and evaluation layer:

- replay sessions
- compare topology-jump strategies
- measure whether proof-backed branch pruning reduces turns
- find ranking failures where formal facts were ignored or overused
- test graph algorithms before they become Python runtime behavior

Julia should not become the runtime for proof checks. It should evaluate the
strategy and feed better algorithms back to Python.

* Why Native Parsers Still Matter

AXLE is strongest for Lean artifacts. ASP's advantage is broader:

- native language harnesses can locate stable selectors and semantic owners
  inside Rust, Python, Julia, Gerbil, TypeScript, and future languages
- those selectors can generate proof obligations over stable models
- Python can connect proof obligations to cross-language topology
- Policy/Scenarios can decide which obligations are worth creating

The proof layer is therefore downstream of native parsing. ASP should not
replace parser-owned evidence with generic text extraction. It should use native
parsers to decide what is stable enough to formalize.

* What To Learn From AXLE

** Statement and candidate must be separate

ASP should model formal verification as:

#+begin_src text
trusted statement + candidate artifact -> proof receipt
#+end_src

The trusted statement should usually be generated from a schema, protocol, or
architecture contract, not from the same code that is being edited.

** =check= is lower trust than =verify_proof=

Compilation is useful for feedback, but it is not enough to prune
architecture branches. Branch legality requires the stronger receipt when a
proof-backed claim is involved.

** Declaration extraction is graph material

=extract_decls= exposes declaration-level dependencies and complexity metrics.
For ASP, equivalent fields should enter the graph as proof topology:

- declaration selector
- local type dependencies
- local value dependencies
- external dependencies
- type hash
- proof complexity
- heartbeats and timings
- messages and failed declarations

These facts are useful for ranking proof obligations and for finding expensive
or fragile proof nodes.

** Repair tools are not policy

=repair_proofs= is useful as a candidate generator or diagnosis tool, but ASP
must not treat repair output as automatically high quality. Repair output should
flow through Policy/Scenarios and, when relevant, another proof receipt.

** Version and environment must be explicit

AXLE's behavior depends on Lean version, Mathlib environment, import handling,
and tool defaults. ASP receipts must pin:

- checker
- checker version or source commit
- Lean environment
- import handling mode
- formal statement digest
- candidate digest
- trust level

* Initial Integration Plan

** Phase 0: Research artifact

Completed by this RFC:

- clone AXLE into =.data/axiom-lean-engine=
- record commit and version
- identify proof-check, declaration-extraction, and proof-manipulation tools
- map them into ASP evidence/topology/policy theory

** Phase 1: Schema draft

Add shared schemas:

- =semantic-proof-obligation.v1.schema.json=
- =semantic-proof-receipt.v1.schema.json=
- optional =semantic-formal-fact.v1.schema.json=

These schemas should be language-neutral and selector-owned.

** Phase 2: Scenario contract

Add Org scenarios that test the decision behavior, not just proof execution:

- consumer fallback should be rejected when producer invariant has a proof
  receipt
- missing proof receipt should open an obligation rather than silently weaken a
  policy
- stale proof receipt should not prune a branch after schema version changes

** Phase 3: Experimental Lean lane

Add an experimental lane that can call AXLE or a local Lean checker for small
model proofs. This lane must not become required for normal ASP use.

It should support:

- produce formal statement artifact
- produce candidate proof artifact
- run checker
- write receipt
- attach receipt to EvidenceGraph

** Phase 4: Topology integration

DynamicTopology should consume FormalFacts as branch-legality inputs:

- legal branch
- illegal branch
- unproven branch
- stale branch

GraphRoute can then avoid branches that are already illegal or proven
unnecessary.

** Phase 5: Julia replay

Replay sessions to answer:

- Did proof-backed branch pruning reduce search/edit turns?
- Did it prevent defensive fallbacks?
- Which proof obligations were too expensive or too abstract?
- Which policies should request formal facts earlier?

* Non-Goals

- Do not build an agent MCP server around AXLE.
- Do not require Lean or AXLE for every ASP workflow.
- Do not formalize fast-changing code directly.
- Do not expose full proof artifacts to the agent unless requested.
- Do not replace Policy/Scenarios with proof repair tools.
- Do not treat online proof verification as the only trust path.

* Open Questions

1. Which ASP invariants should be first-class proof obligations?
2. Should proof obligations be generated from schemas, Org contracts, Rust type
   models, or all three?
3. Which trust level is enough for internal ASP engineering invariants?
4. How should stale receipts be invalidated when a schema or topology slice
   changes?
5. How should Python rank proof obligations against scenario facts and policy
   facts?
6. Which Julia replay metrics best show that proof-backed branch pruning reduces
   turns without hiding necessary evidence?

* Current Recommendation

ASP should adopt formal verification as a branch-legality evidence layer.

The first implementation target should not be arbitrary program correctness.
It should be ASP's own stable contracts:

- selector identity
- schema packet invariants
- renderer round-trip invariants
- cache/index state transitions
- topology branch legality

This is the highest-leverage path because these invariants are exactly where
defensive engineering currently appears. Proving or explicitly disproving them
lets the agent choose the right boundary instead of scattering fallbacks across
the codebase.
