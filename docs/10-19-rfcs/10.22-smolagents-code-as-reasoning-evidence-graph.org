* One-sentence summary
=smolagents= is useful to ASP as an external reference for code-as-reasoning: it shows how model-written code actions, tool calls, observations, errors, state, and plans can become structured run evidence. ASP's core theory and improvement scheme live in [[file:10.23-asp-reasoning-evidence-graph-theory.org][10.23 ASP reasoning evidence graph theory]], not in this reference note.
* Scope
This RFC is intentionally a reference note for =smolagents=.

It owns:
- the local source snapshot used for comparison;
- the specific =smolagents= concepts that are useful to ASP;
- the limits of =smolagents= as an ASP architecture model;
- the mapping from =smolagents= implementation concepts to ASP core theory.

It does not own:
- ASP =ReasoningTree= / =EvidenceGraph= / =GraphRoute= theory;
- ASP DB Engine and Turso state architecture;
- ASP compact render contract;
- Python production graph-route algorithms;
- Julia offline topology research.

Those live in:
- [[file:10.23-asp-reasoning-evidence-graph-theory.org][10.23 ASP reasoning evidence graph theory]]

* Core finding
The important lesson is not "LLM writes code." The important lesson is:

#+begin_src text
reasoning can become an executable artifact
#+end_src

In =smolagents=, one action step is not an opaque chat turn. It can contain:
- model input messages;
- model output text;
- parsed code action or tool calls;
- execution logs;
- observations;
- errors;
- token usage;
- final-answer marker;
- timing.

That is close to an ASP reasoning-event model. The difference is authority. =smolagents= uses Python AST/runtime structures to execute code actions. ASP should use the stronger theory in [[file:10.23-asp-reasoning-evidence-graph-theory.org][10.23]]: provider-owned native parsers, selector identity, dependency/test/Org topology, DB Engine receipts, dynamic overlays, Python production graph ranking, Julia offline replay, and compact render.
* Relationship to core ASP reasoning theory
This RFC uses =ReasoningTree=, =EvidenceGraph=, and =GraphRoute= only as mapping terms into [[file:10.23-asp-reasoning-evidence-graph-theory.org][10.23 ASP reasoning evidence graph theory]].

The mapping from =smolagents= is:
- =AgentMemory= and step serialization map to ASP run evidence.
- =TaskStep=, =PlanningStep=, and =ActionStep= map to reasoning-event classes.
- Code blocks executed by =CodeAgent= map to ASP reasoning actions.
- Python AST execution in =LocalPythonExecutor= maps to ASP's general action parsing layer.
- Tool schema metadata and tool validation map to ASP command/provider contracts.
- Observations, logs, errors, timing, and final-answer markers map to receipt and replay evidence.

The mapping is useful, but it is not the architecture owner. ASP's architecture owner is the core theory RFC plus the DB/search architecture RFC.
* What smolagents gets right
** Code action as a better action language
The code-action surface is useful because it supports:
- composability;
- object management;
- generality;
- branching and recovery;
- a representation that matches model training data better than deeply nested JSON.

ASP should learn from this. An ASP reasoning action should be a small program over approved evidence/search/tool interfaces, not a plain grep instruction and not a raw JSON dump.
** Memory as a typed run log
=AgentMemory= stores typed steps:
- =SystemPromptStep=;
- =TaskStep=;
- =ActionStep=;
- =PlanningStep=.

It supports succinct step views, full step dictionaries, replay, and full-code return. ASP should push this further by making the run log graph-addressable and selector-aware.
** Planning as explicit evidence
=PlanningStep= stores planning input messages, planning output, plan text, timing, and token usage. ASP should preserve this idea, but link planning evidence to selected anchors, rejected routes, graph frontiers, recommended next commands, and subsequent action steps through the core theory in 10.23.
** Tool schema as executable contract
=Tool= carries name, description, input schema, output type/schema, code export/import, and requirements extraction. =tool_validation.py= uses AST checks to ensure tool source is reconstructable and self-contained.

This maps well to ASP provider and command contracts: a tool or ASP command should emit enough schema, provenance, and receipt data to become evidence.
** AST-level execution trace
=LocalPythonExecutor= parses code with Python AST, maintains state, enforces authorized imports, calls tools, captures logs, and detects final answers.

For ASP, the lesson is not to adopt this executor as-is. The lesson is to normalize actions into structured events and then attach stronger native-parser proof where ASP has provider authority.
* Limits of smolagents for ASP
=smolagents= is an agent framework. ASP is an evidence/search protocol and codebase intelligence substrate.

Important limits:
- Python AST is not enough for multi-language codebase reasoning.
- The local executor is explicitly not a security sandbox.
- Memory is typed but still mostly list-shaped, not graph-shaped.
- Code actions are executed, but not automatically linked to repository selectors, dependency facts, tests, Org decisions, DB generations, or dirty overlays.
- Tool/action traces are not a durable cross-session graph read model.
- Search is not the center of the architecture.

ASP should absorb the run-evidence model, not copy the framework boundary.
