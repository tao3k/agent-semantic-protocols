* One-sentence summary

After Rust, Python, TypeScript, and Julia can all emit parser-owned
=field=/=type=/=collection= facts, ASLP moves from fact extraction to a
closed loop:

#+begin_src text
parser-owned facts -> precise frontier -> exact query -> feedback memory -> benchmark
#+end_src

* Problem

The current graph-turbo path can rank parser-owned facts and project compact
frontiers.  That is no longer enough.  Agent failures now come from a more
specific gap: the system can expose relevant facts, but it does not yet
standardize how those facts drive fewer source reads, better test selection,
failure evidence, and session memory.

The next protocol layer must answer these questions across languages:

- which field/type/collection fact is the best next evidence boundary?
- which exact selector should the agent read first?
- which test target validates the selected owner or changed field?
- which failure assertion, stack frame, or hot block explains the failure?
- which frontier items were already explored, ignored, read, edited, or
  validated in this task session?
- did the agent actually use the frontier, or did it fall back to raw reads and
  repeated windows?

This is a context precision problem, not a graph visualization problem.

* Non-goals

- Do not add a new default agent-facing =asp graph= command.
- Do not make graph-turbo a prompt renderer with hidden semantics.
- Do not switch to GNN/PyG before deterministic sparse graph metrics show a
  need.
- Do not optimize token count by removing parser-owned semantic evidence,
  exact selectors, frontier actions, test links, omission facts, or avoid
  guidance.
- Do not introduce language-private fact meanings that cannot be represented in
  the shared ontology.

* Protocol direction

The next ASLP research line is:

#+begin_src text
cross-language semantic facts
  -> relation-specific sparse graph profiles
  -> frontier and capsule projections
  -> read/failure/test feedback memory
  -> context precision benchmark
  -> optional lightweight learned scoring
#+end_src

Graph-turbo remains an internal ranking and projection layer consumed by
=search=, =query=, =check=, read-plan, and failure-frontier surfaces.  It
should keep producing schema-owned packets and compact decision frontiers with
=rank=, =frontier=, =frontierActions=, =profiles=, =omit=, and =avoid=.

* Semantic ontology

Four-language =field=/=type=/=collection= facts must converge on one shared
ontology.  Provider-owned parser facts stay native at extraction time, but the
graph facts consumed by graph-turbo use shared semantic fields:

#+begin_src text
field.ownerKind      struct | class | interface | dataclass | module | object
field.name           provider-owned field symbol
field.ownerPath      source owner path
field.access         read | write | append | mutate | construct | validate

type.name            provider-owned type text
type.element         sequence/set/iterator element type
type.key             map key type
type.value           map/result/optional value type

collection.family    sequence | map | set | iterator | optional | result
collection.impl      Vec | list | Array | Vector | HashMap | dict | Map | Dict | ...
collection.mutation  append | insert | remove | update | clear | replace

fact.provenance      parser | build | test | failure | receipt | heuristic
fact.confidence      exact | inferred | heuristic
fact.freshness       fresh | stale | unknown
#+end_src

The same user intent should produce homologous graph facts across languages:

#+begin_src text
Rust       Vec<T>        -> collection.family=sequence impl=Vec
Python     list[T]       -> collection.family=sequence impl=list
TypeScript Array<T>      -> collection.family=sequence impl=Array
Julia      Vector{T}     -> collection.family=sequence impl=Vector

Rust       HashMap<K,V>  -> collection.family=map impl=HashMap
Python     dict[K,V]     -> collection.family=map impl=dict
TypeScript Map<K,V>      -> collection.family=map impl=Map
Julia      Dict{K,V}     -> collection.family=map impl=Dict
#+end_src

** P0 contract surface

The P0 ontology contract is schema-first:

- =schemas/semantic-fact-ontology.v1.schema.json= owns the shared
  =field=/=type=/=collection= node vocabulary, =has_type= and
  =collection_of= relations, provenance/confidence/freshness fields, and
  collection families.
- =schemas/semantic-fact-graph.v1.schema.json= owns the runtime
  =search semantic-facts --json= provider packet.  Providers must advertise
  this packet schema for fact graph output instead of =semantic-graph.v1=;
  =semantic-graph.v1= remains the embedded search packet graph vocabulary.
- Relation vocabulary closure is explicit at P0.  Relations that exist only in
  the ontology schema are =calls=, =imports=, =mutates=, =reads=, and =writes=.
  Relations that exist only in the runtime fact graph are =affects=,
  =covered_by=, =depends_on=, =packages=, =targets=, and =tests=.  Relations
  that are profile-internal to graph-turbo are =checks=, =gates=, =relates=,
  =repairs=, =selects=, =split=, and =uses=.  Profile relation sets must be
  schema-declared or explicitly profile-internal.
- Confidence/freshness closure is an explicit mapping, not an implicit enum
  merge.  Ontology =exact= maps to runtime =exact=; =inferred= maps to
  runtime =high=, =medium=, or =low=; =heuristic= maps to runtime
  =heuristic=.  Ontology =fresh= maps to runtime =fresh= or =cache-hit=;
  =stale= maps to =stale=; =unknown= maps to =unknown=.
- =schemas/semantic-fact-ontology.fixtures.v1.json= is the executable
  cross-language fixture matrix for Rust, TypeScript, Python, and Julia
  sequence/map facts.
- =schemas/semantic-language-registry.providers.v1.json= advertises the
  ontology schema for the four source-language providers so future provider
  output alignment has a registry-visible contract.
- =packages/python/asp_graph_turbo/src/asp_graph_turbo/ontology.py= projects the
  fixture catalog into a schema-valid graph-turbo request by moving
  ontology-specific fields into graph node =fields= and preserving top-level
  request fields for selectors, paths, symbols, and actions.
- =tests/unit/test_semantic_fact_ontology_schema.py= validates the schema, the
  fixture matrix, and the provider registry declaration.
- =tests/unit/test_asp_graph_turbo_ontology.py= validates that the projected
  ontology facts are rankable as field-first graph frontiers.
- =tests/unit/test_semantic_fact_protocol_vocabulary.py= validates RFC number
  uniqueness, relation boundary closure, graph-turbo profile-internal
  relations, and confidence/freshness mapping coverage.
- =languages/rust-lang-project-harness/src/search/semantic_facts.rs= emits
  ontology-shaped Rust =field=/=type=/=collection= graph facts from =syn=
  parser facts while preserving the existing provider metadata fields used by
  compact rendering.
- =crates/agent-semantic-protocol/tests/unit/provider_command/facade/pipe/provider_ontology.rs=
  validates that ASP keeps provider-emitted ontology fields when it builds a
  =semantic-graph-turbo-request.v1= packet.

The Rust, Python, TypeScript, and Julia providers are the first concrete
emitters for this contract.  Other providers may still roll out incrementally,
but they must map their parser-owned facts into this shared contract before
profile scoring consumes those facts.

* Flow profiles

Graph-turbo should grow profile-owned flow views before adding new public
commands.  Existing =search=, =query=, =check=, failure-frontier, and read-plan
surfaces may select these profiles internally.

Required initial profiles:

- =field-impact= :: owner -> field -> type -> hot -> test
- =type-impact= :: type -> constructors/methods/usages -> tests
- =collection-impact= :: collection -> element/key/value -> read/write/mutate
  sites -> tests
- =failure-evidence= :: failure -> assertion -> hot/field/type/collection ->
  tests
- =test-selection= :: changed owner/fact -> smallest covering test frontier
- =affected= :: changed owner/fact -> package/build/test/dependency frontier

Profile output must stay compact and action-oriented:

#+begin_src text
[graph-frontier] profile=collection-impact alg=typed-sparse-frontier
F=field:struct-field(entries: Vec<Entry>)@src/cache.rs:20:20!code
Y=type:field-type(Vec<Entry>)!evidence
C=collection:sequence(Vec<Entry>)!evidence
H=hot:fn(write_entry)@src/cache.rs:80:104!code
T=test:case(cache_replay_roundtrip)!tests
frontier=F.code,H.code,T.tests
frontierActions=F.code=>asp rust query --selector src/cache.rs:18:24 --workspace . --code
profiles=collection-impact,test-selection
omit=source,full-callgraph,unrelated-fields
avoid=raw-read,manual-window-scan,broad-fzf
#+end_src

* Sparse matrix bank

The Python graph-turbo engine should move from one adjacency view toward a
relation-specific sparse matrix bank.  Relation matrices are deterministic
algorithm inputs, not prompt text:

#+begin_src text
A_matches
A_contains
A_has_type
A_collection_of
A_reads
A_writes
A_mutates
A_calls
A_imports
A_covers
A_fails
A_explains
A_points_to
A_validates
A_belongs_to
A_builds
#+end_src

Profiles compile a weighted sparse graph from those matrices:

#+begin_src text
field-impact       = has_type + reads + writes + mutates + covers
collection-impact  = collection_of + reads + writes + mutates + covers
failure-evidence   = fails + explains + points_to + touches + validates
test-selection     = covers + validates + belongs_to
affected           = calls + imports + belongs_to + builds + covers
#+end_src

The first implementation remains SciPy/NumPy based.  Learned scoring may
adjust profile weights later, but relation construction and profile semantics
must remain inspectable.

Profile-level calibration is a schema-owned sparse-bank input, not a hidden
prompt hint. =schemas/semantic-graph-turbo-calibration.v1.schema.json= records
per-profile =kindDeltas= and =relationDeltas= derived from feedback receipts.
=asp-graph-turbo calibrate <feedback>... <request>= builds this packet, and
=asp-graph-turbo rank --calibration <packet>= applies the deltas before sparse
backend construction. Calibrated =kindBonus= values drive node scoring and rank
explanations; calibrated =relationWeightMultiplier= values drive CSR edge
weights, relation-channel weight mass, oriented-edge weights, and the rank
fingerprint so cached backends cannot reuse a stale profile.

The P1 observable contract is =profileMatrices[].relationChannels= and
path-backend metrics in =schemas/semantic-graph-turbo-result.v1.schema.json=.
Each profile matrix summary lists one channel per allowed relation, including
zero-edge channels, with supported/reachable edge counts and supported/reachable
weight mass. =algorithmMetrics.pathBackend= and
=algorithmMetrics.pathFallbackCount= record whether typed paths used
=python-bfs-small=, =scipy-yen=, =scipy-dijkstra=, or =python-bfs-fallback= and
how many backend fallbacks were needed. =algorithmMetrics.pathPairCount= and
=algorithmMetrics.pathCandidateCount= record the source/sink pair width and the
final backend candidate count before budgeted typed-path projection. This keeps
the sparse bank and typed-path algorithm choice inspectable before learned
weights or alternative backends are introduced.

The agent-facing compact JSON contract is
=schemas/semantic-graph-turbo-summary.v1.schema.json=.  It projects the active
profile frontier with ranked node locators, selected edges, typed paths,
algorithm trace, algorithm metrics, and the active
=profileMatrices[].relationChannels= row.  It explicitly omits the full score
vector, full node fields, non-active profile matrices, and source code so
sandtables can measure token cost without losing the current frontier's exact
next-action facts.

* Build, test, and package graph

Parser facts are not enough for repository-level agents.  The next fact graph
must include deterministic build/test/package nodes:

#+begin_src text
package
workspace-member
build-target
test-target
test-case
runner
fixture
external-package
package-manager
#+end_src

Required relation vocabulary:

#+begin_src text
owner -> package             belongs_to
package -> build-target      builds
test-case -> test-target     belongs_to
test-target -> owner         covers
test-target -> package       validates
external-package -> owner    imported_by
field/type -> test-case      covered_by
failure -> test-target       failed_in
#+end_src

The agent-facing result is not a full architecture report.  It is a minimal
validation frontier:

#+begin_src text
frontier=T.unit,T2.integration,P.package
frontierActions=T.unit=>asp rust check --test cache_replay_roundtrip .
omit=full-suite,full-build-graph
avoid=run-full-suite,manual-window-scan,raw-read
#+end_src

* Feedback memory

Read-loop and failure-loop memory are not only post-hoc metrics.  They become
ranking inputs through receipt graph facts:

#+begin_src text
frontier receipt
read receipt
failure receipt
test receipt
edit receipt
#+end_src

Each receipt should record:

#+begin_src text
taskFingerprint
commandFingerprint
selector
owner
symbol
range
frontierReturned
frontierFollowed
codeActuallyRead
testCommand
testResult
editTouchedOwner
outputFingerprint
#+end_src

=schemas/semantic-fact-frontier-receipt.v1.schema.json= owns this receipt
contract.  All fields above are required at the top level; values that do not
apply to a specific receipt kind must be recorded as =null= or an empty array
instead of being omitted.  This keeps benchmark conversion able to distinguish
"not applicable" from "not recorded".
=schemas/semantic-fact-frontier-receipt.fixtures.v1.json= records the first
real-project task-session fixture for this contract plus runtime capture
fixtures for =asp-graph-turbo receipt= and the public =asp <language> search
--frontier-receipt-out= facade.
The graph-turbo runtime exposes this capture path through
=asp-graph-turbo receipt=, which ranks a graph-turbo request and emits the
receipt schema using explicit followed/read/test facts from the caller.
The ASP search runtime exposes a receipt trigger through
=--frontier-receipt-out <path>= on graph-turbo frontier search commands, with
optional =--frontier-receipt-follow-node=, =--frontier-receipt-read-selector=,
and =--frontier-receipt-test-*= facts.  This trigger strips runtime receipt
flags before provider execution, writes the receipt as a side effect, and keeps
stdout as the normal compact =[graph-frontier]= output.

Graph-turbo must penalize:

- same selector repeated
- overlapping range repeated
- same owner scanned repeatedly
- raw read after exact frontier
- broad fzf after projected frontier
- frontier returned but ignored

Graph-turbo must promote:

- unvisited high-confidence owner
- exact locator with parser provenance
- nearest assertion/hot block
- covered test target
- field/type/collection flow node with failure evidence

* Query capsule

=query --view capsule= is the preferred next protocol surface once ontology
and flow facts are stable.  A capsule is not source code.  It is a compact
semantic pre-read packet:

#+begin_src text
[query-capsule] profile=type-impact ranker=graph-turbo:field-type-collection
Y=type:struct(CacheRecord)!evidence
F=field:struct-field(entries: Vec<Entry>)!code
C=collection:sequence(Vec<Entry>)!evidence
H=hot:fn(write_record)!code
T=test:case(cache_replay_roundtrip)!tests
frontier=F.code,H.code,T.tests
omit=source,full-field-list,full-callgraph
avoid=raw-read,manual-window-scan
#+end_src

Capsule success means the agent can choose one exact =query --code= selector
or one exact test action without reading wide source first.

* Benchmark contract

ASLP needs a ContextBench-style process benchmark.  It must measure context
retrieval and utilization, not only final patch success.

Required metrics:

#+begin_src text
contextRecall
contextPrecision
contextUtilization
frontierFollowRate
exactCodeSuccess
manualWindowScanCount
rawReadFallbackCount
repeatOwnerCount
duplicateSelectorCount
broadFzfCount
testSelectionPrecision
failureEvidenceHitRate
commandsToFirstUsefulLocator
commandsToPatch
stdoutBytesToPatch
graphTurboLatencyMs
#+end_src

The key metric is:

#+begin_src text
contextUtilization =
  frontier items actually used for edit/test/failure reasoning
  /
  frontier items emitted by graph_turbo
#+end_src

This prevents a false optimization where graph-turbo emits more plausible
nodes but agents still ignore them or fall back to manual source scans.

=schemas/semantic-fact-frontier-benchmark-report.v1.schema.json= owns the
offline comparison report before live sandtable scoring.  Each report joins a
receipt fixture with the graph-turbo benchmark metric surface, records the
scenario's =goldContext=, and derives =contextMetrics= such as
=contextPrecision=, =contextRecall=, =contextUtilization=,
=goldFrontierBestRank=, =goldSelectorActionRank=, =exactCodeSuccess=, and
=testSelectionPrecision=.  The companion
=schemas/semantic-fact-frontier-benchmark-report.fixtures.v1.json= currently
fixes the task-session receipt, graph-turbo runtime receipt, ASP runtime
frontier-only receipt, two ASP runtime followed/read/test receipts, and one
relation-weight followed/read/test receipt in one table.  The three
followed/read/test ASP runtime scenarios are marked ready for weight
calibration; the relation-weight scenario records the fixed hot-companion path
where a matched =item= keeps its backend =hot= range in the returned frontier
and selector action queue.  Because the current calibration-ready receipts all
have full gold recall, the benchmark report defers relation-weight changes
until a new calibration-ready runtime receipt shows a frontier miss or a
missing gold selector action.
=tests/unit/test_semantic_fact_frontier_benchmark_report_schema.py=
recomputes scoring metrics from the source receipt fixtures so report rows
cannot drift from recorded evidence.
=schemas/semantic-graph-turbo-sandtable-summary.v1.schema.json= accepts the
live scoring rows that consume calibration-ready report scenarios through
=asp-graph-turbo sandtable-summary --benchmark-report ... --report-scenario ...=.

* Lightweight scoring

Learned scoring is allowed only after deterministic metrics exist.  The first
version is an interpretable weighted scorer:

#+begin_src text
final_score =
  ppr_score
  + weighted_feature_score
  + action_utility_score
  + feedback_score
  - redundancy_penalty
  - read_loop_penalty
#+end_src

Candidate features:

- exact query token match
- parser-owned confidence
- field/type/collection flow distance
- exact locator availability
- covered_by test
- failure stack distance
- assertion proximity
- read/write/mutation relation
- same-owner repetition
- same-range overlap
- stale cache
- heuristic edge

Training signals come from receipt graph facts: frontier followed, exact code
success, edit touched node owner, test passed after edit, failure evidence
used, manual-window-scan avoided, duplicate read avoided.

* Audit plan

P0 audit should answer:

- Do all four language providers emit homologous =field=/=type=/=collection=
  graph facts for sequence and map fields?
- Does graph-turbo preserve parser provenance, confidence, freshness, exact
  selector, and context locator?
- Are debug-only ranking details separated from agent-facing compact frontier?
- Are read/failure/test receipts available as packet-visible facts or
  sandtable observations?
- Do current live sandtables measure frontier follow rate and utilization, not
  only command counts?

P1 audit should answer:

- Which relation vocabularies are still language-private?
- Which graph-turbo profiles still rely on string heuristics instead of typed
  facts?
- Which agent-facing surfaces still emit too many runnable branch choices?
- Which tests prove lower token cost without precision loss?

* Landing plan

** P0: Ontology and schema closure

- Add or revise shared schemas for semantic fact ontology:
  =field=, =type=, =collection=, =fact.provenance=, =fact.confidence=,
  =fact.freshness=.
- Add a cross-language fixture matrix for Rust/Python/TypeScript/Julia:
  sequence field, map field, optional/result field where supported.
- Gate graph-turbo request/result packets against the new ontology without
  changing default agent behavior.

** P1: Relation matrix bank

- Split graph-turbo relation handling into named sparse matrices.
- =schemas/semantic-graph-turbo-result.v1.schema.json= exposes
  =profileMatrices= as a packet-visible profile matrix bank with relation,
  transition, supported edge, reachable edge, and density summaries.
- =packages/python/asp_graph_turbo/src/asp_graph_turbo/profile_matrix.py=
  compiles the current graph against every registered profile without changing
  the existing default frontier contract.
- =packages/python/asp_graph_turbo/src/asp_graph_turbo/profiles.py= registers
  =field-impact=, =type-impact=, =collection-impact=, and =failure-evidence=
  as internal graph-turbo profiles that can be selected by existing surfaces.
- =tests/unit/asp_graph_turbo_feedback_profiles/= validates
  relation-specific profile compilation for =field-impact=,
  =collection-impact=, and =failure-evidence= flows.
- Preserve existing =owner-query= and =failure-frontier= output contracts.

** P2: Feedback receipt graph

- Extend read/failure memory from observations into packet-visible receipt
  facts.
- =schemas/semantic-graph-turbo-result.v1.schema.json= exposes
  =receiptAdjustments= plus =receiptBoostCount= and =receiptPenaltyCount= so
  receipt effects are inspectable instead of hidden inside scores.
- =packages/python/asp_graph_turbo/src/asp_graph_turbo/receipt.py= applies
  deterministic boost/penalty deltas from =kind=receipt= nodes, receipt edges,
  and seen selector receipts while preserving parser-owned graph reachability.
- =packages/python/tools/src/tools/semantic_sandtable/agent_observation_frontier.py=
  derives =frontierFollowRate=, =contextPrecision=, and =contextUtilization=
  from ASP search frontier selectors and subsequent =query --selector= commands.
- =packages/python/tools/src/tools/semantic_sandtable/expectations.py= gates those
  metrics with =minFrontierFollowRate=, =minContextPrecision=, and
  =minContextUtilization=.
- =packages/python/asp_graph_turbo/src/asp_graph_turbo/receipt.py= derives
  ranking penalties for duplicate selector, overlapping range, same-owner scan,
  and raw-read fallback after exact frontier from packet-visible receipt facts.
- =tests/unit/asp_graph_turbo_feedback_profiles/= validates those receipt
  penalties against schema-valid graph-turbo result packets.
- Add sandtable gates for =frontierFollowRate= and =contextUtilization=.

** P3: Build/test/package graph

- =schemas/semantic-fact-graph.v1.schema.json= accepts =package=, =build=,
  and =dependency= nodes plus build/test/package metadata edges such as
  =belongs_to=, =builds=, =tests=, and =depends_on=.
- =packages/python/asp_graph_turbo/src/asp_graph_turbo/profiles.py= registers
  =test-selection= and =affected= as deterministic profile masks for existing
  search/query/check surfaces.
- =tests/unit/asp_graph_turbo_feedback_profiles/= validates
  =test-selection= and =affected= against deterministic
  owner/field/package/build/test/dependency fixtures.
- =schemas/semantic-language-registry.providers.v1.json= exposes
  =search/semantic-facts= for Rust, Python, TypeScript, and Julia with
  =semantic-fact-graph.v1= packet metadata, so agents can discover the fact
  graph surface from the provider registry instead of hard-coding it.
- =languages/rust-lang-project-harness/src/search/semantic_facts.rs= now emits
  parser-owned Cargo =package=, =build=, =dependency=, and =test= graph facts
  from manifest dependency facts and Cargo test target parsing. Runtime evidence:
  =cargo run --quiet --features search,cli -- search semantic-facts "cargo test dependency" --json .=
  produces schema-valid =semantic-fact-graph.v1= output with
  =builds=/=depends_on=/=tests=/=belongs_to= edges.
- =languages/python-lang-project-harness/src/python_lang_project_harness/_semantic_graph_facts.py=
  now merges parser-owned Python field facts with project-level =package=,
  =build=, =dependency=, and =test= facts from =pyproject.toml= and AST-based

  "pytest dependency", "--json", "."])= produces schema-valid

- =languages/typescript-lang-project-harness/src/cli/semantic-graph-facts.ts=
  now merges parser-owned TypeScript field facts with project-level =package=,
  =build=, =dependency=, and =test= facts from =package.json= and AST-based
  test discovery. Runtime evidence:
  =node dist/src/cli/main.js search semantic-facts "npm test dependency" --json .=
  produces schema-valid =semantic-fact-graph.v1= output with the same
  =builds=/=depends_on=/=tests=/=belongs_to= edges.
- =languages/JuliaLangProjectHarness.jl/src/semantic_graph_facts.jl= now

  =julia --project=. -e 'using JuliaLangProjectHarness; print(render_julia_semantic_graph_facts_json(pwd(), "Pkg test dependency", ""))'=
  produces schema-valid =semantic-fact-graph.v1= output with the same
  =builds=/=depends_on=/=tests=/=belongs_to= edges.
- Provider packets must expose a visible source-to-package bridge instead of
  leaving graph-turbo to infer package reachability. Rust, Python, TypeScript,
  and Julia now emit =belongs_to= from matched =field=/=hot=/=owner= nodes to
  the package node when project metadata is available. This lets
  =test-selection= and =affected= traverse =field -> package -> build/test=
  using schema-owned facts.

** P4: Query capsule

- Define =semantic-query-capsule.v1=.
- Add =asp <language> query --view capsule= only after P0-P3 facts can produce
  enough evidence without source.
- Gate capsule by exact follow-up action availability.

** P5: ASLP context precision benchmark

- Add benchmark fixtures with gold context, gold frontier, gold exact selector,
  and gold test action.
- Keep the offline receipt+benchmark comparison in
  =schemas/semantic-fact-frontier-benchmark-report.fixtures.v1.json= before
  promoting any ranking-weight change.
- Record =contextPrecision=, =contextRecall=, =contextUtilization=,
  =commandsToFirstUsefulLocator=, and =stdoutBytesToPatch=.
- Recompute =contextPrecision=, =contextRecall=, =contextUtilization=,
  =goldFrontierBestRank=, =goldSelectorActionRank=, =exactCodeSuccess=, and
  =testSelectionPrecision= from =goldContext= plus the source receipt fixture
  in schema tests.
- Run live sandtable summary rows that consume calibration-ready report
  scenarios before allowing scoring changes to count as an optimization.
- Keep at least two real-project followed/read/test scenarios as the minimum
  calibration floor, and require a new calibration-ready failing receipt before
  changing relation weights.

** P6: Lightweight learned scoring

- Fit per-profile weights from receipt graph data.
- Keep features and weights inspectable.
- Compile graph-turbo feedback packets into
  =semantic-graph-turbo-calibration.v1= packets with explicit =receiptCount=
  and =reasons= evidence for each kind or relation delta.
- Apply calibration only through profile-owned =kindBonus= and
  =relationWeightMultiplier= fields that are visible in =profileCompatibility=
  and =profileMatrices=.
- Treat PyG/GNN as a lab-only follow-up unless deterministic metrics show a
  concrete failure that weighted sparse scoring cannot address.

* Acceptance criteria

The RFC is considered landed only when:

- the shared ontology schema validates Rust/Python/TypeScript/Julia fixtures;
- graph-turbo has relation-specific profile tests for at least field,
  collection, and failure evidence flows;
- live or recorded sandtables report context precision/utilization metrics;
- read/failure/test receipt facts affect ranking in a tested way;
- graph-turbo feedback can compile into profile calibration packets, and
  =rank --calibration= changes scoring, relation-channel weights, explanations,
  and cache fingerprints through schema-visible profile fields;
- default agent-facing outputs remain compact and preserve exact follow-up
  actions;
- at least one real-project run shows fewer repeated reads or broad searches
  without losing the correct edit/test/failure evidence frontier.
