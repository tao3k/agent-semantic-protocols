* One-sentence summary
ASP should treat agent reasoning as typed, replayable, graph-transforming evidence: =ReasoningTree= records how a run moved, =EvidenceGraph= stores reusable evidence across runs, =DynamicTopology= explains the current architecture state and legal reasoning branches, and =GraphRoute= navigates toward exact selectors, commands, tests, edits, and writing actions.
* Scope
This RFC is the core theory document for ASP reasoning, evidence, topology, and agent-facing navigation.

It owns:
- the user and agent mental model for code-as-reasoning;
- the distinction between =ReasoningTree=, =EvidenceGraph=, =DynamicTopology=, and =GraphRoute=;
- the principle that reasoning actions should become typed evidence;
- the principle that executable code identity must be selector/entity/generation/overlay identity, not line ranges;
- the improvement scheme that connects native parsers, DB Engine state, dynamic overlays, Python graph-route ranking, Julia offline replay, and compact render.

It does not own:
- the external =smolagents= source snapshot, which remains in [[file:10.22-smolagents-code-as-reasoning-evidence-graph.org][10.22 smolagents code-as-reasoning EvidenceGraph]];
- the DB Engine / Turso state layout, which remains in [[file:10.21-asp-db-engine-state-and-search-architecture.org][10.21 ASP DB Engine state and search architecture]];
- the codebase-memory-mcp search comparison, which remains in [[file:10.20-codebase-memory-mcp-code-search-research.org][10.20 codebase-memory-mcp code search research]];
- policy facts and recipes, which remain in [[file:10.24-policy-facts-recipes-schema.org][10.24 policy facts recipes schema]];

* Authoritative correction: DynamicTopology as the fourth layer
The earlier three-layer framing is incomplete. The stable theory is a four-layer model:

#+begin_src mermaid
flowchart LR
  RT["ReasoningTree<br/>How this run got here"] --> EG["EvidenceGraph<br/>What evidence exists and how it connects"]
  EG --> DT["DynamicTopology<br/>What the current target is in the architecture"]
  DT --> GR["GraphRoute<br/>Where the agent should go next"]
  GR --> NA["NextAction<br/>Exact selector / command / recovery"]
  NA --> RT
#+end_src

=ReasoningTree= records how one agent run moved. =EvidenceGraph= stores typed reusable evidence across runs. =DynamicTopology= projects that evidence into the current architecture state. =GraphRoute= is the navigation algorithm that chooses the next jump from the graph plus topology.

This distinction is mandatory:

#+begin_src text
Topology:
  dynamic architecture state for the current project/feature/task
  map + current road conditions + architecture roles + legal/illegal branches

GraphRoute:
  navigation/ranking/jump algorithm over EvidenceGraph + DynamicTopology
#+end_src

Topology is dynamic because every search, query, read, observation, validation, and archive action can add or invalidate topology edges. It is the key to multi-hop search, token control, and long-running agent work because the model should not rediscover the same local architecture every turn.

Topology answers:

#+begin_src text
What is this target?
What owns it?
What does it produce or consume?
Which invariant or boundary does it sit on?
Which tests, docs, scenarios, and prior failures prove that role?
Which reasoning branches are legal here?
Which facts are stale because the task, generation, or feature scope changed?
#+end_src

The minimum =TopologySlice= for an agent turn should include:

- target role: writer, reader, adapter, boundary, renderer, validator, orchestrator, or invariant owner;
- ownership: package/module/language harness/schema owner;
- structural edges: produces, consumes, validates, depends-on, routes-to, renders, persists;
- trust boundary: external input, internal invariant, legacy migration zone, dirty overlay, or unknown;
- pressure: hot path, memory pressure, token budget, concurrency, stale-index risk, or prompt-critical render;
- proof: schema, test, benchmark, snapshot, scenario, prior failure, RFC/ADR, or task archive;
- applicable policy/scenario ids for the current topology slice.

The first public JSON contract for this slice is
[[file:../../schemas/semantic-dynamic-topology.v1.schema.json][semantic-dynamic-topology.v1]].
It is intentionally a compact topology projection, not the full graph and not
the full reasoning history.

The production loop is:

#+begin_src text
ReasoningTree
-> EvidenceGraph / overlay
-> DynamicTopology
-> GraphRoute
-> WhereFrame / recommendedNext
-> HowFrame when evidence is sufficient for an edit group
-> ReasoningTree
#+end_src

The first public JSON contracts are split by responsibility:

- [[file:../../schemas/semantic-dynamic-topology.v1.schema.json][semantic-dynamic-topology.v1]] projects the current architecture/topology state.
- [[file:../../schemas/semantic-where-frame.v1.schema.json][semantic-where-frame.v1]] says how to search, what is confirmed, what evidence is still missing, what to avoid, and when to stop searching and synthesize a write strategy.
- [[file:../../schemas/semantic-how-frame.v1.schema.json][semantic-how-frame.v1]] says how to change, which edit branches are legal or pruned, and how to validate the edit group.

* Branch legality and defensive programming
Code-as-reasoning makes one important failure mode visible: a code branch is often a reasoning branch materialized into source.

A defensive branch such as a fallback, default, catch-all, optional unwrap, broad recovery, or silent compatibility path carries an implicit claim:

#+begin_src text
This failure can happen here, and this recovery is architecturally legal.
#+end_src

ASP should not accept that claim because the code shape looks safe. The branch must connect to topology-backed evidence.

The branch rule is:

#+begin_src text
A defensive branch is allowed only when EvidenceGraph + DynamicTopology prove it
as a boundary recovery, legacy migration, or explicitly accepted compatibility path.
Otherwise it is an unproven reasoning branch and should be pruned.
#+end_src

The topology roles decide the action:

- external boundary: validate, recover, and report with context;
- internal invariant: do not fallback in the consumer; fix the producer, schema, constructor, or writer and add validation evidence;
- legacy migration zone: allow bounded compatibility only with version evidence, expiry, and receipts;
- unknown role: route to missing topology evidence before writing.

This is the theoretical reason ASP can reduce defensive programming without making the model weaker. It does not forbid careful code. It moves uncertainty out of runtime fallback code and into =EvidenceGraph= as an explicit evidence gap:

#+begin_src text
evidenceGap:
  missing proof that selector X is always produced by owner Y

recommendedNext:
  route to producer/schema/test before adding consumer fallback
#+end_src

The model should not be told only "avoid defensive programming". It should be shown why a branch is legal or illegal in the current topology slice.

* Python and Julia ownership
The runtime and research boundary is strict.

Python owns the production graph intelligence runtime:

- build and update production =EvidenceGraph= projections from Rust/artifact inputs;
- compute =DynamicTopology= slices for the active feature/task;
- run graph-route ranking, next-jump selection, diversity, and failure-aware routing;
- assemble compact =QualityFrame= / =WhereFrame= / =HowFrame= projections for the agent;
- emit deterministic, schema-backed packets that Rust can validate and render.

Julia owns offline graph algorithm research:

- replay large session and artifact corpora;
- evaluate which topology jumps reduce turns and token cost;
- compare ranking algorithms, graph features, failure-aware routing, and branch-legality predictors;
- find Python production ranking failures and produce corrected features;
- study whether topology projection reduces defensive branches and repeated searches.

The promotion path is:

#+begin_src text
Agent actions -> artifacts/archives -> Julia replay/research
-> validated graph feature or algorithm -> Python production runtime
-> Rust-rendered compact frame -> Agent action
#+end_src

Julia does not sit on the prompt-critical runtime path. Python does not own open-ended notebook research. Rust remains the fast substrate for CLI, DB, search engine integration, schema validation, and compact render.

The direction of dependency matters. Core ASP theory should not live inside an external reference note. External research notes may point here when they explain how an outside system maps into ASP.
* Core principle
The core principle is:

#+begin_src text
reasoning becomes an executable, typed, replayable evidence artifact
#+end_src

This does not mean that code itself thinks. It means that an agent can express part of its reasoning as a small executable action. The action can call tools, hold intermediate state, branch on observations, handle errors, and produce a final answer. Once it runs, ASP can record what happened as structured evidence instead of losing the reasoning inside a natural-language paragraph.

A prompt-only loop usually looks like:

#+begin_src text
think about intent -> write a tool call -> read output -> think again
#+end_src

The ASP target loop is:

#+begin_src text
intent -> reasoning action -> execution/parse -> observation/error/state
-> typed evidence -> graph-route next action
#+end_src

The important change is not that the action is written in Python, Rust, TypeScript, or any other language. The important change is that the system can lower the action into a structured event with provenance, parser proof, observations, and graph edges.
* Basic code-as-reasoning loop
The basic loop is:

#+begin_src mermaid
flowchart TD
  A["User intent"] --> B["Agent plans next step"]
  B --> C["Agent emits reasoning action"]
  C --> D["Runtime executes or parses action"]
  D --> E{"Observation or error"}
  E --> F["Typed run memory records step"]
  F --> B
  E --> G["Final answer or accepted evidence"]
#+end_src

The useful properties are:
- composability: one action can call several tools in a controlled order;
- object management: intermediate values can be named and reused;
- branching: the action can react to a missing result, empty list, low confidence, or error;
- replayability: action code, normalized calls, observations, logs, errors, and timing can be stored;
- auditability: the system can inspect what was tried, what failed, and what evidence was produced.

Compared with a plain JSON tool call:

#+begin_src text
JSON call: run this tool with these arguments.
Reasoning action: run this tool, inspect the result, branch if weak,
record evidence, then choose the next action.
#+end_src

This makes code-as-reasoning a better local action language for multi-step evidence gathering. It does not remove the need for schemas. It increases the need for schemas, because every executed action should be normalized into typed events and receipts.
* Why generic code-as-reasoning is not enough
Generic code-as-reasoning can record that an action called a tool and produced an observation. That is useful, but it is not enough for ASP.

Agent-grade codebase reasoning also needs to know:
- which provider-owned selector was proven;
- which owner, module, package, or artifact contains the evidence;
- which dependency edge made the route important;
- which test covers the selector;
- which Org/RFC/ADR decision refers to the evidence;
- whether the result came from stable DB state or a dirty overlay;
- whether this route failed in another session;
- whether the next action should read more, edit, test, write, or suppress the route;
- whether the action can be replayed without re-running providers.

ASP should therefore treat a reasoning action as the beginning of evidence construction, not as the final artifact.
* Legacy three-layer framing
This section is retained as historical compression only. The authoritative model is the four-layer model above: =ReasoningTree=, =EvidenceGraph=, =DynamicTopology=, and =GraphRoute=.

#+begin_src mermaid
flowchart LR
  RT["ReasoningTree<br/>How this run got here"] --> EG["EvidenceGraph<br/>What evidence exists and how it connects"]
  EG --> GR["GraphRoute<br/>Where the agent should go next"]
  GR --> NA["NextAction<br/>Exact selector / command / recovery"]
  NA --> RT
#+end_src

=ReasoningTree= is the causal trace of a run. It answers:

#+begin_src text
How did the agent get here?
Which action produced which observation?
Which branch failed?
Which recovery was tried?
Which evidence supports the answer?
#+end_src

=EvidenceGraph= is the cross-session evidence substrate. It answers:

#+begin_src text
What selectors, entities, tests, dependencies, artifacts, observations,
errors, receipts, and route decisions are known?
How are they connected?
Which facts are stable, overlay-local, stale, or accepted?
#+end_src

=GraphRoute= is the navigation policy over the evidence state. It answers:

#+begin_src text
Given this intent and what the system already knows,
which exact owner, selector, test, dependency, writing anchor,
or recovery action should the agent use next?
#+end_src

The relationship is:
- =ReasoningTree= produces dynamic evidence;
- =EvidenceGraph= joins dynamic evidence with stable codebase, artifact, Org, and DB facts;
- =DynamicTopology= projects graph evidence into the current architecture state;
- =GraphRoute= reads the graph plus topology and chooses the next action;
- the next action creates more reasoning evidence.
* Layer boundaries
The boundaries are strict.

=ReasoningTree= is not the graph-route. It is a replayable causal log. It may add dynamic weights such as "this route failed", "this observation was decisive", or "this recovery succeeded", but it must not become the canonical topology.

=EvidenceGraph= is not a transcript dump. It is a typed evidence substrate. It should store or reference facts that can be reused: selectors, entities, dependencies, tests, artifacts, search receipts, route receipts, render receipts, overlay facts, and bounded reasoning metadata.

=GraphRoute= is not the full reasoning history and not the topology itself. It is a ranked navigation result over EvidenceGraph plus DynamicTopology. It should be compact and agent-actionable: exact selector, owner, dependency, test, edit target, writing anchor, or recovery action.

The production loop is:

#+begin_src text
ReasoningTree -> EvidenceGraph/overlay -> DynamicTopology -> GraphRoute
-> compact render/recommendedNext -> ReasoningTree
#+end_src

This prevents two bad designs:
- treating every reasoning branch as durable route topology, which pollutes the DB under fast-changing code;
- treating graph-route as a replay log, which wastes tokens and hides the next actionable selector.
* Code action as graph-transforming evidence
For ASP, the strongest form is:

#+begin_src text
code action -> native parser proof -> evidence node/edge -> graph-route update
#+end_src

One reasoning action should be lowered into graph events:
- =parsed_as=: action code to AST/native parser proof;
- =calls=: action to tool or selector action;
- =reads_state= / =writes_state=: action to memory, overlay, DB, or runtime state;
- =produces=: action to observation;
- =fails_with=: action to error;
- =supports= / =refutes=: evidence to plan, selector, route, writing claim, or final answer;
- =derived_from=: evidence to generation, overlay namespace, source hash, artifact, or provider receipt;
- =ranked_by=: candidate evidence to Python graph scoring receipt;
- =rendered_as=: graph frontier to compact agent packet;
- =recommended_next=: packet to exact next command.

#+begin_src mermaid
flowchart TD
  A["CodeAction"] --> B["AST / native parse"]
  A --> C["ToolCall"]
  A --> D["SelectorAction"]
  A --> E["StateDelta"]

  B --> P["Parser proof"]
  D --> S["Stable selector or overlay entity"]
  C --> O["Observation"]
  E --> M["Memory / overlay state"]

  P --> EG["EvidenceGraph"]
  S --> EG
  O --> EG
  M --> EG

  EG --> R["GraphRoute rank"]
  R --> N["Next exact action"]
#+end_src

This is the point where ASP can exceed generic code-as-reasoning. The action is no longer just executed; it becomes a typed graph transformation. The system can later ask why the route was chosen, which selector was decisive, which branch failed, and which action should be suppressed.
* ASP enhanced loop
The complete ASP loop is:

#+begin_src mermaid
flowchart TD
  I["Intent"] --> P["Reasoning action"]
  P --> X["Execute / parse action"]

  X --> O["Observation / error / state delta"]
  X --> S["Selector action"]
  X --> T["Tool call"]

  S --> NP["Native parser proof"]
  NP --> EG["EvidenceGraph"]

  O --> RT["ReasoningTree"]
  T --> RT
  P --> RT

  RT --> EG

  EG --> GR["GraphRoute"]
  GR --> R["Compact ranked evidence"]
  R --> N["Recommended next action"]
  N --> P
#+end_src

ASP has stronger authority than a generic executor because:
- native parsers can prove selectors, owners, imports, definitions, calls, tests, and language-specific facts;
- DB Engine state can preserve stable generations, search receipts, route receipts, render receipts, and accepted actions;
- dynamic overlays can represent dirty code without forcing every edit into the durable DB;
- Python can run production graph-route ranking over normalized evidence;
- Julia can replay large session batches offline to discover which topology jumps reduce turns.
* User mental model
For users, the practical model is:

#+begin_src text
The agent is not just searching files.
It is building evidence and navigating a map of evidence.
#+end_src

A weak search loop looks like:

#+begin_src text
search -> read -> search again -> read more -> guess
#+end_src

The ASP target loop is:

#+begin_src text
intent -> evidence packet -> exact selector -> edit/test/write
#+end_src

Every meaningful action should become reusable evidence:
- a successful selector read can make future routes shorter;
- a failed route can be suppressed;
- a test edge can make a candidate more authoritative;
- a dependency edge can make a candidate more relevant;
- an Org/RFC edge can connect code search to writing search;
- a dirty overlay can change ranking without polluting stable DB state.
* Agent mental model
For an agent, the rule is:

#+begin_src text
Do not treat code search as line retrieval.
Treat it as evidence navigation.
#+end_src

The agent should prefer:
- selectors over line numbers;
- owners over raw files;
- route receipts over unstructured notes;
- compact evidence over raw JSON dumps;
- prior accepted receipts over cold search;
- graph-route next actions over broad exploration when enough evidence exists.

When more context is needed, the agent should ask for progressive disclosure:

#+begin_src text
compact packet -> more proof -> bounded snippet -> replay/audit
#+end_src

It should not default to:

#+begin_src text
dump whole file -> inspect manually -> repeat
#+end_src
* Search and writing implications
For code search:

#+begin_src text
intent -> reasoning action -> parser proof -> graph/topology expansion
-> compact evidence -> exact selector
#+end_src

For future writing search:

#+begin_src text
writing intent -> Org/RFC/ADR anchors -> code/entity evidence
-> prior reasoning tree -> compact outline/evidence packet -> write action
#+end_src

Reasoning trees become searchable state. An agent can ask:
- "What evidence did I already collect for this selector?"
- "Which route failed and why?"
- "Which action produced this observation?"
- "Which plan branch was abandoned?"
- "Which dynamic overlay changed the result?"

This avoids forcing every prompt into a cold search flow.
* Improvement scheme
The core ASP improvement scheme is:

1. Make reasoning events typed. A plan step, code action, tool call, observation, error, state delta, and final answer should have stable event identities.
2. Attach native parser proof. Search results should become selectors, entities, dependencies, tests, and provider-owned facts instead of line rows.
3. Store reusable evidence. Stable facts and accepted receipts belong in the DB Engine; high-churn dirty facts belong in dynamic overlay JSON until promoted.
4. Route through EvidenceGraph. Search should join stable DB facts, overlays, prior receipts, parser facts, Org/writing evidence, and reasoning events.
5. Rank with GraphRoute. Python production ranking should combine topology, freshness, prior success/failure, dependency/test proximity, novelty, and suppression signals.
6. Render compactly. Agent-facing output should expose the minimum sufficient evidence, exact next command, confidence, freshness, and =more= handles.
7. Replay offline. Julia should replay large session batches to measure which topology jumps reduce turns and which ranking failures should feed back into Python.

The target product behavior is:

#+begin_src text
intent
  -> code-as-reasoning action
  -> selector-first evidence
  -> EvidenceGraph update
  -> graph-route next action
  -> fewer turns to correct edit/test/write
#+end_src

The final purpose is not to store more data. The purpose is to reduce search turns by making evidence durable, typed, ranked, and reusable.
* Runtime and research authority
Runtime authority should remain strict:
- Rust owns high-performance substrate, schema validation, provider orchestration, DB/overlay reads, receipt persistence, and compact render.
- Python owns production graph-route algorithms: fusion, ranking, pruning, stale-route suppression, recovery ranking, and scoring receipts.
- Julia is offline only: large session replay, topology jump evaluation, ranking-failure mining, graph metric design, and candidate algorithm notes for Python.

Julia must not execute runtime route decisions. A route improvement should flow from Julia research to Python production implementation to Rust-validated receipt/render output.
* Core invariants
- Executable identity is selector/entity/generation/overlay id, not line range.
- Raw JSON is a debug/analyzer surface, not the default agent prompt surface.
- A reasoning action is not complete until it produces typed evidence or a typed failure.
- Stable facts and volatile overlays must remain separate until promotion is justified.
- GraphRoute output should be compact, ranked, and directly actionable.
- ReasoningTree output should be causal, replayable, and audit-friendly.
- EvidenceGraph should be selector-first and storage-backed enough for cross-session reuse.
