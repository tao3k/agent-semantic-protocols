#+TITLE: 10.27 Project-Scoped Agent Session Registry
#+PROPERTY: RFC_ID 10.27
#+PROPERTY: STATUS draft

* Intent

Agent session routing is project-scoped registry state. ASP must not create a
session artifact tree just to remember root and child session relationships.
The durable key is the current project plus the agent session ids.

* Contract

ASP records resident child sessions in the agent session registry DB with this
identity:

- =project_id=
- =root_session_id=
- =session_id=
- =parent_session_id=
- =name=
- =role=

The normal-agent entrypoint remains small:

#+BEGIN_SRC sh
asp agent session bootstrap --name asp-explore
#+END_SRC

The =project_id= is implicit from the current workspace. A normal agent never
copies a host child id, message target, or model into a shell command. The
platform adapter projects native lifecycle events into the registry; rollout
data remains recovery evidence, not routing authority.

* Gap Audit

The original gap was that ASP could observe a platform child session id and then
try to infer the root session from Codex rollout metadata. That is useful during
registration, but unsafe as a routing rule:

- an unregistered child could appear to belong to a root and accidentally reuse a
  resident child route
- a drifted child could keep running after its model, role, or agent config no
  longer matched the ASP lane contract
- global state could contain multiple projects, so root-only lookup could route
  a child from the wrong repository
- session artifacts would add another state owner even though the registry DB
  already owns durable session relationships

The fix is to make =project_id= part of the registry identity and to treat
rollout metadata as validation evidence only.

* Routing Rule

Pre-tool and binary gates resolve route state from the current project registry
first:

#+BEGIN_SRC text
project_id + current session id -> registered child/root relation
project_id + root session id + lane name -> resident child id
#+END_SRC

If the child is unregistered, ASP denies search/query/test work and asks the
agent to register the current child or create the configured child. If the child
is registered but validation fails, ASP denies route reuse and asks the agent to
close/delete the drifted child and create a fresh configured child.

* Implemented Enhancements

- =asp_agent_sessions= stores =project_id= and all route reads include that
  project scope.
- Register writes =project_id + root_session_id + child_session_id + name/role=
  into the registry DB.
- Reuse/status/pre-tool route checks no longer use rollout-only root inference
  for unregistered children.
- The platform adapter writes root, child, message target, role, and model from
  native lifecycle events; the normal agent does not supply those values.
- Model, role, root, parent, and explicit agent config path mismatches are hard
  failures.
- Missing rollout =agent_path= is warning-only when role/model/root/parent still
  match the expected ASP agent configuration. ASP compares the effective
  =asp.toml=/deployed agent configuration to rollout actual metadata; it does
  not require humans to pass =agent_path=.
- Sandbox mismatch is warning-only because Codex can inherit the parent sandbox.
- Register failure text tells the agent to close/delete the invalid child and
  recreate a fresh configured child.
- Main-session wrapped test/build commands such as =direnv exec . env ... cargo
  test= route to the testing lane instead of running in the root agent.

The current =project_id= implementation is intentionally minimal: it uses the
canonical project root as the DB scope key and does not create session artifacts.
If a stable repo-id helper already exists, this can be swapped behind the
project-id resolver without changing the registry contract.

* Flow

#+BEGIN_SRC mermaid
flowchart TD
  A["normal agent enters bootstrap pane"] --> B{"resident route ready?"}
  B -- yes --> H["dispatch through stored native message target"]
  B -- no --> C["invoke native managed create or resume"]
  C --> D["host emits SubagentStart"]
  D --> E["adapter validates native type and model"]
  E -- pass --> F["atomically claim project-scoped resident route"]
  E -- mismatch or duplicate --> G["reject new child; retain valid resident"]
  F --> H
  H --> I["host emits SubagentStop"]
  I --> J["retire only the matching native child route"]
#+END_SRC

* Non-Goals

- No =agent-sessions/roots/lanes/children= artifact tree.
- No TOML state files for session routing.
- No rollout-only route reuse for unregistered children.

Checkpoints, evidence summaries, and heartbeats can reuse the same project DB
with session ids as foreign keys. They do not require a separate session artifact
layout.

* Managed profile provenance

The configured managed-child profile is resolved from the canonical
=agents/*_codex.toml= file plus the dynamic model overlay. That resolved
profile, rather than the main session model or reasoning effort, is the only
source for an =asp_explorer= child model and reasoning effort.

Native create and native resume select the managed agent by its configured
role. They must load the resolved child profile. =message-agent= carries
search work and compact receipts only; it must not carry model or reasoning
instructions. A parent follow-up must not be used to emulate profile loading.

Validation compares the host-observed child profile with the resolved
configuration. A missing or mismatched model/reasoning observation leaves the
child in the validation/recovery loop. It is not a reason to inherit the main
session profile, fabricate a registry model, or create a generic replacement.

* Scenario Tests

These scenarios must remain covered as behavior tests, not only as unit-level
helper checks:

| Scenario | Expected behavior | Current coverage |
|---+---+---|
| Register child with no explicit root | Register validates rollout and fills root/parent | =asp_agent_session_register_infers_root_from_codex_child_rollout= |
| Unregistered child tries reuse | Reuse is a miss in the current project; rollout does not route | =asp_agent_session_reuse_in_unregistered_child_does_not_use_codex_rollout_root= |
| Global DB with two project roots | Same DB path does not leak sessions across projects | =asp_agent_session_defaults_to_global_state_root= |
| Drifted model | Register hard fails | =asp_agent_session_rejects_mismatched_codex_model_profile= |
| Drifted agent config path | Register hard fails and remediation says recreate | =asp_agent_session_rejects_mismatched_codex_agent_config_path= |
| Missing rollout agent path | Register warns but passes when role/model/root/parent match config | =asp_agent_session_warns_but_allows_missing_codex_agent_config_path= |
| Drifted registered child in hook path | Main-session search is denied and does not reuse the stale child | =codex_main_session_does_not_reuse_drifted_asp_explore_registration= |
| Wrapped test/build command | Root agent is denied and routed to =asp-testing= | =codex_main_session_routes_wrapped_test_command_to_asp_testing= |
| Installed hook full lifecycle | session-start reuse, main deny route, child allow, post-tool evidence | =codex_installed_hook_full_resident_child_lifecycle_scenario= |

The full lifecycle is covered by a behavior test that drives the complete flow
through installed hooks:

#+BEGIN_SRC text
session-start -> create configured asp-explore -> register -> main ASP query denied
and routed -> child query allowed -> post-tool heartbeat/checkpoint evidence
#+END_SRC

The remaining refinement is packaging this as a reusable declarative scenario
harness case, rather than only as a Rust behavior test.

* Open Gaps

** Codex Agent Metadata

Current Codex multi-agent startup can produce a subagent rollout without
=source.subagent.thread_spawn.agent_path=. ASP treats that missing field as a
warning when the rollout still matches the expected role/model/root/parent from
the ASP configuration. If the field is present and points at the wrong config,
ASP hard-fails the registration.

Resolution options:

- prefer a Codex/host fix that records the configured agent path in rollout
  metadata
- or add a Codex adapter proof that can validate the selected named agent without
  relying on =agent_path=

Until one of those exists, missing =agent_path= remains an auditable warning, not
a routing bypass. Model or role drift still fails registration.

** Stable Project Id Helper

The registry currently scopes by canonical project root. This avoids artifacts
and is deterministic, but it is not the same as the existing =repo-*= state
layout id. The next refinement is to reuse the official project-id resolver, if
one exists, behind the same project scope function.

** Declarative Scenario Harness

The command and hook behavior tests cover the failure modes and the full
lifecycle. A declarative scenario-harness wrapper would make the same lifecycle
easier to reuse for future lanes such as =asp-testing= and future host adapters.
* Native lifecycle transport authority
For Codex, generated native hook schemas are the source of truth. The
=SubagentStart= adapter uses this fixed projection:

| Registry field | Codex native field |
| =root_session_id= | =session_id= |
| =parent_session_id= | =session_id= |
| child =session_id= | =agent_id= |
| =message_target_id= | =agent_id= |
| managed role/profile | =agent_type= |
| observed model | =model= |

=agent_id= is both the value returned by native =spawn_agent= and the address
accepted by native =send_input=. A successful =SubagentStart= therefore closes
the message-target handshake without a model-authored receipt or a second probe.
=session_id= is the parent task and must never be registered as the child.

The adapter ignores non-ASP =agent_type= values so customer subagents remain
outside ASP lifecycle management. For the configured resident type, an atomic
claim admits one resident. A duplicate or model-mismatched start is rejected
through the native hook response while the existing valid resident remains
authoritative. =SubagentStop= retires only the registry row whose child id equals
the native =agent_id=.

The bootstrap pane does not accept child identity as normal-agent output. After
native create or resume, the normal agent re-enters the same pane. =Ready= is
derived only from the registry row written by the native lifecycle hook.

Codex currently does not expose effective reasoning effort in the generated
=SubagentStart= input. ASP hard-validates =agent_type= and =model=. Reasoning
remains managed-profile provenance until Codex adds a native
=reasoning_effort= observation; ASP must not present it as runtime-observed data.

* Project Hook State And Agent Config Projection
** Hook state belongs to the project workspace
Global =state-root/hooks/state= is not a valid owner for activation,
dispatcher TTL, replay, or hook event state. Those records are runtime state for
a concrete project/workspace and must be under the project layout:

#+BEGIN_SRC text
~/.agent-semantic-protocols/
  projects/by-id/<repo-id>/
    workspaces/<workspace-id>/
      live/hooks/cache/
      live/hooks/state/
        activation.json
        events.jsonl
#+END_SRC

The global state root may still own cross-project registries, runtime binaries,
provider locks, Org templates, and canonical agent config sources. It must not
own per-project hook activation or event state.

** Agent configs are ASP-owned, host configs are projections
ASP-owned agent definitions live under:

#+BEGIN_SRC text
~/.agent-semantic-protocols/agents/
  config.toml
  asp-explorer_codex.toml
  asp-testing_codex.toml
  asp-explorer_claude.md
#+END_SRC

The suffix selects the host projection target:

| Source suffix | Projection target |
| =_codex.toml= | =$CODEX_HOME/agents/<name>.toml= |
| =_claude.md= | =$CLAUDE_HOME/agents/<name>.md= |
| =_claude.toml= | =$CLAUDE_HOME/agents/<name>.toml= |

=asp sync= refreshes these projections. The host directories are not canonical
state; they are symlinks or copies that let Codex and Claude discover the
configured ASP agents. Registration validation reads the ASP canonical Codex
config first and falls back to the host projection only when the canonical
source is absent.

The =*_codex.toml= files are Codex-native projection sources and must stay
compatible with the Codex agent schema. ASP-only dynamic model metadata lives in
=~/.agent-semantic-protocols/agents/config.toml=:

#+BEGIN_SRC toml
[platform.codex.models]
primary = "gpt-5.3-codex-spark"
fallback = ["gpt-5.4-mini"]
capacityThreshold = 0.8
#+END_SRC

When a Codex capacity warning requires changing models, ASP should surface an
executable command instead of leaving only a mismatch warning:

#+BEGIN_SRC sh
asp agent session switch-model --name asp-explore --model gpt-5.4-mini
#+END_SRC

For Codex, =switch-model= updates ASP-managed subagent/child-session model
configuration in =agents/config.toml= and the matching Codex agent projection. It
does not switch the main session model. Later ASP-managed child sessions start
with the configured child-session model. A currently running child session still
needs a native message-agent follow-up from the main agent to confirm the
configured child-session model; after that receipt, validation accepts the
configured primary or fallback and must not keep emitting the previous child
model warning.

** Scenario coverage
The regression shape is:

#+BEGIN_SRC text
asp paths --json
  => hookStateDir is under projects/by-id/.../workspaces/.../live/hooks/state
  => activation is hookStateDir/activation.json

asp sync
  => scans ~/.agent-semantic-protocols/agents/*_{codex,claude}.*
  => refreshes host agent projections
  => reports agentConfigs=<n>

asp agent session register
  => validates model/role/root/parent against ASP canonical config
  => hard-fails model drift
  => warning-only for inherited sandbox mismatch
#+END_SRC
