#+TITLE: RFC 10.19 - Global ASP State Root and Artifact Layout
#+AUTHOR: agent-semantic-protocols
#+DATE: 2026-06-28
#+OPTIONS: toc:2

* One-sentence summary
ASP durable state lives under a global State Core root, keyed by stable
repo/workspace/scope identity, while every DB row and artifact remains
traceable to the exact invocation that produced it.

* State root selection
State Core has one executable override:
#+begin_src text
ASP_STATE_HOME
#+end_src

When unset, ASP uses:
#+begin_src text
$HOME/.agent-semantic-protocols
#+end_src

=PRJ_CACHE_HOME= is not part of the State Core v2 contract. The project-local
=<git-toplevel>/.cache/agent-semantic-protocol= path is diagnostic evidence
only. =asp state locate= may report that it exists, but v2 implementations must
not read it as a fallback, copy data from it, or silently migrate it.

* Identity model
Executable identity must not depend on display labels, project names, or line
ranges.

Repository identity:
- is derived from git toplevel, common gitdir/worktree metadata, remote URL
  evidence when available, and root-path fallback hash when git evidence is
  missing;
- is stable across worktrees for the same repository;
- uses display labels only as aliases.

Workspace identity:
- is derived from the concrete checkout/worktree root, gitdir/worktree metadata,
  and repo id;
- differs between worktrees that share a repo id;
- is the default bucket for live DB state, hook runtime events, session registry
  rows, and artifacts.

Scope identity:
- defaults to the workspace scope until a package/language manifest narrows it;
- is where monorepo package, provider, dependency, and analyzer facts attach.

* Canonical layout
The current physical layout is:
#+begin_src text
$ASP_STATE_HOME/
  VERSION
  state.json
  session-registry.turso
  registry/
    events.jsonl
  aliases/
    by-display-name/
  projects/
    by-id/
      <repo_id>/
        project.json
        workspaces/
          <workspace_id>/
            workspace.json
            live/
              client/
                client.turso
                manifest.json
                hooks/
                  events/
                  ttl/
                  dispatcher/
            artifacts/
              org/
              prompt-output/
              provider/
              analyzer/
#+end_src

The trace must remain stable even if the physical layout evolves:
#+begin_src text
repo_id -> workspace_id -> scope_id -> generation_id -> provider_id ->
search_receipt_id -> selector/snippet/artifact
#+end_src

=session-registry.turso= is the top-level ASP lifecycle control-plane
database. It is not a project artifact and MUST NOT be placed under
=projects/by-id/<repo_id>/=, =workspaces/<workspace_id>/=, or
=live/client/agent/=. Project and workspace directories carry project
identity, dynamic search/history data, command artifacts, provider output, and
analysis material consumed by Python, Julia, or other downstream tooling.

* Manifest contract
=live/client/manifest.json= is written by the owning State Core/DB Engine
service, not by command code. It records:
- =stateLayoutVersion= =state-v2=;
- =backend= =turso=;
- =futureBackend= =turso= while the report schema still carries that field;
- repo/workspace/scope ids;
- DB path;
- artifact path;
- generation manifest path;
- schema/report versions.

=stateLayoutVersion= versions the on-disk State Core layout. It is separate
from schema and protocol versions, which remain attached to their individual
JSON contracts.

Commands must call typed services to obtain these paths. They must not
reconstruct =projects/by-id/...= manually.

* Crate ownership
The command crate is a public facade only:
#+begin_src text
agent-semantic-protocol::command -> parse CLI, dispatch, compact render
agent-semantic-runtime::state_core -> state root, repo/workspace/scope identity
agent-semantic-client-core -> client-facing State Core services and reports
agent-semantic-client-db -> DB Engine schema, Turso path, sessions, manifests
agent-semantic-search -> search routing, overlays, source-index lookup
language harnesses -> native parser facts and shared packets
#+end_src

When a command needs new behavior, first define the typed service in the owning
crate, then call it from the command facade.

* Artifact contract
Artifacts are durable evidence, not raw source mirrors. By default, State Core
should store bounded packets, receipts, selector locators, graph facts, route
receipts, and analyzer outputs. Raw source text is excluded unless a schema
explicitly permits a bounded snippet or replay artifact.

Python and Julia analysis should be able to locate:
- DB Engine reports and route receipts;
- source/search generations;
- provider packet artifacts;
- prompt-output and compact render receipts;
- Org plans/checkpoints;
- analyzer experiment outputs;
- dynamic JSON overlay artifacts.

Python owns production graph-route algorithms. Julia owns offline algorithm
research and measurement. Both consume the same State Core identity and DB
Engine paths.

* State locate command
=asp state locate= and =asp state locate --json= are diagnostics over State
Core. They must include:
- state root;
- repo id and display label;
- workspace id and display label;
- scope id;
- client dir;
- manifest path;
- DB path;
- artifact path;
- backend and future backend fields;
- layout version;
- project-local cache detection as a diagnostic-only field.

The CLI must call the State Core service. It must not compose paths itself.

* Non-goals
- no agent MCP server as state owner;
- no project display name as executable identity;
- no line range as search/write identity;
- no =PRJ_CACHE_HOME= in the v2 state-root contract;
- no project-local cache fallback;
- no provider-local cache as authority over Rust-owned client state.

* Validation gates
- =ASP_STATE_HOME= set/unset tests pass;
- git repo, worktree, and non-git identity tests pass;
- same display name does not collide;
- =PRJ_CACHE_HOME= does not affect State Core;
- manifest JSON includes layout/backend/ids/DB/artifact paths;
- =asp state locate --json= validates against
  =schemas/semantic-state-locate-report.v2.schema.json=;
- no command-layer path helper reconstructs State Core v2 paths directly.
* Implementation checkpoint: M0 DB/cache identity closure
The M0 DB/cache closure keeps State Core as the source of path truth:
- active client DB paths resolve through State Core and DB Engine services under =live/client/client.turso=;
- command code must not compose =projects/by-id/<repo_id>/workspaces/<workspace_id>/...= paths by hand;
- external absolute workspace roots may be used by cache probing to choose the effective project state bucket;
- positional path arguments remain part of the CLI/provider forwarding contract unless they are explicit cache identity markers;
- cache request fingerprints must ignore trailing workspace markers such as =.= and absolute workspace directories so equivalent invocations reuse the same generation;
- project-local cache remains diagnostic-only and is still not a v2 fallback or migration source.

The matching validation gates are:
- =cargo test -p agent-semantic-client --locked cli_args -- --nocapture=
- =cargo test -p agent-semantic-client --locked cache_cli -- --nocapture=
- =cargo test -p agent-semantic-protocol --locked provider_command::receipt -- --nocapture=
- =cargo test -p agent-semantic-client-db --locked=
