#+TITLE: RFC 10.21 - ASP DB Engine, State, and Search Architecture
#+AUTHOR: agent-semantic-protocols
#+DATE: 2026-06-29
#+OPTIONS: toc:2

* One-sentence summary
ASP uses Turso-backed DB Engine v1 plus State Core v2 to build a
selector-first, overlay-aware, graph-routed code search substrate whose metric
is fewer agent turns from intent to evidence.

* Scope
This report consolidates the DB Engine, State Core, code-search, Rust
high-performance, language harness, Python graph-route, and Julia research
workstreams.

In scope:
- Turso-backed DB Engine v1 as production storage;
- global State Core under =$HOME/.agent-semantic-protocols= or =ASP_STATE_HOME=;
- source-index and structural-index cold functional gates;
- durable search generations, route receipts, topology rows, and analyzer
  artifacts;
- dynamic dirty-code overlay search;
- thin command facades;
- shared language harness contracts;
- Python production graph-route algorithms;
- Julia offline ranking and topology research.

Out of scope:
- agent MCP server as architecture owner;
- line numbers as executable identity;
- storing every dirty edit as durable generation rows;
- one global tree-sitter bundle replacing native providers;
- direct =tantivy= as a second product backend.

* Terminology
=graph-route= is the production topology-backed search/navigation read model.
It chooses a next command, selector, owner, dependency, test, or recovery route
from current evidence. It must be compact, ranked, and directly actionable.

=reasoning tree= is the causal run trace: intent, plan, action, observation,
error, recovery, evidence, and answer. It can produce route signals but is not
the canonical topology.

=EvidenceGraph= is the join point. Stable codebase facts, DB generations,
dynamic overlays, and reasoning-step metadata become typed nodes and edges.
=graph-route= reads and ranks this substrate; =reasoning tree= appends bounded
new evidence.

=where-frame= answers where to inspect or edit: owner, selector, dependency,
test, artifact, route, or scope.

=how-frame= answers how to change safely: intended edit, risk, validation,
write path, rollback evidence, and follow-up command.

Where and how are separate schemas because search and editing are different
agent tasks.

** Dynamic overlay lanes
=dynamic overlay= is the session-scoped dirty-code overlay domain. It is not a
single route source.

It has two public lanes:
- =query-overlay= serves ASP query-set commands such as =asp fd -query= and
  =asp rg -query=. It projects query clauses, query-pack quality, candidate
  sources, and graph-turbo request seeds for the query mechanism.
- =search-overlay= serves ASP search commands such as =asp ... search ...= and
  =search pipe=. It projects search candidates, search pipe traces, and
  selector/owner seeds for the search mechanism.

Durable Turso generations and source-index facts can precede either lane. A
source-index hit may skip the corresponding overlay lane, but the trace must name
the skipped lane precisely, for example =query-overlay:skipped= for query
wrappers and =search-overlay:skipped= for search pipe. Public receipts must not
collapse both lanes into =finder= or a generic =overlay= token.
* Agent session gated search loop
The main agent owns judgment, editing, checkpoint decisions, and final
synthesis. The resident =asp-explore= child owns parser-backed search/query
execution and compact evidence production. If a runtime session id is present,
restricted search/query commands must pass through the resident validated child
or fail closed before artifact writeback.

Durable evidence from this loop feeds graph-route:
- root/child session relationship and validation status;
- provider search receipts and selector/owner/dependency facts;
- post-tool command/evidence refs;
- Org plan/checkpoint state;
- suppression signals for repeated low-value routes.

* Current implementation checkpoint
Current local evidence:
- =agent-semantic-client-db= is the DB Engine surface.
- Default builds resolve =backend=turso= and =client.turso=.
- DB Engine facades exist for cache generations, syntax replay,
  provider-command selections, artifact events, agent sessions, graph, search,
  overlay documents, source-index, structural-index, and route receipts.
- =agent-semantic-runtime::state_core= resolves =ASP_STATE_HOME=,
  =state-v2=, repo/workspace/scope ids, and State Core client/artifact paths.
- =agent-semantic-client-core= re-exports/consumes State Core through
  =ProjectContext=.
- =asp paths= and =asp state locate= report State Core paths instead of
  project-local cache paths.
- =agent-semantic-search= owns source-index lookup orchestration and dynamic
  overlay routing.
- command-layer search candidate collection is partially thin:
  command code delegates candidate collection to =agent-semantic-search= and
  keeps CLI/render projection.
- Python =asp_graph_turbo.rank_frontier= already owns production graph ranking
  behaviors such as kind budgets, read-memory suppression, query-token
  coverage, and receipt construction.
- Julia analyzers already consume artifact/search flow data, but the artifact
  trail still needs stronger session/prompt/action/parent ids and
  recommendedNext adoption status.

* Milestone plan
** M0 Patchset hygiene
Goal: keep the current patchset auditable before adding new behavior.

Done gates:
- workspace =cargo fmt --check= passes;
- =git diff --check= and cached diff check pass;
- no dead-code suppression attributes;
- no public old backend token or version-suffixed Turso backend token;
- =cargo test -p agent-semantic-client-db= passes;
- =cargo test -p agent-semantic-client-core= passes.

** M1 DB Engine contract cutover
Goal: public DB contracts say Turso, while any pre-cutover validation evidence
stays DB-crate-internal.

Required work:
- public schemas use =backend=turso= and =database.engine=turso=;
- receipts/manifests do not expose old backend names;
- command/client code obtains DB paths through =ClientDbEngine= or State Core;
- source-index, structural-index, cache, syntax, provider selection, artifact,
  session, graph, search, overlay, and route receipt traffic stay behind DB
  crate facades;
- schema docs describe Turso as DB Engine v1, not as a future optional backend.

Validation:
- DB Engine report schema tests;
- client receipt schema tests;
- =asp cache status= includes =|dbEngine backend=turso=;
- =asp state locate --json= includes State Core paths and =backend=turso=.

** M2 Source-index cold functional gate
Goal: a clean workspace can refresh source-index and then search through the
source-index path without silent finder fallback.

Required work:
- scenario fixture starts from missing DB;
- =asp cache source-index refresh= creates the Turso source-index read model;
- subsequent =asp ... search ...= returns =source=source-index= or an explicit
  source-index state;
- missing DB emits =sourceIndex:missing-db= with refresh hint;
- cold gate records elapsed time, provider spawn count, stdout bytes, route
  source, fallback reason, and DB path.

Cold functional gates come before hot-path latency gates.

** M3 Dynamic overlay search adapter
Goal: dirty-code search uses the same lexical semantics family as DB Engine
search without inserting every transient edit into stable generations.

Required work:
- overlay namespace includes repo/workspace/session/base generation;
- overlay facts store dirty hashes, changed/deleted selector ids, bounded
  snippets, parse-delta summaries, active failures, and overlay-local entity ids;
- search router order is receipt anchors -> overlay -> dirty provider parse ->
  stable Turso FTS/BM25 -> semantic/vector fallback;
- lexical is not an agent-facing ranking backend.

Validation:
- dirty edit search returns selector-first evidence;
- stable generation stays clean;
- no line-range executable identity.

** M4 Command layer thin facade
Goal: =agent-semantic-protocol/src/command/*= is CLI parse, dispatch, and render
only.

Required work:
- source-index lookup, owner-items cache, candidate collection, graph-route
  preparation, runtime cache, and session storage live in owning crates;
- command files do not hard-code language ecosystems;
- language behavior comes from provider manifest and harness interface.

Validation:
- command-focused tests prove dispatch/render only;
- shared schema paths cover Rust, Python, TypeScript, Org, and Julia.

** M5 Language harness interface and performance matrix
Goal: most languages implement one shared owner-items/search-seeds contract.

Required work:
- define shared owner-items/search-seeds schema;
- =asp init --language <name>= scaffolds schemas, templates, guide docs, and
  harness skeleton in a chosen folder;
- special language facts are provider-owned extensions, not ASP core branches;
- cold functional gates exist for Rust, Python, TypeScript, Org, Julia, and
  Gerbil Scheme;
- hot gates follow only after cold gates pass.

Receipt fields:
- elapsed time;
- provider spawn count;
- stdout bytes;
- route source;
- fallback reason;
- selected selector/owner count.

** M6 Graph-route, where-frame, and how-frame
Goal: reduce search turns by feeding compact, progressive evidence to agents.

Required work:
- =where-frame= renders the next inspection/edit locations;
- =how-frame= renders edit strategy and validation plan;
- Python consumes DB/overlay/topology receipts to rank production routes;
- Julia mines failures and proposes ranking improvements offline;
- compact render avoids raw JSON dumps and supports progressive disclosure.

Validation:
- route receipts record selected route, rejected routes, evidence ids, freshness,
  and next command;
- Python graph-route tests show ranking uses topology plus receipt memory;
- Julia analysis can reconstruct route quality from DB/artifact data.

* Acceptance criteria
The architecture is accepted when:
- Turso is the only public DB backend token;
- State Core paths are the only v2 durable paths;
- source-index cold functional gate passes before hot gates;
- dynamic overlay search no longer depends on lexical as agent ranking;
- command-layer heavy logic is moved to owning crates;
- language harnesses share the common contract;
- graph-route emits compact where/how evidence that lets agents reach decisive
  snippets in fewer turns.
