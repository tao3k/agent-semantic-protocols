* Terminology bridge: graph-route and reasoning tree
This architecture depends on the terminology defined in [[file:10.22-smolagents-code-as-reasoning-evidence-graph.org][10.22 smolagents code-as-reasoning EvidenceGraph]].

=graph-route= is the topology-backed search/navigation read-model. It chooses a next command, selector, owner, dependency, test, or recovery route from the current evidence state. It should be compact, ranked, and directly actionable by an agent. This is the production home for ASP topology theory: the route layer should use typed topology edges, neighborhood expansion, centrality, bridge nodes, dependency/test proximity, stale-route suppression, and high-value topology jumps to reduce search turns. The DB Engine should persist enough stable graph-route inputs to make repeated searches cheaper across sessions: source generations, selectors, symbols, dependencies, topology edges, search receipts, render receipts, accepted next actions, and suppression signals for repeated low-value routes.

=reasoning tree= is the causal run trace. It records how a specific agent session or subagent branch moved from intent to plan, action, observation, error, recovery, evidence, and answer. It is not the route itself. It can produce route signals, but it remains run evidence. The Turso-backed DB Engine v1 should store bounded metadata for reasoning runs and steps, while volatile or bulky details stay in dynamic JSON overlay artifacts.

=EvidenceGraph= is the join point. Stable codebase facts, DB generations, dynamic overlays, and reasoning-step metadata become typed nodes and edges. =graph-route= reads and ranks this substrate; =reasoning tree= appends new evidence to it.

Topology data belongs on the graph-route side of the architecture. Reasoning trees may add dynamic edge weights such as "this route failed", "this selector produced the decisive observation", or "this recovery branch succeeded", but they should not become the canonical topology. The canonical topology is built from provider-native parser facts, dependency/test/Org/artifact facts, DB generations, and bounded overlay facts.

Python and Julia are both part of the topology-backed route algorithm system, but they must not share runtime authority. Python owns production graph-route algorithms: fusion, scoring, pruning, suppression, recovery ranking, next-action ranking, and compact scoring receipts. Julia is strictly offline research and analysis: graph-algorithm optimization, topology metric design, DB/overlay data analysis, ranking-failure studies, and algorithm-quality experiments. Its concrete deliverables are large session replay, measurement of which topology jumps actually reduce search turns, ranking-failure mining, and candidate algorithm notes that Python can implement and validate in production. Julia does not execute runtime route decisions. A route improvement should flow from Julia research to Python production implementation to Rust-validated receipt/render output.

The production loop is:
#+begin_src text
ReasoningTree -> EvidenceGraph/overlay -> GraphRoute -> compact render/nextCommand -> ReasoningTree
#+end_src

This prevents two bad designs:
- treating every reasoning branch as durable route state, which would pollute the DB under fast-changing code;
- treating graph-route as a full replay log, which would waste tokens and hide the next actionable selector.

* Dynamic search implementation checkpoint
- Dynamic dirty-code search is a separate Rust-side orchestration boundary, not a DB write path and not an ASP-owned language parser.
- High-performance search has a fixed responsibility split:
  - =languages/*-harness= command-line providers own parser truth, language ecosystem facts, owner-items/query/snapshot/diff contracts, stable selectors, display ranges, and language-specific schemas.
  - Rust ASP owns the wrapper around those CLIs: provider discovery, manifest/capability validation, trust gates, process lifecycle, warm worker reuse, timeout/cancellation, batch dispatch, stdout/stderr contract validation, cache/writeback, overlay merge, DB Engine ingestion, graph bridge inputs, receipt emission, and compact agent rendering.
  - Python owns production graph algorithms over normalized ASP facts; Julia owns offline/research algorithms and artifact analysis. Neither replaces harness parser truth.
This split is the core performance architecture. The win is not that ASP reimplements each language faster; the win is that each harness does only the native semantic work, while Rust amortizes process startup, normalizes packets once, caches stable generations, keeps dirty overlays cheap, and renders compact evidence without forcing every agent turn through cold provider startup or raw source scans.

- Provider =workspace-scope= execution and packet parsing are local provider transport semantics. =agent-semantic-client-local-cli= owns =provider_workspace_scope=, =provider_workspace_scope_from_stdout=, schema-id/status handling, and provider-default language/provider inheritance. =agent-semantic-client= may project a supported packet into source-index scope files after path containment and ignore checks, but it must not construct provider CLI requests or parse provider workspace-scope JSON itself.
- Pure provider-manifest path rules are client-core contracts, not client orchestration and not command-owned language tables. =agent-semantic-client-core::provider_scope= owns source-extension matching, ignored-prefix matching, project-relative path normalization, and safe scoped child-path construction because those rules derive from activation/provider manifest identity. =agent-semantic-client= may still perform transient workspace IO for source-index collection and apply these rules, but it must not define its own provider path grammar. Command modules may ask for =language_file_spec=, but they must not define language ecosystems or maintain their own extension tables.
- =search owner <path> items --workspace . --view seeds= no longer requires =--query=. An empty query means "enumerate bounded owner-local structural anchors" for the selected owner, but the anchors must come from a language harness/interface. The provider capability flag may optimize schema routing, but it must not block dispatch to the language harness CLI. If the harness returns no owner-items contract for an existing owner, the route fails closed instead of synthesizing language facts.
- =query <path> items --workspace . --code= should become the code projection over the same harness/interface item set. Until that interface exists for a language, ASP must not use client/protocol hard-coded parsing as a compatibility shortcut.
- Dynamic output should be marked with =alg=asp-dynamic-owner-items-v1= or =source=dynamic-owner-items= only when it is rendering harness/interface facts or overlay facts, not when ASP guessed declarations from raw source.
- Owner-items runtime policy is runtime-owned, not command-owned and not client-owned. =agent-semantic-runtime::run_language_owner_items_dispatch_plan= owns the existing-owner check, provider-only execution, and fail-closed error surface. The same runtime module owns the language-owner-items cache key, cache read/write, compact stdout filtering, and failure rendering because those are provider lifecycle/runtime semantics rather than client aggregation semantics. =agent-semantic-client= must not re-export this surface as its own API and must not regain harness dispatch, cache, compact render, or failure policy. =agent-semantic-protocol/src/command/search_pipe_owner_items_fast.rs= may parse CLI args and supply the provider adapter, but it must not decide that a language-specific inline parser runs before, after, or instead of the language harness. Legacy inline owner-items fallbacks are removed under the new architecture; a language without a provider-owned owner-items interface fails closed until its harness/schema route exists.
- This checkpoint is intentionally narrower than the future Turso/Tantivy overlay search. It proves the architecture boundary first: command facades stay thin; =agent-semantic-search= owns dynamic search, document candidate collection, document auto/finder fallback acquisition, finder acquisition, source-index acquisition policy, and overlay routing; client/cache crates stay service-specific; durable DB generations remain stable; dirty overlay evidence is not synchronously inserted into Turso or SQLite for every edit.
- The dynamic overlay control path must not become a language syntax table inside ASP. Full language semantics, including Rust modifiers such as async/visibility forms, come from =languages/*-harness= command-line interfaces or their emitted schemas/artifacts. ASP may orchestrate those commands, cache their compact results, merge them with overlays, and render the agent packet, but it must not depend on the harness crate or reimplement provider parser rules in the client/protocol crates.
- The next dynamic phase should replace line-oriented lexical matching with the Turso-owned search implementation when that boundary is exposed, plus provider/native parser deltas where available. Direct =tantivy= must remain a control/benchmark dependency unless it is consumed through Turso's search package or DB Engine adapter.
* Millisecond performance gate matrix
High performance is a core product requirement, not a later optimization. The new architecture must make the common agent search path complete in milliseconds once the binary and harness workers are warm; =cargo run= startup time and tool shell startup are not counted as search latency.

Every language/provider lane should have scenario receipts with:
- cold-path functional gates first: an already-built =asp= binary invokes an installed provider/harness from an empty per-scenario cache and still returns schema-valid selector-first evidence in the millisecond class for bounded owner/search routes;
- hot-path gates second: repeated runs may prove cache/worker reuse, but they must not be the only evidence that the route is correct or performant;
- elapsed milliseconds for cold parser run, hot cache run, dynamic overlay run, and source-index warm path;
- provider spawn count;
- stdout byte count and rendered packet byte count;
- route source, such as =source-index=, =dynamic-overlay=, =harness-owner-items=, =turso-fts=, or =graph-fallback=;
- fallback reason and whether a fallback was expected;
- selected candidate count and first actionable selector.

Initial acceptance gates:
- Cold owner-items functional path is the first scenario priority. For each language, the first =search owner <path> items --view seeds= gate must prove one provider-owned harness invocation returns shared owner/item evidence, records =route_source=owner-items-provider=, permits exactly one provider process, avoids native finder fallback, avoids source synthesis in =command/*=, and avoids executable line-range identity. This is the correctness gate before any hot-path cache optimization is credited.
- Org/Markdown no longer need a route bridge from =query --view metadata= to owner-items. Their embedded orgize manifests declare =searchCapabilities.ownerItems=true= and route owner lookup through the shared =asp org|md search owner <path> items --workspace <root> --view seeds= surface. =asp_org_owner_items_cold_functional_path= is the Org cold functional gate; hot-cache work remains a later optimization because the first acceptance requirement is provider-owned correctness on a cold request.
- Rust owner-items hot path: =asp_rust_owner_items_cache_hot_path= is the first hot-cache =search owner <path> items --view seeds= gate. It proves bounded owner-local search replays cached harness output without provider respawn, native finder fallback, source synthesis, or executable line-range identity.
- TypeScript owner-items hot path: =asp_typescript_owner_items_cache_hot_path= extends the same shared owner-items cache/orchestration gate to a non-Rust language through the =ts-harness= manifest/capability route. This is the first proof that the architecture is language-harness generic rather than a Rust-only fast path.
- Python owner-items hot path: =asp_python_owner_items_cache_hot_path= extends the same proof to =py-harness= through the provider manifest route, keeping Python parser authority in the language harness while Rust ASP owns warm cache replay and compact rendering.
- Julia and Gerbil Scheme cold owner-items paths: =asp_julia_owner_items_cold_functional_path= and =asp_gerbil_scheme_owner_items_cold_functional_path= extend the cold functional gate to =asp-julia-harness= and =gslph=. The Gerbil gate specifically proves the route uses the provider-owned owner-items interface rather than the retired Rust inline parser path.
- Source-index warm path: =source=source-index= must be emitted and must not silently fall back to native finder.
- Dynamic overlay root path: workspace walk, language predicate filtering, and overlay projection must stay in low double-digit milliseconds on focused roots and must not produce line-range executable identity. The =asp_dynamic_overlay_search_pipe_warm_path= scenario fixes this as a real CLI gate for =asp rust search pipe ... --source finder= with =route_source=dynamic-overlay=, zero provider processes, bounded stdout, and no fallback.
- Search pipe candidate collection: query tokenization, path-query detection, owner-root resolution, native finder routing, dynamic overlay routing, ignore handling, language file predicates, and provenance projection belong to the dedicated =agent-semantic-search= crate. =agent-semantic-protocol/src/command= may only adapt CLI config into =SearchPipeCandidateRequest= and render returned candidates. =agent-semantic-client= must not become the replacement monolith for command-heavy search logic; existing client search primitives are migration seams to be moved into narrower owner crates as M4/M5 continues. Native fd/rg/exa finder execution is now an =agent-semantic-search= responsibility rather than a client facade responsibility.
- Pipe ingest candidate projection: newline/NUL record splitting, =path:line[:column]:text= parsing, path normalization, and compact symbol projection belong to =agent-semantic-client=. =agent-semantic-protocol/src/command/search_pipe_candidates.rs= may read stdin and convert client candidates into the render model, but it must not own byte-level ingest parsing or path-resolution algorithms.
- Query-wrapper candidate collection: term splitting, identifier-axis expansion, source-index lookup/filter/projection, fallback fd/rg filesystem scanning, package-path augmentation, candidate priority, language-neutral file filtering, provider-manifest file boundary resolution, and query-wrapper quality/risk/noise/package-cohesion analysis belong to =agent-semantic-search=. The command layer may convert protocol query clauses into =QueryWrapperSearchRequest= / quality DTOs and render compact traces, but it must not own recursive filesystem search, package expansion, source-index candidate projection, native finder routing, query-axis construction, cohesive candidate filtering, or query-quality policy. =agent-semantic-client= no longer owns native finder, query-wrapper scan, or language file-spec primitives.
- Cache status/import/flush/invalidate DB paths: DB open/inspect must go through =ClientDbEngine= facade helpers so the Turso cutover can change backend implementation without command-level rewrites.
- Per-language matrix: Rust, TypeScript, and Python now have both cold functional =owner-items-provider= gates and hot cache =owner-items-cache= gates. Julia and Gerbil Scheme now have cold functional gates and still need hot cache gates. Org still needs an owner-items-compatible provider route or schema bridge because its current manifest owner route is =asp org query --selector ... --view metadata= rather than =search owner <path> items --view seeds=. Special language facts may add provider-owned extension fields, but the route, receipt, and performance fields must use the shared contract.

Performance failures should block architecture acceptance even when functional tests pass. Any new search route in =crates/agent-semantic-protocol/src/command/*= must either stay a thin dispatch/render facade or add a scenario gate proving why the heavy work cannot live in the owning client/db/provider crate.

Scenario gates are the acceptance surface for performance-sensitive search work. A focused unit test may protect an internal helper, but it is not sufficient proof for M0-M6 performance closure unless a =tests/unit/scenarios/*/benchmark.toml= fixture records the command, route source, maximum provider process count, maximum stdout bytes, fallback reason, observed timing, and an executable scenario test that validates the real ASP output. Source-index warm paths now assert =route_source=source-index=, =max_provider_process_count=0=, =max_stdout_bytes=8192=, and =fallback_reason=none= in scenario metadata, so DB/search refactors cannot silently regress into provider spawn, native finder fallback, or unbounded output.

The cross-language scenario contract floor is now explicit: Rust, TypeScript, Python, Julia, Orgize, and Gerbil Scheme each must expose at least one scenario benchmark with route source, provider process ceiling, stdout ceiling, and fallback reason metadata. Existing parser/policy scenarios may satisfy only this metadata floor with =route_source=parser-scenario=; they do not replace the remaining M5 work to add true per-language =search owner <path> items --view seeds= hot-path scenarios.

* Search backend decision
Persisted generations should use Turso FTS/BM25 over ASP entity documents.

Dynamic code should not be pushed into the durable Turso DB on every dirty edit. The router should merge:
#+begin_src text
agent memory / prior receipt anchors
-> active overlay search
-> dirty-file provider parse
-> stable Turso FTS/BM25
-> semantic/vector fallback
-> Python graph fusion
-> compact render
#+end_src

Tantivy dependency rule:
- ASP should not directly introduce =tantivy= as a second product search backend.
- Persisted search uses Turso FTS.
- Dynamic overlay search should use the same Turso-owned Tantivy-backed semantics when that boundary is exposed or vendorable from Turso.
- If that boundary is not stable, ASP keeps a small adapter/control implementation for experiments and falls back to current search only as a migration lane.
- =fzf= should stop being agent-facing ranking after parity. It can remain as a human interactive affordance or temporary migration control.

Replacement gates:
- recall@k, MRR, nDCG, false-positive-at-1;
- warm overlay update latency;
- dirty-buffer indexing cost;
- selector stability across edits;
- field-hit metadata quality for graph fusion;
- parity against current =fzf= on path/name discovery.
* FTS/BM25 plus identifier search versus fzf
=fzf= is strong for interactive fuzzy narrowing, path discovery, and human recall. It is weak as an agent search substrate:
- no durable field weights;
- no BM25-style term/document scoring;
- no stable receipt explaining why a candidate won;
- no first-class camelCase/snake_case/path-segment expansion;
- no native concept of generation freshness or overlay visibility;
- no graph-fusion metadata.

FTS/BM25 plus identifier search is better for the agent loop because it can return:
- selector-first entities;
- field hits by name, qualified name, signature, doc text, dependency, and route terms;
- deterministic identifier tokenization;
- current-generation filtering;
- structured rank features for Python graph fusion;
- replayable receipts for Julia research.

The dynamic-code caveat is real: dirty code changes too fast for every edit to become a durable DB write. The answer is overlay search, not line-number search. Overlay documents are keyed by selector/source hash when possible, and by temporary overlay-local entity ids when not.
* Search playbook and turn reduction
=codebase-memory-mcp= wins today on productized few-turn search. Its playbook is direct:
#+begin_src text
search_graph / semantic_query / query_graph / search_code
-> trace_path when relationship evidence is needed
-> get_code_snippet when bounded source evidence is needed
#+end_src

ASP can beat that only if its default command returns a better evidence packet, not a larger JSON blob.

Target ASP playbook:
#+begin_src text
intent + memory
-> router chooses skip | resume | refine
-> selector/owner/dependency/test/org anchors
-> overlay + stable Turso FTS + native parser proof
-> Python graph fusion
-> compact render with recommended next exact actions
#+end_src

One-turn target:
- If memory already contains a valid selector, jump directly to freshness check and exact action.
- If intent maps to an owner/dependency/test/Org anchor, skip broad lexical retrieval.
- If lexical retrieval is needed, return ranked selectors plus a compact proof and the next exact command in one packet.

Two-turn target:
- Turn 1 returns =EvidencePacket= with top selectors, graph paths, proof sections, and bounded snippets when safe.
- Turn 2 executes exact selector action or asks for a specific larger disclosure.

The router must not force every request through cold retrieval. Agent memory and prior receipts are first-class inputs.
* No line-number identity
Line ranges are display hints, not executable identity.

Executable identity must be:
- provider selector;
- native fact id;
- entity id;
- generation id + source hash;
- overlay namespace + overlay-local entity id when the stable selector is unavailable.

The render may include =displayRange= and =sourceLocatorHint= for readability, but follow-up commands must use selectors and provider-owned query forms.
* Tree-sitter plus native query combinator
The Codebase-Memory paper validates the value of tree-sitter-scale structural indexing. ASP should not reject that lesson just because ASP has native providers. The better design is a combinator:
#+begin_src text
tree-sitter-compatible S-expression query
-> coarse structural candidates
-> native provider query-lang proof
-> dependency/test/Org topology expansion
-> Python graph fusion
-> compact render
#+end_src

This can be cheaper than multiple agent turns because the whole combinator is one command and one receipt. It only pays off if receipts preserve the proof boundaries:
- =structuralProof=: grammar id, query AST hash, captures, predicate results, structural spans, candidate counts.
- =nativeProof=: provider id, selector, semantic facts, omitted facts, rejection reasons, source fingerprint, exact locator.
- =fusionProof=: graph seeds, edge types, rank inputs, diversity decisions, confidence, final action source.

The planner should store =PlanTrace= for every considered combinator route:
- route choice;
- internal stage latencies;
- candidate counts;
- proof gain;
- fallback reason;
- graph confidence;
- final action source.

Julia should compute =combinator_win_rate= by comparing one combinator packet against the shortest non-combinator path for the same prompt class.
* Compact render contract
The agent should not read raw JSON unless it asks for debug output. JSON wastes tokens and hides priority.

The render contract should be schema-backed, compact, and progressively disclosed:
- top line: route, confidence, freshness, generation/overlay state;
- ranked selectors with stable ids;
- proof summary per selector;
- graph paths only when they change the decision;
- exact =recommendedNext= command;
- =avoid= and stale/overlay warnings;
- optional =more= handles for larger disclosure.

ASP's existing compact graph ideas such as =frontier=, =routeGraph=, =G1/G2/G3=, and =S1/S2= are directionally right, but they need a unified schema. The command layer should not hand-roll render formats per language.

Render should live in Rust to reduce harness burden. Harnesses should emit facts; Rust should render facts into the agent prompt surface.
* Rust high-performance architecture
The command layer under =crates/agent-semantic-protocol/src/command/*= should become thin: argument parsing, policy checks, and handoff only. Heavy work belongs in focused crates:
- =agent-semantic-state-core= or existing client-core state module: identity, manifests, State Core v2.
- =agent-semantic-db-engine=: Turso backend, schema migration, transactions, receipts, stable generations.
- =agent-semantic-search-core=: lexical/semantic/selector search, overlay merge, field-hit metadata.
- =agent-semantic-router=: evidence-state routing, memory skip/resume/refine, route receipts.
- =agent-semantic-render=: schema-backed compact graph render.
- =agent-semantic-provider-bridge=: provider worker protocol, process pools, packet validation.

The core feature is that a language harness only implements native parsing and ecosystem facts. It does not implement high-performance search, cache invalidation, graph ranking, route planning, or DB layout.

Tokio/process design:
- persistent provider worker pools per =language_id/provider_id= where possible;
- =tokio::process= with length-delimited JSON or JSONL protocol;
- backpressure, cancellation, and timeout scopes;
- bounded concurrency by provider, workspace, and CPU-heavy parse class;
- =Semaphore= gates for DB writers, overlay index updates, provider workers, and render fanout;
- cancellation for abandoned agent branches;
- warm DB connections and prepared statements behind a backend trait;
- =spawn_blocking= only around unavoidable CPU-bound or blocking library calls;
- batch writes for generation imports and overlay checkpoints;
- in-memory hot maps for overlay selector deltas, with durable writes only for replayable receipts and accepted checkpoints.

The performance goal is:
#+begin_src text
harness packet -> Rust validation -> DB/overlay projection -> graph/search router
-> compact evidence render -> exact selector action
#+end_src
* Python and Julia interface
Julia should be able to analyze =DB + dynamic JSON= without forcing every volatile fact into DB.

DB Engine responsibilities:
- stable generations, graph/topology rows, receipts, and metrics;
- =PlanTrace= and =SearchStage= with candidate counts, latencies, route choices, proof gain, fallback reason, and final action source;
- artifact links to generation, session, prompt/action, and provider command ids;
- bounded dynamic overlay JSON when normalization is not worth it.

Julia responsibilities:
- replay route decisions without re-running providers;
- test graph algorithms, combinator thresholds, diversity budgets, and abstention rules;
- measure warm-context skips, overlay wins, false skips, and combinator win rate;
- publish reproducible notebooks.

Python responsibilities:
- implement selected graph algorithms from Julia experiments;
- run production graph fusion and diversity ranking;
- return compact receipts to Rust for storage and render.
* Audit-driven implementation roadmap
The current RFC already has the right architectural direction, but the implementation must be sequenced so that ASP does not create a second monolith while replacing SQLite v1. The audit verdict is:
- the DB Engine facade and State Core v2 path model exist, but active storage is still =sqlite-v1= and =futureBackend=turso= is only a reportable skeleton;
- source-index and structural-index transaction seams are good enough to project into an EvidenceGraph schema, but the current rows are not yet the target read model;
- dynamic search has an owner boundary in =agent-semantic-search=, but overlay search still needs a Turso-owned Tantivy-semantics path or a controlled adapter before =fzf= can be retired for agent ranking;
- =crates/agent-semantic-protocol/src/command/*= is still the main risk area. Command modules may parse CLI args and render receipts, but provider lifecycle, runtime policy, DB state, dynamic search, graph routing, and render schema must live in owner crates;
- Python production ranking and Julia offline replay are directionally defined, but the DB/overlay receipts still need enough =PlanTrace= and =SearchStage= fields to make turn-reduction experiments reproducible.

Owners: =schemas/=, =agent-semantic-client-db=, =agent-semantic-client-core=, =agent-semantic-search=, =agent-semantic-runtime=.

Deliverables:

3. command-thinness contract for every touched file under =crates/agent-semantic-protocol/src/command/*=;

- existing SQLite-backed flows still pass through the DB Engine facade and emit schema-valid =dbEngine= receipts;
- no new public search behavior is allowed without a schema or RFC section for its packet shape.

Owners: =agent-semantic-client-db= and provider harnesses.

Deliverables:

3. build field-weighted entity documents for name, qualified name, signature, doc summary, dependency terms, route terms, and provider-built identifier expansions;
4. expose generation freshness and visibility so stale generations cannot become default actionable evidence.

- a stable selector can be recovered from DB state without line-number identity;
- graph-route inputs include enough field-hit metadata for Python ranking and enough provenance for Julia replay.

** M4: replace agent-facing fuzzy search with stable FTS plus dynamic overlay

** M9: run comparison benchmarks before declaring the new architecture done
Owners: Rust scenario tests, Python ranking metrics, Julia replay notebooks.

Benchmark matrix:
1. codebase-memory-style repository search: turn count, recall@k, false-positive-at-1, token cost, and relationship/path-trace quality;
2. dirty overlay search: update latency, selector stability, stale result suppression, and no forced durable write per edit;
3. large Rust workspace simulation: intent-to-owner, intent-to-selector, dependency/test proximity ranking, and nextCommand correctness;
4. cross-language owner-items: Rust, TypeScript, Python, Julia, Org/Markdown, and Gerbil Scheme cold/hot gates;
5. compact render: rendered byte count, first actionable selector, recommendedNext adoption, and progressive-disclosure correctness.

Exit gate:
- ASP demonstrates a measurable path to fewer turns than the current flow and a plausible route to beat codebase-memory-mcp on selector authority, dirty overlay freshness, and compact evidence quality.
* Open risks
- Turso is still beta. ASP needs a backend trait and a SQLite v1 migration/control lane until Turso FTS, transactions, and multi-process behavior are proven under ASP workloads.
- Turso FTS boundary may not expose a stable reusable Rust package for overlay search. That requires an adapter experiment before replacing =fzf=.
- Dynamic overlays can become too complex if every dirty edit is normalized. Only replayable receipts and accepted checkpoints should be durable by default.
- Native parser coverage is uneven across languages. Tree-sitter-compatible fallback must remain a first-class coarse structural stage, then native providers add proof where available.
- A richer compact render can become its own dialect. It needs a schema and tests, not ad hoc text.

* Turso overlay search adapter checkpoint
The first executable Turso overlay search checkpoint now lives in =agent-semantic-search=, not =agent-semantic-client= or =command/=. The feature-gated =turso-overlay= search adapter exposes search-owned =TursoOverlaySearchDocument= and =TursoOverlaySearchHit= DTOs, wraps DB-owned =bootstrap_turso_client_db=, =upsert_turso_overlay_document=, and =search_turso_documents= internally, and filters results to overlay hits before returning them to callers. This keeps client-db row types behind the search boundary while proving that dirty-code overlay search can consume Turso-owned search semantics through the DB Engine seam.

The cold functional gate =asp_turso_overlay_search_adapter_cold_functional_path= is feature-gated behind =agent-semantic-protocol/turso-overlay=. It bootstraps a fresh Turso local-file fixture, upserts one transient overlay document, searches it through =agent_semantic_search::search_turso_overlay_documents=, requires =routeSource=turso-overlay=, zero provider processes, zero native finder processes, no executable line-range identity, and a 25ms max query budget. This is the first scenario-style Turso overlay performance gate. It deliberately measures the search adapter call, not =cargo run= or command startup.
