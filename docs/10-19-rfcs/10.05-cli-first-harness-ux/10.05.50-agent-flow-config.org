* Agent search flow: structured, not chaotic

The agent should follow this order:

#+begin_example
1. search prime     → provider resolves path context, then emits bounded map
2. search dependency → inspect external dependency source usage when relevant
3. search owner     → understand current file's responsibility
4. search symbol    → find exported symbol definitions
5. search callsite  → find owner-level import/reexport sites
6. search import    → inspect import/reexport owner edges
7. search tests     → locate tests importing the edited owner
8. search lexical       → find broader symbol/path candidates
9. search ingest    → normalize external candidate streams when needed
10. edit            → apply change
11. check           → verify change against policy
12. check --full    → closeout when needed
#+end_example

NOT:

#+begin_example
rg keyword
open many files
guess public API
run big test suite
#+end_example

Example: changing =OrderStatus=.

#+begin_src shell
ts-harness search prime .
ts-harness search dependency react .
ts-harness search deps react/jsx-runtime@19.0.0::jsx .
ts-harness search api OrderStatus .
ts-harness search symbol OrderStatus .
ts-harness search callsite OrderStatus .
ts-harness search import ./order .
ts-harness search tests src/domain/order.ts .
ts-harness search lexical OrderStatus .
ts-harness search owner src/domain/order.ts .
rg -n "\bOrderStatus\b" src tests | ts-harness search ingest .
#+end_src

The agent immediately knows the owner-grouped blast radius.  Raw lexical output
is never pasted into the prompt directly; it is normalized into the shared
semantic-search packet.  TypeScript =callsite= is owner-level in this slice: it
uses parser-owned import/reexport edges, and can later be upgraded with
language-service identifier references without changing the packet schema.

* Custom subagent execution
The ASP search subagent owns cheap exploration work: =search pipe=,
=search owner=, owner/frontier ranking, dependency/test reachability, and
selector discovery. It must not return source bodies, snippets, or line-range
selectors to the parent. The parent does not need a flat list of every owner or
item the subagent touched; it needs the graph route that survived ranking.

The subagent stop receipt is therefore one compact graph-route receipt per
independent branch, not a pile of standalone selector receipts. The receipt
projects the local =EvidenceGraph= into the smallest route that preserves the
next parent decision:

- =intent= is the parent-supplied task slice, not a new analysis paragraph.
- =route= names the graph route selected from parser/finder/test/dependency
  evidence.
- =state= is =selector-ready=, =needs-refine=, or =no-output=.
- =evidence= rows are ranked graph nodes with parser-owned identities and typed
  roles.
- =next= is the single primary executable action for the parent.
- =alt= may carry at most two secondary exact reads or refinements.
- =avoid= and =omit= are compact anti-action facts, not explanations.

Example:
#+begin_src text
[asp-search-subagent]
schema=asp-search-subagent.graph.v1
intent=query-code-file-selector-boundary
route=query-owner/request -> query-owner/runner -> facade-tests
state=selector-ready
evidence=E1 kind=item role=primary owner=crates/agent-semantic-protocol/src/command/query_owner/request.rs selector=rust://crates/agent-semantic-protocol/src/command/query_owner/request.rs#item/method/parse_structural_selector relation=admits-query-selector
evidence=E2 kind=test role=guard owner=crates/agent-semantic-protocol/tests/unit/provider_command/facade/search_output_snapshots.rs selector=rust://crates/agent-semantic-protocol/tests/unit/provider_command/facade/search_output_snapshots.rs#item/function/asp_rust_query_file_selector_code_without_item_is_rejected relation=covers-deny
next=E1 asp rust query --selector 'rust://crates/agent-semantic-protocol/src/command/query_owner/request.rs#item/method/parse_structural_selector' --workspace . --code
alt=E2 asp rust query --selector 'rust://crates/agent-semantic-protocol/tests/unit/provider_command/facade/search_output_snapshots.rs#item/function/asp_rust_query_file_selector_code_without_item_is_rejected' --workspace . --code
avoid=raw-read,file-level-query-code,repeat-same-search
omit=source,line-range,confidence,long-explanation
#+end_src
The receipt shape is a hook-verifiable contract, not only a prompt convention.
When a managed read-only =asp-explore= child emits a =subagent-stop= event, the
hook runtime SHOULD validate that the final message contains at least one
selector-only =[asp-search-subagent]= graph receipt with =schema=, =intent=,
=route=, =state=, at least one =evidence= row, and exactly one primary =next=
action. The hook MUST reject receipts that include source bodies, snippets,
line-range selectors, confidence labels, long explanations, or "not found"
inventories. The recovery action is to ask the same child session to re-emit
compact graph-route receipts; the parent should not replace the child or read
source directly just because the receipt was malformed.
The parent owns intent synthesis, final exact reads, edits, and validation. Once
a subagent returns a primary =next= action, the parent runs one exact =asp
<language> query --selector <parser-owned-selector> --workspace . --code= or
document =--content= command and reasons over the returned code/content itself.
Display line ranges and =sourceLocatorHint= remain metadata; they are not
subagent return identities.


When the client runtime exposes a project custom agent role such as
=asp_explorer=, the parent agent may spawn one read-only worker per independent
=actionFrontier= or =targetActions= branch.  The branch prompt carries the safe
ASP command group, parent-known evidence, missing facts, and risk; the worker
returns one =[asp-search-subagent]= graph-route receipt for parent fan-in and
main-model verification.

The custom role name is not a shared protocol requirement.  If the runtime only
exposes built-in subagent roles, the parent should use the built-in explorer role
with the same ASP Explorer branch prompt.  If subagents are unavailable, the
parent runs the same command groups sequentially and synthesizes the same
receipts.

Codex role visibility is owned by the loaded config layer stack.  Installing the
project role writes both the role file and its project config declaration, but an
already-running thread may need a reload or new thread before the generated
=spawn_agent= tool schema lists =asp_explorer=.

* Config: LLM generates, harness validates

The agent can write:

#+begin_src json
{
  "decisions": [
    {
      "kind": "ownerRuleSeverity",
      "owner": "src/generated",
      "ruleIds": ["TS-AGENT-R013"],
      "severity": "info",
      "evidence": [
        "package.json script generate:schema",
        "src/generated/README.md"
      ],
      "rationale": "Generated code is documented at schema source boundary."
    }
  ]
}
#+end_src

But the harness validates:

#+begin_src text
- owner path exists?
- rule id exists?
- evidence path exists?
- rationale non-empty?
- scope narrow enough (owner-local, not global)?
- not disabling entire agent_policy pack?
- not silently excluding tests?
#+end_src

Bad config produces a compact finding:

#+begin_example
[TS-CONFIG-R004] Warning: Global rule disable lacks evidence
@ .ts-harness/config.json:12:1
fix: replace global disable with owner-local decision and rationale
Help: config disables TS-AGENT-R011 across the project.
Contract: Agent-authored config must be owner-local or include org-level rationale.
#+end_example

* =--changed= fast lane

Not every check needs to be full-project.  =--changed= evaluates only:

#+begin_src text
syntax parse of changed files
export/import shape
owner-local rules
agent policy for touched public surfaces
extension policy for touched React/Effect files
config/package facts (only when config files changed)
#+end_src

Full check is reserved for closeout.

Provider activation MUST carry an explicit =execution= mode from the shared
language registry into generated provider manifests, activation JSON, runtime
profiles, =asp providers= output, and installed skill provider tables. Source
language harnesses use =external-process=. Document providers such as =org= and

ASP dispatches the native document provider in-process rather than spawning an
orgize binary.

* Agent integration bundle structure (future)

For Codex clients, =asp install plugin --codex= installs the
repo-maintained language-bin skill into
=.agents/skills/agent-semantic-protocols/SKILL.md=.  The skill is concise and
agent-facing: it describes =rs-harness=, =ts-harness=, and =py-harness= command
flows, and tells agents not to use =--json= for exploratory searches.

The public binary is =agent-semantic-protocol=.  It exposes:

#+begin_src shell
asp install plugin --codex .
asp hook pre-tool ...
asp hook stop ...
asp ast-patch dry-run --packet <ast-patch.json> .
#+end_src

=agent-semantic-hook= remains the library dependency that implements the hook
subcommands inside =agent-semantic-protocol=.  The migration is a command-surface
consolidation, not a deletion of the hook runtime crate.

=asp install hook= also writes the project runtime profile:

#+begin_src text
.cache/agent-semantic-protocol/runtime/profiles.json
#+end_src

The profile pins the provider argv resolved at activation time and records a
small health result for each activated provider.  =asp rust|typescript|python=
facades and the local-native client backend consume this profile before they
fall back to shell lookup.  The profile is regenerated when activation is
installed or synced, and doctor reports the same profile path and provider
health.  This keeps provider execution aligned across Rust, TypeScript, and
Python without depending on generated =runtime/bin= symlinks or the agent's
current shell =PATH=.

If an agent-client integration bundle is desired, it contains:

* Main-session command intent

The hook classifies ASP evidence commands before it decides a lane. Command
spelling alone is not sufficient: a parser-owned selector read and a broad
query command have different cost, ambiguity, and recovery rules.

| Intent | Command shape | Main-session behavior |
|-
| =control-plane= | =asp agent session ...= and other configured lifecycle controls | Allow |
| =exact-evidence-read= | =asp <language> query --selector <parser-owned item selector> --code= or =--names-only= | Allow |
| =reasoning-flow= | =search prime=, =search pipe=, =search owner=, =rg=, =fd=, =query --term=, a file selector, or any selector that is not parser-owned | Deny and enter the resident =asp-explore= interactive loop |
| =unknown= | Any command outside the explicit classes | Apply the configured control-plane allowlist; otherwise deny |

=search owner <owner> items= remains a reasoning-flow command even when the
owner path is known. It materializes/ranks candidates and is the mechanism
that discovers an exact selector; it is not an exact read. Likewise, a
=query= command is exact only when its selector already identifies one
parser-owned item and its projection is bounded. A term query, file selector,
or default/broad projection must route to =asp-explore=.

The denial contains one action only: enter =asp agent session bootstrap --name
asp-explore=, choose a pane number, and continue until the resident child is
ready. The pane owns create/resume/recover decisions. The hook does not
suggest alternative shell search tools or command-specific fallbacks.

* Failure mode reference

| Scenario                | CLI Behavior                                |
|-------------------------+---------------------------------------------|
| daemon down             | direct parser run, output =daemon=missing=  |
| daemon stale            | auto-rebuild cache, then respond            |
| protocol mismatch       | exit 3, deterministic upgrade message       |
| cache invalid           | direct parser run, output =cache=invalid= with reason |
| project root mismatch   | exit 3, refuse cross-project contamination  |
| request timeout         | retry once, then direct parser run          |
| direct parser run fails | exit 3, compact error message               |
