#+TITLE: Language Evidence Search Namespaces
#+PROPERTY: RFC 10.05.80

* Language evidence search namespaces
:PROPERTIES:
:ID: 10.05.80-language-evidence-search-namespaces
:END:

Agent-facing search must prevent the model from guessing language facts from
training memory.  The public search surface therefore needs programming
language native namespaces that tell the agent which authority owns the answer:
current project code, dependency resolution, language ecosystem extensions,
active compiler or interpreter environment, language rules, standard library
facts, executable patterns, or version comparison.

This RFC extends the =search= namespace model from
=10.05.30-ingest-pattern-views.org=.  It does not replace =search prime=,
=search pipe=, =search owner=, =search deps=, or =search pattern=.  It makes the
authority boundary explicit so language harnesses can converge without adding a
loose natural-language =concept= surface.

* Motivation

Large language models often carry stale or mixed language knowledge.  A model
may confuse Gerbil Scheme with Racket or other Scheme dialects, use an old Rust
nightly API on a stable toolchain, or trust a comment that no longer matches the
code.  The harness must turn these questions into parser, dependency,
extension, runtime, and witness facts before the agent writes code.

The central rule is:

#+begin_src text
Do not let the agent use remembered language facts when an active harness can
resolve the same question from project, dependency, environment, language, std,
pattern, or compare evidence.
#+end_src

* Namespace contract

** =search code=

Current project code.  This namespace owns source-local facts: symbols, owners,
definitions, callsites, imports, exports, tests, docstrings, comments, and
comment-derived claims.

Comments are not ground truth.  They are author-intent evidence that must be
checked against parser facts and executable witnesses when the claim matters.

Parser-owned structural syntax facts also belong to =search code=.  Providers
should expose language-neutral =syntaxFacts= through
=schemas/semantic-native-syntax-fact-index.v1.schema.json= and keep
language-specific meaning in =languageKind= plus provider-owned =fields=.
Higher-order constructs are represented this way: closure forms such as
=lambda= and =case-lambda= use =kind=function= with roles such as
=anonymous-function= or =multi-arity-function=; higher-order invocations such
as =map=, =filter=, =fold=, =cut=, and =for/fold= use =kind=call= with roles
such as =sequence-map=, =sequence-filter=, =sequence-fold=,
=partial-application=, or =loop-fold=.  Compact search renderers may surface
=operandCount=, =arities=, =formals=, and =caller= when available, but must not
invent a provider-private fact kind outside the shared enum.

Example shapes:

#+begin_src shell
asp rust search code symbol OrderStatus --workspace . --view seeds
asp rust search code comments --near enqueue --workspace . --view seeds
asp gerbil-scheme search code comments --near defsyntax --workspace . --view seeds
#+end_src

Comment claim output should distinguish the claim, the code evidence, and the
verdict:

#+begin_src text
commentClaim=macro-preserves-hygiene
codeEvidence=uses-datum->syntax-without-lexical-context
witness=expand-captures-binding
verdict=comment-code-mismatch
next=search lang macro hygiene
#+end_src

** =search deps=

Resolved dependency facts.  This namespace owns package versions, enabled
features, public API usage, dependency type leakage, dependency documentation
facts, and external API guidance.  The local lockfile or package resolver
defines the local dependency scope.

Rust already has this direction through =search deps=.  Other language
harnesses should converge on the same authority boundary instead of adding
language-specific ad hoc names.

The namespace must distinguish local dependency evidence from external
documentation advice.  When the resolver and source-usage pass both miss the
dependency, the provider should return a compact no-output fact and must not
invent a =docs-use= frontier.  A =docs-use= or docs/API follow-up is valid only
when local dependency facts exist, or when the user asked for explicit external
research outside the current workspace resolution.

Example shapes:

#+begin_src shell
asp rust search deps tokio --workspace . --view seeds
asp rust search deps tokio::timeout --workspace . --view seeds
asp gerbil-scheme search deps :std/sugar --workspace . --view seeds
#+end_src

** =search extension=

Language harness ecosystem extensions.  This namespace owns curated framework or
package knowledge that the language harness brings into the agent workflow:
framework APIs, syntax conventions, idioms, examples, and pattern guidance.

=search deps= answers "what dependency is resolved in this project?".
=search extension= answers "what provider-owned ecosystem knowledge package is
available for this language, and how should the agent use it?".  An extension
may be activated by a dependency declaration, but its guidance is provider-owned
and can include source-derived API facts, examples, comments, documentation
claims, and executable witnesses.

Examples:

- TypeScript =effect= extension: Effect API, fiber/concurrency idioms, resource
  safety patterns, and project-specific usage guidance.
- Gerbil Scheme =poo= extension: object system, metaobject protocol, protocol
  syntax, and POO-specific examples.
- Gerbil Scheme =poo= inherited pattern: =gerbil-poo= depends on
  =gerbil-utils=, so provider-owned =search pattern= may expose
  higher-order-control and typed-combinator-style guidance with
  =origin=inherited= when the package closure and active load path prove the
  import.
- Rust =tokio= extension: async runtime APIs, cancellation, timeout, process
  lifecycle, stream drain, and task ownership patterns.

Example shapes:

#+begin_src shell
asp typescript search extension effect api --workspace . --view seeds
asp gerbil-scheme search extension poo syntax --workspace . --view seeds
asp gerbil-scheme search extension poo pattern object-system --workspace . --view seeds
asp gerbil-scheme search pattern higher-order-control gerbil-utils inherited --workspace . --view seeds
asp rust search extension tokio cancellation --workspace . --view seeds
#+end_src

Extension output should expose the activated extension fact first, then route to
the more specific namespace that owns the requested evidence:

#+begin_src text
namespace=extension
extension=poo
authority=ecosystem-extension
evidenceGrade=fact
capabilities=object-system,metaobject-protocol,protocols
next=search pattern poo object-system
#+end_src

When an extension query names a provider-owned package registry entry rather
than an active project dependency, the output must say so explicitly.  The fact
may be =registered=, but it must not imply that the current workspace activated
the extension.  The text renderer should give the agent one clear next action:
run the pattern query that owns the executable writing guidance.

#+begin_src text
namespace=extension
extension=poo
activation=gerbil-poo://
dependencyMode=registered
agentAction=follow-next registeredKnowledge=gerbil-poo:// notProjectActivation=true
next=search pattern gerbil-poo usage
#+end_src

Registered package knowledge must also expose source acquisition interfaces for
client-side indexing.  The harness does not clone packages or build Rust SQL
indexes itself; it reports candidate source locations and repository identity.
For Gerbil packages, clients should probe the installed package source under the
active Gerbil package roots, such as =~/.gerbil=, before considering the git
repository candidate.  When local source is absent, the agent-facing packet must
treat that as a package-manager miss and tell the client to install the package
first, then retry local source/index discovery.  Repository cloning is a fallback
only after the package-manager install/source check remains unavailable, and that
fallback must stay outside the language harness.

#+begin_src text
sourceRef.kind=package-manager-source
sourceRef.localSource.rootHint=~/.gerbil
sourceRef.localSource.package=git.cons.io/mighty-gerbils/gerbil-poo
sourceRef.localSource.status=probe-first
sourceRef.repository.url=https://git.cons.io/mighty-gerbils/gerbil-poo
sourceRef.repository.vcs=git
sourceRef.indexHint.owner=asp-client
sourceRef.indexHint.mode=local-source-before-git
#+end_src

Agent-facing renderers must also expose the install-first fields in the same
packet family:
~sourceRef.localSource.missingAction=install-package-before-repository-fallback~,
~sourceRef.localSource.installHint~,
~sourceLookup.missingLocalAction=install-package-before-repository-fallback~,
and ~sourceLookup.fallbackPolicy=repository-source-after-install-check~.

Pattern output may be direct or inherited.  Direct pattern facts come from the
activated extension itself.  Inherited pattern facts come from a package in the
activated extension's dependency closure and must include an origin, a dependency
chain, and an import/load-path witness so the agent does not confuse inherited
APIs with ambient machine state:

#+begin_src text
namespace=pattern
extension=poo
pattern=gerbil-utils-higher-order-control
origin=inherited
via=git.cons.io/mighty-gerbils/gerbil-poo->git.cons.io/mighty-gerbils/gerbil-utils
importWitness.status=verified
importWitness.module=:clan/base
next=guide --code --topic typed-combinator-style --intent style
#+end_src

Providers must distinguish three cases:

- =direct=: the current project declares the package that owns the pattern.
- =inherited=: a declared dependency proves the owner package through a
  provider-owned package closure.
- =ambient=: the active load path can see the module but no package closure
  proves ownership; this is warning-grade evidence, not a repair target.

** =search env=

Active compiler, interpreter, runtime, toolchain, configuration, and load path.
This namespace owns what the current machine can actually compile or execute.

For Rust, =search env= must derive facts from active toolchain evidence such as
=rust-toolchain.toml=, =rustup show active-toolchain=, =rustc -Vv=, =cargo
metadata=, =Cargo.lock=, =rustc --print cfg=, edition, resolver, and enabled
features.

For Gerbil Scheme, =search env= must derive facts from the active =gxi= binary,
Gerbil home, load path, importable modules, and runnable probes.  The user may
have a concrete runtime such as a Homebrew =gxi= installation; the harness must
discover that active runtime instead of assuming a version from documentation.

Example shapes:

#+begin_src shell
asp rust search env toolchain --workspace . --view seeds
asp rust search env cfg --workspace . --view seeds
asp gerbil-scheme search env gxi --workspace . --view seeds
asp gerbil-scheme search env load-path --workspace . --view seeds
#+end_src

** =search runtime-source=

When the agent needs language, macro, standard-library, or framework facts that
depend on the active compiler/runtime version, memory is not evidence.  The
provider must expose a =runtime-source= namespace that maps the active runtime
to a versioned source reference and an ASP-owned acquisition plan.

For Gerbil Scheme, the provider derives the runtime tag from the active =gxi=
system version and returns a structured packet with =runtime=, =sourceRef=, and
=acquisition=.  The source reference names the upstream repository and exact
checkout policy.  The acquisition plan says ASP owns clone-or-fetch, checkout,
and structural index under an ASP-managed state namespace.  It must not expose
developer-local checkout or cache directories in the public
packet.

The namespace is a bridge, not the final answer.  After acquisition/indexing,
the next agent-facing step must search the version-matched source checkout for
the exact macro, API, or language construct.  A model answer based on training
memory, an unversioned checkout, or raw unindexed source scanning is a policy
failure.

#+begin_example
asp gerbil-scheme search runtime-source macro --workspace . --view seeds
asp cache runtime-source acquire --language-id gerbil-scheme --repository https://git.cons.io/mighty-gerbils/gerbil --checkout v0.18.2 --state-namespace runtime-source/gerbil-scheme --index-owner asp-structural-index
#+end_example

Runtime-owned printer boundaries also belong here.  For Gerbil Scheme,
=writeenv= / printer-hook guidance should return versioned source selectors and
ASP-owned acquisition/index facts before any extension pattern, such as POO
=:wr= serialization, can be promoted from =partial= to =verified=.
Agent-facing =next= values must keep CLI order, for example =search
runtime-source macro sugar module-sugar=, rather than provider-internal phrasing.

The shared packet contract is
=schemas/semantic-runtime-source-acquisition.v1.schema.json=.  Language
providers may add provider-owned details under facts, but the language-neutral
contract keeps the runtime/source/acquisition mapping explicit.

Runtime-source packets may also expose agent-facing code lookup metadata at the
packet top level:

- =selectorResolver= :: the logical source selector scheme, ASP state namespace,
  active runtime version key, structural index owner, and output mode.  This
  tells the agent how a selector such as
  =gerbil-runtime-source://src/std/sugar.ss#defrule= becomes a source-backed
  exact read after ASP has acquired and indexed the checkout.
- =sourceExamples= :: structured minimal forms derived from real source
  selectors.  Examples must be mappings such as head, operands, keywords,
  symbol, role, and selector; they must not be unstructured prose blobs.
- =sourceComments= :: comment extraction contracts for the selected source
  windows.  The provider should expose the extractor and fallback behavior so
  absence of useful comments is itself visible evidence instead of silently
  becoming model memory.

The non-JSON compact render should project these fields as
=|selectorResolver=, =|sourceExample=, and =|sourceComment= lines before an
agent follows =query --selector ... --code=.  The JSON packet remains the
machine contract, while the compact lines are the default agent workflow.  Both
surfaces must avoid developer-local checkout paths.

** =search compiler-evidence=

Compiler evidence is the versioned source-backed bridge between a provider's
internal proof/check layer and the active language implementation.  It is not a
replacement for =search runtime-source=; it is a narrower namespace for
compiler metadata, optimizer relations, local assertion environments, and
source selectors that justify medium-weight proof claims.

For Gerbil Scheme, =search compiler-evidence optimizer subtype assertion= must
expose =optimize-base.ss= selectors for =!signature=, =!class=, =!predicate=,
=!type-subtype?=, and =!class-subclass?=, plus =optimize-ann.ss= selectors for
=fold-assert-type= and =assert-type=.  The packet uses the generic
=semantic-language-evidence= schema because it is an evidence fact packet, not
a runtime-source acquisition packet.

The namespace must explicitly reject full proof-system claims such as general
constraint solving, quantifier reasoning, principal inference, proof-term
calculus, or cross-module theorem proving unless a future typed core supplies
those guarantees through its own schema.

** =search proof=

Proof search exposes medium-weight positive derivation witnesses over
normalized provider type structures.  It answers "what did the harness prove,
with which rule tree, and where is the compiler boundary?" rather than "is this
a complete type theory?"

For Gerbil Scheme, =search proof= owns TypeSpec proof packets generated by the
existing TypeSpec validation and subtyping engine.  The packet must include:

- proof system identity, level, model, relation set, rule catalog, and explicit
  non-goals;
- recursive proof trees with rule, conclusion, detail, and premise fields;
- proof profile metrics such as root rule, depth, node count, and preorder
  rule chain;
- compiler-evidence linkage that routes the agent to =search
  compiler-evidence optimizer subtype assertion= before claiming the proof is
  grounded in Gerbil compiler behavior.

The shared packet contract is =schemas/semantic-type-proof.v1.schema.json=.
The proof system is medium-weight: it can validate normalized contract ASTs,
build alias/subtype/compatibility witnesses, and expose proof cost, but it must
not become an implicit theorem prover.

Executable extension patterns use
=schemas/semantic-extension-pattern-mapping.v1.schema.json=.  Provider
registries must advertise both =search/pattern= and =search/runtime-source=
with their output schema ids so downstream agent interfaces can validate POO
pattern mappings, partial evidence, and runtime-source acquisition packets
without relying on provider-private text rendering.

** =search lang=

Language rules independent of one project but scoped to the active language and
runtime family.  This namespace owns syntax, binding, module rules, macro
systems, phase behavior, type or trait rules, evaluation order, and semantic
constraints.

The namespace is not a documentation dump.  It must cite parser-owned or
runtime-owned evidence and should route to =search env= when the rule depends on
the active compiler or interpreter version.

Gerbil module guidance must cite =module-sugar= runtime source for import and
export set syntax.  Queries such as =search lang rename-in= or =search lang
rename-out= should expose =only-in=, =except-in=, =rename-in=, =for-syntax=, and
=rename-out= selectors, plus failure cases for Racket =require= assumptions,
unchecked import renames, and confusing import rename with export rename.

Example shapes:

#+begin_src shell
asp rust search lang macro_rules --workspace . --view seeds
asp rust search lang lifetime-elision --workspace . --view seeds
asp gerbil-scheme search lang macro hygiene --workspace . --view seeds
asp gerbil-scheme search lang syntax-object --workspace . --view seeds
#+end_src

** =search std=

Standard library and language-bundled module facts.  This namespace owns
standard module exports, built-in procedures, syntax forms, traits, macros, and
version-gated std behavior.

=search std= must expose machine-usable import and capability facts, not only a
human prose answer.  For Gerbil Scheme, a query such as =search std json= should
return the active standard module =:std/text/json=, the =read-json= capability,
and the minimal import form =(import (only-in :std/text/json read-json))= so an
agent can validate JSON packets in native Gerbil instead of adding a Python or
shell parser.

Standard-library macro facts must route back to version-matched source when the
form is not just an importable procedure.  For Gerbil Scheme, =search std
sugar= exposes =:std/sugar=, =defrule=, =defsyntax-call=, =syntax-call=, and
the =for-syntax= import witness from the active runtime source checkout.  The
next step for writing those forms is =search pattern hygienic-macro=, not a
memory-derived Scheme macro answer.

Example shapes:

#+begin_src shell
asp rust search std Iterator::try_fold --workspace . --view seeds
asp gerbil-scheme search std :std/srfi/1 fold --workspace . --view seeds
asp gerbil-scheme search std :std/sugar defsyntax --workspace . --view seeds
#+end_src

** =search capability=

Project-local capability posture.  This namespace owns the compact bridge from
parser-owned project facts to agent steering.  It does not replace
=search structural=, =search extension=, or =search pattern=.  Instead, it
answers: "which language capabilities are active or available in this project,
and which next search/check route should the agent use before editing?"

For Gerbil Scheme, =search capability= derives its facts from the project
index, =gerbil.pkg= / =build.ss= policy, dependency activation, and parser-owned
syntax facts.  It should expose package/module posture, macro posture, POO
posture, higher-order posture, control-flow posture, and configurable-interface
posture.  The compact renderer should include stable =|fact=,
=|agentScenario=, =|capability=, =|failureCase=, and =|qualitySignal= lines.
The JSON surface may keep these as provider-owned evidence facts; language
neutral structural syntax facts remain in the shared structural index schema.

Capability output must stay passive and auditable.  A POO posture fact may say
that =gerbil-poo= is active and point to =search pattern poo class=, but it must
not inline a hand-written object-system recipe.  A macro posture fact may point
to =search runtime-source macro sugar=, but it must not answer from memory.  A
configurable-interface posture fact may report that built-in defaults apply or
that =gerbil.pkg= overrides exist; downstream projects should not be forced to
redeclare built-in defaults just to consume the harness.

Example shapes:

#+begin_src shell
gerbil-scheme-harness search capability posture --workspace . --view seeds
gerbil-scheme-harness search capability poo --json .
#+end_src

** =search pattern=

Executable writing patterns.  This namespace owns recipes that can show a
minimal correct shape, a failure case, and a witness for the active environment.
The agent selects a registered pattern; it does not write a backend DSL.

Pattern output must include:

#+begin_src text
namespace=pattern
evidenceGrade=fact|unknown
quality=verified|partial|insufficient
agentScenario=<why-an-agent-would-ask-this>
intent=<what-writing-task-this-pattern-authorizes>
minimalFormMapping=<head,operands,keywords,selector>
failureCase=<id,risk-kind,corrective-action,selectors>
selector=<source-or-parser-owned-witness>
selectorResolver=<scheme,status,query-selector-policy>
agentReadOrder=<scenario,steering,resolver,forms,failures,quality>
agentAction=<what-to-do-before-editing>
witness=<runtime-or-source-backed-proof>
missing=<required-gap-list-or-->
next=<next-search-or-query-command>
#+end_src

The pattern namespace is specifically responsible for helping an agent that does
not already know a language, framework, or extension.  The provider should emit
structured mappings, not opaque inline snippets: form heads, operands, keywords,
selectors, risk kinds, and corrective actions are separate fields.  A renderer
may print those mappings as text, but the provider-owned evidence remains
structured.

Logical selectors such as =gerbil-poo://object.ss#defclass= are source anchors
inside a provider-owned package registry, not workspace line selectors.  The
renderer must therefore emit a resolver line that tells the agent whether the
selector is directly queryable, package-manager resolved, or only a logical
anchor for the pattern.  If the selector is logical-only, the same output must
include a read order and an action line that tells the agent to use the
=minimalForms= and =failureCases= before editing, instead of trying
=query --selector= or guessing from generic Scheme/Racket memory.

For package-owned logical selectors, =sourceRef= must carry both source lookup
interfaces: a local package-source probe and a repository fallback.  The
language harness owns selector/package identity and acquisition metadata; the
ASP client or Rust indexing layer owns probing =~/.gerbil=, cloning the git
repository if needed, and building any SQL search index.  Text output and
machine packets should make this order visible to agents through
=sourceLookup=, =selectorResolver=, =agentReadOrder=, and =agentAction= fields
so they do not mistake =gerbil-poo://...= for a workspace selector or a fake
activation fact.

For example, a Gerbil agent that does not know POO must receive =defclass=,
generic/method, and protocol source selectors plus failure cases that prevent
Racket-style class syntax.  A Gerbil agent that asks for macro syntax must
receive both macro syntax evidence and the generated-code policy that forbids
blindly emitting =defsyntax= or =syntax-case= in generated code.
If the agent asks for =defrule=, =defsyntax-call=, or =syntax-call=, pattern
evidence must also include =:std/sugar= runtime-source selectors and the
=for-syntax= phase import witness.

Complex POO guidance must also include semantic regression witnesses from real
Gerbil projects.  Class hierarchy guidance should route through C3
method-resolution-order selectors, slot-order checks, and failure cases from
Gerbil's own =:gerbil/runtime/c3= tests instead of stopping at syntactic
=defclass= examples.

POO source-gap guidance should stay explicit about quality.  =compose-proto=
is =verified= only after a runtime instantiation witness exercises
=instantiate-proto=, =compose-proto=, and =compose-proto*=.  IO/JSON guidance
should expose =:pr=, =:wr object=, =:json=, =:write-json=, typed-value =TV=
writer selectors, JSON/pr runtime witnesses, a writeenv method-dispatch witness,
and any missing writeenv roundtrip witness instead of reducing serialization to
a single JSON snippet.  Until a runtime owner exposes a stable writeenv
roundtrip entrypoint, the pattern must warn against direct writeenv construction
or assuming ordinary =write= output roundtrips through POO =:wr=.  Its =next=
action must move to =runtime-source= evidence for =writeenv= / printer hooks
instead of looping back to the same partial pattern query.  Debug
tracing guidance should expose =trace-poo=, =trace-inherited-slot=,
=traced-function=, and =$computed-slot-spec= as =verified= only after a runtime
trace witness proves the wrapped procedure still calls through the computed-slot
superfun chain.

Agent-quality policy should make those evidence gaps actionable.  For Gerbil
Scheme, downstream POO code in =src/= that directly calls =writeenv= or defines
POO IO method overrides must be reported as structural policy findings until
runtime-source-backed writeenv / printer-hook witness coverage is present.  The
rule is about parser facts and extension activation, not text heuristics: a
generated project that imports =:clan/poo/io=, defines =defmethod= for IO, and
constructs =writeenv= directly should produce machine-readable findings and a
stable snapshot that route the agent to =search runtime-source writeenv printer
hook=.

Example shapes:

#+begin_src shell
asp rust search pattern async-timeout --workspace . --view seeds
asp gerbil-scheme search pattern hygienic-macro --workspace . --view seeds
asp gerbil-scheme search pattern defsyntax-syntax-case --workspace . --view seeds
#+end_src

** =compare=

Version and semantic distinction.  =compare= is not text diff.  It compares
facts across environments, dependencies, standard library versions, language
rules, or project baselines.

The shared packet contract for agent-facing comparison output is
=schemas/semantic-compare-packet.v1.schema.json=.  Provider registries must
advertise =search/compare= with that output schema id.  The contract is
language-neutral: provider-owned =authority= values distinguish comparisons
such as Gerbil active-runtime-versus-documented evidence from Rust
active-toolchain-versus-requested evidence.

The initial Gerbil Scheme slice is =search compare env active documented=: it
compares the active runtime fact against documentation or model-memory claims,
returns resolution status booleans rather than local executable paths, and
makes the active runtime the authoritative side for version-sensitive guidance.

The initial Rust slice is =search compare env stable nightly=: it compares the
active toolchain against requested stable/nightly sides.  The output is
=verified= only when the resolver can prove the active toolchain, =rustc=
version, and both requested toolchain sides; otherwise it is a successful
=insufficient= packet with explicit =missing= facts.

Example shapes:

#+begin_src shell
asp rust search compare env stable nightly .
asp rust search compare deps tokio-1.37 tokio-1.45 .
asp gerbil-scheme search compare env active documented .
asp gerbil-scheme search compare std :std/sugar active documented .
#+end_src

* Index ownership contract

Language providers and language project harnesses expose structural index feeds;
they do not own persistent or incremental index state.  A provider feed may
contain parser-owned owners, symbols, dependency usages, source locators, file
hashes, package roots, schema ids, and provider metadata.  It must not contain
raw source text, and it must not require the provider to remember previous
generations in order to answer the current feed.

ASP owns incremental indexing.  The ASP client/cache layer compares incoming
file hashes with the latest stored generation, derives =unchanged=, =changed=,
and =deleted= paths, invalidates stale rows, and decides whether to apply a
partial merge or fall back to full replacement.  Future language harnesses
should therefore implement the same feed interface rather than private
incremental caches.

The incremental path is an ASP orchestration loop: first compute a refresh plan
from provider file hash evidence, then request parser facts only for changed
paths, and finally apply a refresh packet that copies unchanged rows from the
previous generation into the new generation.  A deleted path is represented by
its absence from the incoming file hash set, not by stale provider-owned index
state.

The stream interface is the client write-back boundary.  When a provider emits a
=semantic-structural-index= packet, the client stores that packet as a
=structural-index/= cache artifact, advertises the generation in the cache
manifest, and applies the packet through the ASP-owned structural index refresh
path.  The packet remains a feed: provider processes do not read, merge, or
persist previous generations.

For Rust this means =rs-harness= can continue to emit
=semantic-structural-index= facts, but =*-rs= crates must not become the
incremental index implementation.  For Gerbil Scheme this means the native
parser can emit owner and extension facts, while ASP still owns the index
refresh plan and cache state.

Gerbil exposes the hot path as =search structural --json= and advertises it as
=index/structural= in the provider registry.  That packet is a lightweight
native-parser interface with file hashes, owner handles, workspace-size totals,
owner-level native fact summaries, generation id, =indexMode=interface=, and
=rawSourceStored=false=.  It intentionally leaves =symbols=,
=dependencyUsages=, and =syntaxFacts= empty on the default packet while still
emitting parser-owned totals such as =symbolTotal=, =dependencyUsageTotal=, and
=nativeSyntaxFactTotal=.  It is
not a provider cache and it is not the complete workspace graph.  ASP Rust
consumes the interface, owns unchanged / changed / deleted refresh planning,
builds the structural index, and delegates expensive topology or ranking work
to graph-turbo when needed.

Full native fact projection is owner-bounded.  Providers expose it through the
same view with =search structural --owner <path> --json= and register it as
=index/native-syntax-owner-facts=.  This packet uses the shared
=semantic-native-syntax-fact-index= shape under =facts= and is the fan-out unit
for ASP-side parallelism, caching, and graph construction.  Whole-workspace
=syntaxFacts= materialization is reserved for explicit validation artifacts via
=search structural --json --artifact=.  Full workspace symbol and dependency
rows follow the same rule: they are validation/debug or ASP-owned index
material, not part of the default agent-facing hot path or benchmark gate.

Gerbil structural feeds should also expose native syntax facts when the parser
can prove them without macro expansion side effects.  The first required fact
families are =module-import= facts for import modifiers and phase, =macro= facts
for =defsyntax=, =defrules=, =defrule=, and related hygienic macro surfaces,
=binding= facts for formal and local bindings, =poo= facts for =defclass=,
=defgeneric=, =defmethod=, and =defprotocol= forms, =higher-order= facts for
=lambda=, =case-lambda=, =map=, =filter=, =fold=, =cut=, and =for/fold= style
data transforms, =control-flow= facts for named loops, continuation boundaries,
exception/resource scopes, and branch forms, =dependency-usage= facts for
manifest-backed imports and extension capabilities, =comment= facts for typed
contract and engineering comment quality, and =call= facts with expression-level
source ranges where the native reader exposes them.  These facts are projected
through the shared =semantic-native-syntax-fact-index= owner packet shape.
Search renderers and fuzzy candidate rankers may rank or summarize them, but
they must not invent parser facts by matching rendered strings or source literal
payloads.

Compact text renderers should expose a bounded set of =|syntaxFact= rows for
agent-facing seed views.  These rows are summaries of native parser facts, not
new evidence: each row should include the shared kind, language kind, owner,
source range, and provider-owned fields that prevent dialect guessing.  For
Gerbil POO this means a =defclass= row carries supers, slots, and options, and a
=defmethod= row carries the generic name, receiver, receiver type,
specializers, and dispatch arity.  A =defprotocol= row should surface as an
=interface= fact with =languageKind=defprotocol= and =role=protocol= so agents
can choose protocol/generic dispatch instead of hand-rolled alists or hash
objects.  Compact rows are emitted from owner-bounded facts or explicit
artifacts.  The default structural JSON packet remains the lightweight machine
interface that points ASP to the owner fact commands instead of inlining the
complete graph payload.

* Evidence grades

All namespace output should classify evidence so the agent can reason about
accuracy instead of treating every line as a fact.

#+begin_src text
fact      Parser, compiler, runtime, dependency resolver, or std index fact.
claim     Comment, docstring, README, guide, or external documentation claim.
witness   Compile, expand, run, test, or snapshot proof for a fact or pattern.
mismatch  Claim disagrees with parser/runtime/witness evidence.
unknown   The harness cannot establish the fact under the active environment.
#+end_src

The agent may use =claim= as intent evidence, but it must not use =claim= as a
semantic fact when =fact= or =witness= is available or required.

* Quality feedback loop

Search output must report quality gaps instead of hiding them.  A namespace may
return useful partial facts, but it must explicitly say what is missing before
the agent can treat the output as sufficient implementation guidance.

#+begin_src text
quality=complete      Facts and required witnesses are available.
quality=partial       Some facts are available, but required evidence is missing.
quality=insufficient  The harness cannot support the requested guidance yet.
#+end_src

Partial or insufficient output is a harness improvement signal.  It means one of
the parser, package-resolution, extension-index, runtime-probe, snapshot, or
witness layers is incomplete.  The next action should route to the missing layer,
not to an LLM guess.

Example:

#+begin_src text
[gerbil-search-pattern] query=poo syntax evidenceGrade=fact quality=partial
|pattern id=poo-object-system sourceRef=package-manager-download:gxpkg:git.cons.io/mighty-gerbils/gerbil-poo:runtime-resolved witness=source-backed-poo-mapping
|selector role=class-definition symbol=defclass selector=gerbil-poo://object.ss#defclass
|quality partial missing=runnable-witness,full-source-index-coverage
next=search extension poo syntax
#+end_src

In this state the agent may cite the selector facts, but it should not invent a
complete POO coding recipe.  The correct engineering work is to improve the
Gerbil extension source index and add a runnable =gxi= witness.

* Gerbil Scheme vertical slice

The first Gerbil slice should be narrow and deep:

#+begin_src shell
asp gerbil-scheme search extension poo --workspace . --view seeds
asp gerbil-scheme search extension poo syntax --workspace . --view seeds
asp gerbil-scheme search env gxi --workspace . --view seeds
asp gerbil-scheme search lang macro hygiene --workspace . --view seeds
asp gerbil-scheme search lang syntax-object --workspace . --view seeds
asp gerbil-scheme search std :std/sugar defsyntax --workspace . --view seeds
asp gerbil-scheme search pattern hygienic-macro --workspace . --view seeds
asp gerbil-scheme compare env active documented --workspace . --view seeds
#+end_src

Required facts:

- active =gxi= path and version
- Gerbil home and load path
- importable std module graph for the slice
- active POO extension fact with dependency and capability evidence
- POO API/syntax/pattern query routing without invented facts
- macro syntax forms used by the slice
- hygiene facts for syntax objects, lexical context, and =datum->syntax=
- minimal =defsyntax= or =syntax-case= example
- failure case that demonstrates accidental capture or broken context
- expand or compile witness from the active =gxi=
- snapshot output for the search packets
- project capability posture facts for package/module, macro, POO,
  higher-order, control-flow, and configurable-interface steering

The implementation must remain native Scheme owned for Gerbil language facts.
Rust or ASP may own CLI routing, schema validation, and hook delivery, but Gerbil
syntax, expansion, module, and witness facts belong in the Gerbil harness.
Source-scope discovery is also provider-owned: the Gerbil harness resolves the
active project/package scope from Gerbil package anchors such as =gerbil.pkg=,
installed package metadata, load paths, and package-manager source roots.  It
reports those roots and logical selectors as facts.  Heavy source indexing,
cross-package lookup tables, and replayable source-index persistence belong to
ASP's Rust SQL source-index layer; Scheme code must not rebuild a parallel
database or scan implementation to compensate for a missing Rust index.  If a
package such as =gerbil-poo= is not installed, the provider should report the
missing package-manager fact and the install/source-root next action before any
Git fallback is considered.

The Gerbil =poo= extension has a separate responsibility from harness-internal
dependency use.  =src/extensions/poo.ss= describes downstream project guidance:
activation from =gerbil.pkg=, semantic checks, pattern mappings, selectors, and
agent-facing evidence for writing good POO code.  If the harness itself uses POO
to implement an internal feature, that implementation should live under the
owning source module for that feature and should consume the dependency through
ordinary Gerbil code.  Do not conflate internal dependency usage with the
extension evidence surface.

Gerbil also participates in the shared evidence graph surface.  The provider
owns only the lightweight packet boundary: =evidence graph --json= emits a
portable =semantic-evidence-graph= with owner, invariant candidate, verification
receipt slot, review action, and missing-evidence gap; =evidence analyze --json=
wraps that graph as a =semantic-graph-turbo-request= with
=profile=evidence-quality=.  The Gerbil provider must not implement the complex
ranking, topology, or reliability loop internally.  It exposes native Scheme
owner/witness facts and lets ASP graph-turbo consume the request for
cross-language evidence-quality ranking.

* Agent workflow

When a task touches language features with high ambiguity, the agent should use
the namespace that owns the question before editing.

#+begin_src text
current project structure      -> search code
author intent or comments      -> search code comments
dependency API and features    -> search deps
framework or ecosystem package -> search extension
active compiler/runtime facts  -> search env
language binding or syntax     -> search lang
standard library facts         -> search std
medium-weight type proof       -> search proof
compiler proof boundary        -> search compiler-evidence
known executable writing shape -> search pattern
version distinction            -> compare
#+end_src

High-risk edits include macros, module systems, type systems, generated code,
FFI, async runtimes, feature gates, and version-dependent APIs.  Hooks may
require a namespace receipt for these edits once the namespace output is stable.

* Provider evidence gate levels

Provider evidence conformance has two gate levels.  The levels must not be
collapsed into a single environment-dependent "whatever is active" test because
that hides coverage drift from CI and hides slow providers from local audit.

** Core-fast gate

The core-fast gate is the PR hot path.  It explicitly requires the Rust,
Python, and TypeScript providers to be active, then checks:

- =evidence graph --json= emits a schema-valid graph with non-empty nodes and
  edges.
- =evidence analyze --json= emits a schema-valid graph-turbo request with a
  valid evidence-quality profile, seeds, graph payload, and non-empty summary
  counts.
- =guide= exposes =search prime=, =search pipe=, =query --selector --code=, and
  evidence graph/analyze commands through the language facade.
- =search pipe= returns agent-facing quality evidence: =queryQuality=,
  =actionFrontier=, =recommendedNext=, and =avoid= guidance that keeps the agent
  away from raw source reads.
- The run writes a timing receipt so the agent can see which provider and which
  command consumed time.

The core-fast gate must use an explicit language list rather than only the
currently active provider set.  If a required provider is not active, the gate
fails and tells the agent to run the setup target.

** All-providers gate

The all-providers gate is the full conformance and performance audit.  It
requires every code provider in this RFC slice to be active: Rust, Python,
TypeScript, Julia, and Gerbil Scheme.  It runs the same evidence, facade, pipe,
schema-sync, and timing checks as core-fast, but it uses a looser command-time
threshold because providers such as Julia may still pay runtime startup cost.

The all-providers gate is the right place to catch Julia or Gerbil drift.  CI
may run it as a scheduled or full provider job; local developers can run it
before publishing provider changes.  A slow provider must show up in the timing
receipt rather than being hidden inside an aggregate pytest duration.

For Julia, the all-providers gate is also a compiled-provider freshness gate.
The installed =asp-julia-harness= must be the compiled app, not a shell wrapper
that launches =julia --project=.  If Julia source or =juliac= app files are
newer than the compiled binary, setup must rebuild before installing.  After
setup warm-up, Julia guide, evidence graph, evidence analyze, and pipe commands
must each stay under two seconds.

** Timing receipt

The timing receipt is an agent-facing JSON artifact, not a performance
benchmark.  It records the smoke scope, requested languages, active languages,
per-command durations, and the maximum command threshold used by the gate.
Agents should use it to decide whether to optimize provider startup,
precompute/cache guide output, split a slow provider into the all-providers
gate, or investigate a regression in graph/analyze.

** Provider knowledge-axis close-loop gate

The provider knowledge-axis gate verifies the agent-facing namespace contract
through the public =asp <language> search <axis> --json --workspace <root>=
facade, not through provider-private unit helpers.  It covers Gerbil Scheme,
TypeScript, Python, and Julia across =env=, =runtime-source=, =lang=, =std=,
=capability=, =extension=, =pattern=, =proof=, =compiler-evidence=, and
=compare=.

The gate checks three things:

- each provider guide advertises the axis commands it is about to execute;
- each query returns parseable JSON with an expected =fact= or explicit
  =unknown= frontier gap instead of model-memory guidance;
- each command and the full matrix stay within a visible local latency budget.

Where a provider already owns the shared =semantic-search-packet= envelope, the
gate requires that envelope and validates =languageId= plus =projectRoot=.  A
provider-local schema may still expose richer axis payloads, but it must
normalize to the same evidence fields: namespace/view, evidence grade, fact
count, hit/frontier count, and next-step guidance.  Provider-local shapes are a
transitional compatibility path; new cross-language axes should prefer the
shared semantic search packet.

#+end_src

- =orgize lint= passes for this RFC.
- Existing search namespace tests continue to pass.
- Gerbil =search runtime-source= emits version-matched =runtime=, =sourceRef=,
  and =acquisition= fields without local checkout paths.

  versioned compiler source selectors and rejects full proof-system claims.

  TypeSpec proof packet with recursive proof tree, depth, node count, and
  compiler-evidence linkage.

- Gerbil slice has snapshots for =env=, =lang=, =std=, =pattern=, and =compare=.

- Gerbil =search capability posture= exposes parser-owned posture facts for
  package/module, macro, POO, higher-order, control-flow, and configurable
  interface steering.

  runtime-source selectors without leaking local checkout paths.

  =module-sugar= import/export selectors and failure cases without using Racket
  =require= memory.
- Gerbil =search extension poo= reports activated POO capabilities and routes
  API/syntax/pattern queries without claiming unsupported facts.

  next action that improves parser, extension index, runtime probe, snapshot, or
  witness coverage.

- At least one agent scenario proves the model writes Gerbil POO-style code only
  after receiving =extension/pattern= evidence from the harness.

  runtime witnesses and =verified= quality; IO/JSON/writeenv/typed-value
