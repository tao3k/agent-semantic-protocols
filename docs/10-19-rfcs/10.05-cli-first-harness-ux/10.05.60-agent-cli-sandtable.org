* Agent CLI sandtable

Sandtable may replay an agent CLI, but that replay is not a new public =asp=
command and not a hidden compatibility alias for =command=.  The scenario step
owns this boundary through a canonical =agentCli= object.  A step MUST contain
exactly one execution body: either =command= for ordinary subprocess replay, or
=agentCli= for an agent-client command synthesized by the sandtable runner.

The first supported agent client is Claude Code in print mode.  The generated
argv is explicit:

#+begin_src text
<binary> -p <prompt> --output-format <text|json|stream-json> [...]
#+end_src

Optional fields append only documented CLI flags: =--input-format=,
=--include-partial-messages=, =--include-hook-events=, =--verbose=, and
=--model=.  The runner does not accept arbitrary extra args, infer intent,
choose a model, or translate prompt text.  If the test needs the exact
installed CLI boundary, the scenario writes =binary="claude"=.  If CI only
needs the argv/env contract, the scenario writes a fake executable fixture.

Provider routing is explicit environment plumbing.  DeepSeek-through-Claude-Code
scenarios set the Anthropic-compatible endpoint and token in =agentCli.env=:

#+begin_src json
{
  "ANTHROPIC_BASE_URL": "https://api.deepseek.com/anthropic",
  "ANTHROPIC_AUTH_TOKEN": "${DEEPSEEK_API_KEY}"
}
#+end_src

The =agentCli.env= map expands environment-variable references such as
=${DEEPSEEK_API_KEY}= from the scenario process environment.  It does not use
sandtable capture-token expansion; prompt and argv fields own captures, env
fields own environment references.

Live scenarios SHOULD add =requiredEnv=["ANTHROPIC_AUTH_TOKEN"]= so a missing
secret fails before a subprocess is spawned.  Default sandtable coverage MUST
use fake agent CLI fixtures, not a live network call.

The Python Claude Code SDK is a separate =agentSdk= execution body backed by
the =claude-code-sdk= package in the sandtable tools project.  It is not a
compatibility mode of =agentCli=: the runner synthesizes a Python module command
for the SDK wrapper, while =agentCli= preserves the installed CLI boundary.
Both execution bodies share the same environment-gating rules, but schema and
tests MUST keep them distinct.

Live SDK sandtables that audit ASP command behavior SHOULD set
=allowedTools=["Bash"]= plus =requireAspBashCommands=true=.  The runner then
uses the SDK permission callback to deny non-Bash tools and deny Bash commands
that are not =asp ...= invocations.  This makes command-count receipts measure
the protocol surface under test instead of Claude Code's default Grep/Glob/Read
tooling.

Live agent CLI scenarios that may call a networked model SHOULD live under
=sandtables/fixtures= and set scenario-level =skipUnlessEnv=.  The default
scenario discovery excludes fixtures, and an explicit run without the gate
returns =skip= instead of failing:

#+begin_src json
{
  "skipUnlessEnv": ["ASP_LIVE_CLAUDE_CLI", "ANTHROPIC_AUTH_TOKEN"]
}
#+end_src

** Deep-question large-library audits

Large-library sandtables may define =evidence.deepQuestionCases= for
agent-facing questions that are deliberately deeper than a single symbol lookup.
Each case records the natural-language question, the scenario step ids that
exercise it, the query terms expected to seed parser-owned search, and an audit
budget for semantic command use.  These cases are not answer fixtures; they are
runtime behavior probes for whether Claude, Codex, or a future SDK wrapper uses
compact graph/search/query commands before source reads.

The first Rust coverage set uses ten cases across the Tokio, bytes, and ignore
large-library matrices.  A live Tokio case may run under the Tokio 0.6.0
workdir with a prompt such as:

#+begin_src text
tokio 0.6.0 的 Vec<scalar> collection fields 意义是什么？
#+end_src

The live scenario MUST stay env-gated with =skipUnlessEnv= and SHOULD request
hook events so command traces can be converted into receipts.  After a run, the
operator builds a receipt from the command trace with:

#+begin_src sh
semantic-sandtable --build-receipt-from-trace <trace-root-or-jsonl> \
  --scenario-id rust.tokio-claude-deep-question-flow \
  --language rust --project-name tokio --project-source registry \
  --intent "tokio Vec<scalar> collection fields"
#+end_src

The receipt summary owns the audit counters used by deep-question budgets:
=aspCommands=, =searchCommands=, =queryCommands=, =directReadCommands=,
=repeatedCommands=, and =repeatedSearches=.  A good run should spend most of its
budget on compact =search= and targeted =query= commands, avoid repeated argv,
and keep =query --selector --code= behind an exact selector chosen from parser-owned
frontier output.  =directReadCommands= is a compatibility total, not an
absolute deletion target: a bounded exact-window fallback after parser-owned
frontier evidence may be acceptable, while broad or unbounded
=direct-source-read= commands should fail the scenario.  New reports and
receipts therefore split the total into =directReadBoundedCommands=,
=directReadBroadCommands=, =directReadUnboundedCommands=, and
=directReadRiskCommands=.  Scenario expectations should prefer
=maxDirectReadRiskCommands= when the intent is to reject token-waste loops
without banning the low-priority bounded fallback.
