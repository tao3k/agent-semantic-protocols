* The two-phase search model: path-context prime → multi-pipe

Phase 1 starts with =<lang-harness> search prime=.  The provider first resolves
the supplied path or project hint using its native project manager, then emits a
bounded graph packet for the resolved context.  There is no shared
=search workspace= surface: package membership, project references,
source/test roots, virtual environments, and Julia project source dependencies
are provider-owned path-context facts.

Prime remains a purpose-built, deep but bounded graph packet designed to give
the LLM a package/project-level search map.  Multi-pipe is Phase 3+, used by
subagents after the parent has selected an owner, dependency axis, or provider
reported context branch.

#+begin_src text
Phase 1a: <lang-harness> search prime       (provider resolves path context)
Phase 2:  agent parent reasons about axes   (LLM task, not CLI)
Phase 3:  focused searches                  (dependency/symbol/owner/ingest)
Phase 4:  parent synthesis → edit plan      (LLM task, not CLI)
Phase 5:  <lang-harness> check --changed    (validation)
#+end_src

Public TypeScript sandbox runs calibrate this model against large real
repositories.  =vitejs/vite= requires the TypeScript provider to interpret
=pnpm-workspace.yaml= and project references before prime, otherwise the parent
agent starts from an incorrect single-package map.  When a package pattern also
includes fixture packages such as =packages/**/__tests__/**=, the provider path
context resolver must rank source packages before fixture packages so the
capped context list still gives the parent useful follow-up choices.
=microsoft/TypeScript= requires prime to degrade through parser-version and
deep-AST pressure while preserving per-file import resolution, otherwise the
next =owner= / =import= / =tests= pipes lose graph edges after fallback.
Text-flow evidence such as =surface=source|test=, =ownerRole=, and bounded
=text= snippets exists so follow-up filters do not re-search the original query
token.

** What path context covers (and what it does not)

Path context is provider-owned setup for search packets, not a standalone
agent-facing search route.  It outputs only enough compact facts inside prime
or receipts for the agent and cache to understand what the provider resolved.
It never outputs raw hits, source lines, docs prose, or full per-owner graphs:

#+begin_src text
resolved package/project scale
candidate package or project contexts (ranked/capped)
provider-internal package/project edges
external dependency axes across packages
finding groups by package
next hints (context/dependency axes, not hits)
#+end_src

Path-context invariants:

#+begin_src text
context id     provider-owned stable id, often a root-relative package path
context name   native package/project name from metadata, when present
context role   root | package | project | source-scope | single-package
context config native config path, e.g. package.json, Cargo.toml, Project.toml
context edges  provider-internal import/reference/member edges only
findings       package-scoped counts/groups, not full diagnostics
owners/items   emitted by prime/owner packets, not the resolver itself
hits           emitted by search packets, not the resolver itself
#+end_src

Path-context summary format inside prime:

#+begin_example
[search-prime] mode=package pathContext=multi-package context=@repo/root candidates=12 shown=12 edge=18 external=7 find=3
|context . name=@repo/root role=root packages=12 patterns=packages/* dependencies=7
|context packages/core name=@repo/core role=package pattern=packages/* type=module config=packages/core/tsconfig.json diagnostics=0 next=prime:packages/core
|context apps/web name=@repo/web role=package pattern=apps/* type=module config=apps/web/tsconfig.json diagnostics=0 next=prime:apps/web
|edge P:. -internal-> P:packages/core importKind=type-import via=package-name package=@repo/core from=src/index.ts
|next prime:packages/core,prime:apps/web
#+end_example

** What prime covers (and what it does not)

Prime outputs package-level structure, signals, and next candidates.  It never outputs
raw hits, source lines, or docs prose:

#+begin_src text
native dependency / feature / cfg or package/config signals
owner graph (ranked, capped)
source_large owner warnings
public API owners
test/bench/example surfaces
finding groups
docs candidates
next hints (axes, not hits)
#+end_src

Large-library =search prime --view seeds= may run as a structural map instead
of a policy/full-native-facts collection.  When a provider takes this optimized
path it must say so in returned packet fields, not by relying on compatibility
or absence inference.  The prompt-facing synthesis SHOULD include explicit
entries such as =analysis=structure=, =nativeSyntaxFacts=skipped=, and
=policyFindings=skipped=.  =search policy= and =check= remain the policy finding
surfaces; optimized prime must not pretend that skipped findings were evaluated.

Owners are ranked, not listed alphabetically:

3. owners with findings
4. owners with external dependency edges

** Graph-derived search synthesis

Search providers may run bounded graph algorithms over their native reasoning
tree to make LLM planning cheaper.  The graph is an internal computation input;
the prompt-facing packet gets only compact graph evidence and synthesis inside
the existing =search= result.  The graph vocabulary is owned by
=schemas/semantic-graph.v1.schema.json= and embedded into search packets through
compatible =nodes=, =edges=, and =searchSynthesis= fields.  Providers must not
add a separate =search graph= lane for the agent, and must not hide graph
semantics in language-private =fields= maps:

#+begin_src text
algorithm        named algorithm, e.g. owner-rank-frontier or bounded-reachability-depth1
scope            prime | owner | dependency | query-set
highImpactOwners ranked owner paths that explain the selected axis
frontierOwners   owner paths one graph step beyond the current packet
editFrontier     source owners the next edit/read window should consider
testFrontier     test owners tied to the selected source frontier
windowSet        typed owner/test/read window targets selected before source windows
findingOwners    owner paths with policy findings in the current graph slice
seeds            next actions derived from graph frontier, not text coincidence
#+end_src

Graph synthesis must never replace parser-owned facts.  It ranks, trims,
explains, and proposes next actions from owner/dependency/test edges already
owned by the language harness.  If the provider cannot prove an edge from
parser, package metadata, schema registry, or test ownership facts, the graph
algorithm must not present it as an impact path.
RFC 006 defines how prompt-facing =seed= and =synthesis= semantics are folded
back into the graph feature: seeds are selected frontier actions, and synthesis
is rendered through =alg=, =rank=, =frontier=, and =avoid= rather than a
separate prompt-facing protocol.

For =search prime --view seeds=, providers should still use their native parser
and reasoning tree.  A seed packet built from directory/file discovery alone is
too close to a path list and causes the LLM to fan out across unrelated files.
The compact seed packet should run the same owner graph ranking as full prime
and derive its frontier actions from =searchSynthesis=:

Prime is a /decision primer/, not a candidate answer.  Prompt-facing prime output
MUST say that explicitly with a compact decision row carrying at least:
=purpose=decision-primer=, =answer=false=, =code=false=, the available search
capabilities, the first-hop ladder, history fields owned by ASP artifacts, risk
guidance, and the canonical next command shape
=search pipe '<question-or-feature-term>' --workspace . --view seeds=.  The shared ASP graph
renderer and any provider-local fast-prime renderer MUST emit the same
decision-primer contract before owner/dependency/test graph facts.
Provider-local prime may expose owner/dependency/test graph facts, but
artifact-history counters such as =directReadRisk=, =repeatedPrime=,
=repeatedPipe=, and =bestPath= are ASP client facts and should be identified as
such rather than inferred by language providers from local source.

Prime prompt-output cache identity MUST include a render ABI fingerprint derived
from the decision-primer contract.  A cached =search-prime= stdout artifact
without =|decision purpose=decision-primer= is not replay-safe, even when source
file hashes are fresh.  This prevents old locator/frontier stdout from
surviving renderer-contract changes; the preferred replay path remains a
structured search packet rendered by the current ASP graph renderer.

#+begin_src text
highImpactOwners  ranked owners shown in the packet
frontierOwners    adjacent owners one parser-owned graph step outside the packet
findingOwners     owners carrying grouped findings
seeds             owner/test/dependency frontier actions derived from the graph
#+end_src

If prime skips a collection axis for latency, =findingOwners= may be empty only
when the packet also returns the corresponding collection status in
=searchSynthesis.fields=.  Agents must treat =policyFindings=skipped= as a
typed next-step signal to run =search policy= or =check= when a task needs rule
evidence, not as evidence that no findings exist.

For =search owner <path> --view seeds=, providers should emit
=bounded-reachability-depth1=: incoming owner count, outgoing owner count,
frontier owners, finding owners, and owner/test seeds.  This makes the next
agent step a graph decision instead of a new round of text fragments.

RFC 006 owns the compact rendering.  The old single =|synthesis= line is
superseded by graph-integrated rank and frontier facts:

#+begin_src text
[search-graph] mode=prime root=. alg=owner-rank-frontier
legend: aliases ID:kind; node ID=kind:role(value)!next; edge SRC>{DST:rel}; frontier ID.next
aliases=G:search,L:owner,T:owner
L=owner:path(src/lib.rs)!owner
T=owner:path(src/tests.rs)!owner
L>{T:frontier}
rank=L,T frontier=L.owner,T.owner
#+end_src

Agent-facing graph feature semantics are specified separately in RFC 006,
=docs/10-19-rfcs/10.06-agent-compact-graph-feature.org=.  RFC 005 owns the CLI surface and
search/query command lifecycle; RFC 006 owns the LLM-oriented graph facts,
rank, compact projection, and next-action semantics shared by all =search=
surfaces.

** How the agent client uses prime to build the subagent flow graph

Agent guide:

```md
## Phase 1: deep prime

Always start complex Rust tasks with:

`rs-harness search prime`

Read mode, package, feature, cfg, owner, large-owner, api-candidate,
test-surface, and next fields.

Do not spawn subagents before reading prime unless the user provides an
exact file and symbol.

## Phase 2: parent planning

Use prime output to choose one or more search axes:

- feature/cfg axis
- dependency/API docs axis
- owner/item axis
- public API/callsite axis
- tests/bench/examples axis
- harness finding axis

Spawn read-only subagents only after selecting axes.

Example: if prime shows `feature io-uring ... requires_cfg=tokio_unstable`:

feature subagent:
  `rs-harness search features io-uring cfg owners tests`
cfg subagent:
  `rs-harness search cfg tokio_unstable owners tests`
source subagent:
  `rg -n "io_uring|tokio_unstable" tokio/src tokio/tests | rs-harness search ingest items tests`
```

** =next= placement rule for prime

=next= on owner lines targets that owner implicitly.  Global =|next= only
for candidates without a clear owner context.  Never duplicate an
owner-specific next into the global =|next= line.

** Confidence modeling in prime

When facts cannot be confirmed from parser/Cargo data alone, prime uses
explicit confidence markers rather than guessing:

#+begin_src text
reason=public-api-carrier         confirmed from parser facts
reason=possible-public-api-carrier uncertain, needs public-api pipe
reason=likely-parser-owner        heuristic, needs owner pipe
public_external_allowed=true      confirmed from cargo metadata
source=workspace|path|crates-io|git  dependency origin
#+end_src

This prevents prime from misleading the agent client with false certainty on
cross-crate or macro-heavy codebases like rustc.  The =source= field on
=cargo= lines distinguishes workspace-internal from crates-io
dependencies, which affects search strategy and docs-source selection.

** Strong signal: =source_large=true=

When prime marks an owner as =source_large=true=, the agent client must NOT open the
file directly.  Instead:

```sh
rs-harness search owner <file> items
```

This rule prevents token explosion from files like =file-search/src/lib.rs=
(1192 lines, ~12 public items) or =tokio/src/runtime/mod.rs= (668 lines).

** Shared flags on all =search= views

#+begin_src text
                 default agent output (compact line protocol)
--json                    tool / IDE / flowhub output
#+end_src

Agent clients must use compact output for interactive exploration.  =--json=
is only for schema-contract tests, sandtable receipts/replay, IDE/Flowhub
consumers, or a local validator that consumes the JSON directly.  It is not a
prompt-facing discovery mode: repeated commands such as
=search fzf <query> owner tests --json .= waste tokens, hide the intended line
protocol guidance, and should be rejected by agent hooks with a safe compact
route.

When a provider does emit =--json= for a machine consumer, the renderer should
prefer minified JSON.  Pretty JSON is easier for humans but produces the same
facts with extra prompt-visible whitespace; schema-contract sandtables and
receipts should validate the structured payload instead of relying on
line-oriented JSON formatting.

When an agent asks for JSON during exploration, the hook denial should preserve
the semantic route, explain that agent search commands must not use =--json=,
and remove only the output-mode error.  For example:

#+begin_src text
denied: asp typescript search fzf location.path owner tests --json .
guide:  asp typescript search fzf location.path owner tests --workspace . --view seeds
#+end_src

If the rejected JSON-output command already carries a concrete owner path, the
hook should not fall back to a generic guide.  It should route to the
provider's owner query command and derive the compact query from the path or
provider-owned query fields, using this shape:

#+begin_src text
asp <language> search owner <path> items --query <derived-query> .
#+end_src

For =search fzf=, the first required query positional is owned by the query,
even when the literal text begins with a dash.  For example,
=ts-harness search fzf --json --workspace . --view seeds= searches for the literal
=--json= token.  A structured output request must put the output flag after the
query, e.g. =ts-harness search fzf OrderStatus --json .=.  This prevents a
flag-like source token from silently becoming a prompt-facing JSON packet.
The same literal ownership applies to explicit query flags:
=search fzf --query --language owner tests --workspace . --view seeds= searches for the
literal =--language= token, while =owner= and =tests= remain search surfaces.
Providers must not route =search fzf --query <term>= into owner item-query
state; it is the fzf query surface.

If a daemon is running, the CLI internally routes through the daemon socket.
If no daemon is running, the CLI runs the native parser directly.  The agent
never knows which path was taken.

#+begin_example
public contract:
  CLI args + stdout compact/json + exit code

private optimization:
  local daemon, TypeScript Program cache, file watcher, socket IPC
#+end_example

* Daemon failure matrix

| Daemon State          | CLI Behavior                          |
|-----------------------+---------------------------------------|
| running, cache fresh  | fast incremental result               |
| running, cache stale  | auto-rebuild cache, then respond      |
| not running           | direct parser run (same output)       |
| protocol mismatch     | deterministic upgrade message, exit 4 |
| crashed mid-request   | CLI retries once, then direct run     |

The agent never receives "server unavailable" as an error — only as a
=daemon=missing= diagnostic field when a provider chooses to expose daemon
status, which is informational.

* Compact format specification: line protocol + ASCII graph

=<lang-harness> search= output is a /line protocol/, not a Markdown report.
It inherits the existing Rust harness compact contract: stable tags, single
locator syntax, short action lines, minimal fact lines, no audit headers, no
prose.

Rules:

#+begin_src text
- no blank lines
- no Markdown headings
- no bullet indentation
- no source excerpts in default search packets
- no Help / Contract unless it is a check finding
- no prose paragraphs
- paths relative to project root
- every line independently readable
- every block default ≤ 12 lines
- no Unicode arrows, no box-drawing, no DOT/Mermaid/Graphviz
#+end_src

Path values are canonical project-root-relative paths, not display locators.
They must not include rank prefixes, node slots, URI schemes, line ranges,
column numbers, or command prefixes.  In particular, values such as
=0:src/components/WorkflowExecution.tsx=, =owner:src/lib.rs=, and
=src/lib.rs:42:90= are not paths.  Put rank/score in separate fields
(for example =rank=0= or =score=2), put line/column data in =locations=, and
render compact action labels only inside =next= values.

Those labels are ARI prompt hints backed by typed packet actions.  Client and
provider code must materialize executable commands from schema-owned action
objects, registry capabilities, and workspace policy; they must not recover
argv by parsing =next= text, owner lines, rank lines, or frontier strings.

This rule also applies when the provider internally works with package-local
paths.  Agent-facing owner paths, hit paths, query selectors, and next-action
arguments remain project-root-relative.  If a provider needs to expose a package
name, package root, package-local path, rank, graph slot, or display locator, it
must use a separate field such as =package=, =packageRoot=, =packagePath=,
=rank=, or =node=.  It must not shorten =packages/typescript/src/foo.ts= to
=src/foo.ts= in command-ready fields, because agents will copy that value into a
read/search/edit command against the workspace root.

Edge endpoints are graph node ids, not path values.  Owner node ids use the
typed locator =O:<projectPath>=; they must not use numeric prefixes such as
=0:<projectPath>=.  If the provider wants to render ordering, it should add
=rank=N= instead of changing the owner/path identity.

Line shape:

#+begin_src text
[search-kind] k=v k=v k=v          ← header line, always first
|node KIND:id k=v                   ← typed graph node
|edge KIND:id -kind[:label]-> KIND:id  ← ASCII edge, always " -kind-> "
|owner path k=v next=...             ← owner with inline next actions
|item name kind=... k=v next=...     ← intra-file item with inline next
|hit path=<path> line=N column=N ... ← hit location uses fields, not path:line
|find RULE xN path=<path> node=KIND:id ← finding path first, graph node second
|next kind:arg,kind:arg(scope=x)     ← global next actions (rare, only when
                                       no owner/item context exists)
|note k=v                           ← exceptional info (keep ≤ 1)
#+end_src

=|item= lines only appear when the request includes the =items= pipe or an
advertised provider method returns item facts by default.
=|item= lines must not repeat fields already encoded by their read locator:
=read=path:start-end= carries the owner path and line range.  Providers should
also suppress normal boolean defaults such as =public=false=, =doc=false=,
=exported=false=, and =typeOnly=false=; emit them only when they carry positive
or exceptional information.
=|owner= lines should suppress healthy/default validation fields such as
=public=false=, =parserOwner=false=, =valid=true=, =syntaxDiagnostics=0=, and
=semanticDiagnostics=0=.  Emit diagnostics, invalid states, parser ownership,
or public/export facts only when they change the agent's next action.
=|edge= lines should suppress normal boolean defaults such as =typeOnly=false=
and =test=false=.  Emit =typeOnly=true=, =test=true=, or exceptional resolution
facts when they affect the edit/test boundary.
=|hit= lines must render the path as =path=<projectPath>= and line/column as
separate fields.  They must not use =src/lib.rs:42:7= or
=src/lib.rs:42:90= as the positional value because agents will copy that token
as a file path.
=|hit= lines should suppress =owner=<path>= when =owner= equals =path=, and
should suppress true/default provenance fields such as =realOwner=true= unless
the value is exceptional.
=|next= is an inline field on owner/item/package/cargo lines, not a separate
line type.  A standalone =|next= line is only used when no owner/item
context exists to attach to.

Next actions render compact action labels:

#+begin_src text
next=dependency:reqwest
next=callsite:create_order
next=pattern:public-error-boundary
next=text:timeout(scope=tests)
next=owner:src/api/orders.rs
next=tests
next=bench:parse
#+end_src

If the next action targets the current owner, the owner is implicit.  If it
targets a different owner, it is explicit: =next=owner:src/api/orders.rs=.
The source JSON packet still carries the typed action target, capability, and
locator.  The rendered label is allowed for agent navigation and strict
golden/backcompat validation; it is not a production action API.

Node prefixes:

#+begin_src text
P:  package
O:  owner / module owner
C:  dependency / crate
T:  test / bench target
F:  finding group
S:  symbol
#+end_src

Edge kinds (all ASCII, no Unicode):

#+begin_src text
-mod->          module child edge
-path-mod->     #[path] mod edge
-include->      include! edge
-crate->        local owner dependency (same package)
-workspace->    workspace-internal dependency (different member)
-crates-io->    crates.io external dependency
-path-dep->     path-based external dependency
-ext->          external dependency (generic, when kind unresolved)
-test->         owned test / bench edge
-find->         finding at owner
-call->         symbol / callsite edge
-import->       import root edge
-reexport->     facade re-export edge
#+end_src

Distinguishing =workspace->= from =crates-io->= matters in monorepos like
rust-lang/rust and agent-client repos where cross-crate edges are workspace-internal
but still cross-package.

A provider-private diagnostic method, if advertised, also uses this line
protocol rather than prose:

#+begin_src text
[search-plan] view=dependency q=reqwest
|step deps dep=reqwest
|step native use_root=reqwest
|step fd scope=src,tests
|step rg pat=\breqwest\b
|rank owner,public,direct,test
#+end_src

* Graph-first defaults

=search= views are split into two modes.  The default is /graph/ —
structure before hits.  =hits= is opt-in per query.

| View          | Default Mode | Output                                  |
|---------------+--------------+-----------------------------------------|
| =prime=       | graph        | =|node= + =|edge= + =|find= + =next=  |
| =owner=       | graph        | =|node= + =|edge= + =next=             |
| =dependency=  | graph        | =|dep= + =|edge= + =|owner= + =next=  |
| =deps=        | hits         | =|dep= + =|owner= usage + optional =|api= |
| =features=    | graph        | =|feature= + =|dep= + =|target= + =next= |
| =targets=     | graph        | =|target= list                            |
| =symbol=      | hits         | =|def= + =|call= + =next=              |
| =callsite=    | hits         | =|call= + =next=                        |
| =import=      | graph        | =|node= + =|edge -import->= + =next=    |
| =pattern=     | hits         | =|owner= + =locations= + =next=         |
| =patterns=    | graph        | =|pat= list                              |
| =tests=       | graph        | =|edge -test->= + =next=               |
| =text=        | hits         | =|owner= + =locations= + =next=         |

Override with =--view graph|hits|both|seeds=.

* Output mode: compact is the default

=rs-harness= is agent-first.  All commands render compact text by default.
There is no =--compact= flag.  =--json= is for structured consumers.  Provider
descriptors may also advertise narrower text projections such as =--code= for
already-selected parser item queries.

#+begin_src text
rs-harness search prime            → compact packet (default)
rs-harness search prime --json     → structured JSON
rs-harness search owner src/lib.rs items --query 'run' --code → compact code only
rs-harness check --changed         → compact finding text (default)
rs-harness check --changed --json  → structured JSON report
#+end_src

For agent exploration, =--json= is a hook-deniable output-mode error, not a
better search.  Agents should request compact =--view seeds= when they need the
smallest prompt-facing packet and reserve JSON for tests, receipts, or
machine-to-machine validation.

* Graph-first view controls

#+begin_src text
--view graph|hits|both|seeds
--package <pkg>
#+end_src

=--view= selects the final render mode for a search packet.  Package/project
selection is provider-owned path-context resolution, usually by passing a
package or project path as the final project hint.  Depth, edge, hit, owner,
line, feature, target, and seed caps are provider-owned deterministic limits
unless a future =methodDescriptor= explicitly advertises them for that method.
They are not shared semantic-language flags.

There is no shared =--backend= flag today.  Execution-route selection is fully
internal unless a method descriptor later advertises bounded text match modes.
Even then, the agent client chooses intent-level match semantics, not raw tool
execution.  RFC 008 extends this boundary to the cross-language =asp= client:
local native provider CLI, Cloud Flight, and hybrid route selection are client
configuration and diagnostics state, not search/query flags agents must pick
during normal exploration.
* Native fast-prime latency contract
The public =search prime --view seeds= contract is still the decision-primer
packet defined above. A provider-local or ASP-client fast path may render that
packet without invoking the full language provider only when it preserves the
same decision-primer row, next-command shape, owner frontier vocabulary, and
explicit algorithm label.

The fast path is a latency optimization, not a semantic downgrade. It may use
external Rust search tools such as =fd= to enumerate bounded owner candidates,
but it must not output source lines, full files, or stale cached source text.
It should identify itself with an =alg= value, for example
=native-fd-prime-frontier-v1=, so tests and agent receipts can distinguish the
optimized structural frontier from a full provider semantic packet.

For normal repositories, =search prime --view seeds= is part of the interactive
search gate and should complete under one second on the installed release
binary. Julia provider startup and language-runtime cold paths may use a looser
gate, but the ASP Rust client should still keep filesystem discovery, wrapper
dispatch, and compact rendering in the millisecond-to-subsecond range.
