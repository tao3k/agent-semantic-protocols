* Gerbil Scheme style conventions from gerbil-utils
:PROPERTIES:
:ID: 10.05.90-gerbil-scheme-style-conventions
:END:

The executable rule matrix, structural quality facts, and acceptance gates are
owned by =10.05.91-gerbil-scheme-code-quality-contract.org=. This document
keeps the reference observations and agent-facing rationale.

The Gerbil Scheme harness should learn from the official =gerbil-utils=
project without copying its code.  The useful contract is not a set of inline
snippets; it is a set of structural expectations that the provider can expose as
policy findings, search facts, and agent-facing guidance.

This audit used =https://git.cons.io/mighty-gerbils/gerbil-utils= as the
reference project.  Local checkout paths are evidence only and must not appear in
project configuration, generated docs, or user-facing guidance.

* Reference observations

** Package and owner shape

=gerbil-utils= uses package name =clan= and keeps most public owners at the
package root, such as =base.ss=, =concurrency.ss=, =generator.ss=,
=path-config.ss=, and =building.ss=.  Deeper folders are used when the domain
needs ownership, for example =pure/dict/t/assq-test.ss= for dictionary tests.

The owner names are domain names, not generic buckets.  This matches the harness
policy direction that rejects vague owner segments such as =utils=, =helpers=, or
=common= when a more precise owner exists.

** Test layout

Tests live under =t/=, not top-level =test/=.  Subdomains may also own their
tests under a local =t/= directory, for example =pure/dict/t/assq-test.ss=.
The root =t/base-test.ss= suite in =gerbil-utils= is also a real-project
witness for package-level tests under =t/=.

The important distinction is directory convention versus test granularity.  The
harness should use =t/= for Gerbil project convention, but still keep explicit
unit tests.  Each test owner should export one named suite such as
=base-test= or =assq-test= and build that suite with =:std/test=.

** Build and test exclusion

=building.ss= treats =t= as a default exclude directory for ordinary module
build discovery.  This is the right boundary: tests are first-class owners, but
they should not be mixed into runtime module enumeration.

** Naming for data flow

=generator.ss= uses directional names such as =generating<-list= and
=list<-generating=.  This is a useful Scheme convention for agent guidance:
conversion direction belongs in the function name when both sides are common
domain values.

This naming is especially valuable for high-order and stream-like APIs because
it prevents vague names such as =convert=, =process=, or =handle=.

** Control flow

=base.ss= uses =case-lambda= for arity-specific behavior such as composition and
pipeline helpers.  =concurrency.ss= separates shutdown, race, retry, and
parallel mapping behavior into named functions instead of embedding ad hoc
thread management at call sites.

The harness should teach agents to prefer named local helpers and small
continuation boundaries for complex control flow.  A long inline macro body or
deep anonymous control-flow block should be treated as a quality risk unless the
owner has a parser-backed reason for it.

* Harness decisions

** Adopt =t/= for Gerbil tests

The Gerbil Scheme harness stores its tests under =t/= and keeps unit helpers
under =t/unit/=.  The load path for local verification is therefore =src:t=.

Top-level =test/= is a retired layout for this harness.  The policy layer should
warn when a Gerbil source file is discovered under =test/= and should recommend
=t/= while preserving explicit unit tests.

Top-level =tests/= is not an alias for =t/=.  By default the provider treats
=test/= and =tests/= as non-native test directories and reports
=GERBIL-SCHEME-MOD-R006=.  This rule is intentionally strict because a
self-applying harness must not give an agent an easy escape hatch: renaming or
creating a second test tree to avoid =t/= would hide coverage from the native
verification lane.

Projects that truly cannot use =t/= must opt in through parser-owned package
policy in =gerbil.pkg=.  The policy must list the non-=t= directory explicitly
and must include a clear explanation for why =t/= cannot be used.  A missing,
empty, or placeholder explanation keeps the finding active.  The override only
affects policy acceptance; it must not trigger any automatic directory rewrite.

** Keep unit tests explicit

Moving to =t/= does not mean copying =gerbil-utils= wholesale.  This harness has
schema, snapshot, parser, checker, policy, search, and extension unit surfaces.
Those should remain separately owned and runnable through native Gerbil
=:std/test=.

** Prefer structured guidance over inline examples

When the provider teaches a style rule, it should expose structured fields:
owner, selector, rule id, risk kind, corrective action, and witness.  Renderer
text can summarize the result, but the underlying evidence should remain
machine-checkable.

This does not forbid source-backed teaching examples.  For Gerbil Scheme, the
structured teaching surface is =asp gerbil-scheme guide= plus
=asp gerbil-scheme guide --code=, not an inline snippet inside a warning or a
new search subcommand.  The non-code guide packet exposes the exemplar topic,
selector, comment selector, code selector, and progressive follow-up command.
The code guide itself writes only the extracted source comment and the complete
function, macro, or use block needed to demonstrate the idiom.  It must not
render =primaryExemplar= metadata, =|code begin= markers, compressed pseudo-code,
or hand-copied inline source.  When the agent needs more contrast or transfer
examples, the non-code guide packet should expose an explicit =--more= or
=--limit N= command.  Multiple examples are therefore a progressive choice, not
the default token cost.

The Gerbil guide selection grammar should stay on =guide --code= instead of
adding topic-specific subcommands.  =--topic= chooses the idiom family,
=--intent= or =--role= narrows the agent task, =--rule= and =--finding= route
known policy findings to source-backed repair examples, and =--level= plus
=--more= controls progressive depth.  A POO repair example must be extracted
from parser-owned POO source.  A macro witness example should connect back to
the runtime-source acquisition path so the agent learns to cite version-matched
runtime facts before editing macro transformers.

The type-commented combinator style is also a guide topic, not a new command.
=asp gerbil-scheme guide --code --topic typed-combinator-style --intent style=
should emit one source-backed excerpt by default: adjacent intent/type comments
and the compact helper or transform block that demonstrates the idiom.  =--more=
may add another source-backed compact function, but it must not duplicate the
first excerpt or copy examples from external libraries into renderer text.
Policy findings should point to this topic with a command field when the agent
needs style guidance; the actual teaching payload remains extracted source.
Because this style is the default teaching target, bare =guide --code= should
route to =typed-combinator-style=.

** Govern language capabilities instead of banning them

Gerbil Scheme macros and higher-order forms are language capabilities, not
mistakes.  The harness should not globally ban =defsyntax=, =syntax-case=,
=syntax-rules=, =case-lambda=, =cut=, =for/fold=, or POO protocol patterns.
Those constructs are the right way to express local DSLs, arity-polymorphic
APIs, structured aggregation, and extension dispatch when ordinary first-order
functions would produce repetitive or low-quality code.

The harness boundary is governance, not suppression.  Harness-owned source can
use advanced Gerbil features when the owner is explicit and tests cover the
behavior.  Generated or agent-authored macro forms require a POO
=macro-governance= policy override with a clear user explanation and a witness,
such as an expansion test, roundtrip test, or selector-backed review note.
Short placeholder explanations remain findings.  This keeps the language
expressive while making cheap agent escape paths visible and auditable.

** Turn capabilities into agent steering

Agent steering should not tell a model merely that the file is Scheme.  It
should expose the Gerbil-specific contract that the parser has already proven:
package modules keep =package:=, =namespace:=, import/export, and owner-local
test shape; POO dependencies and imports prefer =defclass=, =defgeneric=,
=defmethod=, and =defprotocol= over loose alist/hash object encodings; macro
edits require a runtime-source witness before changing transformer shape; pure
data transforms prefer higher-order forms such as =map=, =filter=, =fold=,
=cut=, and =for/fold=; and raw runtime IO overrides require a witness test
instead of treating =writeenv= or printer hooks as ordinary functions.

These directions belong in machine-readable policy and search facts before they
belong in prose.  The provider should render them through =check= findings,
=search= structural facts, =guide= exemplars, and the =info= packet's
=agentSteering= surface.  Documentation remains the reviewable contract, but an
agent should receive the steering passively from the harness when the project
declares or imports the relevant Gerbil capability.

** Use POO as the policy extension mechanism

POO is the right extension system for policy because it can model policy as a
protocol instead of a hard-coded ban list.  A policy implementation should be
able to answer:

- whether the rule applies to this owner and source kind
- whether user override is allowed
- which explanation and witness are required
- how the finding should render for the agent
- which search/json facts make the decision auditable

The default harness implementation stays strict for generated or agent-authored
code.  User-level policy may override that default, but the override must remain
parser-owned and visible in search, JSON, and self-apply snapshots.  This gives
users enough room to program in a non-rigid way while preventing agents from
turning policy into a cheap bypass.

** Prefer explicit precise imports

Gerbil reference code such as =gerbil-poo/cli.ss= uses =only-in= imports even
for common standard-library and local dependency surfaces. The harness should
teach this as an agent-facing default: when code imports a runtime library,
dependency package, or owner-local helper, the imported surface should be
explicit instead of relying on a broad module import.

The parser already records =moduleImportFacts= with the module, phase,
modifier, and imported symbols. Policy should consume those facts directly and
warn when a runtime source owner imports a governed module with modifier
=direct=, or with =only-in= but no imported symbols. The finding should tell the
agent to replace the broad import with =(only-in <module> <symbols...>)= after
checking actual owner usage. It must not infer missing symbols from raw text.

This rule is guidance for precision and future edits, not a runtime performance
optimization. =only-in= makes the dependency boundary auditable for agents; hot
path latency still belongs to CLI/module-loading design and parser-owned
indexing.

* Policy mapping

| Reference practice | Harness policy |
|--------------------+----------------|
| POO protocol and generic dispatch | =GERBIL-SCHEME-AGENT-R008= checks method/generic shape and requires protocol or class evidence. |
| Higher-order data transforms | =GERBIL-SCHEME-AGENT-R009= advises functional forms for pure data transforms instead of manual loops. |
| Macro transformer edits | =GERBIL-SCHEME-AGENT-R011= requires runtime-source witness evidence before agent-authored macro edits. |
| Protocol implementations | =GERBIL-SCHEME-AGENT-R012= requires declared protocol evidence for protocol-shaped owners. |
| Package/module conventions | =guide= and =info.agentSteering= expose package-module, import/export, test-root, and override policy guidance. |
| =t/= test owners | =GERBIL-SCHEME-MOD-R006= warns on non-native =test/*.ss= and =tests/*.ss= unless explicit package policy allows the directory with explanation |
| Domain owner names | agent policy rejects vague owner segments |
| Facades are thin | modularity policy rejects facade implementations |
| Runtime build excludes tests | build discovery should not include =t/= as runtime source |
| Directional conversion names | search guidance should prefer =A<-B= names for converter APIs |
| Vague helper names | =GERBIL-SCHEME-AGENT-R004= warns on =helper=, =process=, =handle=, =convert=, and similar definition names |
| No runtime effects at source top level | =GERBIL-SCHEME-AGENT-R005= warns on executable calls at =src/*.ss= top level |
| Macro capability governance | =GERBIL-SCHEME-CHECKER-W002= permits harness-owned macros but requires POO =macro-governance= policy, explanation, and witness for generated or agent-authored macro forms |

- POO protocol and generic dispatch maps to =GERBIL-SCHEME-AGENT-POLICY-008=:
  method/generic shape requires protocol or class evidence.

- Higher-order data transforms map to =GERBIL-SCHEME-AGENT-POLICY-009=: pure data
  transforms should prefer functional forms over manual loops.

- Macro transformer edits map to =GERBIL-SCHEME-AGENT-POLICY-011=: agent-authored
  macro edits require runtime-source witness evidence.

- Protocol implementations map to =GERBIL-SCHEME-AGENT-POLICY-012=: protocol-shaped
  owners require declared protocol evidence.

- Typed combinator style maps to =GERBIL-SCHEME-AGENT-POLICY-013=: runtime source

  owners with definitions should carry concise =;;; := signature comments near
  compact helper or transform definitions.

- Package/module conventions are exposed through =guide= and
  =info.agentSteering=: package-module, import/export, test-root, and override
  policy guidance should be passive agent input.

* Agent-facing guidance

An agent writing Gerbil Scheme should first identify the owner and then choose a
small native test owner under =t/=.  It should not create Python wrappers,
top-level =test/= folders, or generic =utils.ss= modules when a domain owner can
be named.

For complex control flow, the agent should introduce named helper functions,
use =case-lambda= for arity-specific public behavior, and keep retry, shutdown,
or generator state transitions in owner-local functions with tests.

When an agent needs a concrete model before writing code, it should ask for a
single focused exemplar first, for example =asp gerbil-scheme guide --code
--topic higher-order-control= or =asp gerbil-scheme guide --code --topic
poo-policy=.  Style guidance for concise typed comments plus small composable
functions may use =asp gerbil-scheme guide --code --topic
typed-combinator-style --intent style=.  Finding-driven repair may use =asp
gerbil-scheme guide --code --rule GERBIL-SCHEME-AGENT-POLICY-009 --intent repair=,
and macro governance review may use =asp gerbil-scheme guide --code --topic
macro-runtime-source --intent witness=.  The rendered excerpt should be small
enough to copy the pattern
mentally: adjacent source comment if present and one complete block.  Selector
metadata and copyable follow-up commands belong to =asp gerbil-scheme guide=
without =--code=.  The code form should not require a trailing =.= argument
unless project-local context is explicitly supplied through =--workspace=.

The policy layer should prefer parser-owned evidence over style prose.  It can
prove vague definition names from definition facts, and it can prove top-level
runtime effects from call facts whose caller is absent in =src/=.  Those facts
are good enough to warn the agent before low-quality Scheme lands in the
project.

Self-apply uses the default policy set.  All agent policies are enabled unless a
project-local =gerbil.pkg= policy explicitly uses =agent-policy enabled-rules=
or =disabled-rules= to select a different rule set.  This keeps the harness
strict by default while still giving downstream users a high-level configuration
interface for project-specific policy maturity.
