* Shared search runtime performance and Native acceleration

This module records the performance diagnosis, target architecture, and the
admission boundary for Native acceleration across language search providers.
It is a living RFC surface: benchmark receipts update the evidence, while the
ownership boundaries and performance requirements remain normative.

** Current decision

Native acceleration is not the default implementation strategy for language
harnesses.  The shared Rust search runtime must first own query validation,
scope, planning, indexing, ranking, output budgets, and hard deadlines.
Language harnesses publish parser-owned, query-free projection artifacts through
a bounded adapter contract; they do not reproduce the shared search pipeline.

The first Native optimization target is therefore a shared Rust read model,
not a C FFI implementation in every language directory.  A C ABI remains an
optional final-stage adapter for an acceleration or import kernel whose
release-mode profile proves that it, rather than orchestration or index access,
dominates the end-to-end latency.  It is never a replacement for a harness
native parser.

** Root causes to remove

1. Query grammar and guidance can disagree.  An exact owner item query such as
   =run_language_command= resolves, while a guided alternative form such as
   =run_language_command|preflight|provider= can miss and recommend a much more
   expensive lexical route.  The shared query parser must either implement the
   advertised alternative semantics or reject the form before search begins.
2. Planner recommendations can route a low-quality or missed structural query
   into an unbounded lexical/query-overlay path.  A miss must not silently buy
   a workspace scan.
3. The Turso durable source index is used directly as a hot search read model.
   Snapshot candidate lookup performs metadata checks and a term/owner query
   before ranking, and the sampled query remains inside Turso B-tree traversal.
4. The lexical/query-overlay path lacks an effective wall-clock deadline at
   the public command boundary.
5. Build and installation provenance do not currently guarantee that the
   installed CLI is a current release artifact.
6. Existing fast unit gates do not cover the real binary, realistic index
   sizes, WAL state, cold/warm behavior, fallback state, or output volume.

** Target architecture

#+begin_example
ASP shared Rust search runtime
  query grammar and preflight
  scope, budgets, planner, and deadlines
  incremental structural index
  ranking and compact rendering
  durable-state adapter
    Turso generations, recovery, and evidence
  acceleration read model
    immutable or memory-mapped token/owner/selector facts
  language artifact importer
    shared-schema validation and generation import
language harnesses
  native parser -> query-free L0/L1/L2 projection artifacts
#+end_example

The durable database remains the owner of generations, recovery, provenance,
and canonical persisted facts.  The query hot path uses a compact read model
derived from the current generation.  A generation mismatch or missing read
model returns a typed =cold-index-required= or =index-busy= state; it does not
trigger an implicit full-workspace scan.

The acceleration read model should support artifact-digest incremental import
and reuse, token-to-owner postings, owner-to-selector lookup, bounded intersections, and
deterministic ranking inputs.  Candidate implementations include Rust =fst=,
Roaring bitmaps, or an immutable memory-mapped layout.  The selected format is
an implementation detail behind the shared search contract.

** Owner-language admission
Before ASP resolves a provider or starts an out-of-process language binary, the
parent proves that an owner path belongs to the requested language and declared
workspace. A =.org= owner sent to =gerbil-scheme=, a missing owner, or an owner
outside the workspace returns =owner-language-mismatch= with the requested
language, path class, and suggested route. It starts no provider, refreshes no
index, and never falls back to lexical or full-workspace collection. The compact
form is the =admission= node in =semantic-owner-item-evidence.v2=: it records
=status=, =requestedLanguage=, the owner extension, declared provider
extensions, and an optional suggested language. =owner-language-mismatch= is a
terminal route state, not empty evidence.

The Rust search facade repeats the cheap boundary: canonicalize a regular owner
path, require a registered language extension, and reject invalid owners before
search planning. Owner-item terms are a bounded structured set; pipe-delimited
input must not become an open-ended provider regex. This second admission check
prevents a parent dispatch regression from becoming an unbounded provider
allocation.
The owner-item command materializes direct declaration and selector evidence;
it does not recursively construct call facts and apply a limit afterward. Call
evidence is an explicit evidence/search route, so an item selector query cannot
turn a bounded result request into an unbounded syntax walk.

** Language adapter levels

| Level | Responsibility | Required runtime cost |
|-------+----------------+-----------------------|
| L0 | File identity, bounded lexical tokens, and source-kind facts. | Harness artifact publication; no language process on indexed reads. |
| L1 | AST owners, symbols, item kinds, dependencies, and structural selectors. | Native harness projection publication; never a search fallback. |
| L2 | Optional type, macro-expansion, compiler, or language-server facts. | Explicit harness projection lifecycle; never a warm-read activation. |

A new language may ship L0 or L1 without implementing a complete provider CLI.
The harness publishes the shared projection schema; the Rust runtime owns
search-query parsing, fallback policy, ranking, receipts, and rendering.  This
is the primary way to reduce the cost of adding complex language support.

** Harness projection, ASP import, and search separation

Language parsing and searching are separate planes.  Every
=languages/*-language-project-harness= is the sole owner of its native parser,
AST, structural selector dialect, module semantics, and parser-owned language
projection.  A harness may share its grammar and AST foundation internally
between parser-facing commands, but it does not implement search planning,
query composition, ranking, EvidenceGraph, GraphRoute, source-index lifecycle,
or a language-local search CLI for ASP.

The parser-facing projection is a query-free artifact governed by a shared
schema.  It carries only parser-owned language facts: language and parser ABI,
selector dialect, source paths, owners, exact selectors, and typed semantic
relations.  It MUST NOT carry a user query, candidate list, view, rank, next
action, rendered search result, source digest, workspace identity, import
generation, or artifact identity.  Rust consumes this artifact as data; it
does not accept a source snapshot, define a language parser request, construct
an AST, or infer language semantics from source text.

ASP Rust owns the generic lifecycle every harness can share: shared-schema
validation and decoding, normalized workspace anchor and source-digest
calculation, artifact identity comparison, parser-ABI reuse, incremental
generation selection, transactional import, durable Turso storage, and
construction of the immutable acceleration read model.  This lifecycle manages
normalized facts, not language parsing.  A changed or absent projection is
=projection-stale= or =cold-required=; an agent-facing search miss never starts
a parser, rebuilds a projection, or invokes a provider search command.

Turso stores the canonical normalized copy of harness artifacts.  A warm
=asp <language> search= reads only the current generation and its acceleration
read model.  It performs token postings, bounded intersections, ranking,
EvidenceGraph construction, GraphRoute selection, and rendering with
=providerProcessCount=0= and no native finder or language-parser process.  The
only parser-facing read is an exact structural =query --code= request: the
facade forwards an already-materialized selector to the harness and consumes
the parser-owned exact-selector receipt.  That request is not a search fallback
and receives no query-set or graph-routing arguments.

=gslph fmt <path>= is an independent formatting command.  Its implementation
and cache are private to formatting and MUST NOT share ASP artifacts, Turso
state, source-index generations, lifecycle caches, or implicit import side
effects.  The historical =gslph source-index refresh= command is removed rather
than folded into =fmt=.  The only permitted common implementation is the
harness-internal grammar/AST foundation; command adapters, cache namespaces,
and artifact lifecycles remain separate.

This division is a cross-language requirement, not a Gerbil exception.  It
lets every registered harness retain native semantic authority while ASP
provides one high-concurrency, read-only search and graph runtime.

** EvidenceGraph owner-item route
An owner-item search is an EvidenceGraph route, not a source-text or selector-string route. Given a language, normalized owner path, and bounded query terms, ASP reads only matching =selector= nodes and typed edges from the current Turso generation. The node carries the parser-owned symbol, kind, exact selector, path, and normalized query keys. Ranking consumes those structured fields; it MUST NOT recover a symbol or kind by splitting a selector string, inspecting source text, or invoking a language provider.

The result is compact graph evidence: exact selector identities plus their owner and typed relation context. Source-index postings remain useful for broad owner discovery, but they are not an owner-item answer. A source-index candidate without a bounded parser selector is not eligible for this route. The GraphRoute chooses the next exact =query --code= or a typed graph-neighbor route such as dependency or test evidence.

Projection publication is an explicit ASP lifecycle action. It may invoke the registered harness's bounded, query-free projection adapter for one admitted owner, then validates, digests, imports, and persists the result. A normal search never invokes that adapter. If its generation is missing, stale, or busy, it returns the corresponding typed state with =providerProcessCount=0= and an ASP projection-import action; it never falls through to a provider =search= or =semantic-facts= command.

The canonical lifecycle action is =asp <language> projection import --owner <relative-owner-path> --workspace <root>=. It invokes exactly one registered harness adapter as =projection <owner> --workspace <root> --json=, accepts only the shared query-free projection schema, and imports it through the common ASP lifecycle. The receipt reports =status=imported|reused=, =parserProcessCount=1=, and persisted graph counts. The action rejects absolute or parent-traversal owners, query and ranking flags, language-mismatched projection payloads, and projection payloads which do not contain the requested owner source. It is the sole parser-process exception; it does not expose a generic provider-command forwarding surface.

** Native role and boundary

Native has three legitimate roles:

1. The shared Rust acceleration read model used by all languages.
2. Shared-schema validation, normalized-fact import, and immutable read-model
   construction for artifacts already produced by language harnesses.
3. An optional batch C ABI for a proven acceleration or import kernel when no
   safe Rust implementation meets the measured target.

Native does not own language parsing, ASTs, selector dialects, parser requests,
the public search contract, agent guidance, fallback policy, provider registry,
or language-specific output format.  It consumes normalized artifacts and
returns shared search state or a typed failure.  Per-symbol FFI calls, hidden
workspace scans, and Native-only query semantics are not allowed.

** C ABI admission gate

A C ABI implementation may be proposed only when all of the following are
true:

1. The command runs as a current release artifact and passes the real CLI
   benchmark matrix.
2. Query preflight, read-model access, fallback deadlines, and output budgets
   already satisfy their gates.
3. A release-mode profile shows one acceleration/import kernel consuming more than
   40 percent of end-to-end p95 latency and at least 50 ms per representative
   batch.
4. The interface is deterministic, batch-oriented, ownership-safe, and does
   not expose language runtime objects across the ABI.
5. macOS and Linux release packaging, ABI versioning, failure isolation, and
   memory-safety tests are defined before adoption.
6. The same normalized fact contract remains available to non-Native adapters.

Failing any gate keeps the implementation in Rust or in the existing language
adapter.  C FFI is a measured final compression step, never a fallback for an
unbounded shared search path.

** Performance requirements

All timing gates execute the real current release binary.  Debug binaries may
be used for development diagnostics but never satisfy or populate these
receipts.

| Query class | Required p95 | Additional invariant |
|-------------+--------------+----------------------|
| Invalid or over-broad query | <= 20 ms | No database or provider activation. |
| Exact owner/item, warm | <= 20 ms | No lexical fallback. |
| Exact owner/item, cold | <= 75 ms | Return =cold-required= or =projection-stale=; no parser. |
| Indexed symbol/search, warm | <= 30 ms | No provider process. |
| Indexed symbol/search, cold | <= 100 ms | Bounded read-model load or typed cold state; no parser. |
| Incremental one-file artifact import | <= 50 ms | No source-language parse or unrelated import. |
| Any implicit fallback attempt | <= 200 ms hard deadline | Return one focused =nextCommand= after the deadline. |

Performance tests must also bound candidate count, provider process count,
native finder process count, output bytes, peak RSS, index state, and fallback
state.  A fast empty result from a broken index is not a passing benchmark.
Parser projection publication is a separate harness performance contract.  It
may measure native parsing and its own cache, but it must never be charged to
an agent-facing search receipt or used as a search fallback.

** Improvement order

The implementation and tracking order is normative:

1. Make release provenance explicit and prohibit debug artifact installation.
2. Enforce query grammar, broad-query rejection, and public wall-clock budgets.
3. Remove structural-miss-to-unbounded-lexical fallback.
4. Build the shared generation-bound acceleration read model.
5. Move languages to the L0/L1/L2 query-free projection artifact contract.
6. Re-profile release scenarios and evaluate the C ABI admission gate.

The next Native decision occurs only after steps 1 through 5 have fresh
scenario receipts.  Until then, the recorded decision is =shared-rust-read-model=
and the C ABI status is =not-admitted=.

** Search hot-path latency is a hard contract

Interactive =search owner=, exact selector query, resident dispatch, and cache
lookup are millisecond-scale operations.  A slower timeout is not a recovery
mechanism: increasing a deadline MUST NOT be used to convert a latency failure
into an apparent success.

Every hot-path stage MUST emit a terminal receipt within its own fixed budget.
The receipt records =elapsedMs=, =stage=, =queryIdentity=, =selector= (or
=null=), cache state, provider-process count, and
=result|empty|timeout|cancelled=.  The first missed stage budget cancels its
owned work and returns that receipt; callers MUST NOT silently retry with a
larger deadline or start a duplicate resident worker.

The performance gate measures warm and cold paths separately.  Warm exact
owner/selector queries MUST be served from the Rust read model without a
provider process.  A cold path may refresh bounded missing state, but MUST NOT
perform an unconditional workspace refresh or spawn a language provider merely
to discover that cached state is current.  Tests fail on latency-budget breach
even if a later result is semantically correct.

Timeout policy is operation-specific and contract-visible.  A global provider
timeout environment variable remains only a safety ceiling for explicitly
long-running provider commands; it MUST NOT define or relax interactive search
latency.  Search quality work therefore removes the blocking stage or process
boundary that caused the delay instead of increasing the wait.

** ASP provider process memory containment

Every =asp <language>= native provider process runs behind the shared Rust
transport boundary. The boundary applies a default 1 GiB process memory limit
on Unix (an inherited address-space limit on Linux and parent-observed RSS
monitoring with process-group termination on macOS), and accepts an override through
=ASP_PROVIDER_MEMORY_LIMIT_BYTES=; =0= explicitly disables the memory limit.

The parent MUST survive provider failure and preserve an abnormal termination
receipt. The receipt records configured memory bytes, whether enforcement was
active, exit code or Unix signal, timeout state, output byte counts/digests,
and a stable termination reason. A signal compatible with allocation failure
under an active limit is reported as =memory-limit-suspected= rather than being
silently collapsed to exit code 1. Timeout receipts also remain present in the
client receipt instead of being discarded while converting transport errors.
- =gslph evidence graph/analyze= may read package identity, but MUST NOT build a
  source index merely to choose the project owner node.
- A source-backed guide exemplar already has an exact owner path.  It MUST parse
  only that owner and MUST NOT call =collect-project=.

** Turso SQL-owned incremental source-index refresh

The durable source-index refresh path MUST compute membership changes inside
Turso.  The Rust client may retain the incoming provider import and the bounded
set of changed or removed owner paths, but it MUST NOT load a second complete
snapshot membership map merely to compare file hashes.

Each refresh stages the incoming =(owner_path, file_hash)= membership on the
active connection and uses indexed SQL joins to derive the write set:

- incoming =LEFT JOIN= canonical owners yields added and hash-changed owners;
- canonical owners =LEFT JOIN= incoming yields removed owners;
- only added or changed owners are upserted;
- only changed or removed owners have token postings rebuilt or deleted;
- generation and layout metadata publish in the same transaction after those
  writes succeed.

The staging relation is connection-local, keyed by =owner_path=, cleared before
every import, and populated from the already-canonical incoming membership
packet.  This keeps refresh memory proportional to the incoming packet plus the
actual change set, rather than the incoming packet plus a second full database
snapshot.

Rust retains that incoming membership as a borrowed lookup over the provider
import; it MUST NOT duplicate every owner path and file hash into another owned
map.  Canonical selector JSON and semantic-term projection MUST be constructed
only after the SQL joins return the change set, and only for added or changed
owners.  Validating selector-to-owner membership may inspect every selector,
but unchanged selector bodies, source text, and term vectors MUST NOT be cloned
or serialized during an incremental refresh.

The incremental performance gate MUST include a representative large snapshot
followed by a one-percent change refresh.  Its receipt records elapsed time,
changed owners, removed owners, posting writes, and peak RSS.  Correctness tests
also cover additions, hash changes, removals, and a projection-rebuild request.

The canonical Scenario is
=asp-source-index-incremental-refresh-one-percent=: 1,193 owners followed by
exactly 12 changed file hashes, no additions, and no removals.  Its release
Benchmark MUST assert =changedOwnerCount= of =12=, =removedOwnerCount= of =0=,
and =postingWriteCount= of =204= in addition to the 500 ms incremental-refresh ceiling
and 128 MiB scenario RSS budget.  Refresh reports expose these write-set counts;
a generation-reuse report exposes zero for all three.  This structural gate
prevents an all-owner selector or posting rebuild from passing only because the
benchmark host is fast.

The Turso read model applies its own defensive query bounds even when the
caller already passed shared parser admission.  Query text is limited to 16
KiB, token projection to 32 terms, and returned candidate requests to 256.
These limits bound the per-token posting fan-out and candidate materialization
inside the Rust client DB process; a language provider or malformed internal
caller cannot expand read memory by supplying an unbounded term list or
=u32::MAX= candidate limit.

In-process source-index read/write exclusion is scoped by normalized Turso DB
path, not by one process-global lock.  Independent repositories MUST NOT return
=Busy= merely because another repository is refreshing its own index.  The
lock registry stores weak references and prunes entries whose DB lock is no
longer active, so opening many projects does not create an unbounded resident
lock map.  Cross-process serialization remains owned by the existing
DB-specific operation lock.
