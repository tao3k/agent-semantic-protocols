* One-sentence summary
ASP treats platform session identifiers as identity facts, but makes the
=asp= binary gate the authority for search/query routing, resident child
validation, heartbeat, TTL, expiration, and GC records.

* Problem
ASP now routes expensive search and query work through a resident
=asp-explore= child session. Hook messages can help the main agent recover
early, but they are not a sufficient enforcement boundary: a hook can be
missing, stale, or bypassed by another runtime. The =asp= binary must therefore
check the session before any search/query class command runs or writes
artifacts.

That requires durable state:

- the root agent session id, usually from =CODEX_THREAD_ID= in Codex
- the child session id returned by the subagent runtime
- the child role, name, and parent/root relationship
- whether the child was registered and validated against the expected
  =asp-explorer.toml= profile
- whether the child is still usable for search routing
- when stale child rows should stop being routed to
- when historical rows can be archived or removed

Codex and other agent runtimes can provide deterministic session ids, but they
do not provide one portable, stable active-session truth source. ASP therefore
separates three concerns: environment ids identify the current runtime session,
the registry owns root/child relationships, and optional platform adapters
validate child profile/status when the host exposes a stable JSON status
contract.

* Source Model
Session identity and session lifecycle are separate contracts.

| Contract | Owner | Examples | Stability |
|----------+-------+----------+-----------|
| Identity | agent runtime | =CODEX_THREAD_ID=, =CLAUDE_CODE_SESSION_ID=, =AGENT_SESSION_ID=, =SESSION_ID= | process/session scoped |
| Registry | ASP DB | root id, child id, parent id, name, role, status | durable |
| Binary gate | =asp= runtime/dispatch | restricted command class, current role, child validation | authoritative before query/search |
| Liveness | ASP DB | heartbeat, last_seen, expires_at, last_tool_event | ASP-owned |
| Profile validation | platform adapter + expected config | Codex agent status JSON, =asp-explorer.toml= fingerprint | required before routing child search/query |
| Reconcile | optional platform adapter | Codex CLI session status, future Claude status | best-effort |

ASP must never require a platform active-list API for normal routing.

* Platform Facts
** Codex CLI
The current Codex CLI exposes saved-session operations such as =resume=,
=archive=, =delete=, =unarchive=, and =fork=. =codex resume --all= can show saved
sessions, but it is not an active-session list and does not prove that a thread
is currently executing.

A Codex adapter should use a stable Codex CLI status JSON command when one is
available. If no such public CLI command is available, the adapter reports
=unknown= and leaves binary gate routing to the registry plus artifact
evidence.

** Codex Desktop host
The Codex desktop host can expose thread summaries with statuses such as
=active=, =idle=, and =notLoaded= through app-owned tooling. ASP may use that
data for optional reconciliation when such a connector is available. ASP must
not make this connector mandatory because hooks and CLI commands must remain
valid in Codex CLI, Claude Code, and future runtimes.

** Claude and future runtimes
Claude Code and future runtimes may expose their own session environment
variables. ASP should accept them as identity facts, not as lifecycle truth.
If a runtime later provides a stable active-list API, ASP can add a reconcile
adapter without changing the core DB contract.

* Session Identity Resolution
ASP resolves the current session id from the current process environment.

Resolution rule:

1. Collect supported runtime session environment variables, currently
   =CODEX_THREAD_ID=, Claude Code session variables, =AGENT_SESSION_ID=, and
   =SESSION_ID=.
2. If exactly one supported id exists, use it as the current =SESSION_ID=.
3. If multiple supported variables exist with the same id, use that id.
4. If multiple supported runtime ids exist and disagree, fail closed instead of
   guessing priority.
5. If no supported runtime id exists, ASP is outside an agent session and must
   not infer one from CWD, transcript path, or saved-session files.

This deliberately avoids priority ordering between Codex and Claude. A single
agent process should have one runtime session identity. Root/child identity is
not resolved from an ASP-owned environment variable. If the current session is a
registered child, ASP discovers its root by registry lookup; otherwise the
current runtime id is the root candidate.

ASP must not introduce an =ASP_ROOT_SESSION_ID= public contract. That would
make ASP responsible for proving which platform runtime is active, while the
real authority is the host runtime environment plus the registry relationship.

** Codex Thread Signal And Child Discovery

A non-empty =CODEX_THREAD_ID= is a positive Codex runtime signal. It identifies
the current host thread and enables Codex-specific reconciliation, but it is not
by itself lifecycle truth. The registry remains the routing authority.

When Codex thread APIs are available, ASP MAY use them as an adapter for child
discovery. The parent should enumerate threads with =list_threads=, select
threads whose =source_thread_id= or delegation source matches the current root
thread, and then use =read_thread= to inspect recent turn status, errors, and
update timestamps. Thread title, tags, delegation payload, or configured agent
name MAY be used as secondary evidence to identify a resident role such as
=asp-explore=.

Codex child discovery is additive evidence. A discovered child thread can
confirm that a registry row is still associated with a host thread, expose the
host-reported model, and clarify whether a child is active, idle, completed, or
silent. Missing, stale, or partial Codex thread results MUST NOT cause raw
source fallback and MUST NOT override a valid registry row without explicit
non-routable evidence.

The Codex adapter SHOULD persist the discovered host facts in the session
activity snapshot keyed by =sessionId=. It MUST NOT encode dynamic activity in
the session id string itself. Stable identity belongs in the session key;
changing activity, validation, model, sandbox, and liveness observations belong
in the associated registry/activity record.

** SessionStart Topology Validation

Session start is the earliest reliable point to prevent nested resident-worker
creation. When a new Codex-backed session starts and =CODEX_THREAD_ID= is
present, the ASP hook SHOULD run a bounded topology validation before allowing
restricted ASP search/query work to proceed.

The validation input is the current root thread id, the starting child thread
id, the configured resident role, and any host facts available from Codex thread
APIs. If the host exposes =source_thread_id=, the hook SHOULD enumerate sibling
threads under the same parent and count children with the same resident role. If
the host API only exposes known-thread reads, the hook MUST fall back to
registry/delegation evidence and persist the parent relationship when the child
is registered.

For =asp-explore=, the validation result MUST answer whether exactly one
routable resident child exists for the parent root session. A child session MUST
NOT create another =asp-explore= child for itself. If validation finds that the
current session is already a resident child, restricted ASP commands MUST route
within that child and MUST NOT request another nested resident worker.

The hook MAY require a topology receipt before continuing. The receipt SHOULD
include the parent thread id, child thread id, discovered sibling count, matched
resident role, validation status, and next action. Failure to obtain host thread
facts is not raw-source fallback permission; it is a bounded reconciliation
state that must route through registry status, resume, or deny.

SessionStart MUST also check registration before allowing a child session to
run restricted ASP work. A newly opened subagent has a child session id, but it
may not have the parent thread id in its local environment. The hook therefore
uses the child session id to query the ASP session registry and recover the
registered project/root context.

The registration gate is:

1. Look up the child session id in the registry.
2. Read the registered =projectId= and, when present, the =rootSessionId=,
   =parentThreadId=, role, model, and sandbox validation facts.
3. If no registry row or project context exists, deny SessionStart with a
   registration instruction for the child:
   =asp agent session register --name <resident-name> --child-session-id <child-session-id> --role <role>=.
4. After registration succeeds, write or refresh =activitySnapshotShort= and
   continue the lifecycle validation loop.

This gate makes registration the handoff between Codex child creation and ASP
lifecycle ownership. =projectId= is used here to locate the project-scoped
registry record and configuration. It is still not the resident ownership key;
the routable resident key remains the root/parent/child/role relationship.

The topology MUST NOT use project id as its primary key. A subagent may be
opened in a host thread that is not represented under the same project folder
view as the parent, and users may have many projects, parent threads, and
subagents open at the same time. =projectId= is useful for repo/workspace
scoping, but resident-child ownership is keyed by =rootSessionId=,
=parentThreadId=, =childThreadId=, and =role=.

write it during SessionStart into =activitySnapshotShort=. That snapshot is the
fallback source of truth when =list_threads= only exposes user-visible thread
summaries.

* Binary Gate Model

** Resident Agent Policy Configuration And Guardrails

The resident =asp-explore= child is a policy-configured worker, not an
untyped background thread. Its model, agent role, root thread, parent thread,
and sandbox expectations MUST be validated against the configured agent
profile before the child is considered routable.

The model is configurable by policy. Runtime validation MUST compare the
expected model and role from the ASP agent profile with the host-reported child
thread facts. A mismatch is a validation failure unless the profile explicitly
allows that migration. The validation receipt SHOULD record both expected and
actual values so an operator can distinguish a stale child from an intentional
model update.

Sandbox posture is part of the same guardrail. A child that is expected to be
read-only but is reported as write-capable MUST NOT be silently accepted. The
registry MAY keep the child as a warning state for diagnosis, but routing
decisions MUST treat the warning as policy evidence and MAY require reconcile,
resume with corrected configuration, or deny before dispatching search/query
work.

Duplicate worker policy is scoped by =(rootSessionId, role, name)=. For a given
root session, creating a second active =asp-explore= worker is forbidden while a
routable child exists. Timeout by itself is not replacement evidence. The parent
MUST first inspect registry status, child thread activity, and rollout activity,
then resume or reconcile the existing child before considering replacement.
The =asp= binary gate is the authoritative enforcement point. Hook decisions are
helpful recovery and UX hints, but every search/query class command must pass
the binary gate before provider execution or artifact writeback.

** Lean Recovery Obligations

The agent-facing recovery flow is modeled as a small state machine in
~tests/fixtures/agent_session_recovery/agent_session_recovery.lean~.

The proof obligations are intentionally focused on failure handling and
multi-step trace safety:

- parent wait deadline does not imply child replacement
- active-running heartbeat does not imply child replacement
- send failure resumes the existing child before replacement
- resume failure queries session status before replacement
- completed previous turn is read/reused, not replaced by itself
- model abnormal close is distinct from completed turn and may replace
- provider timeout records an internal timeout receipt and process-group cleanup without replacing the agent child
- replacement only follows host-not-found, validation failure, model abnormal close, or bounded no-progress exhaustion
- any trace with no replacement evidence cannot contain a replace action
- bounded retry preserves non-replacement while budget remains
- an underspecified hook deny receipt permits raw fallback in the model; the sound receipt must include requiredAction, nextAction, completionReceipt, and a raw-fallback prohibition
- the standalone recovery policy never emits raw fallback; every modeled failure maps to an ASP protocol action

* Registry Model

** Thread Status And Liveness Taxonomy

ASP registry state is the durable owner of routing, while host thread and
rollout facts are liveness observations. The status vocabulary below is the
shared policy surface for resident children:

- =active=: a current turn is running or recent heartbeat/activity exists.
- =idle=: the child thread is loaded and routable, but no current turn is
  running.
- =completed=: the latest delegated turn completed and its result can be read.
- =timeout=: a wait deadline expired, but the child has not been proven dead.
- =stale=: no recent heartbeat, artifact update, or host event exists within
  the configured freshness window.
- =dead=: the child is non-routable because validation failed, the host reports
  the thread/process is gone, or the registry row can no longer be reconciled
  with host evidence.

=timeout= and =stale= are routing constraints, not replacement commands. A
parent wait deadline, such as a 180 second tool wait, is not enough to classify
a child as stuck. The parent MUST query the child by session/thread id and read
the rollout last event, last heartbeat, current turn id, and terminal event
before assigning =stale=, =dead=, or replacement status. Long-running searches
that continue to emit rollout events are =active= or =running= even when the
parent wait call times out.

=timeout= and =stale= MUST trigger bounded status checks, resume attempts, or
reconciliation before a new worker is spawned. =dead= MAY permit replacement,
but only after the parent records the validation or host evidence that made the
old child non-routable.

Duplicate-worker detection MUST aggregate all known children under the same
root session and role. Counting only the current child row is insufficient; the
parent must consider registry rows, Codex child threads when available, and
rollout activity before deciding whether another worker would violate the
single-resident-child contract.
ASP stores the durable relationship in an agent session registry DB.

Canonical row intent:

| Field | Meaning |
|-------+---------|
| root_session_id | Stable root agent session id. |
| session_id | The represented session id. For child rows this is the child session id. |
| parent_session_id | Direct parent session id. For =asp-explore= this is normally the root id. |
| name | Human and routing name, for example =asp-explore=. |
| role | Machine role, for example =root= or =asp-explore=. |
| status | Lifecycle status used by routing and GC. |
| created_at | First registration time. |
| updated_at | Last registry mutation time. |
| last_seen_at | Last ASP-owned observation of this session. |
| last_heartbeat_at | Last explicit heartbeat. |
| expires_at | Timestamp after which this row is not routable unless renewed. |
| archived_at | Timestamp for soft archival. |
| last_tool_event | Last hook event that touched this session. |
| last_command | Compact command surface, not full source output. |
| last_evidence_ref | Compact evidence/checkpoint reference. |
| expected_profile_path | Expected resident child profile, for example =~/.codex/agents/asp-explorer.toml=. |
| expected_profile_fingerprint | Stable digest of the profile ASP expected at registration time. |
| platform_profile_fingerprint | Stable digest of the profile observed from platform status when available. |
| validation_status | =passed=, =failed=, =unknown=, or =not-required=. |
| validation_reason | Compact mismatch or adapter reason. |
| validated_at | Last successful or failed validation timestamp. |
| metadata_json | Provider-specific metadata that is not part of routing. |

Identity fields are stable. Liveness fields are monotonic observations.
Validation fields are routing inputs for resident search/query children.
Metadata must not be required to decide routing.

** activitySnapshotShort Lifecycle Loop

The registry row is durable identity. =activitySnapshotShort= is the bounded
activity fact attached to that identity. It is the right place to store
short-lived host facts such as parent thread id, child thread id, role, model,
current turn id, lifecycle state, last event time, last heartbeat time,
terminal event, last artifact update, close request, destroy request, and
reconciliation status.

Every Codex-backed child session SHOULD write an =activitySnapshotShort= record
at session start before it is allowed to run restricted ASP search/query work.
The record MUST include the root session id, parent thread id when known, child
thread id, resident role, expected model, actual model when available, sandbox
posture, and initial lifecycle state. This write is the topology receipt used by
SessionStart validation.

The lifecycle loop is:

1. =start=: create or discover the child thread, write
   =activitySnapshotShort=, validate topology, then mark the child routable only
   after the receipt is present.
2. =resume=: reuse a known child when the registry row and host facts still
   reconcile; refresh =lastSeenAt=, current turn id, last event, last
   heartbeat, terminal event, and activity state.
3. =close=: mark the child =closing= or =closed= when work is complete or the
   user closes the thread. Closing prevents new work but keeps enough evidence
   for recovery and audit.
4. =destroy=: mark =destroyRequested= and then =destroyed= when the host thread,
   process, or worker has been torn down. Destroyed children are not routable.
5. =reset=: clear or archive stale =activitySnapshotShort= facts for a root
   session when a workday, thread, or task boundary ends. Reset MUST preserve
   durable audit evidence while removing active routing claims.

Close and destroy are different operations. Closing a thread is a lifecycle
state transition; destroying it is host cleanup. A closed child MAY still be
read for result recovery. A destroyed child MUST NOT be resumed unless the host
explicitly reports that the same thread has been restored.

The daily or task-end cleanup loop SHOULD enumerate all known children for the
root session, close routable but idle children when no task needs them, destroy
or archive dead children, and reset resident-name claims that would otherwise
block future work. This cleanup MUST be idempotent so repeated SessionStart or
thread-close hooks do not create duplicate workers or lose recovery evidence.

* Resident Child Resume Contract
ASP must prefer an existing validated child session over creating a new one.

Before any main/root agent starts a new =asp-explore= worker, it must ask ASP
for the routable child under the current root session:

#+begin_src sh
asp agent session resume --name asp-explore --json
#+end_src

If a routable =asp-explore= child exists, the main/root agent must resume or
send work to that child session id. It must not spawn another =asp-explore=
child for the same root.

Creating a new child is allowed only when:

- no =asp-explore= row exists under the current root
- the existing row is =stale=, =expired=, =archived=, =invalid=, or otherwise
  not routable
- the platform-specific resume/send operation reports that the child session no
  longer exists, and ASP records the old row as stale or expired before a
  replacement can become active

This rule is mandatory for token control and memory consistency. A root session
should have at most one routable child named =asp-explore= at a time.

* Registration Validation Policy
The public registration surface only accepts the new child id:

#+begin_src sh
asp agent session register \
  --name asp-explore \
  --child-session-id <child-session-id> \
  --role asp-explore
#+end_src

Registration must validate the child before it writes an active routable row.
The validation is internal to ASP and must not require an agent-visible
=--old-child-session-id= or separate =validate= step.

Validation checks:

- resolve the current root session id from runtime env and registry lookup
- load the expected profile, for example =~/.codex/agents/asp-explorer.toml=
- query the platform status/config adapter for the new child session id when
  available
- compare expected profile fields and fingerprints against the observed child
  status/config
- reject registration when the child was created with the wrong model, role,
  tools, permissions, or config fingerprint

When validation fails, ASP must not create an active row. It returns compact
instructions to close or discard the wrong child, create a new =asp-explore=
using the expected profile, and run the same =register= command again with the
new child id.

When validation succeeds, ASP may internally mark older routable children with
the same =(root_session_id, name)= as =replaced= or =invalid= before activating
the new row. The old child id is registry history, not public agent-facing CLI
input.

* Lifecycle States
ASP uses soft lifecycle states.

| Status | Meaning | Routable |
|--------+---------+----------|
| active | Recently observed, within active TTL. | yes |
| idle | Not currently observed but within idle TTL. | yes, with refresh |
| stale | Past active/idle TTL but not archived. | no |
| expired | Explicitly expired or past hard TTL. | no |
| invalid | Registration or later validation failed. | no |
| replaced | Superseded by a newer validated child under the same root/name. | no |
| archived | Kept for memory/recovery only. | no |
| deleted | Physically removed by GC. | no |

Routing must only use =active= or =idle= rows whose =expires_at= is absent or in
the future and whose resident-child validation status is =passed= when the row
is used for ASP search/query routing.

* TTL Policy
TTL is an ASP policy, not a platform fact.

Recommended defaults:

| Role | Active TTL | Idle TTL | Hard retention |
|------+------------+----------+----------------|
| root | 30 minutes | 24 hours | 30 days |
| asp-explore | 10 minutes | 2 hours | 14 days |
| worker | 10 minutes | 2 hours | 14 days |
| audit/archive worker | 30 minutes | 24 hours | 30 days |

These defaults should be configurable, but hook safety must not depend on user
configuration being present. Missing config uses defaults.

* Heartbeat Contract
Heartbeat is the mechanism that turns identity into routable liveness.

ASP updates session liveness when:

- =asp agent session register= creates or updates a row
- a hook observes a root or child session in =session-start=
- a pre-tool hook evaluates ASP routing for a root or child session
- a post-tool hook records a compact command/evidence receipt
- =asp agent session heartbeat= is called explicitly by a long-running worker
- =asp agent session list --active= refreshes expired statuses before printing

Heartbeat must update =last_seen_at= and =updated_at=. Explicit heartbeat also
updates =last_heartbeat_at=.

Heartbeat must not store full command output, source dumps, or large search
packets. It stores compact evidence refs and command summaries only.

* Hook Stage Responsibilities
Hook stages are recovery and UX layers. They should reduce wasted turns by
telling the main agent what to do early, but they are not the authoritative
security or routing boundary. The =asp= binary gate must repeat the session
check before every search/query class command.

** session-start
Resolve the current root =SESSION_ID=. If no active =asp-explore= child exists
for that root, return a compact instruction for the main agent to create one
and register it:

#+begin_src sh
asp agent session register \
  --name asp-explore \
  --child-session-id <child-session-id> \
  --role asp-explore
#+end_src

The root id is inferred from the environment. The command does not require a
visible =--root-session-id= in the normal same-session path.

If a routable =asp-explore= child already exists, session-start must not ask the
main agent to create a new one. It should return the existing child session id
and instruct the main agent to resume that child.

** pre-tool
If the current session is the root/main session and it attempts ASP search or
query exploration, the hook may check for an active or idle =asp-explore= child
and return an early routing instruction.

- If a routable child exists, return the child id and resume instruction.
- If no child exists, instruct creation/registration.
- If the child exists but is stale/expired/invalid, instruct recreation or
  heartbeat recovery.
- If the current session role is =asp-explore= and it is routable, let the
  command proceed; the binary gate still validates before provider execution.

The pre-tool message for an existing child must be resume-first. It should say:

#+begin_example
Resume registered asp-explore child session <child-session-id>.
Resume or send this search/query to that child; do not spawn another asp-explore
for this root unless resume fails and the old child is expired.
#+end_example

** post-tool
Post-tool records compact evidence:

- session id
- root session id
- child session id when present
- command family
- provider/language id when present
- evidence refs
- next command or next action when available
- success/failure classification

Post-tool should refresh heartbeat for the executing session and attach compact
checkpoint evidence. It should not turn the main session into the search owner.

* Routing Rules
Main/root session:

| Command family | Behavior |
|----------------+----------|
| =asp agent ...= | allow |
| =asp hook ...= | allow |
| =asp install ...= | allow |
| =asp org capture/lint/guide/archive/recall= | allow unless configured otherwise |
| =asp rg= | deny and route to =asp-explore= |
| =asp query/search/pipe= | deny and route to =asp-explore= |
| =asp org query/search= | deny and route to =asp-explore= |
| =asp <language> query/search= | deny and route to =asp-explore= |

Child =asp-explore= session:

| Command family | Behavior |
|----------------+----------|
| ASP search/query | allow when active or idle |
| ASP install/hook state mutation | deny unless explicitly allowed |
| destructive source mutation | deny |

* CLI Contract
Agent-facing commands:

#+begin_src sh
asp agent session register \
  --name asp-explore \
  --child-session-id <child-session-id> \
  --role asp-explore

asp agent session resume --name asp-explore --json

asp agent session status --name asp-explore --json
#+end_src

Administrative and diagnostic commands may include:

#+begin_src sh
asp agent session heartbeat [--child-session-id <id>] [--role <role>]
asp agent session list --active
asp agent session list --all
asp agent session show --name asp-explore
asp agent session expire --ttl <duration>
asp agent session gc --older-than <duration> [--dry-run]
asp agent session reconcile --provider codex [--dry-run]
#+end_src

The normal same-session path should not require users or agents to pass
=--root-session-id=. That flag remains for cross-session administration and
debugging.

=asp agent session resume --name asp-explore --json= should return a compact
routable child receipt when one exists. If no child is routable it should
return a machine-readable miss that session-start, pre-tool, and the binary gate
can use to emit the creation/register instruction.

* Reconcile Contract
Reconcile is optional and best-effort.

Codex Desktop can expose thread status in host tooling. If available, ASP may
use it to mark rows stale faster or annotate diagnostics. Reconcile must not:

- create a root session when no runtime session id is present
- delete rows only because a platform active list is unavailable
- override a fresh ASP heartbeat
- be required for hook routing to work

Reconcile can safely:

- mark rows =stale= when platform says the thread is not loaded and ASP
  heartbeat is already past idle TTL
- report that a root has no routable child
- report that multiple active children share the same name
- annotate diagnostic output with platform status

* GC Contract
GC is two-phase.

1. Soft transition:
   - active/idle rows past TTL become =stale=
   - stale rows past hard TTL become =expired=
   - expired rows with useful evidence become =archived=
2. Physical deletion:
   - only rows older than the retention policy are removed
   - rows with checkpoints or evidence refs are archived before deletion
   - =--dry-run= reports counts and affected ids before mutation

GC must never remove the current session row during the current command.

* Memory And Checkpoint Integration
Session lifecycle powers memory recall but does not replace memory ranking.

Recall should prefer:

1. current root session id
2. active/idle child sessions under that root
3. plans/checkpoints linked to those sessions
4. stale/expired rows only for recovery and audit
5. cross-session records only when explicitly requested

For a single root session with several plans, recall should rank by:

- session id match
- plan id / task id match
- recent checkpoint evidence
- unfinished checklist items
- next action
- evidence refs

* Failure Modes
| Failure | Required behavior |
|---------+-------------------|
| No runtime session id | Do not infer identity; allow only non-session commands or report unsupported session context. |
| Multiple conflicting runtime ids | Fail closed and ask for explicit session id or environment cleanup. |
| Root has no child | Deny main search/query and instruct child creation/register. |
| Child exists but expired | Deny main search/query and instruct recreation/heartbeat recovery. |
| Child exists and is routable | Deny main search/query and instruct resume; do not spawn a replacement. |
| Duplicate active child registration | Reject or require explicit replacement; do not silently create another routable child. |
| Child profile mismatch at register | Reject registration; return recreate-with-expected-profile instructions. |
| Child profile mismatch after register | Mark row invalid and deny routing until a correctly configured child registers. |
| Child role mismatch | Deny child search/query until role is corrected. |
| Platform active list missing | Continue using ASP heartbeat. |
| DB unavailable | Fail closed for main search/query routing; allow diagnostic commands. |

* Security And Privacy
The session registry stores routing and checkpoint metadata. It must not store:

- full shell output
- source file contents
- model reasoning text
- raw transcript dumps
- secrets from environment variables

Evidence refs should point to durable artifacts with their own access policy.

* RFC Closure Gates
- A root session without =asp-explore= denies main ASP search/query and emits
  the register instruction.
- A root session with active =asp-explore= denies main ASP search/query and
  emits the child id with resume instructions.
- Session-start with an active =asp-explore= child does not ask the main agent
  to spawn a new child.
- Registering an =asp-explore= child with the wrong platform profile is rejected
  before the row becomes routable.
- Registering a valid =asp-explore= under the same root internally replaces or
  invalidates older routable rows without requiring an agent-visible old child
  id.
- A stale/expired child is not routed to.
- An =asp-explore= child can run ASP search/query while active or idle.
- The =asp= binary gate rejects restricted search/query commands even when hook
  pre-tool did not trigger.
- =asp agent session list --active= never returns expired rows.
- =asp agent session gc --dry-run= reports affected rows without mutation.
- Post-tool receipts update heartbeat and compact evidence refs without storing
  raw source or output dumps.

* Open Questions
- Should =idle= child rows be routable by default, or should routing force an

- Should =asp-explore= have a shorter active TTL than other worker roles?
- Should reconcile adapters live in the hook runtime crate or in provider-owned

- Should archived rows remain in the same Turso database or move to an

- Should =post-tool= write one receipt per command or one compact rolling

** Codex Rollout Source Of Truth
*** Activity And Resident Role Facts
The rollout parser owns both topology and liveness facts. It SHOULD parse =session_meta=, =turn_context=, spawn =function_call_output= records, and structured event records into a root-scoped index keyed by session id. Each indexed session SHOULD expose an activity report with =status=, =lastEventAt=, =lastHeartbeatAt=, recent heartbeat records, terminal-event evidence, running-child evidence, and scanned-line count.

Activity status is evidence, not a wall-clock shortcut. A long-running child is not stale merely because an arbitrary timeout elapsed. ASP MUST compare the last parsed rollout event and terminal/running markers before classifying a session as =active=, =silent=, or =closed=. If no exact rollout file exists for a known session id, that is a Codex lifecycle/logging invariant failure, not permission to fall back to raw source access.

Resident-child identity MUST come from the configured resident child name and resident agent role. Defaults may describe the ASP explorer lane, but topology validation and register instructions MUST NOT treat =asp-explore= as a protocol constant.


Codex rollout JSONL under =CODEX_HOME/sessions= is the deterministic source of
truth for agent lifecycle topology. The local environment can identify the
current thread through =CODEX_THREAD_ID=, but it does not reliably expose the
parent thread, resident role, project id, or spawn depth. Those facts MUST be
recovered from rollout =session_meta=, =turn_context=, and parent
=function_call_output= spawn records.

Given a known session id, rollout lookup is a strong invariant:

1. Locate the rollout file by exact session id, for example with a file-index
   query equivalent to =rg --files --glob "**/rollout-*<session-id>.jsonl"
   "$CODEX_HOME/sessions"=.
2. If no file is found for that exact session id, treat the state as a Codex
   lifecycle/logging invariant failure. Do not silently create another resident
   child, do not fall back to recursive directory scans, and do not infer a date
   shard by decoding the session id in ASP.
3. Parse the first matching rollout =session_meta= to recover
   =rootSessionId=, =parentThreadId=, =threadSource=, =agentRole=,
   =agentNickname=, and spawn =depth=.
4. Parse the parent rollout spawn ledger to discover child ids from
   =function_call_output= records that contain =agent_id=. Root-scoped lifecycle
   indexing is therefore =root rollout -> child ids -> exact child rollout=,
   not global scan then filter.

For =asp-explore=, a rollout-derived record with =depth > 1= or
=parentThreadId= pointing at another =asp-explore= child is a nested resident
violation. Registry rows and =activitySnapshotShort= MUST be reconciled from
the rollout-derived topology instead of treating registry miss or stale
multi-agent state as permission to spawn a replacement.
