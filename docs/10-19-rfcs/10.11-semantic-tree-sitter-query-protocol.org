* One-sentence summary

=semantic-tree-sitter-query.v1= is the shared syntax query and capture ABI for
language providers: it uses tree-sitter-compatible query shape, captures, node
types, fields, predicates, and ranges as the portable grammar layer, while
compiler/native parser providers remain the authority for language syntax,
project semantics, cache provenance, and edit safety.

* Problem

The current search and query contracts already separate agent-facing compact
output from machine JSON packets, but syntax-level facts still appear in several
provider-shaped forms:

#+begin_example
nativeSyntaxFacts[].languageKind
query.projection.nodes[].kind
search.items[].kind
readPlan.symbols[].itemKind
prompt-facing nodes=...
provider-private fields
#+end_example

Those forms are useful, but they do not give agents and caches one stable syntax
coordinate system.  They also make cross-language query behavior harder to
compare: Rust, TypeScript, Python, and Julia can all prove richer native facts,
but the shared packet only sees the later semantic summary.

Tree-sitter's query model is the useful base to reuse here.  It already has a
compact S-expression pattern language, node types, field names, captures,
predicates, directives, named and anonymous nodes, =ERROR= and =MISSING= nodes,
and grammar-owned node type catalogs.  LLMs have strong prior training on this
shape, and engineers can inspect it without learning a provider-private node
encoding.

The goal is not to turn this repository into a tree-sitter wrapper.  The goal is
to use the mature tree-sitter query schema as the portable syntax ABI for a
stronger native-parser-backed language syntax search system.

* Design principles

1. /Tree-sitter-compatible shape is the portable syntax ABI./  The shared
   schema owns query source, patterns, captures, fields, predicates, directives,
   ranges, match ids, capture ids, grammar ids, and node type spelling.

2. /Native parser and compiler authority is the core product./  Rust =syn= or
   compiler-facing facts, the TypeScript Compiler API and =tsconfig= facts,
   Python =ast=, =tokenize=, and =symtable= facts, JuliaSyntax facts, and future
   provider-native parser stacks own language syntax authority.  They can prove
   facts that tree-sitter alone cannot: type-only imports, cfg gates, macro
   boundaries, include/module ownership, scope, project references, owner/test
   reachability, and edit safety.

3. /The schema is for the whole language syntax surface, not only owners./  It
   covers declarations, imports, exports, types, traits, interfaces, classes,
   macros, attributes, decorators, calls, bindings, assignments, patterns,
   match/case arms, control flow, literals, injected languages, tests, cfg or
   feature gates, and syntax diagnostics.

4. /ASP coordinates; language providers project./  The Rust =asp= client
   dispatches, validates, caches, renders, and records receipts.  It does not
   implement TypeScript, Python, Julia, or Rust language semantics.  Each
   provider emits =semantic-tree-sitter-query.v1= captures from its native parser
   or from an explicitly declared hybrid/runtime mode.

5. /There is one query surface./  Agent-facing =asp <language> query= is the
   syntax query surface.  There is no separate =ts-query=, =syntax-query=, or
   raw-query command family.  Typed selectors, code-shaped terms, and
   S-expression query source all normalize into the same tree-sitter-compatible
   query kernel and packet contract.

6. /Query catalogs are configurable protocol assets./  Tree-sitter-compatible
   query patterns live in versioned catalog files, not hard-coded prompt text.
   ASP owns catalog loading, normalization, validation, cache keys, and common
   selector-to-query routing.  Language providers own the thin grammar profile
   that says which catalog queries they support and how native parser facts map
   to those captures.

7. /Cache rows never invent facts./  ASP cache entries are derived from
   provider-owned packets or provider-owned exports.  Cached captures can speed
   repeated queries, but they cannot become a semantic fallback that outranks
   the provider.

8. /Selectors are source identity, not fuzzy search./  A selector that resolves
   to a real file or directory is compared by canonical source identity: file
   equality for files and containment for directories.  A selector that does not
   resolve on disk may only match an exact project-relative path.  Providers and
   cache replay must not use basename, suffix, or contains-style matching as the
   syntax query ABI, because that can silently redirect a query to the wrong
   owner.

9. /Normalized row replay is keyed by query AST/ABI./  SQLite syntax rows may be
   replayed only when the request query compiles to the same tree-sitter ABI
   fingerprint as the stored provider packet, the selector identity matches, and
   the row generation's file hashes are fresh.  Exact forwarded argv,
   artifact id, stdout text, and compact renderer version are receipt and
   provenance signals, not the semantic boundary for row replay.
   =semantic-tree-sitter-query= generation/artifact ids should be derived from
   the same AST/ABI identity plus grammar/profile, selector identity, and
   source file hashes.  Exact request fingerprints remain provenance only.
   When a client probes =query/tree-sitter= without an exact request
   fingerprint, an unrelated generation with the same provider/export method
   must not decide row freshness.  Artifact replay may still require the chosen
   generation to be fresh, but normalized row replay must check the matched row
   generation's own AST/ABI identity and file hashes.

* Current v1 boundary

The first vertical slice is a tree-sitter-compatible native projection.  It is
not a claim that every provider links a complete tree-sitter runtime or executes
the full tree-sitter query VM.  A typical current packet should be read as:

#+begin_example
sourceAuthority = native-parser-adapter
adapterMode = native-projection
compatibilityLevel = native-only | approximate | exact
#+end_example

This is intentional.  The public ABI reuses tree-sitter query shape because it
is the best portable syntax vocabulary, while the product authority remains in
provider-native parser and compiler facts.  Rust may prove a
=function_item/name= capture through =syn= facts; TypeScript may prove a
=function_declaration/name= capture through the TypeScript Compiler API;
Python may prove a =function_definition/name= capture through =ast= and
=tokenize=.  These are tree-sitter-compatible facts, not necessarily
tree-sitter-runtime facts.

The current implementation baseline includes:

- ASP-compiled plan facts for captures, node types, fields, and structured
  predicates;
- predicate operations =eq=, =any-eq=, =any-of=, =match=, =any-match=,
  =not-eq=, and =not-match= with explicit string or capture operands;
- compact non-=--code= locator/frontier output;
- pure-code stdout only after exact selector or unique-match selection;
- Rust native projection for item and selected call frontiers;
- cache and receipt provenance that references validated packets or rows rather
  than raw source inference.

Provider guides and docs must describe this as
=tree-sitter-compatible native projection=.  They must not imply complete
tree-sitter runtime compatibility unless the packet uses
=adapterMode=tree-sitter-runtime= or =adapterMode=hybrid= and the provider
records that proof.

* Agent command-selection calibration

The syntax ABI is not adopted until real agent command traces use it.  Artifact
history under each provider root is therefore an acceptance surface, not only a
debug cache.  A 2026-06-06 audit of
=languages/*/.cache/agent-semantic-protocol/artifacts= showed the current
failure mode:

- Rust recorded 90 provider commands: 49 =query/direct-source-read= commands,
  36 =search/owner= commands, 4 other query commands, 1 =search/prime= command,
  and only 4 =semantic-tree-sitter-query= packet artifacts.
- Python recorded 25 provider commands: 8 =query/direct-source-read= commands,
  15 =search/owner= commands, 2 =search/prime= commands, and no
  =semantic-tree-sitter-query= packet artifacts.
- TypeScript recorded owner/search commands but no syntax-query artifacts in
  the sampled cache.

This is a protocol guidance failure even when each individual direct read is
bounded and provider-owned.  Agents follow the cheapest executable command in
hook recovery and skill text; if those surfaces make
=query --from-hook direct-source-read --code= more prominent than syntax
locate, owner-local item query, read-plan frontier, or tree-sitter-compatible
S-expression capture, the public workflow collapses back into manual range
scanning.

Provider guides, hook recovery text, and the installed =SKILL.md= must therefore
present normal =query --selector <path-or-range> --workspace <workspace-root>
--code= as the last-mile source-preserved transport after a parser-owned
frontier has selected an exact locator.  =--from-hook direct-source-read=
remains a low-priority exact-window fallback, not the normal discovery or public
recovery route: it is acceptable only after =search pipe= or equivalent
parser-owned frontier output, with an exact bounded selector, and with a span
small enough for the current policy threshold.  Discovery should route through
=search prime=, =search fzf=, =search owner ... items=, normal =query
--names-only=, or =query --treesitter-query= locator/frontier output.  Wide or
unbounded source recovery should return read-plan or syntax frontier refs before
any inline code.  The graph-turbo history audit must keep
=query/direct-source-read=, =query/code=, and =query/tree-sitter= as distinct
action methods so future evidence can detect regressions directly instead of
folding them into plain =query=.

The initial exact-window threshold is history-derived, not aesthetic.  The local
artifact audit over 1,033 cached =direct-source-read= prompt outputs showed a
median window of 76 lines, p75 of 158 lines, p90 of 243 lines, 487 outputs over
80 lines, 251 over 160 lines, and 108 over 240 lines.  The first policy
therefore allows bounded fallback windows of 80 lines or fewer after parser-owned
frontier evidence, while treating wider reads as read-plan/frontier work.  This
threshold is hook policy and receipt feedback, not part of the syntax ABI.

* Next projection layers

The next layer is not a separate =ts-query= or =syntax-query= family.  It is a
stronger ASP-compiled query plan and native provider executor behind the same
=asp <language> query= surface.

** Pattern graph plan

The flat =captures/nodeTypes/fields/predicates= plan should evolve into a
compiled pattern graph owned by ASP:

#+begin_src json
{
  "patterns": [
    {
      "id": "p0",
      "root": { "nodeType": "function_item" },
      "children": [
        { "field": "name", "nodeType": "identifier", "capture": "function.name" },
        { "field": "parameters", "nodeType": "parameters", "capture": "function.parameters" },
        { "field": "return_type", "nodeType": "_", "capture": "function.return_type", "optional": true }
      ],
      "predicates": [
        {
          "op": "match",
          "capture": "function.name",
          "values": [{ "kind": "string", "value": "^render_" }]
        }
      ],
      "outputTarget": "enclosing-item"
    }
  ]
}
#+end_src

ASP still compiles catalog-id and S-expression inputs.  Providers execute the
typed pattern graph through native parser/compiler facts and should not parse a
second provider-private S-expression language.

** Multi-capture match model

The current row projection is sufficient for single-capture navigation such as
=@function.name=.  It is not enough for full tree-sitter predicate semantics
where a predicate compares captures within the same match:

#+begin_src scheme
(pair
  key: (property_identifier) @key.name
  value: (identifier) @value.name
  (#eq? @key.name @value.name))
#+end_src

Before providers claim support for capture-to-capture predicates, packets and
cache rows need an explicit multi-capture match model:

#+begin_example
SyntaxQueryMatch {
  captures: Vec<SyntaxCapture>,
  nativeRelations: Vec<NativeRelationRef>
}
#+end_example

The predicate executor then evaluates =@key.name= and =@value.name= from the
same match row instead of inferring related capture text from independent rows.

** Field structural selectors

Fields are not only labels for rendering.  In the stronger native projection
layer, fields become structural selectors that providers map onto native parser
shape:

#+begin_src scheme
(function_item
  name: (identifier) @function.name
  parameters: (parameters) @function.parameters
  return_type: (_) @function.return_type)
#+end_src

For Rust this maps to =syn::ItemFn.sig= fields.  For TypeScript it maps to
compiler AST declaration members.  For Python it maps to =ast.FunctionDef=
attributes plus token ownership where indentation or decorators matter.  If a
field cannot be structurally verified, the provider must return
=compatibilityLevel=approximate= or reject the pattern as unsupported instead
of pretending it has exact syntax proof.

* Terminology

** Tree-sitter-compatible

A packet is tree-sitter-compatible when its query and match model can be
understood in terms of tree-sitter query concepts:

#+begin_example
query source     S-expression pattern source, when present
node type        grammar node type such as function_item or call_expression
field            child field such as name, body, function, argument
capture          named capture such as function.name or call.target
predicate        typed capture filter such as eq, any-eq, any-of, match, any-match, not-eq, not-match
directive        capture metadata such as set or strip
range            project-relative source range
ERROR/MISSING    syntax recovery nodes when the provider can expose them
#+end_example

The provider may produce this shape without linking tree-sitter.  For example,
TypeScript can use the Compiler API and still emit a
=tree-sitter-typescript=-compatible =function_declaration/name= capture.

** Source authority

=sourceAuthority= records what proved the match:

| Value | Meaning |
|-------+---------|
| =native-parser= | The language provider's native parser/compiler stack proved the capture. |
| =native-parser-adapter= | Native facts were projected through a tree-sitter-compatible mapping table. |
| =tree-sitter-runtime= | A real tree-sitter parser/query runtime proved the capture. |
| =codeql= | A CodeQL database/query proved the semantic relation that was projected into this frontier packet. |
| =hybrid= | Tree-sitter-compatible candidates were verified or enriched by the native provider. |
| =cached-provider-export= | ASP replayed a previously validated provider-owned export. |

Only provider-owned native modes may claim compiler/project semantics.  A
=tree-sitter-runtime= capture can be useful syntax evidence, but it does not
prove type, owner, dependency, test, policy, or patch-safety facts by itself.

** Execution backend

=executionBackend= records which engine executed or replayed the compiled
query plan.  It is deliberately separate from =sourceAuthority=: a packet can
be rendered through the same tree-sitter-compatible frontier ABI while the
backend is native parser, tree-sitter runtime, CodeQL, a hybrid path, or cached
replay.

| Backend | Purpose |
|---------+---------|
| =native-parser= | Default provider path.  Native parser/compiler facts execute the ASP-compiled plan. |
| =tree-sitter-runtime= | A real tree-sitter query runtime executes the pattern. |
| =codeql= | Optional semantic backend for local flow/path frontiers. |
| =hybrid= | More than one engine contributes candidates or verification. |
| =cached-replay= | ASP replays a validated provider-owned packet or normalized rows. |

CodeQL is a backend, not a new prompt-facing command family.  CodeQL is an
optional semantic backend.  It must emit the same frontier-first packet shape
as other engines, should start with =flow-lite= local source/sink/path
frontier, and must not require every project or provider to maintain a CodeQL
database before basic syntax query continues to work.

Compact contract: =flow-lite= local source/sink/path frontier.

Provider registry descriptors may advertise supported engines through
=executionBackends=.  Current Rust query descriptors should declare
=native-parser= and must not advertise =codeql= until an actual CodeQL-backed
executor exists.

** Adapter mode

=adapterMode= records how the provider produced the packet:

| Mode | Purpose |
|------+---------|
| =native-projection= | Primary path.  Native parser facts are projected into the shared syntax ABI. |
| =hybrid= | Tree-sitter-compatible query narrows candidates; native parser verifies and enriches them. |
| =tree-sitter-runtime= | Bootstrap/debug path.  A provider directly runs tree-sitter query. |
| =codeql-query= | CodeQL query results are projected into the shared frontier ABI. |
| =cached-replay= | ASP renders from validated cache rows or artifacts without starting the provider. |

** Compatibility level

=compatibilityLevel= records the strength of the mapping:

| Level | Meaning |
|-------+---------|
| =exact= | Native fact maps exactly to a tree-sitter node, field, capture, and range. |
| =approximate= | The shared capture is close enough for syntax navigation, but provider fields carry the exact native shape. |
| =native-only= | The provider has a valid language fact with no useful tree-sitter-compatible capture. |
| =tree-only= | The capture is syntax-only and has not been verified by the native parser. |

Agents must not treat =approximate= or =tree-only= as edit-safety evidence.

* Packet contract

The new shared schema is:

#+begin_example
schemas/semantic-tree-sitter-query.v1.schema.json
schemaId = agent.semantic-protocols.semantic-tree-sitter-query
protocolId = agent.semantic-protocols.semantic-language
#+end_example

The core packet shape is:

#+begin_src json
{
  "schemaId": "agent.semantic-protocols.semantic-tree-sitter-query",
  "schemaVersion": "1",
  "protocolId": "agent.semantic-protocols.semantic-language",
  "protocolVersion": "1",
  "languageId": "typescript",
  "providerId": "ts-harness",
  "projectRoot": ".",
  "grammarId": "tree-sitter-typescript",
  "grammarProfileVersion": "typescript-compiler-api-2026-06",
  "executionBackend": "native-parser",
  "sourceAuthority": "native-parser",
  "adapterMode": "native-projection",
  "compatibilityLevel": "exact",
  "query": {
    "input": "collectToolActions",
    "dialect": "tree-sitter-query",
    "compiledSource": "(function_declaration name: (identifier) @function.name)"
  },
  "matches": [
    {
      "id": "m1",
      "patternIndex": 0,
      "captures": [
        {
          "id": "c1",
          "name": "function.name",
          "nodeType": "identifier",
          "field": "name",
          "named": true,
          "range": {
            "path": "src/tool-action.ts",
            "lineRange": "24:80"
          },
          "nativeFactRefs": [
            "typescript:src/tool-action.ts:24:function:collectToolActions"
          ]
        }
      ]
    }
  ],
  "nativeSyntaxFacts": [],
  "semanticHandles": [],
  "truncated": false
}
#+end_src

The packet should not store raw source by default.  Capture text may be omitted
or summarized unless an explicit machine consumer requests bounded exact text
and the provider policy allows it.  Source windows remain =semantic-read-packet=
or provider =query --code= output, not the default syntax query packet.

* Query input model

The agent-facing path is the normal query surface and the default
tree-sitter-compatible query kernel.  Agents usually provide typed or
code-shaped selectors:

#+begin_src shell
asp typescript query collectToolActions .
asp rust query 'fn collect_tool_actions' .
asp python query 'def render_packet' .
#+end_src

ASP resolves the language facade and provider registry, checks cache, and on
miss calls the provider-owned command.  The input is normalized through the
query catalog into a tree-sitter-compatible query model before the provider
returns =semantic-tree-sitter-query.v1= captures.

S-expression source is also part of the same =query= surface.  It is not a
separate command family:

#+begin_src shell
asp typescript query \
  '(function_declaration name: (identifier) @function.name)' .
#+end_src

Provider guides and registry descriptors may advertise which input forms are
accepted: =selector=, =code-shaped=, =catalog-id=, and =s-expression=.  All forms
route through the same query kernel, schema, cache, and receipt path.  Providers
may reject a particular S-expression when the grammar profile cannot map it or
the native parser cannot verify it, but that rejection is a normal query miss or
unsupported-pattern response, not a fallback to another protocol.

* Query catalog files

The shared syntax ABI should be configurable through tree-sitter-style query
catalogs.  Catalog files use the familiar =.scm= extension used by tree-sitter
query ecosystems.  =.scm= is the only repository and registry catalog filename
extension for this ABI.  Scheme-like S-expression query text is an accepted
input form when a provider advertises =queryInputForms=s-expression=, but
=.scheme= catalog filenames are not a compatibility surface.

Recommended layout:

#+begin_example
languages/typescript-lang-project-harness/
  tree-sitter/
    tree-sitter-typescript/
      grammar-profile.json
      corpus-profile.json
      queries/
        declarations.scm
        imports.scm
        calls.scm
        control-flow.scm
        tests.scm

languages/rust-lang-project-harness/
  tree-sitter/
    tree-sitter-rust/
      grammar-profile.json
      corpus-profile.json
      queries/
        calls.scm
        cfg.scm
        declarations.scm
        imports.scm
        injections.scm
        macros.scm
        tags.scm
      query-corpus/
        calls.txt
        declarations.txt
        imports-macros-cfg.txt

languages/python-lang-project-harness/
  tree-sitter/
    tree-sitter-python/
      grammar-profile.json
      corpus-profile.json
      queries/
        declarations.scm
        imports.scm
        calls.scm
        decorators.scm
        control-flow.scm
#+end_example

The catalog is not another semantic engine.  It is a configurable query layer
that lets providers and ASP agree on capture names and grammar shapes:

#+begin_src scheme
;; declarations.scm
(function_declaration
  name: (identifier) @function.name)

(class_declaration
  name: (type_identifier) @class.name)
#+end_src

ASP responsibilities:

- treat registry catalog paths as canonical catalog handles, not as downstream
  filesystem requirements
- normalize query ids and capture names
- validate catalog metadata, source delivery declarations, packet references,
  and cache fingerprints
- include catalog id, grammar id, grammar profile version, catalog fingerprint,
  and grammar profile fingerprint in cache keys
- route typed selectors such as =query collectToolActions= to catalog query ids
- embed registry-declared canonical catalog sources into the ASP binary for the
  public =asp <language> query --catalog ...= path; downstream clients must not
  depend on provider package source files being installed
- compile catalog-id and S-expression inputs into the shared query model in the
  ASP wrap layer and forward only internal =--asp-syntax-query-*= plan facts to
  providers
- execute tree-sitter runtime matching only when the provider declares
  =adapterMode=tree-sitter-runtime= or =hybrid=; native-projection remains
  provider-owned

Provider responsibilities:

- maintain the language-local =tree-sitter/<grammar-id>/queries/*.scm= catalog
  as the canonical source in the provider repository, matching the upstream
  tree-sitter query layout where one exists
- embed or otherwise publish those catalogs through the provider binary/export
  path for direct provider debugging and release provenance; public ASP wrapper
  execution uses the ASP-binary embedded catalog copy for ABI planning
- maintain =grammar-profile.json= mapping native parser facts to capture names
  and deliver that profile through the same provider-owned channel
- track upstream grammar test corpus knowledge through lightweight development
  assets such as =corpus-profile.json=; corpus files guide catalog coverage and
  drift detection, but are not required runtime inputs
- maintain provider-local =query-corpus/*.txt= fixtures for ASP capture
  precision. Providers store these fixtures beside their catalogs, but the
  validation, query compilation, cache keys, and replay semantics remain owned
  by the main ASP workspace. These fixtures should cite upstream corpus files
  for grammar shape provenance while only testing ASP-specific capture
  granularity and avoiding duplicate upstream parser grammar tests
- record the ASP workspace git revision that validated the grammar profile and
  query corpus in =grammar-profile.json= only as provenance. It is not a
  current-HEAD equality gate because the main ASP repository naturally advances
  after a language profile is written
- gate grammar-profile compatibility with =aspWorkspace.contractFingerprint=,
  computed from the ASP tree-sitter query ABI/schema/validator contract files.
  Ordinary main-repo commits do not require profile churn when this fingerprint
  is unchanged; ABI/schema/compiler/validator changes do
- validate =grammar-profile.json= against the main workspace
  =semantic-tree-sitter-grammar-profile.v1= schema so other language providers
  can adapt the same profile/query-corpus contract without inventing local
  shapes
- do not import editor-oriented query assets such as =highlights.scm= by default;
  vendored upstream queries should have an explicit syntax ABI calibration role
  such as definition/reference tags or embedded-language injection boundaries
- declare which catalog queries are supported by native projection
- return =compatibilityLevel=exact=, =approximate=, =native-only=, or =tree-only=
  honestly
- keep language-only semantic facts in provider-owned fields or native fact refs
- do not link grammar-specific tree-sitter runtime crates for this ABI; the
  provider catalog is =.scm= plus native projection, and ASP owns any shared
  tree-sitter runtime/query loading path

This keeps language maintenance lightweight: adding a stronger native parser
capability usually means adding or refining a small catalog query plus a grammar
profile mapping, while ASP continues to own shared schema validation, cache,
artifact replay, and compact rendering.

* Provider adapter contract

Each language provider owns a grammar profile that maps native syntax facts to
tree-sitter-compatible captures:

#+begin_example
native kind -> nodeType / field / capture / relation / compatibilityLevel
#+end_example

The profile's =nativeFactProjection= table is the convergence contract.  Native
parser/compiler facts remain the authority, but any fact that can be represented
as a stable syntax anchor should be projected through a catalog capture in
=tree-sitter/<grammar-id>/queries/*.scm=.  Provider-private native fields are
reserved for facts that are genuinely =native-only=.

Examples:

| Language | Native authority | Grammar profile examples |
|----------+------------------+--------------------------|
| Rust | =syn=, compiler-facing facts, Cargo metadata | =function_item/name=, =impl_item=, =trait_item=, =use_declaration=, =macro_invocation=, =attribute_item=, =match_expression= |
| TypeScript | TypeScript Compiler API, =tsconfig=, project references | =function_declaration/name=, =class_declaration/name=, =interface_declaration/name=, =type_alias_declaration/name=, =import_statement=, =call_expression/function= |
| Python | =ast=, =tokenize=, =symtable= | =function_definition/name=, =class_definition/name=, =decorated_definition=, =import_statement=, =call/function=, =match_statement=, =with_statement= |
| Julia | JuliaSyntax, package/module/include environment | =function_definition/name=, =macrocall=, =module=, =using_statement=, =import_statement=, =struct_definition=, =call_expression/function= |

Providers can expose facts that have no tree-sitter equivalent by using
=compatibilityLevel=native-only= and provider-owned fields.  They must not force
project, type, policy, or edit-safety facts into a fake syntax capture.

* Relationship to existing packets

=semantic-tree-sitter-query.v1= is additive.  It does not immediately replace
=semantic-search-packet.v1=, =semantic-query-packet.v1=, or
=semantic-read-packet.v1=.

Existing packets should gradually reference syntax captures instead of
duplicating provider-private syntax fields:

#+begin_example
syntaxQueryRef       packet or artifact id for semantic-tree-sitter-query.v1
syntaxMatchRefs      match ids selected by this packet
syntaxCaptureRefs    capture ids selected by this packet
syntaxAnchor         small inline nodeType/field/capture/range summary
#+end_example

Initial consumers:

| Packet | First integration |
|--------+-------------------|
| =semantic-search-packet.v1= | Add refs for search items, hits, and graph nodes that come from syntax captures. |
| =semantic-query-packet.v1= | Add refs from matches and projection nodes to capture ids. |
| =semantic-read-packet.v1= | Add refs from read-plan symbols, windows, and code frontier entries. |
| =semantic-native-syntax-fact-index.v1= | Add optional refs from facts to syntax captures and grammar profile version. |

Prompt-facing search output should use a compact field named =syn= when the capture
helps the agent choose a next action:

#+begin_example
I=item:symbol(collect_tool_actions)@216:421!outline syn=function_item/name
H=custom:range(command_source_paths@397:401)!code syn=call_expression/function
#+end_example

Do not use a field named =ts= for this purpose; in this repository =ts= already
commonly means TypeScript.

* ASP cache model

This RFC makes syntax captures a first-class cache key family.  The cache goal
is not just artifact replay; it is to reduce repeated native provider startup
and repeated project parsing for =asp <language> query/search=.

The cache execution shape is:

#+begin_example
asp <language> query/search
  -> resolve project root, provider, grammar profile, and request fingerprint
  -> probe local SQLite generation
  -> on exact prompt-output hit:
       replay prompt-output artifact
  -> on packet artifact hit:
       load a previously validated semantic-tree-sitter-query packet
       render compact output or linked search/query/read packet
  -> on normalized-row hit:
       query syntax captures and native fact refs from SQLite
       render through ASP shared renderer
  -> on miss or stale generation:
       call native provider CLI or provider export
       validate packets
       write artifacts and normalized rows
       render compact output
#+end_example

For syntax query replay, the cache boundary is the query AST/ABI fingerprint,
plus grammar/profile/catalog provenance and provider generation.  Stdout is not
the syntax fact and must not be the normal invalidation boundary.  Cache hits
render the current agent-facing locator/frontier surface from a validated
=semantic-tree-sitter-query.v1= artifact or normalized rows.  Stored stdout is
allowed only for explicit prompt-output artifacts or another documented special
case whose provenance points back to the structured packet or row generation.
=semantic-tree-sitter-query= generation/artifact ids are also AST/ABI-derived:
they should not include exact argv strings, stdout text, or volatile packet
bytes except through validated source file hashes.  Exact request fingerprints
can be stored for receipts and provenance, but they are not the row or artifact
identity.
Because =query/tree-sitter= generation probes intentionally avoid exact request
fingerprint lookup, a stale unrelated generation must not block a fresh row hit.
Clients should keep artifact replay behind the artifact generation freshness
gate, then let normalized row replay perform its own AST/ABI, selector, and row
file-hash check.
Freshness uses non-empty file hashes supplied by the validated packet when
available.  If a syntax packet omits =/cache/fileHashes=, the client may derive
path+sha256 freshness rows from validated locator fields such as =selector=,
=read=, =path=, and capture/match ranges; it must not derive semantic facts from
raw source text or rendered stdout.
Renderer or row-projection changes use an explicit syntax row ABI flush, not a
SQLite schema-version bump.  The client records the current syntax row ABI in
cache metadata, for example =syntaxQueryRowAbiVersion=, and when it changes the
client deletes only the =syntax_query_*= row family.  Provider artifacts, normal
cache generations, and unrelated search/query rows remain intact and can refresh
syntax rows from validated packets or a native provider miss.
The explicit maintenance command =asp cache flush syntax-rows= performs the same
scoped row-family reset without deleting manifest generations or artifacts.
Rows are replayable only when they carry the ASP query AST/ABI fingerprint and
the AST/ABI-derived capture and item node types.  The row path does not guess or
fallback from legacy rendered text or incomplete packet fields.

** Agent-facing render contract

The syntax query stdout contract is intentionally smaller than the JSON packet.
Non-=--code= output is a locator/frontier surface for choosing the next exact
query, read, or code extraction; it must not inline source windows or cache
debugging state.  =--code= output is the only default prompt-facing syntax query
mode that writes source text, and it must write only the selected code.

Providers may choose one of two render profiles while sharing the same
=semantic-tree-sitter-query.v1= packet and ASP-compiled query plan:

| Render profile | Current providers | Required facts |
|----------------+-------------------+----------------|
| =compact-graph-frontier= | Rust reference | Header, query alias, capture aliases, enclosing item aliases, =rank=, =frontier=, =omit=, =avoid=, and =ts=node/field= atoms. |
| =corpus-locator= | TypeScript, Python | Repeated =path:start[:end]= locator blocks plus capture label/text, separated by blank lines for multiple paths. |

Both profiles are valid only when they are rendered from provider-native
projection over an ASP-compiled plan.  They are not separate query protocols and
must not create =ts-query=, =syntax-query=, or raw-query command families.
Both profiles must suppress =artifactId=, SQLite path hints, cache roots,
receipt JSON, raw source windows, full node lists, and unbounded capture text in
default non-JSON output.

The code contract is stricter than the locate contract:

#+begin_example
asp <language> query --treesitter-query <query> --selector <exact-locator> --code
  -> stdout contains only provider-selected source code

asp <language> query --treesitter-query <query> --selector <range-or-path>
  -> stdout contains locator/frontier facts only
#+end_example

When a non-=--code= query matches many captures, the renderer should cap prompt
output and state the cap with =matches=, =shown=, =omitted=, and =omit= rather
than printing hidden source or JSON.  When =--code= would select multiple code
windows, providers should reject the request or require a more exact selector or
predicate.

=python -m tools tree-sitter validate frontier-code-contract= is the
package-local rollout gate for syntax locate/code output and Rust, TypeScript,
and Python =native-projection= packets: each JSON packet must retain
=nativeFactRefs=, =rawSourceStored=false=, tree-sitter-compatible capture
=nodeType=/=field=, and native parser enrichment such as =nativeNodeType= while
default stdout stays compact.  =python -m tools tree-sitter validate search-read-plan-frontier-contract= is the paired gate for hook recovery and
search/read-plan output: Rust, TypeScript, and Python must return graph
locator/frontier evidence for search-style recovery without inline source or
cache debugging state, and Rust's reference read-plan must remain
=mode=range-frontier= with =omit=code=.

Recommended SQLite families:

#+begin_example
syntax_query_generation
syntax_query_pattern
syntax_match
syntax_capture
syntax_capture_range
syntax_capture_native_fact_ref
native_syntax_fact
owner_index
query_key_index
artifact_ref
#+end_example

Cache keys must include:

#+begin_example
normalized project root
language id
provider id and provider version
schema version
grammar id
grammar profile version
adapter mode
source authority
normalized query input or compiled query source
ASP-compiled query AST/ABI fingerprint for inline query source
catalog fingerprint when the request uses a registry-declared catalog id
grammar profile fingerprint when the provider registry declares grammar-profile.json
workspace config digests such as tsconfig, Cargo.toml, pyproject.toml, Project.toml
file hash or provider generation
#+end_example

The default cache must not store raw source files.  It may store project-relative
ranges, capture names, native fact refs, structural fingerprints, and artifact
ids.  It must prefer structured packets or rows over rendered stdout so renderer
changes cannot replay stale agent-facing text against fresh query semantics.

* Compact graph and search behavior

Syntax captures feed compact graph decisions, but they do not replace graph
semantics.  RFC 006 still owns prompt-facing graph grammar, rank, frontier,
relations, and avoidance facts.  A graph relation such as =calls=, =covers=,
=imports=, or =contains= must come from provider-owned parser/project facts.

The syntax ABI helps compact output explain why a node is a good next action:

#+begin_example
[search-owner] owner=src/tool_action.rs selector=items terms=3 view=seeds alg=syntax-frontier
legend: aliases ID:kind; node ID=kind:role(value)!next; edge SRC>{DST:rel}; frontier ID.next
aliases=G:search,O:owner,I:item,H:custom
O=owner:path(src/tool_action.rs)!owner
I=item:symbol(collect_tool_actions)@216:421!outline syn=function_item/name
H=custom:range(command_source_paths@397:401)!code syn=call_expression/function
G>{O:selects,I:matches,H:matches}
O>{I:contains,H:contains}
rank=H,I,O frontier=H.code,I.outline
omit=code,full-node-list
avoid=raw-read,repeat-owner
#+end_example

The =syn= atoms are explanatory syntax anchors.  The executable next action
still comes from =!code=, =!outline=, and =frontier=.

* Read-plan behavior

Wide reads should use syntax captures to produce better frontier entries, but
the packet remains =semantic-read-packet.v1=.  Providers should prefer
parser-owned symbols or syntax windows before mechanical range splits.

Example behavior:

#+begin_example
readPlan.mode=range-frontier
readPlan.algorithm=symbol-frontier
readPlan.syn=function_item/name
frontier[0].kind=symbol
frontier[0].action=code
frontier[0].read=src/tool_action.rs:397:401
frontier[0].fields.syntaxCaptureRef=c37
#+end_example

Providers must not synthesize fake symbols such as =local-window=.  If no native
symbol exists, use a window frontier with a syntax capture or a mechanical range
split marked as such.

* Acceptance matrix

This matrix is the release boundary for RFC 011.  A provider or guide may only
advertise the =v1 accepted= behaviors by default.  The =v2/deferred= behaviors
require a schema, registry descriptor, provider implementation, corpus/fixture
coverage, and sandtable or real-project evidence before they can appear in
default guide output.

| Capability | V1 accepted | V2/deferred |
|------------+-------------+-------------|
| Public command surface | =asp <language> query= with =--treesitter-query=, catalog id, selector, and =--code= modes | New public families such as =ts-query=, =syntax-query=, or raw provider query surfaces |
| Runtime dependency | Native provider projection without grammar-specific tree-sitter runtime crates | Complete tree-sitter runtime compatibility claims for every provider |
| Query plan | ASP-compiled flat plan facts: captures, node types, fields, predicates, selector, catalog/profile fingerprints | ASP-compiled pattern graph with nested children, optional children, anchors, alternation, quantifiers, and directives |
| Match row | Single-capture rows that can locate an enclosing item or syntax frontier | Multi-capture match rows with capture-to-capture predicates and native relation groups |
| Predicate ABI | Structured =eq=, =any-eq=, =any-of=, =match=, =any-match=, =not-eq=, and =not-match= operands | Full tree-sitter predicate/directive compatibility, including cross-capture text semantics beyond validated multi-capture rows |
| Field semantics | Field names as capture metadata and native projection hints | Fields as required structural selectors over native parser AST/compiler shapes |
| Output | Non-=--code= locator/frontier only; =--code= pure source after exact selector or unique match | Broad multi-match code dumps, inline source in search/read-plan, cache/debug ids in default output |
| Provenance | =sourceAuthority=, =adapterMode=, =compatibilityLevel=, grammar/profile/catalog/cache refs | Treating syntax-only captures as type, owner, dependency, policy, or edit-safety proof |
| Cache | Replay validated artifacts or normalized rows keyed by AST/ABI, selector identity, grammar/profile/catalog, and file freshness | Deriving semantic facts from rendered stdout, raw source inference, basename matching, or stale unrelated generations |
| Corpus | Provider-local query corpus for ASP capture precision plus upstream corpus citations | Duplicating upstream parser grammar tests or requiring upstream corpus files at runtime |

** Promotion rule

A deferred behavior moves into v1 only after the contract stack changes in this
order:

#+begin_example
RFC -> shared schema -> registry descriptor -> provider projection ->
provider tests -> sandtable/real evidence -> guide output
#+end_example

Until that stack lands, docs and guides must say =unsupported-pattern=,
=approximate=, or =native-only= rather than teaching agents to rely on a
stronger tree-sitter behavior.

** Explicit non-goals for v1

- Full tree-sitter runtime parity across all providers.
- Grammar-specific runtime crates in language harnesses solely for this ABI.
- Edit-safety decisions from tree-sitter captures alone.
- Type, dependency, cfg, macro expansion, or test reachability facts inferred
  from syntax-only nodes.
- Source-dump search output as a convenience for syntax queries.
- Cross-provider query behavior that depends on provider package source files
  being installed downstream.

* Language rollout plan

** Phase 0: RFC and schema

1. Land this RFC.
2. Add =schemas/semantic-tree-sitter-query.v1.schema.json=.
3. Add schema tests for valid native-projection, hybrid, tree-sitter-runtime,
   cached-replay, exact, approximate, native-only, and tree-only examples.
4. Update =schemas/README.md= with the packet role and its relationship to
   search/query/read/cache.

** Phase 1: Rust reference adapter

1. Add Rust =tree-sitter/tree-sitter-rust/queries/*.scm= query catalogs,
   upstream standard query snapshots, and a grammar profile that maps existing
   provider-native item and call facts to tree-sitter-compatible captures.
2. Add provider JSON export for =query= and selected =search owner items= paths.
3. Add compact output =syn= atoms only where they reduce ambiguity.
4. Validate =asp rust query= cold-native and cache-write paths.
5. Run =python -m tools tree-sitter validate runtime-boundary= and then
   =python -m tools tree-sitter validate frontier-code-contract= after Rust,
   TypeScript, and Python are available.  These package-local gates own the
   shared boundary and agent-facing render contract: language providers must
   not depend on tree-sitter runtime packages; Rust may render
   =compact-graph-frontier= and TypeScript/Python may render =corpus-locator=,
   but all three must prove ASP-compiled query input, non-=--code=
   locator/frontier output without cache noise, and pure-code =--code= output.
   The composite =python -m tools tree-sitter validate contracts= command also
   runs =python -m tools tree-sitter validate json-abi-corpus= so the JSON
   packet ABI proves ASP-compiled catalog plans and honest native-projection/no-runtime behavior alongside stdout corpus locators.

Rust is the reference because it is already the strongest local provider and can
exercise native parser facts without requiring another runtime boundary.

** Phase 2: ASP cache integration

1. Add catalog loading and fingerprinting to ASP so query catalog id, grammar
   id, and grammar profile version become part of cache provenance.
2. Add artifact family =semantic-tree-sitter-query/<id>.json= or a compatible
   query artifact namespace, and replay exact non-=--code= syntax locator
   output from validated packet artifacts.
3. Normalize syntax query rows into SQLite from validated
   =semantic-tree-sitter-query.v1= packets; rows store project-relative ranges,
   capture names/text, native fact refs, and artifact refs, never raw source.
4. Add exact artifact replay for identical request fingerprints.
5. Add normalized-row replay for repeated syntax query requests when the packet
   artifact is unavailable or cannot be rendered.
6. Add receipts for cold native, warm provider, local cache hit, stale, and
   invalidated paths.

This phase proves that =asp <language> query/search= can avoid repeated provider
CLI startup when validated captures and native fact refs are fresh.

** Phase 3: TypeScript provider

1. Add TypeScript =tree-sitter/tree-sitter-typescript/queries/*.scm= catalogs.
2. Implement =ts-harness= native projection through the TypeScript Compiler API.
3. Project TypeScript declarations, imports, exports, calls, classes,
   interfaces, type aliases, and type-only import/export facts into captures.
4. Use =tsconfig= and project references in native fields and cache digests.
5. Route =asp typescript query "xx"= through ASP cache first, then
   =ts-harness= on miss.
6. Prove repeated query speedup on a real TypeScript workspace.

** Phase 4: Python provider

1. Add Python =tree-sitter/tree-sitter-python/queries/*.scm= catalogs.
2. Implement =py-harness= native projection through =ast=, =tokenize=, and
   =symtable=.
3. Preserve indentation-sensitive ownership through native fields and avoid
   claiming edit safety from syntax captures alone.
4. Add read-plan syntax frontier coverage for wide direct-source-read recovery.
5. Prove repeated query speedup and fewer manual range scans.

** Phase 5: Julia provider

1. Align Julia on the =asp julia= facade backed by compiled
   =asp-julia-harness= and cacheable runtime profiles.
2. Add Julia =tree-sitter/tree-sitter-julia/queries/*.scm= compatible catalogs
   where the grammar profile has stable names.
3. Export JuliaSyntax-owned captures and native fact refs for cache import.
4. Make ASP serve repeated query/search from Rust-side cache without starting
   Julia when the generation is fresh.
5. Validate module/include/macro boundaries as native facts, not tree-sitter-only
   guesses.

** Phase 6: Cross-language convergence

1. Add package-local schema copies where providers require them.
2. Add registry descriptor capabilities for =semantic-tree-sitter-query.v1=
   export, query input forms, grammar profile id, and cache replay support.
3. Carry ASP-compiled predicate ABI as structured operands:
   ={op,capture,values:[{kind:string|capture,value}]}=.  Providers execute this
   typed predicate model directly; they should not accept a second string
   encoding such as =op:capture=value=.  The =op= label is part of the
   query AST/ABI fingerprint; ASP must preserve =any-eq= and =any-match=
   rather than folding them into =eq= and =match=.
4. Add sandtable scenarios for cold native, cache hit, provider miss, stale
   cache, and real-trigger repeated syntax queries.
5. Feed real-project metrics back into RFC/schema refinement before widening
   renderer behavior.

* Registry descriptors

Provider registry descriptors should advertise:

#+begin_src json
{
  "method": "query",
  "packetSchemas": [
    "semantic-tree-sitter-query.v1"
  ],
  "capabilities": {
    "adapterModes": ["native-projection"],
    "sourceAuthorities": ["native-parser"],
    "grammarId": "tree-sitter-typescript",
    "grammarProfileVersion": "typescript-compiler-api-2026-06",
    "queryCatalogs": [
      {
        "id": "typescript.declarations",
        "path": "tree-sitter/tree-sitter-typescript/queries/declarations.scm",
        "sourceDelivery": "provider-binary-embedded",
        "captures": ["function.name", "class.name", "interface.name"]
      }
    ],
    "queryInputForms": ["selector", "code-shaped", "catalog-id", "s-expression"],
    "cacheReplay": true
  }
}
#+end_src

The descriptor must describe which input forms the same =query= surface accepts.
A provider can support selector and code-shaped input before it accepts arbitrary
S-expression source, but it must not expose that as a separate agent-facing
command.

** Registry conformance contract

Registry descriptor is the only provider syntax capability advertisement source.
Guide text and README examples are secondary views and must not claim more than
the descriptor claims.

Required descriptor fields for RFC 011 syntax query support:

| Field | Required meaning |
|-------+------------------|
| =method= | Must be =query=.  Syntax query is not a new command family. |
| =packetSchemas= | Must include =semantic-tree-sitter-query.v1= before the provider emits that packet. |
| =queryInputForms= | Names accepted input forms: =selector=, =code-shaped=, =catalog-id=, =s-expression=. |
| =queryCatalogs= | Lists registry-owned catalog ids, capture names, canonical paths, source delivery, and fingerprints. |
| =grammarId= | Names the tree-sitter grammar profile vocabulary such as =tree-sitter-rust=. |
| =grammarProfileVersion= | Names the provider grammar profile version used in packet/cache provenance. |
| =adapterModes= | Names supported projection modes such as =native-projection= or =cached-replay=. |
| =sourceAuthorities= | Names what can prove a packet, such as =native-parser-adapter= or =cached-provider-export=. |
| =executionBackends= | Names execution engines such as =native-parser=, =cached-replay=, or =codeql=. |
| =renderProfiles= | Names prompt render contracts such as =compact-graph-frontier=, =corpus-locator=, or =flow-lite-frontier=. |
| =cacheReplay= | States whether ASP may replay validated packets or normalized rows without starting the provider. |
| =unsupportedPatternBehavior= | Must be =diagnostic= or =empty-frontier=; it must not silently fall back to raw search. |

Descriptor consistency rules:

1. =queryInputForms=s-expression= requires a deterministic ASP-compiled plan
   path and an unsupported-pattern response for grammar shapes outside the
   profile.
2. =adapterModes=tree-sitter-runtime= requires =executionBackends= to include
   =tree-sitter-runtime= and packet =sourceAuthority= to record runtime proof.
3. =adapterModes=codeql-query= requires =executionBackends=codeql= and the RFC
   012 promotion gates before guide output advertises the lane.
4. =cacheReplay=true= requires cache keys to include the query AST/ABI,
   selector identity, grammar/profile/catalog fingerprint, execution backend,
   and file freshness.
5. =renderProfiles=compact-graph-frontier= may be Rust-only at first; other
   providers may use =corpus-locator= while sharing the same packet/schema
   contract.
6. Guide output must derive syntax route availability from the descriptor:
   if =s-expression= is absent, =query guide treesitter .= may show catalog
   templates and explain that arbitrary S-expression input is not available.
7. Registry descriptor examples must not include provider package source paths
   as downstream runtime requirements.
   Catalog delivery: provider/ASP binary embedding or provider exports.

* Validation

Schema-level validation:

- packet shape, ids, refs, and range syntax
- enum coverage for source authority, adapter mode, and compatibility level
- no raw source text in default packets
- valid native fact refs when provided
- cache key and generation fields present for cacheable packets

Provider validation:

- Rust reference adapter fixture
- TypeScript Compiler API projection fixture
- Python indentation/decorator fixture
- Julia module/include/macro fixture
- package-local schema copy equality where the provider keeps schema copies

ASP cache validation:

- cold native provider call writes packet artifact and SQLite rows
- identical request replays artifact without provider execution
- repeated compatible request uses normalized rows without provider execution
- stale generation invokes provider and refreshes rows
- invalidated cache reports visible miss or warm-provider path

Real-project evidence:

- command count
- provider process count
- packet size
- cache hit rate
- cold and warm latency
- repeated-trigger reduction
- read-plan repeated range scan reduction
- confusing next actions after compact render

* RFC closure gates

RFC 011 is closed for the native-projection v1 lane only when these gates pass.
These gates are intentionally narrower than the full language rollout plan: they
prove the shared syntax ABI, prompt output contract, and provider advertisement
rules before richer v2 projection layers move into default guide output.

** Schema and docs gates

1. =semantic-tree-sitter-query.v1= validates native-projection, hybrid,
   tree-sitter-runtime, codeql-query, cached-replay, exact, approximate,
   native-only, and tree-only packet examples.
2. =semantic-tree-sitter-provenance.v1= remains the shared base for
   search/query/read syntax refs; no packet envelope redefines those refs in
   provider-private shapes.
3. =schemas/README.md= names the current v1 implementation boundary as
   tree-sitter-compatible native projection and names pattern graph,
   multi-capture match rows, and field structural selectors as next-layer
   capabilities.
4. Active RFCs, generated skills, and provider guides do not advertise retired
   syntax command families such as =ts-query= or =syntax-query=.

** Provider advertisement gates

1. Provider registry descriptors for =query= list emitted packet schemas,
   =queryInputForms=, =grammarId=, =grammarProfileVersion=,
   =queryCatalogs=, =adapterModes=, =sourceAuthorities=, and
   =executionBackends=.
2. A provider may list =native-parser= in =executionBackends= only when it can
   execute the ASP-compiled plan through native parser/compiler facts.
3. A provider may list =tree-sitter-runtime= only when a real tree-sitter
   runtime proves the pattern and the packet records that backend honestly.
4. A provider may list =codeql= only when a real CodeQL-backed executor emits
   the advertised frontier packet.  RFC 012 owns the native relation catalog,
   flow-lite packet plan, artifact policy, and CodeQL promotion gates; RFC 011
   only carries the shared backend/projection vocabulary.
5. Providers must not promote =pattern graph=, =multi-capture=, or
   =field structural selector= support in guide output until the descriptor and
   tests name those capabilities explicitly.

** Prompt and cache gates

1. =asp <language> query --treesitter-query <pattern> .= prints locator/frontier
   output by default.
   Prompt-noise ban: without source windows, cache ids, SQLite paths, receipt JSON, or artifact ids.
2. =asp <language> query --selector <exact> --treesitter-query <pattern> --code .=
   prints pure source code and rejects broad multi-match code extraction unless
   the provider can prove a unique match.
3. Search and read-plan outputs may attach =syn= anchors, but they must not
   inline code or treat syntax-only captures as edit-safety evidence.
4. Cache replay uses validated packets or normalized rows.
   Cache replay key: query AST/ABI, selector identity, grammar/profile/catalog fingerprint, execution backend, and file freshness.
   It must not derive facts from raw source, rendered stdout, basename
   matching, or stale unrelated generations.

** Evidence gates

1. Rust passes the reference native-projection corpus and stdout contract gates.
2. TypeScript and Python pass the shared stdout contract gate before their
   guide output advertises the syntax locate/code flow as generally available.
3. Julia passes the cacheable runtime-profile/export gate before ASP claims it
   can replay repeated syntax queries without starting Julia.
4. At least one real-project benchmark records command count, provider process
   count, cold/warm latency, packet bytes, cache hit rate, and repeated range
   scan reduction before cache benefits are used in marketing or guide text.
5. Artifact history evidence reports =query/direct-source-read= and
   =query/tree-sitter= separately.  A native-projection guide is not promoted
   while real agent traces show direct-read dominating syntax locate for the
   same language/root without a matching read-plan or syntax-frontier reason.

* Provider conformance lanes

Each provider adopts RFC 011 through one explicit lane.  A lane is a public
claim about what the provider can execute and what the =asp <language>= facade
may advertise.  Registry descriptors, provider guide output, query corpus
fixtures, and sandtable evidence must agree on the same lane.

** Lane R: Rust reference native projection

Rust is the reference lane for =native-projection= because it already has
provider-owned native parser facts and compact graph integration.

Required claims:

#+begin_example
languageId=rust
providerId=rs-harness
executionBackends=native-parser
adapterModes=native-projection
sourceAuthorities=native-parser-adapter,native-parser
renderProfile=compact-graph-frontier
queryInputForms=selector,code-shaped,catalog-id,s-expression
#+end_example

Required gates:

- Rust query catalogs live under =tree-sitter/tree-sitter-rust/queries/*.scm=.
- =grammar-profile.json= maps native facts to tree-sitter-compatible captures
  and records the ASP contract fingerprint.
- Query corpus fixtures cover declarations, calls, imports, macros, cfg, and
  selected tests without duplicating upstream grammar parser tests.
- =python -m tools tree-sitter validate runtime-boundary= proves the Rust
  provider does not require a grammar-specific tree-sitter runtime package for
  this lane.
- =python -m tools tree-sitter validate frontier-code-contract= proves non-=--code=
  locator/frontier output and pure =--code= output.

Rust may advertise =s-expression= support only for patterns mapped by the ASP
compiled plan and grammar profile.  Unsupported pattern graph, multi-capture,
or field-structural patterns remain =unsupported-pattern= until v2 gates land.

** Lane T: TypeScript compiler projection

TypeScript proves the compiler/native-parser value of the ABI.  Its authority
is the TypeScript Compiler API, =tsconfig=, and project references, not a
tree-sitter runtime.

Required claims:

#+begin_example
languageId=typescript
providerId=ts-harness
executionBackends=native-parser
adapterModes=native-projection
sourceAuthorities=native-parser-adapter,native-parser
renderProfile=corpus-locator
queryInputForms=selector,code-shaped,catalog-id
#+end_example

Required gates:

- Catalogs live under
  =tree-sitter/tree-sitter-typescript/queries/*.scm= and are embedded or
  exported through the provider/ASP release channel.
- Grammar profile maps declarations, imports, exports, calls, classes,
  interfaces, type aliases, and type-only facts into capture names.
- Cache and receipt digests include =tsconfig= and project-reference inputs
  that affect native compiler facts.
- The provider may add =s-expression= to =queryInputForms= only after the
  grammar profile can reject unsupported patterns deterministically.
- The shared stdout contract gate proves corpus-locator output without source
  windows, cache noise, or JSON in default mode.

** Lane P: Python AST/token projection

Python proves indentation-sensitive native projection.  Its authority is
=ast=, =tokenize=, and =symtable=, with native fields preserving decorators,
block ownership, and indentation-sensitive ranges.

Required claims:

#+begin_example
languageId=python
providerId=py-harness
executionBackends=native-parser
adapterModes=native-projection
sourceAuthorities=native-parser-adapter,native-parser
renderProfile=corpus-locator
queryInputForms=selector,code-shaped,catalog-id
#+end_example

Required gates:

- Catalogs live under =tree-sitter/tree-sitter-python/queries/*.scm=.
- Query corpus fixtures cover functions, classes, decorators, imports, calls,
  with/match blocks, and indentation-sensitive ownership.
- Wide direct-source-read recovery returns read-plan or syntax frontier refs,
  not inline code.
- The provider must not claim edit safety from AST shape alone; edit safety
  stays tied to native parser evidence plus patch-verification receipts.

** Lane J: JuliaSyntax export and cache replay

Julia is the cache-payoff lane.  Its native authority is JuliaSyntax plus
workspace-managed project/module/include context.  The public claim is not
that ASP understands Julia semantics; the claim is that ASP can replay
validated Julia exports when fresh.

Required claims:

#+begin_example
languageId=julia
providerId=julia-lang-project-harness
executionBackends=native-parser,cached-replay
adapterModes=native-projection,cached-replay
sourceAuthorities=native-parser-adapter,cached-provider-export
renderProfile=corpus-locator
queryInputForms=selector,code-shaped,catalog-id
#+end_example

Required gates:

- Catalogs live under =tree-sitter/tree-sitter-julia/queries/*.scm= when the
  grammar profile has stable names.
- Provider exports carry JuliaSyntax-owned captures and native fact refs.
- ASP cache replay proves repeated syntax queries can render from fresh
  validated exports without starting Julia.
- Module, include, macro, and package-environment facts remain native fields or
  native fact refs; they must not be flattened into syntax-only captures.

** Lane Q: CodeQL or flow-lite semantic backend

CodeQL and =flow-lite= are semantic backend lanes above the syntax ABI.  They
do not change the public query command family and do not make CodeQL required
for syntax locate/code.

Required claims:

#+begin_example
executionBackends=codeql
adapterModes=codeql-query
sourceAuthorities=codeql
renderProfile=flow-lite-frontier
queryInputForms=catalog-id
#+end_example

Required gates:

- RFC 012 defines the native relation catalog, flow-lite packet plan, artifact
  policy, and CodeQL promotion gates before provider guides advertise this
  lane.
- Registry descriptors list =codeql= only when a real executor can produce the
  advertised local source/sink/path frontier.
- Default syntax guides keep =native-parser= lanes available without requiring
  a CodeQL database.
- Flow/path results must reference syntax captures or native fact refs where
  useful, but they must not pretend to be tree-sitter runtime matches.

** Native-projection v1 real-project evidence gate

The =pending=real-project-benchmark= marker is removed only after at least one
representative Rust workspace and one representative TypeScript workspace record
the syntax query flow through the public =asp <language>= facade.  The benchmark
is not a micro-benchmark of parser internals; it is an agent workflow evidence
record for discovery, syntax location, exact code extraction, and hook recovery.

Required evidence record:

#+begin_example
[syntax-real-evidence] language=<id> provider=<id> project=<name>
commands=search-prime,syntax-frontier,exact-selector-code,hook-recovery
metrics=commandCount=<n>,providerProcessCount=<n>,packetBytes=<n>,coldElapsedMs=<n>,warmElapsedMs=<n>
metrics=syntaxQueryCount=<n>,exactCodeCount=<n>,manualRangeScanCount=<n>,repeatedTriggerReduction=<n>
metrics=treeSitterArtifactCount=<n>,directReadCount=<n>
outputs=frontier-no-code,pure-code-stdout,registry-descriptor,query-corpus
cacheClaim=none|warm-provider|hit
#+end_example

=python -m tools syntax real-evidence= renders this record after the measurements
are collected.  It is a review helper, not a packet schema and not a benchmark
runner: the caller supplies measured counts and elapsed times from the real
workflow.  It rejects =cacheClaim=hit= unless a cache receipt or normalized-row
replay proof is supplied.

Evidence rules:

1. Search/read-plan recovery remain discovery surfaces, and =search prime= remains a discovery surface; all of them
   must not inline source code.
2. =query --treesitter-query= without =--code= must render locator/frontier
   evidence, not artifact ids, SQLite paths, receipt JSON, or raw source windows.
3. =query --selector <exact> --treesitter-query <pattern> --code= must print only source text for the selected exact or unique match.
4. Cache speedup may be claimed only when the receipt reports =cacheClaim=hit= or
   a validated normalized-row replay path.  =warm-provider= is not a cache hit.
5. =directReadCount= must be explained by exact-locator follow-up or
   source-preserved hook recovery.  It must not exceed syntax locate volume for
   the same workflow unless the evidence record includes the unsupported syntax
   pattern, missing catalog, or read-plan reason that forced the fallback.
6. Julia cache payoff and CodeQL backend execution remain out of scope for this
   native-projection v1 evidence gate.

* Non-goals

- Do not require every provider to link tree-sitter.
- Do not move language parser implementations into the Rust ASP client.
- Do not introduce a separate =ts-query=, =syntax-query=, or raw-query command
  family; =query= is the tree-sitter-compatible syntax query surface.
- Do not replace =semantic-search-packet=, =semantic-query-packet=, or
  =semantic-read-packet= in the first implementation.
- Do not treat cache availability routing as semantic fallback.
- Do not store raw source in the default cache.
- Do not claim edit safety from syntax captures without native parser,
  fingerprint, and patch-verification evidence.

* Resolved decisions for native-projection v1

1. Syntax query artifacts use the dedicated
   =semantic-tree-sitter-query/<id>.json= namespace when the artifact is a
   =semantic-tree-sitter-query.v1= packet.  Existing =query/<id>.json= artifacts
   remain for =semantic-query-packet.v1= output.
2. =semantic-tree-sitter-provenance.v1= is the shared base for syntax query refs,
   matches, captures, and the optional short =syntaxAnchor=.  Search, query, and
   read packet envelopes stay separate.
3. Grammar profile versions are provider-owned stable strings, but each profile
   must carry the shared =semantic-tree-sitter-grammar-profile.v1= shape and the
   ASP contract fingerprint used for compatibility.
4. Arbitrary S-expression input is gated per provider through
   =queryInputForms=s-expression=.  Rust may advertise it for ASP-compiled plan
   coverage; TypeScript and Python keep catalog-id/selector/code-shaped support
   until their grammar profiles can reject unsupported S-expressions
   deterministically.
5. Catalog filenames are =.scm= only.  ASP does not accept =.scheme= catalog
   filenames as an ABI compatibility spelling.  Scheme-like S-expression query text is an input form, not a catalog filename policy.
6. TypeScript follows Rust as the first compiler/native projection target because
   it proves =tsconfig=, project references, declarations/imports/exports, and
   type-only facts through the TypeScript Compiler API.  Python follows for
   indentation-sensitive AST/token projection.  Julia remains the cache-payoff
   lane after validated exports and replay gates land.
