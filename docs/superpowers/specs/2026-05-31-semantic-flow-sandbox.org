* Scenario

The drill replays the TypeScript CI failure mode where the provider package ran
from an independent checkout while the shared JSON Schema files lived only in
the protocol repository root.

The expected flow is:

#+begin_src text
Spec -> Scope -> Plan -> Implement -> Check -> Test -> Verify -> Close
registry -> workspace -> prime -> focused search -> synthesis -> check
#+end_src

This is a sandbox drill, not a new protocol surface.  The CLI still owns
deterministic facts only; the agent client owns task intent, axis selection,
subagent planning, synthesis, and edit decisions.

* Findings

1. The registry/schema design is strong enough for independent provider CI.
   Provider-local schema copies plus submodule sync tests close the CI failure.

2. =search workspace= works as the RFC now describes: it is a router, not a
   whole-repo search.

3. =search prime= gives a useful bounded package map and identifies the
   semantic-search implementation owners before broad search.

4. =search owner <test-file>= now returns an explicit parser-visible module
   owner when the file is selected by the project.  This is weaker than full
   reasoning-owner graph evidence by design, and stronger than path-only because
   the packet carries =source=parser-visible-module=, =parserOwner=false=,
   role/layer metadata, validity, line count, diagnostics, and semantic
   next-actions.

5. =search fzf= now indexes parser-visible TypeScript source text, including
   test files selected by the project, while docs/schema JSON and other
   non-parser text remain =rg | search ingest= candidates.

2. Current TypeScript provider behavior:
   - parser-visible modules outside the reasoning graph resolve as module owners
   - existing non-parser paths resolve as path-only owners
   - parser-visible source text is searchable through search fzf
   - non-parser text misses emit explicit next=ingest guidance

3. Agent guide:
   - use rg | search ingest for docs/schema-copy text until broader
     provider text indexing is advertised
