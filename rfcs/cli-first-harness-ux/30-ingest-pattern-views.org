* External candidate adapters: =search ingest=

When the agent client deliberately needs wider lexical or file-path candidates, it
calls =rg= or =fd= directly — but always pipes output back into
=rs-harness search ingest= for owner grouping, dedup, ranking, and compact
packet rendering.  Raw =rg= / =fd= output is never pasted into the chat.

There is exactly one ingest command — no sub-kinds, no =--query=:

#+begin_src shell
rs-harness search ingest
#+end_src

Shared controls (output / scope control only, no source declaration):

#+begin_src text
--json
--package <pkg>
--view graph|hits|both|seeds
#+end_src

No =--query=, no =ingest rg= / =ingest paths= / =ingest fd= sub-kind.
Ingest auto-detects the format from the first few KB of stdin.
Owner, scope, hit, item, line, and per-owner caps are provider-owned
deterministic limits unless a method descriptor explicitly advertises them.

** ASP-owned seed-query frontier: =search pipe=

=search pipe= is the single public wrapper for cases where the agent needs ASP
to turn an LLM-generated seed query into a ranked locator frontier.  It is not an
operator DSL, not a second =ingest= command family, and not a public
=graph_turbo= command.  Simple external shell search stays explicit:

#+begin_src shell
rg -n "contentBlocks" tests | asp python search ingest items tests --view seeds .
#+end_src

When the query spans multiple roots, repeated command history shows slow fzf
patterns, or the agent would otherwise hand-write a brittle pipe, the agent uses
=search pipe=:

#+begin_src shell
asp rust search pipe "cacheRoot ClientReceipt cacheManifestPath" \
  --source auto \
  --view seeds \
  crates/agent-semantic-client/src crates/agent-semantic-protocol/src
#+end_src

Contract:

#+begin_src text
search pipe <question-or-feature-term> [--source auto|provider|finder|ingest] [--view seeds|graph-turbo-request] [scope...]
#+end_src

- =search pipe= accepts a =question-or-feature-term= that the LLM already
  distilled from the task and previous seed output into a code-search axis.  It
  is not a natural task intent parser.  ASP does not model-expand it into
  adjacent concepts.  The query language must not become public syntax.
- =--source= answers only one question: where candidates come from.  =auto=
  tries provider/parser facts first and falls back to finder recall when
  coverage is weak; =provider= does not run lexical finder and must recommend a
  source flip when coverage is insufficient; =finder= uses bounded lexical
  recall and normalizes results into owners/selectors/tests; =ingest= consumes
  stdin or an external candidate stream and normalizes it through the same
  frontier.
- =--source= never chooses output surfaces.  Prompt-facing surfaces such as
  selector, owner, tests, deps, failure evidence, and read-plan windows are
  selected by graph-turbo from the seed query and candidate shape.
- =--view seeds= returns a =search-pipe= compact frontier with =query=,
  =queryTerms=, =coverage=, =sourceTrace=, =handles=, =nextClasses=,
  graph-turbo =rank= and =frontier=, executable =frontierActions=, optional
  exact-selector =nextCommand=, =omit=, and =avoid=.  It must not inline source,
  raw finder output, full candidate lists, or any operator expression.  Omitted
  =--view= defaults to this same compact frontier.
- =--view graph-turbo-request= returns a schema-owned graph request for the
  graph-turbo ranking boundary.  It is a validation/debug view, not the default
  agent-facing output.  The request includes =surface=, =queryTerms=, =source=,
  =candidateSources=, and =sourceTrace=, but it must not include =pipeExpr=,
  =operator=, or =stageCompose=.
- =search suggest=, not =search pipe --view commands=, returns replacement
  command templates and slow-pattern advice.  It does not execute shell
  commands and does not spawn providers.
- =--pipe=, =pipeExpr=, =|>=, =lit(...)=, =rg(any(...))=, and public
  command-pipeline DSL output are removed from agent-facing =search pipe=.  In
  =asp rg -query=, =asp fd -query=, and owner-local =--query=, =a|b|c= has
  exactly one public meaning: an LLM-generated grouped semantic query-set / OR
  hint.  It is not a shell pipe, not exact-regex syntax, and not a complex DSL.
- Provider-native semantic enrichment may run after the bounded candidate set
  exists, but direct source reads and cheap lexical discovery must not pay a
  project-wide semantic parse cost first.  When the query names data-shape
  concepts such as fields, types, collections, maps, sets, lists, or concrete
  collection APIs such as =Vec=, the bounded enrichment should project typed
  =field=, =type=, and =collection= facts into the graph-turbo request.  The
  default seeds view should then expose field-level selectors through
  =frontierActions= and =nextCommand= instead of asking the agent to run another
  owner search, broad fzf, or tree-sitter guide probe after graph-turbo already
  selected the top branches.

The intended large-repo loop is two-stage when the first seed frontier is not
enough:

#+begin_src shell
asp typescript search pipe "Effect concurrency Fiber Queue Stream Scope" --view seeds .
asp typescript search fzf "Fiber Queue Stream Scope" owner tests --view seeds .
asp fd -query "Fiber|Queue|Stream|Scope|Runtime|internal" .
asp rg -query "Fiber|Queue|Stream|Scope|fork|interrupt|runFork|Runtime" .
asp typescript query --selector <path:start-end> --workspace . --code
#+end_src

=search fzf= remains the explicit lexical/fuzzy candidate command; it is not
replaced by =search pipe --source finder=.  =asp fd -query= searches
owner/path/module recall.  =asp rg -query= searches content/hot-block recall.
In both wrappers, =-query= belongs to ASP ranking context; flags after =--=
belong to the native tool and are accepted as pass-through context without
changing query-set semantics.  =--source finder= is a recovery or advanced source
override for =search pipe= when seeds show provider coverage is weak.

** Search fast path boundary

The cheap search surface is ASP-owned across all language facades.  This is not
a Python-specific optimization: Rust, TypeScript, Python, Julia, and future
providers share the same failure mode if =search= discovery falls through to a
provider's check-grade project report before candidate bounds are known.

ASP owns the first-pass frontier for these shapes:

#+begin_src text
search ingest [items|owner|tests...] --view seeds
search pipe <question-or-feature-term> [--source auto|provider|finder|ingest] --view seeds
fd -query <owner-or-path-term-a|term-b|term-c> [scope...] [-- native-fd-argv...]
rg -query <content-or-error-term-a|term-b|term-c> [scope...] [-- native-rg-argv...]
search fzf <term-or-error> [owner|tests|items...] --view seeds
search reasoning owner-query --owner <path> --query <term> --view seeds
search reasoning owner-tests --owner <path> --view seeds
search owner <path> items --query <symbol-or-a|b|c> --view seeds
#+end_src

These commands return compact locator/frontier output without spawning the
language provider.  They may read bounded candidate stdin, bounded owner files,
or bounded lexical matches under the selected root.  They must not construct a
language provider's full project report, policy/check result, verification
reasoning tree, package graph, or native parser fact index as a prerequisite.
All emitted locators are relative to the agent invocation workspace, not to a
scoped search root.  A locator printed from =asp rust search fzf ... crates/foo=
must therefore replay as =asp rust query --selector crates/foo/... --code .=
without requiring the agent to remember the original scoped root.

Provider binaries remain the authority for native facts.  They are invoked for:

- =check= and policy/report commands.
- =query --code=, exact native reads, and tree-sitter/native syntax projection.
- JSON/schema integrations that explicitly require provider packets.
- Provider-advertised semantic enrichment after ASP has already produced a
  bounded candidate frontier.

The ASP facade also owns the cheap-search configuration surface.  =asp.toml=
may declare ignored search directories, hidden-directory allowlists, language
enable/disable flags, and provider binary overrides:

#+begin_src toml
[search]
ignoreDirs = ["generated"]
includeHiddenDirs = [".github"]

[languages.rust]
enabled = true
bin = "/path/to/rs-harness"
#+end_src

This configuration is applied before provider spawn.  It does not move parser
facts into ASP: it only controls the facade's routing, discovery bounds, and
provider process selection.

If a provider keeps a local =search ingest= or =search reasoning= command for
debugging or direct package use, that implementation must obey the same
boundary: no check-grade full project report before a cheap candidate frontier
exists.  The public =asp <language>= facade enforces this invariant first so
agent-facing latency and command behavior are language-neutral.

The target invariant is that the following slow pattern:

#+begin_src shell
asp rust search fzf owner-items --view seeds crates/agent-semantic-protocol
#+end_src

can be replaced by:

#+begin_src shell
asp rust search pipe "owner items" --source auto --view seeds crates/agent-semantic-protocol
#+end_src

without exposing finder DSL and without requiring the agent to paste raw =rg= or
=fd= output into the prompt.  The public executable wrapper surface is
=search pipe= plus =asp fd -query= / =asp rg -query= for LLM-generated grouped
keyword packets.  Advisory planning remains =search suggest= and must not
execute the candidate pipeline.

#+begin_src text

** Recency-biased ranking

When a search view has multiple same-class candidates after semantic rank,
provider score, and deduplication, project-internal path recency is the next
tie-breaker.  Newer last-modified paths sort before older paths because agents
usually edit the files they are currently reasoning about.  This applies to
owner-grouped hits, ingest candidates, dependency usage owners, and equal-rank
prime owners.

Recency is only a ranking signal.  It must not replace semantic relevance,
public/API/root owner priority, exact-match scoring, or version-scope rules.
Providers only stat paths that resolve inside the selected project or package
root; missing or non-project mtimes fall back to deterministic path/name order.
Default compact output does not expose mtime fields.

[search-ingest] src=paths ...
[search-ingest] src=rg-json ...

CLI executes it.

** =search docs= / =search api=

Dependency API docs are first-class search objects — not prose blobs, but
native-parser API facts connected to local callsites, features, public API
leakage, and tests.  The Rust harness must treat its native parser as the
authority for local source shape; registry and rustdoc sources can enrich facts
only after parser-owned workspace/path facts are exhausted.

Baseline distinction: =search deps= owns local usage for the dependency version
range declared by the current workspace; =search docs= owns external/reference
API facts for an explicitly requested version.  A future composite docs+usage
packet should be added only after a provider can join those two fact sources
without hiding provenance; it is not part of the baseline schema or TypeScript
provider.

#+begin_src shell
rs-harness search docs tokio::runtime::Builder
rs-harness search docs bytes::Buf --package tokio
rs-harness search api tokio::sync::mpsc::Sender::send
rs-harness search public-external-types --package tokio
#+end_src

Distinction:

| Command                    | Returns                                          |
|----------------------------+--------------------------------------------------|
| =search docs=              | doc facts + API shape, minimal prose             |
| =search api=               | API shape only: kind, signature, async, unsafe,  |
|                            | feature, receiver, return, error, trait impls    |
| =search public-external-types=| external dependency types appearing in         |
|                            | public API signatures                            |

Data source priority:

#+begin_src text
1. workspace / path dependency source (native parser facts + rustdoc comments)
2. Cargo registry source (downloaded crate source)
3. local rustdoc JSON cache (target/doc/)
4. docs.rs rustdoc JSON (optional cache/download)
5. HTML docs (last resort, not default)
#+end_src

For Rust, MVP implementation starts with parser-native facts only:
=kind=, =signature=, =params=, =async=, =unsafe=, =receiver=, =return=, and
=error=.  The CLI renderer must not infer these from compact text, regex-only
matches, or prose docs.
This MVP includes public free functions, public inherent impl methods, and trait
impl methods, so receiver shape, impl type, and trait path are parser-owned.
The CLI renderer must project those facts without inferring association itself.

Implementation rule: every new Rust API fact lands first in the native parser
fact model with parser-level tests.  =rs-harness search= may only project those
facts into compact text or JSON; it is not the source of truth.

The header line declares the data source for trust calibration:

#+begin_src text
[search-docs] q=tokio::runtime::Builder crate=tokio source=native-parser
[search-docs] q=bytes@0.6.0::Buf crate=bytes requestedVersion=0.6.0 versionScope=external source=registry-source
[search-docs] q=ignore@0.4.23::WalkBuilder::require_git crate=ignore requestedVersion=0.4.23 versionScope=external source=docs-rs-json
#+end_src

If the external source is not available, the packet admits it instead of
presenting current workspace parser facts as external-version docs:

#+begin_src text
|note docsSource=registry-source missing=true
#+end_src

Dependency-prefixed docs/API queries such as =serde::Serialize= use
=source=registry-source= unless the provider has parser facts for a local
workspace/path dependency source.

Output format (API facts, not docs prose):

#+begin_example
[search-docs] q=fallible pkg=. docs=1 source=native-parser
|api path=src/lib.rs line=11 kind=fn name=fallible next=owner:src/lib.rs source=native-parser docs=local-parser signature=fn(input:String)->anyhow::Result<Thing> params=input:String async=true unsafe=true receiver=- return=anyhow::Result<Thing> error=anyhow::Result impl=- trait=-
#+end_example

=search deps ignore@0.4.23::WalkBuilder::require_git= remains the local usage
entry point.  It may point to =search docs ignore::WalkBuilder::require_git=
when an external docs provider is available, but it must not require docs to
answer local usage.

=search docs= never outputs long prose by default.  Doc-comment excerpts are
not part of the baseline contract; providers can expose them only through an
advertised detail method or method descriptor field.

** =agent= — guide and hooks lifecycle

#+begin_src shell
rs-harness agent install
rs-harness agent doctor
rs-harness agent doctor --json

=install= writes/updates the generated =SKILL.md=, hooks, and config into the configured
agent workspace.  =doctor= checks daemon health, protocol version, cache
