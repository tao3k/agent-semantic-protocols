* CLI is the protocol: search plus query surface

Language-neutral examples use =<lang-harness>=.  The TypeScript implementation
ships this as =ts-harness=; Rust/Cargo-specific lanes use =rs-harness=.
Julia is no longer a bin-parity exclusion.  The provider identity is the
compiled =asp-julia-harness=, with =julia-project-harness.jl= kept as a
source-provider development fallback.  RFC 008 lets =agent-semantic-client=
cache validated Julia provider exports in the local SQLite cache and serve
repeated searches without restarting Julia when the generation is fresh.  That
cache is an execution optimization; JuliaSyntax remains the parser authority and
Rust must not invent Julia parser facts.

The agent sees the language facade commands, with =guide= as the interactive
entrypoint and =agent doctor --json= as the machine/registry diagnostic surface:

#+begin_src shell
asp <language> guide ...                 # provider-owned agent command menu
asp <language> search <view> ...         # all exploration / location / evidence
asp <language> query ...                 # parser-owned compact code/query reroutes
asp <language> check ...                 # policy / style / changed / full
asp <language> agent doctor --json ...   # registry/diagnostic packet
#+end_src

The root =asp search= and =asp query= surfaces are thin language-facade
routers, not a second provider grammar.  They may remove an explicit
=--language <id>= or infer one language from the active provider activation:
=coverage.sourceExtensions= may match an unambiguous path/selector suffix such
as =src/lib.rs=, and =coverage.configFiles= may match an unambiguous project
root.  The root facade must not keep a hard-coded Cargo/tsconfig/pyproject
language table, and it must not parse Cargo workspaces, TypeScript project
references, Python virtual environments, Julia project layout, or package
membership; those remain provider responsibilities.  When a root command cannot
infer one language from activation facts, it must fail with a compact diagnostic
that asks for =--language= or the explicit =asp <language>= facade.

Unsupported or inactive language facades are a fail-closed discovery boundary,
not a reason to search through a different active provider.  When a user runs a
language facade that is not active or not a protocol language id, including root
forms such as =asp search --language <id> ...= and language forms such as
=asp <id> search ...= or =asp <id> query ...=, the facade must stop before
provider execution and emit a structured advisory: the requested facade, the
active language facades, known protocol facades, and bounded next commands.
Only an exact or explicitly mapped active facade may be recommended as the next
=asp <language>= command.  An explicit mapping is a protocol-owned or
configuration-owned alias, not a derived substring, token, suffix, or hyphen
component match; =scheme= must not be inferred as =gerbil-scheme= while the
Gerbil/Scheme provider is still being implemented.  Otherwise the recovery
route must use
provider-neutral finder commands such as =asp fd -query ...= or
=asp rg -query ... <bounded-scope>=, or ask the user to install/activate a
provider.  Agents must not switch a Gerbil/Scheme, Scheme, or other unsupported
target-language question into an unrelated TypeScript/Rust/Python search merely
because that is the only active facade.

#+begin_src shell
asp search --language rust prime --view seeds .
asp query --language typescript --treesitter-query '<pattern>' .
asp query crates/agent-semantic-protocol/src/lib.rs --query run --code .
#+end_src

The root facade is intentionally additive.  Existing provider-owned guide text,
registry descriptors, cache receipts, and hook recovery routes continue to use
the language facade as the canonical shape whenever the active language is
known.

Document languages are a separate facade class.  =asp org= and =asp md= expose
parser-owned document structure, not code harness policy.  The executable
runtime is =asp= with a native =orgize= dependency.  The =orgize= binary remains
a standalone development/debug CLI, but =asp org= and =asp md= do not require it
on PATH.  The root =asp= facade owns this stable command routing; Org facts are
owned by the =orgize= parser and Markdown facts are owned by =comrak= through
the native =orgize= document provider.

#+begin_src shell
asp org guide .                 # native orgize document provider through asp
asp org search prime --view seeds .
asp org query --term CUSTOM_ID --view metadata .
asp org query --selector docs/a.org:10-20 --content .

asp md guide .                  # native orgize/comrak document provider through asp
asp md search prime --view seeds .
asp md query --term Project --view metadata .
asp md query --selector README.md:1-20 --content .
#+end_src

=org/md= support only =guide=, =search=, and =query=.  =check=,
=ast-patch=, =evidence=, and policy gates are intentionally unsupported for
document languages.  Their JSON contracts are document-specific:
=semantic-document-search-packet.v1= for metadata discovery and
=semantic-document-query-packet.v1= for metadata or filtered element-content
projections.
=search= emits document facts such as headings, Org properties, drawers,
tables, source or export blocks, Markdown links, and Markdown code fences, but
document content recovery does not use =search owner= or source owner/items
routes. =query --term <term> --view metadata= returns a bounded document
metadata frontier. Adding =--content= to an explicit =--selector=, =--term=,
=--kind=, or =--field= query changes stdout to the pure content of the matched
document elements; it is still query output and must not become a whole-file
read. Document text recovery uses =query --selector <path-or-range> --content=
and stays a filtered element-content projection, not a whole-file read.
=--code= remains reserved for source-language providers.

=guide= and =query guide= for document languages must expose the parser-owned
query vocabulary, not just command names.  The prompt-facing guide should
include query axes (=--term=, =--selector=, =--kind=, =--field=, =--content=),
the element map, field maps for each element kind, and common recipes such as
TODO headings, checked tasks, property lookup, source blocks, paragraph
content, and range-local element frontiers.  Agents should be able to choose a
document query from the guide without guessing field names or falling back to
raw reads.

=search toc= is the document-first repository orientation view.  It uses the
same provider discovery as =search prime=, but returns only heading facts,
grouped by document path, with =level=, =title=, line range, Org TODO/priority
or tag fields when present, and a selector-query next action for each heading.
Agents should use it before broad document reads when the task is to understand
repo documentation structure, README sections, design docs, RFC outlines, or
Org task hierarchies.  =search toc --json= emits the normal
=semantic-document-search-packet.v1= shape with =method=search/toc= and
=view=toc=, while compact output stays optimized for prompt navigation.

=search fzf <term...> --view toc= is the document keyword-to-outline
composition view.  It uses the same provider-owned document walk as =search
toc=, applies fzf-style keyword discovery to document facts, then renders the
TOC only for documents that contain all supplied keywords somewhere in that
document.  The keywords are document-level selectors for this view, not a
request to dump matching paragraphs.  Compact output must stay heading-only
and include selector-query next actions; =--json= emits the same document
search packet family with =method=search/fzf-toc= and =view=fzf-toc=.  A
trailing positional =toc= may select the same view for shell-friendly command
composition, but =--view toc= is the canonical spelling.

Document query no-hit output is still a protocol surface.  A query with
=hit=0= must not stop at a count-only receipt.  It must state that combined
terms/kinds/fields are an intersection, then return bounded next actions such
as =search fzf= for one term, a single-term =query --term= retry, =query guide=,
and the selector requirement for =query --selector ... --content=.  These rows are
agent-facing recovery guidance; they must not include raw source snippets or
pretend that document recovery can run without a selector.

Document facts are a navigation element map, not a full document AST dump.
The shared packet keeps language-neutral element categories in =kind= and
parser-native node identity in =sourceKind=.  Provider-specific facts stay in
=attributes=.  This lets agents query TODO lists, checklists, tables, headings,
property drawers, links, images, and code blocks without inheriting code search
owner semantics.

#+begin_src text
documentFact.kind =
  heading | property | planning | table | block | list | listItem | task |
  link | image | frontMatter | thematicBreak | selector

documentFact.sourceKind =
  Org wrapper or comrak NodeValue variant, e.g. Headline, PropertyDrawer,
  SyntaxPlanning, SyntaxListItem, NodeValue::Heading, NodeValue::TaskItem
#+end_src

Org maps its org-elements-like syntax wrappers to the shared element map:
=Headline= to =heading=, =PropertyDrawer= entries to =property=,
=SyntaxPlanning= to =planning=, =OrgTable= to =table=, =SourceBlock= and
=ExportBlock= to =block=, =SyntaxList= to =list=, =SyntaxListItem= to
=listItem= or =task= when a checkbox is present, and =SyntaxLink= to =link= or
=image=.  Headline attributes include =level=, =title=, =todo=, =todoType=,
=priority=, and repeated =tag= values.  List/task attributes include =bullet=,
=indent=, =checkbox=, =checked=, and optional descriptive =tag=.

Markdown maps comrak nodes to the same shared categories: =Heading=,
=List=, =Item=, =TaskItem=, =Table=, =CodeBlock=, =Link=, =Image=,
=FrontMatter=, and =ThematicBreak=.  Markdown paragraphs are intentionally not
emitted by default search/query metadata because they turn navigation into full
text indexing; exact paragraph content stays available through selector
queries.

RFC 007 adds the shared =asp ast-patch= verification path.
Language providers still own AST adapters, but agents should use the shared
binary for schema validation, receipts, and Codex-safe dry-run behavior.

** =search= — discovery and candidate acquisition

Every read-only discovery step goes through =<lang-harness> search=.  Search
returns compact evidence, owner/test seeds, dependency facts, and candidate
frontiers.  It should not be the primary path for parser-owned code extraction.
Common views:

#+begin_src shell
<lang-harness> search prime

<lang-harness> search owner <path-or-owner>
<lang-harness> search dependency <crate-or-import-or-package>

<lang-harness> search symbol <name>
<lang-harness> search callsite <name>
<lang-harness> search import <root>
<lang-harness> search pattern <recipe>
<lang-harness> search tests <owner-or-path>
<lang-harness> search fzf <query>
#+end_src

=search fzf= is the preferred agent-facing lexical candidate command.  The
name is intentional: LLMs already know =fzf= means fuzzy filtering/ranking over
a candidate stream, so the command communicates the search behavior better than
a provider-invented =text= abstraction.  The provider still owns the pipeline:
=rg= or an equivalent scanner builds bounded candidates, =fzf= or an
equivalent scorer ranks them headlessly, then the provider groups by owner,
deduplicates, caps, and renders compact seeds.

Do not expose the old text-named compatibility view.  Exact lexical matching,
fixed-string matching, and fuzzy matching are modes of =search fzf= or provider
finder pipelines, not separate public views.  =search fzf= is not a language
query: code-shaped questions, known owners, symbols, declarations, parser
items, and hook read reroutes should use top-level =query= first.

Agent-facing =search fzf= grammar keeps search composition and fzf-native
options separate:

#+begin_src text
<lang-harness> search fzf <query> [owner|tests|items|...] [--scope <path-or-kind>] \
  [--pipe <surface>] [--view seeds|hits|graph|both] [PROJECT_ROOT] \
  [--fzf-arg <fzf-option> ...]
<lang-harness> search fzf <query> [search-options...] [PROJECT_ROOT] --fzf <fzf-options...>
#+end_src

=--scope=, =--pipe=, =--view=, =--query-set=, and the optional project root
belong to the harness search grammar.  =--fzf-arg <value>= passes one
provider-approved fzf-native option without changing grammar ownership.
=--fzf= is a boundary marker: it must be the last harness option and all
following tokens are fzf-native options, so agents should place =PROJECT_ROOT=
before =--fzf= or omit it and let the provider discover the project root.
Providers must reject interactive or shell-shaped fzf options on agent paths
and record accepted/rejected options in the search packet =finder= field.

Path scopes passed to =search fzf= are search scopes, not extra project roots.
Providers must normalize them against the discovered project/package root and
its ancestors.  For example, when the package root is =crates/demo= inside a
workspace, =search fzf run owner crates/demo/tests .= must scope to package-local
=tests= instead of resolving =crates/demo/crates/demo/tests=.

In fuzzy mode, a plain query containing multiple whitespace-separated tokens may
be promoted to the same homogeneous query-set machinery as explicit
=--query-set= terms.  The provider must mark the output with =querySet=N= and a
query-set selector so agents can see that the single fzf phrase was decomposed
into bounded same-axis lexical terms.  Exact mode keeps the query as one literal
fzf pattern.

=search owner= resolves semantic owners first, then parser-visible modules,
then existing project paths.  A parser-visible module that is not a reasoning
owner should return =source=parser-visible-module=, =parserOwner=false=, and
module facts such as role, layer, line count, validity, and diagnostics.  A path
outside the parser module set may return =source=path-only= and
=next=ingest:<path>= so agents can keep the path in the protocol instead of
falling back to raw shell output.

** =query= — first-class parser-native extraction and reroutes

Provider-native item query is the default code extraction path for agents:

#+begin_src shell
<lang-harness> query <project-path> --query <symbol-or-a|b|c>
<lang-harness> query <project-path> --term <symbol-a> --term <symbol-b> --names-only
<lang-harness> query --from-hook direct-source-read --selector <exact-source-path[:line-range]> --code
<lang-harness> query --from-hook direct-source-read --selector <glob> --term <intent-term>
#+end_src

The language provider owns parser/AST lookup, exact item matching, multi-term
query expressions, hook reroute normalization, and compact code rendering.
Julia owner-local item query uses this same shared =semantic-query-packet=
contract for JSON output through the language facade:

#+begin_src shell
asp julia query src/CliExample.jl --term run --json languages/JuliaLangProjectHarness.jl
#+end_src

For an exact =direct-source-read= selector with =--code=, the provider should
return a bounded source-preserved local window for that path/range, not owner
metadata and not compact projection text.  The selector already carries the
path/range, so =--code= must not repeat =line= or =endLine= metadata or return
owner seeds.  The rendered lines must preserve the source/formatter shape for
the selected window, including indentation, blank lines, attributes, semicolons,
macro payloads, enum variants, and visible block braces.  Parser projection may
select or repair a read frontier, but it must not be used as the exact-read text
for this surface.  Rust, TypeScript, Python, and Julia all use their native
parser facts to bound or repair the selector; Julia uses the =asp julia=
facade backed by the compiled =asp-julia-harness= and cacheable provider
artifacts.

=--workspace <workspace-root>= is the only public way to override the
workspace root for source-language query/code commands.  =--code= is a boolean
output selector and must never consume or be followed by a project root.  The
stale shape =query ... --code .= or =query ... --code <root>= is invalid for
owner item queries, tree-sitter syntax-code queries, selector-code exact reads, and =search owner ... items --query ... --code=.  A selector query
without =--workspace= resolves against the invocation workspace, selector
absolute path, or provider-owned workspace inference.

If the selector is too wide for bounded code transport, the provider returns a
=read-plan= frontier, not a denial-only outline.  The active contract is
=semantic-read-packet=: =readPlan.mode= is =range-frontier= or =search-repair=,
=readPlan.frontier= names executable next actions, and =readPlan.avoid= names forbidden next actions such as =repeat-wide-read=,
=manual-window-scan=, and =raw-read=.  A range-only plan must use =windows= and
=frontier= entries; it must not synthesize a fake =symbol= such as
=local-window=.  Parser-owned symbols are preferred when available, and
mechanical range splits are only the fallback.  Every frontier =code= action
must map back to:

#+begin_src shell
<lang-harness> query --from-hook direct-source-read [--workspace] --selector <path[:line-range]> --code .
#+end_src

ASP owns the compact output-mode boundary across every language facade:
=frontier= is the default prompt-facing mode, =code= is selected only by
=--code=, =read-packet= is selected only by =--json --view read-packet=, and
plain =json= is any other structured packet mode.  In =frontier= mode, providers
may emit metadata rows such as =|read=, =|item=, =|owner=, omit/avoid facts, and
exact =read== locators, but they must not emit =|code= rows, =text= fields, or
any inline source payload.  The ASP client validates this before cache
write-back and replay so a provider cannot accidentally turn compact frontier
output into lossy pseudo-source.

For exact selectors without =--code=, the provider stays in =frontier= mode. It
may return a bounded =read-owner= frontier, return a =read-plan= frontier, or
normalize to =search owner <path> items= rather than owner seeds.  A
=read-owner= response must state the selected path, item, line range,
truncation status, omitted source status, and executable read locators; it must
not include =|code= rows, =text= fields, cache ids, SQLite paths, receipt JSON,
or unbounded adjacent source.  A =read-plan= or item response must keep code
omitted and provide executable read locators.  If an agent needs source text,
the next action is an explicit =--code= request or structured
=--json --view read-packet=.  For exact selectors with a query term, the
provider should keep the normal item query/code projection path.  Wildcard or
broad selectors still require concrete terms and should remain bounded
candidate/seed searches.

Hook route construction must preserve exact line-range selectors when the agent
already supplied one.  The hook may strip =:line[-end]= for language profile
matching, but the provider-facing route must keep the bounded selector:

#+begin_example
rtk read src/lib.rs:10:20
=> <lang-harness> query --from-hook direct-source-read --selector src/lib.rs:10:20 --code .
#+end_example

Dropping the range turns a precise local-read intent into a larger owner query
and encourages follow-up raw reads.  That is a protocol bug, not a user
preference.

The language provider must make that bounded selector useful.  A query such as
=asp <language> query --selector src/lib.rs:10:20 --workspace <workspace-root> --code=
should return the exact bounded source window for that range, not whole-owner
item names and not a lossy ancestor projection.  Ranges inside an inline module,
impl, function, or other containing item must preserve the selected source
lines instead of collapsing to only the ancestor declaration.  This is required for
test/assertion edits, import/header inspection, enum/impl variant inspection,
and hook recovery after a denied =sed= / =rtk read= / tool-native read.  The
root hook only preserves the selector; native parser providers own safe
selector validation, read-plan repair, and exact source-window rendering.

Providers should exact-match item names first; if no exact item exists, they may
fall back to contains matching.  A default hit emits parser-owned item metadata,
read locators, and explicit next actions, but it must not inline compact code:

#+begin_example
[search-owner] q=src/search/owner_view.rs pkg=. own=1 item=1 itemQuery=render_search_owner
|query itemQuery=render_search_owner status=hit match=exact item=1 reason=parser-item-exact next=query-code
|item render_search_owner kind=fn next=symbol:render_search_owner read=src/search/owner_view.rs:22:40
omit=code,projection-nodes,large-item-text
avoid=inline-code-in-search,repeat-owner,raw-read
#+end_example

After an agent has already selected a concrete owner and item query, it may ask
for pure compact code:

#+begin_src shell
<lang-harness> search owner <path> items --query 'fun1|fun2|fun3' --code
#+end_src

The =--code= projection emits only compact parser-owned code projection text.
It omits the =|owner= / =|item= / =|code= line protocol metadata and is mutually
exclusive with =--json=.  Providers should keep the default =--query= output
metadata-bearing because =status=hit=, =match=exact=, and visible fallback
markers are the accuracy contract for exploration and hook recovery.  =--code=
is therefore a post-selection token optimization, not a replacement for search
evidence.

For explicit item query code results, the compact parser-owned projection must
be produced by the same renderer regardless of transport.  Prompt-facing search
line protocol must not add =|code text="..."= during default discovery; code
text belongs to explicit =--code=, =query --code=, or JSON/artifact projection
surfaces after the agent has selected a concrete item/frontier node.  Providers
must not apply a second whitespace-only compaction layer at line-render time;
that hides AST structure and pushes agents back into repeated local reads.

Provider compact code must also preserve literal payload semantics.  Whitespace
inside Rust raw strings, Python triple-quoted strings, JavaScript template
literals, Julia string macros, and equivalent language-owned literal regions is
not syntax trivia.  A provider may compact whitespace only outside such
payloads, or it must summarize/omit the payload and route the agent to an exact
read locator.  JSON query packets expose this decision as
=projection.compactSafety=; line protocol =--code= users see the same compact
text and must not infer original indentation from it unless
=compactSafety.whitespacePolicy= is =exact-source=.

Line-protocol =|code= rows may include a parser-owned =nodes= field before
=text=.  The field is a comma-separated compact node list:
=id:kind:role:depth:start:end=.  The JSON query renderer must prefer this
parser-owned node list for =projection.nodes[]= and =expandActions[]=, using
compact =text= classification only as a fallback when a legacy provider does
not emit =nodes=.  This keeps visible line protocol compact while preventing
JSON projections from being reconstructed by renderer heuristics.
Formatter-structural compact may use =role=delimiter= rows for parser-owned
block boundaries; such rows are not formatter residue when their =nodeId= is
present in =projection.nodes[]=.

Compact code is not raw source minification.  It is an agent-facing AST
projection produced by the language-native parser.  The projection may omit
trivia such as blank lines, comments, and indentation, but it must preserve the
semantic structure an agent needs to reason about ownership and edits:

- item boundary and exact read locator
- block nesting and branch ownership
- return / throw / panic / await / unsafe / mutation markers where the parser
  can expose them
- helper calls and hot-block locators when a full item is too large

Rust and TypeScript providers may use token/braces compact syntax because block
ownership survives in ={ ... }=.  Python, Julia, and indentation-sensitive
languages must not flatten raw source indentation into ambiguous one-line code.
They should emit a parser-owned block projection such as =def f(x) { if x > 0
{ return x } return 0 }= or a semantic outline with hot-block locators.  This
projection is for agent understanding, not an editable source replacement.
Exact source remains available through the read locator.

The query response is a progressive reasoning-tree traversal, not a set of
human display modes.  Providers should select the smallest layer that preserves
the next decision:

#+begin_example
owner tree -> item names -> item outline -> compact hot blocks -> exact read for edit
#+end_example

Compact projection is bidirectional.  The forward direction is native parser
facts to agent-facing projection: declarations, branches, calls, exits,
mutation/effect markers, and hot blocks.  The reverse direction is projection
back to exact provider actions: every important node should carry a stable
node id plus a read locator or expansion action so the agent can ask for the
exact block instead of restarting broad search.  The compact text may be lossy
for token savings, but the projection metadata must explain what was omitted
and how to expand it.

#+begin_example
parser AST -> projection.nodes[] -> compact/outline text -> expandActions[] -> exact read
#+end_example

Formatter-assisted compaction is allowed only after the native parser has
proved the structure.  The provider may use the language formatter to choose a
workspace/style-owned source dialect before projection or after an edit
dry-run, because formatter-normalized text lets the compact renderer remove
whitespace, blank lines, and local trivia without guessing semantics.  The
formatter is not the source of AST facts.  It is a safety gate for printable
text that still follows the native AST.

This intentionally separates formatter language from formatted source.  The
original purpose of formatters is human readability: indentation, vertical
spacing, alignment, and style conventions make code comfortable for people to
scan.  Those same choices waste tokens for an agent.  However, LLMs are trained
mostly on formatted code, so the formatter's canonical spelling is still a
useful prior for code understanding.  Compact AST should therefore reuse the
formatter dialect, not the full formatted layout.  Providers may normalize
syntax with the formatter, then project the parser facts into a shorter
agent-facing form that drops human-only layout while keeping grammar,
ownership, and edit locators.

The protocol must not assume one fixed formatter style.  Different Rust,
Python, TypeScript, JavaScript, and future provider workspaces may choose
different formatter tools, versions, line widths, import styles, or plugin
settings.  Those choices are user/workspace policy.  The provider records the
formatter profile it discovered and treats the exact style as opaque; the
shared invariant is that the formatted output still parses to the same
language-native structure.  The shared query projection schema must not expose
formatter names or styles as a cross-language enum; formatter profile belongs
to provider registry/capability metadata or provider-local diagnostics.

Formatter alignment is fail-closed.  If a provider declares that compact output
uses formatter-normalized syntax, the original parser facts and the formatted
parser facts must align through native node identity, symbol path, structural
role, and structural fingerprints.  Token fingerprints may help diagnose
alignment, but they are not the primary contract because formatter styles are
not stable across workspaces.  If AST alignment fails, the provider must not
silently return parser-only compact output as a fallback.  It should emit a
structured failure or diagnostic receipt and stop the formatter-assisted path.
Parser-only compact projection is valid only as an explicitly declared
capability mode before formatter normalization is attempted; it is not a
recovery path after formatter/core AST disagreement.

The formatter stage is language-owned:

- Rust uses =rustfmt=.
- Python uses =ruff format= for formatting; =ruff check --fix= is a separate
  lint-fix or organize-imports capability, not formatting.
- TypeScript and JavaScript use the workspace-declared formatter from
  =package.json= or provider registry facts, such as =prettier=, =biome=, or
  =oxfmt=.
- Julia uses =JuliaFormatter.jl= as a provider-declared formatter capability
  behind the Julia facade.

Compact AST output must not break the normal agent edit loop.  The agent reads
compact projection to understand and choose a target, then expands an exact
read locator before writing code.  The compact projection is not an editable
source replacement and must never be the only path back to source.  It is also
not a text patch context: compact renderers may remove indentation, punctuation
lines, comments, or formatter layout.  =apply_patch= preimages and other
text-hunk contexts must come from an exact source window selected by the
projection's =read=/=exactRead= locator.  A provider that relies on
formatter-normalized text for maximum compression must still return stable node
ids, locators, omitted facts, and expansion actions.

Standalone layout punctuation is never a useful compact line.  Lines such as
=}=, =})=, =});=, =];=, or =,= are formatter/source-layout residue, not
agent-facing facts.  A provider may keep punctuation inside a semantic line
when it helps expression understanding, such as =.map((value) => ({=, but it
must not emit punctuation-only lines as compact code.  If block structure
matters, it belongs in parser nodes, depth, edges, or exact expand actions.

Compact rendering must also be node-unique: one native AST node id produces at
most one rendered row in a given projection.  Nested owners, local classes,
closures, or visitor methods may be summarized under their parent or exposed as
expandable descendants, but the renderer must not emit the same descendant both
inline and as a sibling row.

**** AST projection identity and uniqueness

AST projection uniqueness is a protocol invariant, not a renderer preference.
Within one query/search packet, every projected native AST node must have one
stable projection identity and at most one primary rendered row.

The provider owns the concrete native identity, but the shared projection
identity must be derived from parser facts, not text formatting.  A projection
node identity should include enough provider-owned material to distinguish the
node within the source fingerprint:

- owner path and owner/item identity
- native node kind
- structural role
- parent projection id or structural parent path
- source range or parser-owned stable span
- same-kind ordinal when the language parser requires it
- optional structural fingerprint for diagnostics

In the packet contract this is not optional metadata.  Every compact projection
node must carry =nativeId=, =role=, =depth=, =read=, and
=structuralFingerprint= so the agent can distinguish parser-native identity,
rendering hierarchy, exact source expansion, and stale/ambiguous projections
without guessing from formatted text.

Rendered compact rows are views over projection nodes:

- one rendered row has one primary =nodeId=
- one native AST node may not be rendered as two primary rows
- child nodes may be referenced inline only as labels, edges, or summaries
- repeated semantic references such as callee, type, binding, or symbol use are
  references, not duplicate projection nodes
- =omitted[]= and =expandActions[]= must reference existing projection node ids
  or exact read locators instead of re-rendering hidden code
- owner-local name discovery, fallback candidates, and routing actions such as
  =owner-names= belong to query/candidate output, not to compact AST
  =projection.expandActions=
- every projection action must be executable without guessing: =exact-read=
  and =hot-block= actions carry a =read= locator; =node-query= actions carry
  provider =argv= and must target an existing projection node rather than a
  read locator
- if an =exact-read= or =hot-block= action also emits provider =argv=, its
  =--selector= argument must be the same canonical =read= locator; it must not
  widen back to an owner path or glob

Nested owners need an explicit mode.  A provider may either summarize a nested
function/class/closure under its parent or expose it as a child projection node
with =parentId= and an expansion action.  It must not do both in the same
projection.  If a provider cannot assign unique parser-owned projection ids for
the requested item, it must not claim compact-AST output for that item.  It may
return a weaker, explicitly declared surface such as owner names or outline,
but not a duplicate or ambiguous compact AST projection.

This is the core parser loop.  Native parsers should improve the compact
algorithm and the expansion metadata together: if a provider learns to omit
large match cases, helper bodies, closures, decorators, or type clauses, it
must also emit the omitted fact and the next exact read/query action.  A
projection without reverse navigation is only a display optimization; a
projection with nodes and expansion actions is an agent reasoning interface.
Therefore =mode=compact= requires =sourceFingerprint=, =losslessStructure=,
=exactRead=, =nodes[]=, and =renderedNodeIds[]=.  A provider that cannot
populate those fields must use a weaker mode such as =outline= or =names=.
For compact mode, =exactRead= is the same canonical locator as the match-level
=read= field, and =sourceFingerprint= must bind that locator so stale compact
projections can be diagnosed without comparing arbitrary formatted text.

An exact small item hit should produce compact AST code.  A broad, mixed, or
missed query should produce parser-owned names/candidates.  An exact but large
item should produce semantic outline plus hot-block locators before full compact
code.  The agent should not have to choose among these layers blindly; provider
certainty, item size, and token budget decide the projection.  =--code= is the
explicit override after the agent has selected the item to read.

Python selector queries with an exact file selector plus terms use the
same parser compact projection as owner item queries.  They must not render a
second ad-hoc truncated source snippet for matched functions.  Small matched
items return =save-token-ruff= rows backed by =projection.nodes[]=,
=renderedRows[]=, =exactRead=, and =expandActions[]=; large or ambiguous ranges
return a read plan with exact item locators.  Literal-heavy schema fixtures
should expose parser-owned data-shape rows such as dictionary keys and bounded
literal values instead of raw =ast.unparse= truncation.

*** Parser compact capability plan

The Rust, TypeScript, and Python compact lanes must be advanced as native parser
implementations, not as renderer tricks.  The renderer may choose line protocol,
JSON, or pure =--code= transport, but it must serialize parser projection nodes
that already exist in the language provider.  A provider that can only trim
whitespace has not implemented compact code.  Julia participates through
JuliaSyntax-owned compact projections and the compiled =asp-julia-harness=
facade path.

The mainline compact dialects are =save-token-rustfmt= for Rust,
=save-token-typescript= for TypeScript/JavaScript, and =save-token-ruff= for
Python.  All three dialects share
=compactSafety.whitespacePolicy=formatter-structural= and must carry
=projection.nodes[]=, =renderedRows[]=, =omitted[]=, and =expandActions[]= in
JSON packets.  Formatter names, versions, plugins, line widths, and workspace
scripts remain provider-owned metadata; the shared syntax value names the
compact projection dialect, not a global formatter choice.  Providers that can
only emit names, outline rows, or formatter-shaped text without reverse
navigation must report =mode=names= or =mode=outline= rather than partial
=mode=compact=.

The shared capability ladder is:

1. /Declaration boundary./  Item name, visibility, signature, type-only status,
   and exact read locator.
2. /Control flow./  =if=/=else=, =match=/=case=, loops, and branch-local
   ownership.
3. /Terminal and effects./  =return=, =break=, =continue=, =throw=/=raise=,
   =panic=, =await=, =try/?=, =unsafe=, and equivalent language-owned exits.
4. /Mutation and binding./  Assignment targets, low-value ignored bindings,
   summarized initializer shape, and member/field mutation when the parser can
   expose it.
5. /Calls and macros./  Stable callee names plus bounded argument evidence for
   high-signal calls without dumping whole expressions.
6. /Expansion metadata./  Stable node ids, parent ids, exact read locators,
   omitted facts, and next query/read actions for hot blocks.

The durable plan is:

1. /Snapshot the current parser contract./  Each provider keeps a small fixture
   project and a stable compact-query snapshot in the language harness that owns
   the parser output.  The snapshot records the agent-facing =code= projection,
   =projection.nodes[]=, =omitted[]=, and =expandActions[]=.  It excludes temp
   roots, timings, and other runtime-only fields.  A compact algorithm change
   must update this fixture intentionally.  The root repository no longer
   mirrors parser-compact fixtures or runs provider snapshots; it owns the
   shared schema, RFC contract, and cross-field validators that language
   harnesses use to prove compact line output, =semantic-query-packet.v1= JSON,
   and =parser-compact-token-cost.v1= measurements against the same protocol
   boundary.

2. /Strengthen native parser facts before renderer rules./  New compact
   behavior should come from native parser nodes: Rust =syn=, the TypeScript
   compiler API, Python =ast=, or JuliaSyntax.  Parser facts own branch
   ownership, call labels, return/raise/throw exits, await/effect markers,
   mutation targets, decorators, class/member boundaries, and type/signature
   facts where the language exposes them.  Regex post-processing may normalize
   display text but must not invent structure.

3. /Keep projection bidirectional./  Every hot node that affects editing must
   carry a stable =id=, =parentId= when nested, an exact =read= locator, and an
   executable =expandAction= whose =--selector= equals that exact locator.
   Selector-code queries with line ranges must return the bounded parser
   item or source window, not owner top-items.

4. /Add capability tests by language feature./  After the baseline snapshot,
   add one language-harness snapshot per parser feature class: nested branches,
   loops, terminal statements, mutation, call/macro summaries,
   async calls/effects, class/member bodies, decorators or modifiers, large
   item outline truncation, and selector-range expansion.  These snapshots are
   the parser compact acceptance tests.  Rust should lock both the pure
   =--code= projection and the line-protocol =|code text="..."= transport so
   renderer changes cannot drift away from parser facts.

5. /Measure search-flow impact after snapshots are stable./  Sandtable and
   real-project runs should compare command count, repeated owner/test search,
   direct-read reroutes, and query packet size before and after compact parser
   improvements.  Performance optimizations are accepted only when the snapshot
   contract stays stable or changes intentionally.

Root hooks must not maintain a parallel read/query engine for the normal flow.
The =|query= line is the stop/accuracy contract for agents: =status=hit= with
=match=exact= means the parser selected the requested item; =fallback-contains=
means the provider had to broaden after no exact item existed, including
language-owned function-body matching when the requested evidence lives inside
the code rather than in the symbol name; =status=miss= means the query should
be corrected rather than followed by raw file search.

Owner-local item discovery has a names-only phase.  Agents should use it when a
selected owner is known but the exact item names are still uncertain:

#+begin_example
[search-owner] q=src/command/raw_search.rs pkg=. own=1 item=6 itemQuery=parse_
|query itemQuery=parse_ status=hit match=fallback-contains item=6 reason=parser-item-fallback output=names next=select-item
|item parse_ripgrep_like kind=fn read=src/command/raw_search.rs:97:201
|item parse_fd kind=fn read=src/command/raw_search.rs:311:358
#+end_example

In names-only mode, providers must not emit =|code= lines.  They should emit
read locators and a visible =output=names= field so the agent can choose the
exact follow-up query before requesting compact code.  Names-only hit or mixed
results should use =next=select-item=.  Misses or weak fallback top-item
inventories should use =next=revise-query= unless the provider has a
high-confidence parser-owned replacement item.

When an owner-local item query misses, the provider should emit bounded
parser-owned candidates if they can be inferred from the selected owner's item
index:

#+begin_example
|query itemQuery=parse_ripgrep_scope status=miss match=none item=0 reason=parser-item-miss output=names candidates=parse_ripgrep_like next=revise-query
#+end_example

Candidate names must come from parser-owned item names in the requested owner.
They are query repair evidence, not textual search hits.  A shell-safe repeated
=--term= form is preferred over unquoted pipe expressions because
=--query RawSearchScope|terms_for_raw_search= is interpreted by shells as a
pipeline unless the whole expression is quoted.

Agents must also quote literal query text that contains shell metacharacters.
Backticks are command substitution in common shells, so a copied documentation
string such as =Start with `asp <language> guide .`= must be passed as a
single-quoted argv literal:

#+begin_src shell
asp rust search fzf --query-set 'Start with `asp <language> guide .`' owner tests --view seeds .
#+end_src

If the query text itself contains single quotes, the agent should narrow the
query into separate safe terms or use a provider-documented file/stdin input
surface rather than interpolating raw prose into a shell command.  Providers
must treat the received argv term as literal query text and must not ask agents
to rely on shell escaping side effects.

When provider-native =query --json= is requested for an owner item query, it
emits =semantic-query-packet.v1= rather than =semantic-search-packet.v1=.  The
JSON packet uses =queryCoverage= and =candidateItems= for the same repair
evidence that the line protocol exposes through =|query ... candidates=...=.
Parser compact snapshots must use this =query --json= form; using
=search ... --json= records the wrong packet class and hides compact-query
cost behind search-packet fields.

Julia search uses the shared search packet for search JSON through the facade:

#+begin_src shell
asp julia search fzf run owner tests --view seeds --json languages/JuliaLangProjectHarness.jl
#+end_src

The hook wildcard form uses the same packet class when a client needs a cached
search frontier rather than compact text:

#+begin_src shell
asp julia query --from-hook direct-source-read \
    --selector '**/*.jl' \
    --term render_julia_search_packet \
    --surface owners,tests \
    --view seeds \
    --json languages/JuliaLangProjectHarness.jl
#+end_src

That packet is for validators, IDEs, and Rust-client cache reuse.  It should
mirror the compact search line protocol with =owners=, =hits=, =nextActions=,
=querySet= and =queryCoverage= when terms were provided, =searchSynthesis=,
optional =nativeSyntaxFacts=, and optional =semanticHandles=.  It does not
replace the agent-facing compact default, and it must not invent a
Julia-private search payload.

Providers may additionally expose a =read-packet= query output mode for exact
selected source windows:

#+begin_src shell
<lang-harness> query --from-hook direct-source-read --selector <project-path> --view read-packet .
#+end_src

That mode emits =semantic-read-packet= under the normal
=agent.semantic-protocols.semantic-language= protocol.  It is provider-owned
parser transport, not a root hook command.  The packet should only carry
bounded exact windows that the provider can justify with parser-owned owner,
item, or syntax facts.  Broad selectors must remain seed/candidate queries.

When the target file has separate staged and unstaged states, =direct-source-read=
is still the recovery surface.  Providers may accept a source selector such as
=--source worktree|index|head= and should record the selected layer in the
=semantic-read-packet= with =sourceVersion=.  =sourceVersion= defaults to
=worktree=.  =sourceVersion=index= reads the Git index blob for the resolved
repository-relative path, and =sourceVersion=head= reads the =HEAD= blob.
Packets should include =repositoryRoot= for nested repositories,
=gitBlobOid= for Git object reads, and =worktreeHash= for bounded worktree
source.  Agents must not replace this with raw =git show :path=, raw =git diff=,
or shell source dumps; if the provider cannot produce the requested source
version, it should fail with an explicit provider error or emit a read-plan
repair action rather than silently falling back to a different layer.

=--workspace= is a selector scope flag, not a content-version flag.  Agents use
it when the selector is workspace-relative, for example
=crates/<member>/src/file.rs:1:40= while the project root argument is the
workspace root.  Providers must resolve that path against the workspace root
before applying source-version selection.  =--source= remains limited to
choosing =worktree=, =index=, or =head= bytes for the resolved path and must not
be overloaded to mean workspace-relative lookup.
The =asp <language>= facade accepts and forwards =--workspace= as a shared query
flag, but it must not parse language package membership; Rust, TypeScript,
Python, and Julia providers own the native path-context resolution.

When an owner-scoped item query has no exact or contains match, the provider may
return a bounded owner-local fallback instead of an empty packet.  The fallback
must stay parser-owned, scoped to the requested owner, and visibly marked in the
header, for example =fallback=owner-top-items=.  This is intended for weak
path-derived hook queries such as =types|Types= or =packet-base|PacketBase=:
the hook supplies a cheap route, and the language provider chooses a small set
of top-level declarations that are most useful for compact code inspection.
Providers that implement this behavior should advertise it on the =search/owner=
registry descriptor:

#+begin_src json
{
  "fallbacks": [
    {
      "name": "owner-top-items",
      "trigger": "item-query-miss",
      "appliesToPipes": ["items"],
      "maxItems": 4
    }
  ]
}
#+end_src

Top-level =query= is also the provider-native syntax intent surface.  It is
for code-shaped input where the language parser can prove intent before text
search: =pub use rules=, =fn format_field=, =struct PacketCollections=,
=#[cfg(test)]=, =import {Foo}=, =def run=, and equivalent language-native
forms.  Future syntax-intent variants must live under =query= rather than
=search fzf=.  Providers must route these inputs through a Native Syntax Fact
Index instead of treating them as plain text.  The compact packet should expose
the normalized intent and facts:

#+begin_example
[search-query] q=pub use rules pkg=. intent=syntax.import own=1 fact=1
|query intent=syntax.import term=rules status=hit routed=native-syntax selected=1
|fact id=rust:src/lib.rs:4:reexport:rules kind=reexport source=native-parser owner=src/lib.rs line=4 visibility=public name=rules languageKind=use
[search-graph] mode=query root=. alg=native-syntax-query
legend: aliases ID:kind; node ID=kind:role(value)!next; edge SRC>{DST:rel}; frontier ID.next
aliases=G:search,L:owner,F:fact
L=owner:path(src/lib.rs)!owner
F=fact:symbol(rust:src/lib.rs:4:reexport:rules)
L>{F:contains}
rank=L,F frontier=L.owner
#+end_example

The root protocol owns the shared =semantic-native-syntax-fact-index.v1=
schema.  It defines language-neutral fact identity, kind, owner/location,
visibility, query keys, relations, and provider extension fields.  Rust,
TypeScript, Python, Julia, and future providers each own their concrete fact
builders and language-specific fields under the common schema.  The shared
schema is intentionally not a lowest-common-denominator AST; it is the
agent-facing index of facts that can answer search/query flows without raw text
scans.

The shared fact kind vocabulary must be broad enough for providers to expose
useful native facts without leaking their AST wholesale.  It therefore includes
portable items such as =owner=, =module=, =function=, =method=, =struct=,
=type=, =import=, =export=, =call=, =test=, =doc=, =include=, =field=,
=binding=, =constant=, =argument=, and =macro=.  Language-specific categories
stay in =languageKind=, while provider detail, parser text, tags, and
language-only fields stay under =fields=.

=search fzf= covers fuzzy lexical candidates: source text, owner paths, string
literals, comments, docs, schema files, generated artifacts, config keys,
error messages, and other surfaces where parser-native facts are unavailable or
insufficient.  The default pipeline is =rg+fzf=: =rg= or an equivalent scanner
builds candidates, then =fzf= or an equivalent headless scorer filters/ranks
them.  The provider, not the backend, owns project scope, path filtering, owner
grouping, per-term fairness, deduplication, ranking integration, and compact
packet rendering.

If the query is code-shaped, =search fzf= must not execute the native syntax
API on behalf of the caller.  It should return a visible =next=query= action so
the agent retries through the explicit parser-native API.  If text lies outside
the provider's local index, the provider may return =next=ingest:<query>= and
the agent must route external candidates through =search ingest=.

Hook wildcard direct-read recovery uses the =query= surface as the wrapper
entrypoint, not raw =search fzf=.  When =query --selector= carries
a broad selector and a code-shaped term such as =ClientReceipt {=, the provider
must normalize syntax delimiters out of the lexical fallback term set before
rendering a compact frontier.  The delimiter remains provider-owned query
context; it must not become an independent fuzzy term or cause prompt-facing
=|seed= rows to reappear.

=search fzf= must remain lexical/fuzzy search.  It may support match modes
such as =literal=, =fixed=, =regex=, =word=, =case-folded=, and =fuzzy=, and
backends/pipelines such as =rg=, =rg+fzf=, =ugrep=, or a provider-maintained
trigram/indexed lexical engine.  Backend selection should be visible in packets
through runtime cost/provenance fields, but it should not change the semantic
contract: parser-owned extraction is still =query=, and lexical candidate
expansion is still =search fzf= or =search ingest=.

=ast-grep= is not the fuzzy text backend for this lane.  It is a structural
search/rewrite engine: useful behind =search pattern= recipes or provider-owned
native-fact builders, but not a replacement for =rg= plus approximate lexical
matching.  Agents should never be asked to author raw =ast-grep= DSL as a
general search fallback.

*** Roadmap: agent-facing fzf lexical search

The lexical-search roadmap is intentionally separate from =query=:

1. =search fzf= becomes the preferred public lexical/fuzzy search view for
   agents.  The command name matches the mental model LLMs already have:
   fuzzy filtering over a candidate list.
2. The first =search fzf= backend should be =rg+fzf=.  =rg= owns fast source
   scanning, ignore/glob semantics, and structured hit provenance; =fzf= owns
   fuzzy ranking over candidate lines, paths, or owner labels.  Agent-facing
   runs must use non-interactive filtering/scoring, not a terminal picker.
   Human-facing diagnostics may optionally expose an interactive preview later.
3. =ugrep= or a provider-owned lexical index remains a later alternative when
   fuzzy matching, Boolean combinations, case folding, or cold large-workspace
   pruning cannot be expressed efficiently as =rg+fzf=.
4. Shared schemas should add backend provenance only after the RFC is stable:
   =textBackend=, =matchMode=, =sourceSearchPasses=, =candidateFileBasis=,
   =fuzzyFilter=, =queryTerms=, and bounded runtime cost.  These fields explain
   how text candidates were found; they do not make text results parser-owned
   facts.
5. Provider guides should teach the decision ladder:
   =query= for parser-owned extraction, =search owner/tests/deps= for
   structured discovery, =search fzf= for broad lexical/fuzzy candidates, and
   =search ingest= for externally generated candidate streams.

This keeps the competitive advantage clear: native syntax parsing plus
first-class =query= gives precise code slices and read locators with fewer
tokens and less semantic drift.  =search fzf= is the widening tool, not the
authority layer.

*** Shared finder tools schema

The protocol repository owns =semantic-finder-tools.v1= as the common contract
for provider-approved lexical and structural finder pipelines.  It is not a
new command surface and it does not expose raw shell argv to agents.  It
describes:

- the approved tool catalog, such as =rg=, =fzf=, =fd=, =ast-grep=,
  =ugrep=, or a provider index
- tool roles, such as source scan, fuzzy filter, path list, structural recipe,
  candidate normalization, or scoring
- approved pipelines, such as =fzf-rg-lines=, =text-rg-json=, or
  =pattern-ast-grep-recipe=
- provenance fields that search packets can later embed, such as
  =finderPipelineId=, =matchMode=, =candidateBasis=, =sourceSearchPasses=,
  =fuzzyFilter=, candidate counts, and runtime cost

The first target pipeline is:

#+begin_src text
search fzf <query>
  -> provider-owned rg candidate scan
  -> provider-owned fzf headless filter/rank
  -> provider normalize
  -> owner/test seeds
#+end_src

=rg= owns fast lexical candidate generation.  =fzf= owns fuzzy filtering and
ranking over candidate lines, paths, or owner labels.  The language provider
owns the semantic boundary after that: path normalization, owner resolution,
nearest-item resolution, test frontier selection, deduplication, caps, and
packet rendering.

Agents do not choose the =rg= command, stage order, or a shell pipe shape.
Agents may request provider-approved fzf match options only through
=--fzf-arg= or the terminal =--fzf= boundary.  The provider owns validation,
normalization, defaulting, and provenance; unsupported options are rejected
with a compact error or recorded as rejected finder arguments in JSON.

=search fzf= may reuse the same finder schema with
=surface=search-fzf=, but the preferred agent-facing surface is
=surface=search-fzf= and view =fzf=.

Agent-facing finder runs must be headless and provider-owned.  Human-facing
diagnostics may later expose interactive =fzf= preview, but that is outside
the agent protocol path.

*** ast-grep boundary

=ast-grep= has a useful place, but it is not a substitute for provider-native
=query=.  The provider's native parser and query implementation is stronger for
that language because it owns project configuration, package/workspace scope,
compiler or parser facts, item identity, test reachability, provider-specific
handles, and compact code projection.  For Rust, =rs-harness query/search= is
the authority for Rust semantic extraction; =ast-grep= must not replace it as a
parallel Rust query engine.

The correct =ast-grep= boundary is structural evidence where native provider
facts are absent, incomplete, or intentionally recipe-scoped:

- =search pattern <recipe>= can use provider-approved =ast-grep= recipes for
  structural patterns without asking the agent to write raw =ast-grep= DSL
- a Rust provider may combine =rs-harness= owner/query evidence with
  =ast-grep= structural matches when the structural recipe adds non-native
  evidence, but the final packet must mark =ast-grep= as structural provenance,
  not parser-owned truth
- for languages without a mature provider, =ast-grep= can supply bounded
  structural candidates that are then normalized by the generic finder pipeline
  and clearly marked as non-native structural evidence
- cross-language investigations may combine one language provider's native
  query results with =ast-grep= structural candidates from unsupported
  languages, as long as the synthesis preserves source/provenance boundaries

In short: native harness =query= is the precision layer; =ast-grep= is a
structural adapter and recipe runner; =rg+fzf= is the lexical fuzzy candidate
pipeline.

** Source owner context is provider query transport, not root read transport

The durable discovery path is parser-native =search= output.  Providers should
return compact parser-owned evidence: owners, items, tests, dependency edges,
findings, and =searchSynthesis=.  Once the hook has one exact selected source
path, the root runtime must route back to the owning provider's parser query
surface.  Root-level read recovery is removed and must not be advertised as the
recovery route.  If
=semantic-read-packet= is useful, it is emitted by the language provider as an
optional =query/*= output schema after that route has been followed.  The
read-plan frontier shape is part of the active provider-owned read-packet
contract; providers must not advertise a separate legacy recovery route.

Direct source reads and exact wrapper reads should be denied and rerouted to a
compact code window:

#+begin_src shell
asp <language> query --from-hook direct-source-read --selector <project-path[:line-range]> --code .
#+end_src

The hook decision packet must label that recovery route as =kind=query=.
Provider output may still contain owner, item, test, and read-locator facts, but
the executable route is the provider query surface, not a =search owner= route.
This keeps exact source paths, wrapper line locators such as =rtk read
src/lib.rs:10:20=, and cross-language direct reads on the same semantic
transport.

Broad selector reads should instead route to provider =query --from-hook=,
=query --term ...=, =search fzf ... owner tests=, or =search ingest ...=
depending on the evidence already available.  The provider owns the parser
facts, frontier selection, owner item projection, and broad discovery; the hook
only classifies and routes.

The hook-facing =query --from-hook= form is for broad selectors where no single
owner is known, for example =Read *.ts= or =less **/*.tsx= paired with concrete
terms from the agent's intent:

#+begin_src shell
asp <language> query --from-hook direct-source-read \
  --selector '**/*.{ts,tsx,js}' \
  --intent inspect-cli-search \
  --term parseSearchArgs \
  --term querySets \
  --surface owners,tests \
  --view seeds .
#+end_src

It is a compact route into the provider's parser-owned query layer.  It should
emit bounded query evidence: query coverage, owner candidates, item candidates,
test seeds when requested, and any visible runtime prefilter cost.  It must not
expand the matched files into source text.  =query --from-hook= is the only
agent-facing wildcard direct-read recovery route.  Providers must not advertise
or accept the retired search-query hook form as a compatibility alias; keeping both
forms creates route drift and breaks guide-quality replay.

When the selector is one exact project-relative source path and the agent has no
reliable item query yet, top-level =query --selector <path> --workspace
<workspace-root>= without =--code= may return owner =items= context directly.
This is still parser-owned compact context, not source rendering: it lists item
names, line ranges, read locators, and owner/test frontiers so the next step can
be a precise item query or focused edit.  The =--from-hook direct-source-read=
form remains available only as a low-priority bounded fallback after parser-owned
frontier evidence has selected the exact small window.

Rust adds Cargo-native views under the same search namespace:

#+begin_src shell
rs-harness search prime
rs-harness search targets
rs-harness search deps
rs-harness search deps <dep[/subpath][@version][::api]>
rs-harness search deps <dep[/subpath][@version][::api]> public-api
#+end_src

=search dependency= is a high-level composite view: native package/dependency
facts + source usage + owner graph in one compact output.  For Rust,
=search deps= is the practical dependency-version API usage view: manifest
version/feature/target facts and local API usage owners for the version range
declared by the current workspace.  If a query asks for =dep@version= that does
not match the workspace declaration, the packet must mark it as
=versionScope=external= and avoid presenting current owner usage as evidence
for that external version.  Current workspace versions are reported as
=currentWorkspaceVersion= when a lockfile or manifest exact version proves the
value.  For current workspace versions,
=::api= narrows owner hits to dependency owners that also mention the API token.
=search prime= is the low-token entry point with graph structure.

** Multi-pipe: deterministic packet composition

A provider can optionally chain multiple pipe stages in one invocation.  This
is an RFC target for providers that can keep the packet final-only and
deterministic; agents must still treat the registry's =methods= and
=methodDescriptors= as the callable truth:

#+begin_src shell
<lang-harness> search <source> <pipe1> <pipe2> ...
rg -n "<query>" <roots> | <lang-harness> search ingest <pipe1> <pipe2> ...
#+end_src

Source stages (produce the initial packet):

#+begin_src text
prime                 project reasoning-tree
dependency <dep>      dependency fact + source usage + owner graph
features <feature>    feature -> optional-dep -> target cfg chain
deps <dep>            native manifest dependency facts
symbol <symbol>       symbol definition + callsites
callsite <symbol>     all usage sites
pattern <recipe>      structural recipe search
ingest                stdin auto-sniff (rg -n / fd / git diff)
#+end_src

Candidate pipe stages (enrich the packet, all optional):

#+begin_src text
owners      resolve to owner graph nodes
items       resolve to intra-file items
tests       find owned tests/benches/examples
examples    find owned examples
benches     find owned benches
docs        attach API docs facts (kind, signature, feature, async, ...)
public-api  mark whether dependency types leak into public API
cfg         attach cfg / target facts
features    expand feature graph
dependents  who depends on this owner/dep
#+end_src

Each pipe is a deterministic transform.  The CLI renders only the final
packet.  The agent client selects the pipe combination based on the task; the CLI
never infers intent.

Examples:

#+begin_src shell
# Dependency change with docs and test coverage:
rs-harness search dependency ignore items docs tests --package file-search

# Feature/cfg change across owners and docs:
rs-harness search features io-uring cfg owners tests docs --package tokio

# Public API leakage check:
rs-harness search dependency bytes public-api docs tests --package tokio

# External candidate stream with grouping:
rg -n "require_git|git_ignore|parents" agent-rs/file-search | rs-harness search ingest items tests
#+end_src

When a provider advertises diagnostic tracing, it should return a compact
stage list rather than full intermediate packets:

#+begin_src text
[search-trace] source=dependency query=ignore pipes=items,docs,tests
|stage dependency cargo=1 owners=1
|stage items owners=1 items=4
|stage docs docs=1
|stage tests tests=1
#+end_src

*** Dev-mode command log

Providers may record every =*-harness= CLI invocation when development mode is
explicitly enabled:

#+begin_src shell
SEMANTIC_PROTOCOL_DEV_MODE=1
#+end_src

This is a provider debugging surface, not an agent prompt surface.  Its purpose
is to measure real command count, latency, repeated search loops, view/pipeline
selection, and hook reroute behavior without asking the agent to expand source
or request JSON during normal exploration.

The canonical cache location is:

#+begin_src text
$PRJ_CACHE_HOME/semantic_protocol/<language>/<provider>/commands/YYYY-MM-DDTHH-MM-SSZ-000001-<event-id>.jsonl
#+end_src

If =SEMANTIC_PROTOCOL_TRACE_DIR= is set, providers should write under that
directory instead of deriving =PRJ_CACHE_HOME=.  If neither location can be
resolved, the log write is silently disabled.  Command execution must not fail
because a dev-mode log directory cannot be created or appended.

Each JSONL row must validate against
=semantic-dev-command-log.v1.schema.json=.  Providers record argv, cwd,
project root, language/provider identity, normalized command method,
=startedAtUtc=, =finishedAtUtc=, elapsed time, exit code, and stdout/stderr
byte counts.  Providers must not record full stdout/stderr content by default;
those streams can contain source text, private paths, or model-visible prompt
material.  Obvious secret-bearing arguments such as =--token=, =--api-key=,
=--password=, and =--secret= must be redacted in the recorded argv.

The file name carries a UTC second timestamp for human directory inspection,
but analysis tools should sort by row fields rather than file names.  Within a
session, providers should allocate a best-effort monotonically increasing
=sessionOrdinal= before command execution starts.  The ordinal is provider
local; fan-out analyzers that combine Rust, TypeScript, Python, and other
providers should sort by =sessionId=, =startedAtUtc=, =sessionOrdinal=, and
=eventId= rather than assuming a single global counter.  When a client can pass
context directly, it should set =SEMANTIC_PROTOCOL_SESSION_ID=,
=SEMANTIC_PROTOCOL_PARENT_EVENT_ID=, and =SEMANTIC_PROTOCOL_HOOK_RUN_ID=.
Because real agents often remember a hook warning and then invoke =*-harness=
directly without any hook-provided environment, hook runtimes may also write a
short-lived active-context marker:

#+begin_src text
$PRJ_CACHE_HOME/semantic_protocol/dev-context/<project-root-hash>.json
#+end_src

Providers should read that marker only as a best-effort development trace aid,
with a short TTL.  The marker must validate against
=semantic-dev-active-context.v1.schema.json=.  Providers should mark inherited
command events with =fields.contextSource= =active-context=.  When no
environment or active marker exists, the provider may fall back to a
project-scoped session id.  This makes the trace useful for reconstructing
which model step fanned out into which harness commands, even when commands
overlap, finish out of order, or run after the original hook event has already
returned.

The command log is implemented at each provider CLI entrypoint.  The root hook
may pass hook context such as a hook run id through environment variables, but
it is not the primary log owner: direct CLI usage, sandtable replay, and
hook-rerouted commands should all land in the same provider-owned log stream.

Internal model:

#+begin_src rust
struct SearchPacket {
    header: SearchHeader,
    packages: Vec<PackageFact>,
    cargo: Vec<CargoFact>,
    owners: Vec<OwnerFact>,
    items: Vec<ItemFact>,
    docs: Vec<DocsFact>,
    tests: Vec<TestFact>,
    edges: Vec<EdgeFact>,
    next: Vec<NextAction>,
}

trait SearchPipe {
    fn apply(&self, ctx: &SearchContext, packet: SearchPacket) -> SearchPacket;
}

** Wire contract: semantic language registry + shared =semantic-search= JSON Schema

Every provider first advertises through the semantic language registry, then
maps native search packets to the shared =semantic-search= wire contract when
=--json= is requested.  The registry and packet schemas live at:

#+begin_src text
schemas/semantic-language-registry.v1.schema.json
schemas/semantic-search-packet.v1.schema.json
schemas/semantic-query-packet.v1.schema.json
schemas/semantic-read-packet.v1.schema.json
schemas/semantic-source-location.v1.schema.json
schemas/semantic-tree-sitter-query.v1.schema.json
schemas/semantic-tree-sitter-provenance.v1.schema.json
#+end_src

The protocol repository's =schemas/= directory is the source of truth.
Provider packages that run their own CI from independent checkouts may carry
package-local copies at the same relative paths.  Their tests should validate
against the package-local files, and monorepo/submodule checkouts should compare
the package-local copies with the repository source schemas to prevent drift.

Tree-sitter query is the portable syntax ABI for query/search/read provenance,
but it is not a reason to collapse the three packet envelopes into one generic
packet.  =semantic-query-packet=, =semantic-search-packet=, and
=semantic-read-packet= keep separate required fields and consumer semantics.
Shared path/range fields such as =projectPath=, =lineRange=, =location=, and
=sourceLocator= converge through =semantic-source-location.v1.schema.json=.
Shared syntax provenance fields such as =syntaxQueryRef=, =syntaxMatchRefs=,
=syntaxCaptureRefs=, and =syntaxAnchor= must converge through
=semantic-tree-sitter-provenance.v1.schema.json= so providers do not maintain
parallel copies of the same tree-sitter reference shape.

Agent-facing guides must expose the syntax ABI as routes, not as another
packet family or a cache inspection workflow.  The compact guide route rows are
the public decision contract:

#+begin_src text
|route syntax-locate selectors=S:tree-sitter-query,R:range returns=locator,capture,frontier code=false
|route syntax-code selectors=S:tree-sitter-query,R:exact-selector returns=code code=pure
|route read-plan selectors=R:selector,T:term returns=owners,tests,window-set code=false
|route query-code selectors=O:owner,Q:symbol returns=code code=pure
#+end_src

Provider guide output should be split by use, not by implementation owner.  The
agent-facing line protocols are:

#+begin_src text
[guide] lang=<language> provider=<asp-provider> protocol=guide.v1 root=.
[query-guide] lang=<language> provider=<asp-provider> protocol=query-guide.v1 root=.
[treesitter-query-guide] lang=<language> engine=tree-sitter protocol=treesitter-query-guide.v1 root=.
#+end_src

=guide= is the provider's default agent-facing guide command and emits the
=guide.v1= protocol.  It teaches surfaces and flows, and should point
code-shaped syntax work to =query guide treesitter .= before an agent writes an
inline =--treesitter-query=.  =query guide= teaches stdout contracts, read-plan
contracts, and pure-code requirements.  =query guide treesitter= teaches
tree-sitter S-expression shape, captures, predicates, native enrichment fields,
catalog templates, and the rules for multi-match =--code=.  Hook recovery
messages should refer to these guides when the agent used raw language-source
tools: for example, a Rust =PreToolUse= denial for =Read=, =cat=, =sed=, or
=rg= over =.rs= source should explain that raw Rust source access is replaced by
=asp rust guide .= as the main progressive tool map, with =asp rust search guide
.=, =asp rust query guide .=, and =asp rust query guide treesitter .= listed
only as specialized reference guides when that decision is needed.
The concrete recovery route is the normal selector/code surface:
=asp rust query --selector <path-or-range> --workspace <workspace-root> --code=.
=--from-hook direct-source-read= is reserved for low-priority exact-window
fallbacks after parser-owned frontier evidence has already selected a bounded
small range.

The agent-prefixed guide spelling is retired for prompt-facing flows.
Providers may keep compatibility aliases for old tests or manual debugging,
but registry output, generated skills, hook recovery messages, and new RFC
examples must use =guide= as the main command name and keep
=query guide treesitter .= as a low-frequency reference.
This keeps the low-frequency tool map stable and prevents agents from reading
multiple guide bodies before they know whether they need search, query, syntax,
or verification detail.

Hook recovery reason text is a compact Markdown prompt, not a second JSON
schema and not a line-protocol template.  The authoritative
platform-independent envelope remains the existing
=agent.semantic-protocols.hook.decision= packet in =additionalContext=; the
=permissionDecisionReason= / =systemMessage= string is the low-token agent
instruction.  It must avoid inline guide bodies and use a progressive decision
ladder:

#+begin_src text
# ASP Hook Recovery

The pre-tool hook blocked `<reason>` on language source.

## Stop
Do not retry `Read`, `cat`, `sed`, `rg`, or source-dump commands on the matched source.

## Run Next
```sh
asp <language> query --from-hook direct-source-read ...
```

## Agent Flow
Follow this flow after the hook recovery command gives you a frontier or exact locator.

1. Start from the language guide when you need the agent tool map: `asp <language> guide .`.
2. Map the project with `asp <language> search prime --view seeds .`.
3. Choose an owner, query, dependency, test, or syntax profile from the user intent and seed frontier.
4. When syntax location is needed, read `asp <language> query guide treesitter .`.
5. Execute `asp <language> query --treesitter-query '<pattern>' .` for a capture/frontier result.
6. Select one exact locator from the frontier.
7. Extract pure code with `asp <language> query --selector <path-or-range> --treesitter-query '<narrow-pattern>' --code .`.
8. Treat stdout from `query --code` as pure source code only.
9. Patch with `apply_patch` by default, or use provider `ast-patch` only when hook config enables semantic AST patching and a provider dry-run receipt validates the structural/mechanical edit.

## Rules
- Search is for discovery and should not inline code.
- Query with `--code` is for exact or unique code extraction.
- Tree-sitter query is the syntax base; native parser facts enrich the capture/frontier.
- Do not read full guide bodies unless the current step needs that guide.
#+end_src

The renderer must branch on the loaded hook config.  Generated Codex hook
configs set =experimental.semanticAstPatch.enabled = false=, so
=permissionDecisionReason= must tell agents to patch with =apply_patch= and not
route ordinary edits through provider =ast-patch= unless that option is enabled.

Project discovery must ignore hidden directories by default across Rust,
TypeScript, and Python providers.  This includes heavy local state directories
such as =.devenv=, =.direnv=, =.cache=, and generated tool caches.  A hidden
directory may be scanned only when project config explicitly allows it.  The
shared ASP config surface is =asp.toml=:

#+begin_src toml
[discovery]
ignoredDirNames = ["vendor"]
includeHiddenDirNames = [".agent-fixtures"]

[providers.rust]
enabled = true
binary = ".bin/rs-harness"

[providers.typescript]
enabled = false
#+end_src

Providers may keep language-specific policy config in their existing files, but
file discovery must merge the nearest =asp.toml= discovery settings before
walking source trees or running fd/rg prefilters.  =ignoredDirNames= is an
additive deny-list.  =includeHiddenDirNames= is the only supported exception to
the default hidden-directory prune rule.

Hook activation must also merge =asp.toml= provider settings before writing
=activation.json=.  =providers.<language>.enabled = false= removes that language
from hook source matching and from the =asp <language>= facade for the activated
project.  =providers.<language>.binary= pins the detected executable used for
provider runtime profiles; relative paths resolve from the project root.  The
activation keeps the manifest-owned =binary= identity stable and records the
detected executable in =providerCommandPrefix=, so manifest digest checks remain
stable while runtime output can show the actual bin path.

To keep guide prompts stable, =guide= is the low-frequency agent tool map.  It
should be generated from a typed provider catalog rather than hand-written
inline prose.  =guide= should not inline the full =query guide= or
=treesitter-query-guide= bodies; it should reference them:

#+begin_src text
[search-guide] protocol=search-guide.v1
profiles:
  overview-prime:
    command=search prime --view seeds
    returns=owner/query/dependency/test/finding handles
  owner-items:
    args=owner:path query:term
    returns=symbol/item frontier
  owner-tests:
    args=owner:path
    returns=test frontier
  query-deps:
    args=query:term dependency:pkg
    returns=dependency usage owners/tests
  path:
    args=from:typed-node to:typed-node
    returns=shortest relation path
  read-frontier:
    args=range:path@start:end
    returns=symbol/window frontier without code
avoid=raw-read,manual-window-scan,full-json,natural-language-intent
#+end_src

The paired prime output should expose the current handles and available profile
follow-ups, not code or full item lists:

#+begin_src text
[search-prime] root=<project-root> alg=budgeted-prime-frontier-v1 budget=handles:12
legend: ID=kind:role(value)!next; profiles P(args); frontier ID.next
profiles=owner-items(O,Q),owner-tests(O,T),query-deps(Q,D)
omit=items,blocks,code,full-test-list
avoid=raw-read,full-json,broad-fzf
#+end_src

=--treesitter-query= is precise syntax locate/capture/frontier.  Adding
=--code= to an exact selector turns it into pure code extraction.  Search and
read-plan flows must return locators/frontier rather than inline code.  Plain
=query --code= writes only code to stdout, and =--workspace <workspace-root>=
is the independent workspace override when a root must be specified.  JSON
artifacts, SQLite cache ids, and receipts are debugging/integration surfaces
and should not appear in the default guide output or non-JSON query output.
=python -m tools tree-sitter validate search-read-plan-frontier-contract= is
the package-local executable gate for this search/read-plan rule across Rust,
TypeScript, and Python.  =python -m tools tree-sitter validate exact-direct-read-contract= is the paired gate for exact selector-code:
non-=--code= output must be bounded =read-owner= source window or =read-plan=
frontier protocol output, must keep code omitted and provide executable read locators, and =--code= output must be pure source text.  The repository runs the
composite =python -m tools tree-sitter validate contracts= command from
=check-tree-sitter-query-contracts= in the local provider gate and from the CI
schema/provider gate after the root =asp= binary and TypeScript provider are
built.

Agent-facing guide acceptance:

#+begin_src text
guide-main        command="asp <language> guide ." lowFrequency=true inlineSubguides=false
guide-help        command="asp <language> guide --help ." output=help code=false json=false
search-prime      output=handles/profiles/frontier code=false json=false
syntax-locate     command="query --treesitter-query <pattern>" output=locator/frontier code=false
syntax-code       command="query --selector <exact> --treesitter-query <pattern> --code" output=pure-code
hook-recovery     output=markdown-prompt runNext=query-from-hook detectedBinaries=runtime-profile
debug-surfaces    json,receipts,cachePaths,artifactIds hiddenFromDefault=true
retired-routes    agent-prefixed-guide,search-wrapper-hook-query,ts-query,syntax-query rejectedFromGuides=true
#+end_src

The guide acceptance rule is intentionally prompt-facing.  It does not require
every provider to emit identical internal diagnostics, but it does require that
the command text copied by an agent is stable, low-token, and executable through
the =asp <language>= facade.  Default guide output must not include full
schema bodies, cache inspection recipes, or multiple language-specific guide
bodies.  It should point to specialized guides only when the current decision
needs them.

Schema evolution rules:

#+begin_src text
Additive v1 change     add optional fields, enum members, or descriptors
Breaking change        new schema file and schemaVersion, e.g. *.v2.schema.json
Provider adoption      update package-local copies and sync tests together
Registry advertisement advertise every emitted schema id/version/path
Client behavior        accept known schema versions; reject unknown versions with a compact diagnostic
#+end_src

Never change the meaning of a v1 field in place.  If an agent could interpret
old and new packets differently, the change is breaking and needs a new schema
version plus registry advertisement.

Registry identity fields:

#+begin_src text
registryId      = agent.semantic-protocols.semantic-language-registry
registryVersion = 1
protocolId      = agent.semantic-protocols.semantic-language
protocolVersion = 1
languageId      = rust | typescript | julia | python | ...
providerId      = rs-harness | ts-harness | jl-harness | ...
binary          = rs-harness | ts-harness | jl-harness | ...
namespace       = agent.semantic-protocols.languages.<languageId>.<providerId>
methods         = search/prime | search/dependency | check/full | agent/doctor | ...
methodDescriptors = method + command + view/input + output modes + schemas
#+end_src

=methods= is the callable truth.  The shared packet schema can contain a
cross-language =view= enum, but that enum does not advertise provider
capability.  Agents should only call methods present in the registry and use
=methodDescriptors= to discover the command grammar: whether a method is a
=search= view, which schema it emits, whether it requires one query positional,
whether it consumes stdin, whether =--package= is accepted, and whether
compact/JSON output is supported.  A =search/*= descriptor must carry its
=view=, emitted schema ids, and CLI input semantics; a =check/*= descriptor must
not pretend to be a search view; an =agent/*= descriptor only covers machine
surfaces such as =agent/doctor= or =agent/registry= and only requires
=outputSchemaIds= when =supportsJson=true=.  Text-only helper methods such as
=guide= can be registry-described without inventing a JSON schema they do not
emit.

A =query/*= descriptor must use =command="query"=, must not carry a search
=view=, and must advertise the schema it emits.  For owner-local item lookup,
providers should expose a descriptor such as =query/owner-items= with
=input="owner-path"=, =requiredOptions=["--term"]=,
=outputSchemaIds=["agent.semantic-protocols.semantic-query-packet"]=,
=querySetScopes=["owner"]=, and =outputModes= such as =compact=, =json=,
=code=, and =names=.  That makes query repair discoverable through
=agent doctor --json= rather than hidden in hook prose.

When a search view embeds a shared sub-schema inside the search packet, the
descriptor must advertise both the packet schema and that sub-schema.  For
example, =search/public-external-types= emits the stable
=semantic-search-packet= envelope and may populate =typeSurfaces= entries owned
by =semantic-type-surface=, so providers list both schema ids in
=outputSchemaIds=.  This keeps client validation and cache keys contract-visible
without turning provider-local type systems into shared protocol fields.

Registry invariants are deliberately strict:

#+begin_src text
languageId   stable lowercase source language id, not a file extension list
providerId   stable provider implementation id, e.g. ts-harness
binary       exact executable name the agent should invoke
namespace    agent.semantic-protocols.languages.<languageId>.<providerId>
methods      complete callable set
descriptors  exactly one descriptor for every method, no extras, no duplicates
#+end_src

Providers advertise exactly one semantic-language identity in the registry.
Extra local entry points, if a package has any, are outside the protocol and
must not appear in registry output, generated guides, schemas, or RFC examples.
For TypeScript, the only advertised provider and binary is =ts-harness=.

The descriptor command must match the method prefix.  For =search/<view>=,
=view= must be the method suffix, =outputSchemaIds= must include
=agent.semantic-protocols.semantic-search-packet=, and the descriptor owns CLI
input semantics such as =requiresQuery=, =acceptsStdin=, and
=supportsPackageScope=.  CLI parsers should consume this provider metadata
instead of maintaining a second hard-coded view registry.  Each provider's
tests should compare =methods= and =methodDescriptors= as sets and reject
duplicate descriptor methods.

Descriptor grammar is intentionally small in v1:

#+begin_src text
command              search | check | agent
view                 search method suffix only
requiresQuery         exactly one query positional before PROJECT_ROOT
acceptsStdin          stdin is part of the method input
supportsPackageScope  --package <package-id> is accepted
supportsQuerySet      --query-set <term> may replace the single query positional
acceptedQuerySetSelectors exact-set | prefix-set | stdin-path-set
querySetScopes        project | package | owner contexts accepted by --query-set
acceptedPipes         final-only pipe names accepted after the method query
capabilities          direct searchable surfaces for this method
ingestRequiredFor     surfaces that must route through search ingest
fallbacks             bounded provider fallbacks for weak or missing query hits
clients               agent clients or hooks with specialized command shapes
requiredOptions       options required by the specialized input shape
input                 compact human-readable grammar for non-default input
supportsJson          --json is accepted
supportsCompact       compact text is accepted/default
outputModes           descriptor-advertised output modes, e.g. compact, json, code, names, outline
outputSchemaIds       JSON schemas emitted when supportsJson=true
#+end_src

If a provider needs additional public controls, they must be added to the
descriptor schema first.  Provider-private flags may exist for local debugging,
but they are not semantic-language protocol until they are registry-described.
This keeps agent clients from learning accidental flags or inferring CLI shape
from the shared packet =view= enum.

Search capability descriptors answer the two questions agents need before they
spend tokens expanding context:

#+begin_src text
capabilities      what the provider can search directly
ingestRequiredFor what needs rg/fd/external expansion then search ingest
fallbacks         what bounded provider-owned recovery can run after a miss
#+end_src

The common registry schema owns only the descriptor shape:
=languageId=, =namespace=, and =name=.  It must not maintain one global enum for
TypeScript, Rust, Python, Julia, and JavaScript capability names.  Each language
harness owns its language-specific capability schema under that language
repository's =schemas/= directory and can advertise that schema in its provider
registry.

The protocol repository may keep language-specific schema templates under root
=schemas/=, for example
=schemas/typescript-semantic-capabilities-template.v1.schema.json= and
=schemas/python-semantic-capabilities-template.v1.schema.json=.  Those templates
document the expected active schema shape and vocabulary, but the provider's own
repository still carries the active schema it advertises and validates in CI.

For example, a TypeScript provider can advertise
=languageId=typescript namespace=typescript name=parser-visible-source-text-search=
for =search/fzf= while also advertising TypeScript-scoped
=ingestRequiredFor= records for docs text, schema JSON, generated artifacts,
and other non-parser text.  Agents should use that registry fact before reading
packet notes or running external search.

Minimal multi-pipe support is also registry-described.  A provider advertises
=acceptedPipes= on the source method, for example TypeScript =search/fzf=
can advertise =owner,tests= and accept:

#+begin_src shell
ts-harness search fzf OrderStatus owner tests .
#+end_src

The output remains one final packet; intermediate pipe packets are not rendered.

Search packet identity fields:

#+begin_src text
schemaId        = agent.semantic-protocols.semantic-search-packet
schemaVersion   = 1
protocolId      = agent.semantic-protocols.semantic-language
protocolVersion = 1
languageId      = rust | typescript | julia | python | ...
providerId      = rs-harness | ts-harness | jl-harness | ...
binary          = rs-harness | ts-harness | jl-harness | ...
namespace       = agent.semantic-protocols.languages.<languageId>.<providerId>
method          = search/workspace | search/prime | search/dependency | search/deps | ...
view            = workspace | prime | owner | dependency | deps | symbol |
                  callsite | import | cfg | patterns | pattern | docs |
                  api | public-external-types | tests | text | ingest
#+end_src

The schema owns the stable packet envelope: =header=, =packages=, =nodes=,
=edges=, =owners=, =items=, =hits=, =findings=, =nextActions=, =notes=, and
optional =inputDetection=.  Optional =typeSurfaces= carries shared public API
and dependency type surface facts when a view has parser-owned type evidence.
Language-specific facts must go under typed
=fields= maps instead of changing the envelope.  Search views are consumers of
native parser facts, not substitutes for them: TypeScript package dependency
facts come from the TypeScript JSON AST for =package.json=, source usage comes
from TypeScript import-resolution facts, Rust dependency facts come from Cargo
metadata/TOML parsing, Julia facts come from JuliaSyntax and project metadata,
and Python facts come from =ast= plus project metadata.  This lets TypeScript,
Rust, Julia, Python, and later harnesses preserve compiler-native facts while
giving agent clients one JSON shape to validate and cache.

Compact text remains the default prompt surface.  JSON is the cross-language
artifact and test contract.

Semantic handles are the shared layer for non-code facts that agents still need
to query precisely.  They cover stable things that are not necessarily parser
items: policy rule ids, schema fixtures, test cases, config keys, CLI command
surfaces, dependency API handles, and provider capability names.  The
=schemas/semantic-handle.v1.schema.json= contract owns the cross-language
fields: stable id, kind, source, title, aliases, labels, owner path,
implementation owner, test paths, locations, relations, and provider-owned
=fields=.  Search packets and query packets can embed these facts through
optional =semanticHandles=.

This is deliberately not a Python-only policy shortcut.  Rust harness policy
ids, TypeScript schema fixture names, Python rule ids, Julia package checks, and
future provider-specific facts should all use the same handle shape when the
fact is stable and useful as an agent query target.  Provider implementations
remain language-native; the shared protocol only makes the handle observable.

Policy-rule lookup is the motivating example:

#+begin_src shell
<lang-harness> search policy <rule-id-or-alias> owner tests --view seeds .
#+end_src

Expected compact facts:

#+begin_example
|handle kind=policy-rule id=PY-PROJ-R001 title="src layout" aliases=src-layout,packaged-project-layout owner=src/python_lang_project_harness/_project_policy_catalog.py implementation=src/python_lang_project_harness/_project_policy_layout.py tests=tests/unit/harness/project_policy/test_layout.py
|next owner:src/python_lang_project_harness/_project_policy_layout.py,tests:tests/unit/harness/project_policy/test_layout.py
#+end_example

Owner item query should not pretend these ids are function names.  If a selected
owner is a catalog, the provider may expose a handle-scoped pipe or query mode:

#+begin_src shell
<lang-harness> search owner <catalog-path> handles --query <rule-id-or-alias> .
#+end_src

That output should return =semanticHandles= first and code only when the agent
asks for the implementation owner or exact item.  For broad owner-local
discovery such as =test=, =minimal_packet=, or =parse_=, providers should return
=outputMode=names= or =outline= before code windows.

Type surface facts are the first shared sub-schema extracted from the search
envelope.  =schemas/semantic-type-surface.v1.schema.json= owns only the
language-neutral facts that an agent can use across providers: type name, kind,
role, owner path, visibility, member shape, external origin, and version scope.
It intentionally does not model full Rust, TypeScript, Python, Julia, or future
provider type systems.  Language-specific type-checker details remain
provider-owned =fields= facts or provider-local schemas.  Search packets can
embed these shared facts through optional =typeSurfaces=, primarily for
=search api=, =search public-external-types=, and parser-native query outputs
that need contract-visible type surfaces instead of stringly =fields= only.

This mirrors the useful part of Language Server Protocol naming discipline:
=protocolId= identifies the semantic-language protocol, =namespace= identifies
the concrete provider method space, =languageId= identifies the source
language, =providerId= identifies the implementation, and =binary= names the
executable.  Agent clients should route by registry identity and method, not by
hard-coded platform-specific argument structs.
