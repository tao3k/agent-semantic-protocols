* One-sentence summary

=agent-coding-quality.v1= is the cross-language policy canon for code that an
agent can reliably locate, explain, edit, and verify: language providers keep
parser authority, while this repository owns the shared signal names, evidence
model, severity promotion policy, and provider adaptation sequence.

* Problem

The current policy vocabulary is too narrow if it stops at control-flow shape.
Deep nesting, literal dispatch chains, broad linear functions, and manual loops
are valuable early signals, but they are only one part of the larger agent
quality problem.

The target is not aesthetic lint.  A finding should help the next agent recover
the engineering state of the code:

#+begin_example
owner -> intent -> input domain -> decision model -> algorithm phase
-> failure/resource/concurrency boundary -> verification evidence
#+end_example

If a policy says only "function has high complexity", the agent still has to
infer the hidden predicate stack, the domain model, the error boundary, and the
safe repair path.  This RFC makes those concepts schema-visible as
language-neutral quality signals.

* Plan audit

The proposed direction is correct: the first layer should be named
=agent-coding-quality=, not =control-flow policy=.  Control-flow becomes the
first high-value signal domain because Rust, Python, and TypeScript already
have parser surfaces that can prove enough facts for useful findings.

The plan needs three corrections before implementation:

1. /Separate canon from findings./  The shared RFC and schema define signal
   identity, evidence shape, risk wording, repair recipes, and promotion rules.
   Provider findings remain provider-owned packets until each language maps its
   parser facts into the shared signal catalog.

2. /Treat severity as evidence-weighted./  Agent coding quality starts as
   =info= or advisory.  A signal becomes a blocking candidate only after
   parser facts, official language guidance, community practice, and local
   fixtures agree.  This keeps the policy out of subjective style wars and
   avoids encouraging config allowlists.

3. /Stage provider adaptation by readiness./  Python, Rust, and TypeScript
   already expose agent-policy or control-flow surfaces.  Julia has registry and
   snapshot surfaces, but the type-stability, mutation-contract, and
   function-barrier policy facts still need a provider-owned implementation
   pass.

* Scope boundary

This RFC owns:

- shared =agent-coding-quality.*= signal taxonomy
- the =schemas/agent-quality-signal.v1.schema.json= catalog shape
- language-neutral evidence kinds and severity promotion rules
- provider mapping requirements for Rust, Python, TypeScript, Julia, and future
  languages
- compact vocabulary that provider findings should use when they reference a
  shared signal

This RFC does not own:

- Rust project structure configuration, already owned by RFC 004
- provider-private parser rules or fixture implementation
- final blocking defaults for any language
- client feature flags such as multi-agent execution or tool parallelism
- raw source heuristics outside provider-owned parser facts

* Evidence model

Agent coding quality signals need three evidence layers.

** Official language guidance

Official manuals and language-owned guides define what "native quality" means
for each language:

- Python: explicit, readable, flat-enough code; =match/case= for finite pattern
  dispatch; comprehensions and generators for native collection transforms.
- Rust: exhaustive =match=, iterator pipelines for collection transforms,
  meaningful error types, type-visible domain models, and API clarity.
- TypeScript: control-flow should help compiler narrowing; discriminated
  unions and =never= exhaustiveness are first-class quality surfaces.
- Julia: functions clarify steps, inputs, and outputs; performance-sensitive
  code should be function-local, generic, type-stable, and explicit about
  mutation.

** Community engineering practice

Community practice defines long-lived engineering boundaries that are directly
useful to agents:

- Tokio: async/sync boundaries, IO-bound concurrency, message passing, and
  type-state examples keep runtime modes explicit.
- Effect: typed error channels, resource scopes, structured concurrency, and
  pattern matching make failure and dependency requirements visible.
- SciML: generic code, trait/interface boundaries, mutation contracts, domain
  errors, type-stability, and type-space tests define high-quality Julia code.

** Agent quality research

Agent and LLM-code quality papers support the core product requirement: passing
tests is not enough.  Agent-computer interfaces, localization -> repair ->
validation flows, static analysis feedback loops, code-smell evidence, and
skill-fit results all point to the same policy shape: findings should serve
localization, repair, and validation rather than produce generic style advice.

* Signal taxonomy

=agent-coding-quality.v1= starts with these domains:

| Domain              | Purpose                                           |
|---------------------+---------------------------------------------------|
| =control-flow=      | Make decisions, traversal, and phases recoverable |
| =native-idiom=      | Prefer language-native constructs agents know     |
| =domain-type=       | Encode finite domains and modes in visible types  |
| =error-resource=    | Surface failure, cleanup, and resource lifetime   |
| =concurrency=       | Make async, actor, and shared-state boundaries    |
| =owner-boundary=    | Keep module/API ownership visible                 |
| =verification=      | Attach test, fixture, and feedback evidence       |

Control-flow is the v1 rollout domain, not the whole canon.

* Control-flow v1 signals

** =control-flow.decision-stack=

Risk: the agent must retain nested predicates and terminal paths before editing
one branch.

Typical parser facts:

- max nesting depth
- branch count
- terminal else count
- match/switch/case count
- return or throw distribution

Repair recipes:

- guard clauses
- named predicates
- typed dispatch
- =match= / =case= / =switch= with exhaustiveness where supported
- smaller functions around one decision model

** =control-flow.traversal-knot=

Risk: traversal, filtering, accumulation, and error handling are interleaved,
so local edits can silently change algorithm meaning.

Typical parser facts:

- nested loop count
- loop-with-branch count
- accumulator mutation count
- early continue/break/return count
- error handling inside traversal

Repair recipes:

- split traversal, predicate, accumulator, and failure phases
- use iterator/comprehension/generator constructs when they preserve the domain
- move failure handling to a named boundary

** =control-flow.literal-dispatch-chain=

Risk: a finite domain is encoded as a string, number, or bool branch chain
without a visible domain model.

Typical parser facts:

- repeated literal comparisons
- branch chain length
- lack of enum/tag/discriminant/match construct
- public API parameter or field using bool/string semantic modes

Repair recipes:

- Rust enum plus exhaustive =match=
- Python =match/case= for finite patterns
- TypeScript discriminated union plus =never= exhaustiveness
- Julia dispatch, symbols, or explicit domain structs where appropriate
- dispatch table only when it preserves validation and error reporting

** =control-flow.broad-linear-phase=

Risk: a long function hides several algorithm phases in a flat sequence, so the
agent cannot isolate one repair boundary.

Typical parser facts:

- high statement count
- repeated local mutation groups
- multiple owner/API calls from unrelated phases
- mixed parse, validate, transform, render, and verify steps

Repair recipes:

- extract named phases
- keep public API shape stable while moving implementation steps behind private
  helpers
- attach phase-specific tests or fixture rows

** =native-idiom.manual-transform-loop=

Risk: hand-written loops encode a common map/filter/count/sum/collect transform
in mutable state, making off-by-one and state-reset edits likely.

Typical parser facts:

- loop with push/append/count/sum accumulator
- no intervening domain-specific side effect
- equivalent language-native transform exists

Repair recipes:

- Python comprehension/generator/any/all/sum
- Rust iterator pipeline
- TypeScript array iterator or Effect collection helper where appropriate
- Julia broadcast, map, filter, reduce, or generator expression when type
  stability is preserved

** =error-resource.hidden-boundary=

Risk: error, resource, or cleanup behavior is hidden inside control-flow shape
instead of being visible as an API or type boundary.

Typical parser facts:

- broad catch/except swallowing
- public unwrap/panic/throw without typed boundary
- acquire/release split across branches
- async or task state mixed with local branch logic

Repair recipes:

- typed error/result/effect channel
- domain-specific error type
- resource scope helper
- actor/message boundary for shared async state

* Finding vocabulary

Provider findings that map to this RFC should speak in the shared signal name
first, then language-specific evidence.

#+begin_example
agent-coding-quality/control-flow.decision-stack

Risk:
  Agent must retain nested predicates and terminal paths in hidden context.

Evidence:
  max_nesting_depth=4
  terminal_else_count=2
  match_count=0

Repair:
  Extract named predicates.
  Convert finite branch domain to match/discriminated union/enum.
  Split traversal and accumulator phases.
#+end_example

The shared wording is not a prompt-only convention.  The signal id must be
schema-visible so compact text, JSON artifacts, provider fixtures, and future
cache/replay rows can compare the same finding.

* Schema implications

=schemas/agent-quality-signal.v1.schema.json= is a catalog schema, not a
provider finding packet.

It defines:

- =signalId=, for example =control-flow.decision-stack=
- =domain=, such as =control-flow= or =domain-type=
- =agentRisk=, the agent-local failure mode
- =sourceEvidence[]=, with kinds such as =official-manual=,
  =community-practice=, =agent-quality-paper=, =parser-fact=,
  =self-fixture=, and =real-project-evidence=
- =parserFacts[]=, language-neutral fact keys that providers can map from
  native parser facts
- =repairRecipes[]=, stable repair vocabulary with optional language hints
- =defaultSeverity= and =promotable=
- =promotionPolicy=, the evidence threshold for stronger severity
- =providerMappings[]=, the staged readiness state per language/provider

Provider finding packets may later reference this catalog by =signalId=.  They
should not duplicate the catalog text or invent compatible aliases.

* Provider adaptation plan

** Phase 0: RFC and schema

This phase lands the shared RFC and =agent-quality-signal.v1= catalog shape.
No provider becomes blocking from this phase.

** Phase 1: package-local schema copies and registry descriptors

Each language provider should copy or reference the shared schema according to
its current package convention, then add registry metadata for the signal
domains it can emit.

Providers must distinguish:

- =ready=: parser facts already support the signal
- =partial=: related parser facts exist but need mapping
- =planned=: provider has syntax/parser access but no rule yet
- =blocked=: provider cannot prove the signal without a new parser capability

** Phase 2: first control-flow fixtures

Ready providers should add the five v1 fixtures before changing default policy
output.  Partial providers should add the subset they can prove from
provider-owned parser facts and keep the missing signal ids out of emitted
findings until the facts exist.  Planned providers should advertise readiness
only through registry/schema metadata and must not emit shared signal ids yet.

| Fixture                       | Signal                                    |
|-------------------------------+-------------------------------------------|
| decision stack                | =control-flow.decision-stack=             |
| traversal knot                | =control-flow.traversal-knot=             |
| literal dispatch chain        | =control-flow.literal-dispatch-chain=     |
| broad linear phase            | =control-flow.broad-linear-phase=         |
| manual native transform loop  | =native-idiom.manual-transform-loop=      |

Each emitted fixture must prove both parser facts and compact finding
vocabulary.  A provider does not become =ready= for a signal by copying the
catalog; it becomes ready only when a stable provider fixture proves the
language-owned facts and the shared signal vocabulary together.

** Phase 3: compact output convergence

Provider compact output should converge on the same signal ids and risk/repair
headings while preserving language-owned evidence values.  Compact text remains
the default agent surface; JSON is for validation, cache, and artifacts.

Ready and partial providers that emit a signal should include the shared
=agentQualitySignals= label in JSON-compatible finding metadata and should make
the same signal id visible in compact/snapshot coverage.  Planned providers
should expose the catalog in registry metadata only.

** Phase 4: real-project evidence and promotion

After fixtures pass, exercise the signals against representative real projects.
Capture command count, packet size, latency, repeated-trigger patterns, false
positives, missed facts, and repair usefulness.

The first ready-provider evidence pass is recorded in
=docs/30-39-research/31.23-agent-quality-real-project-evidence.org=.  It covers
TypeScript, Python, and Rust harness repositories through their public policy
check commands.  The record proves stable localization for ready providers, but
it intentionally does not promote any signal to blocking severity: TypeScript
needed a false-positive refinement before the final run, Python emitted a
28.58 MB JSON packet for one quality finding, and Rust policy findings make the
check command exit non-zero while still producing parseable JSON evidence.

Compact-output parity is recorded in
=docs/30-39-research/31.24-agent-quality-compact-parity-evidence.org=.  Ready
providers now expose the same shared signal ids in default compact text that
they expose in JSON =agentQualitySignals= labels.  The remaining promotion gap is
repair usefulness: a real-project pass must show that starting from a compact
signal line improves localization and edit choice before any signal can become a
blocking candidate.

The first repair-oriented pass is recorded in
=docs/30-39-research/31.25-agent-quality-repair-evidence.org=.  It starts from a
TypeScript compact =native-idiom.manual-transform-loop= line, removes the
parser-owned condition in =ownersForHits= and =uniqueOwners=, and reduces the
TypeScript real-project quality findings from 15 to 13.  This is positive repair
evidence for one provider and one signal; follow-up Python and Rust repair
passes extend this evidence before any severity promotion.

The Python and Rust follow-up repair passes are recorded in
=docs/30-39-research/31.26-agent-quality-python-rust-repair-evidence.org=.  The
Python pass starts from a compact =control-flow.broad-linear-phase= line and
reduces Python real-project quality findings from 1 to 0 by splitting
=run_protocol_cli= into named routing steps.  The Rust pass starts from a
compact =native-idiom.manual-transform-loop= line and reduces Rust real-project
quality findings from 4 to 0 by replacing the =semantic_fact_owners= push loop
with an iterator pipeline, moving the inline semantic-facts test into
=tests/unit=, and splitting the remaining traversal and collection-loop helpers.
These passes establish cross-provider repair usefulness, but broader repair
history is still required before any advisory signal becomes a blocking
candidate.

The final v1 implementation closure is recorded in
=docs/30-39-research/31.27-agent-quality-final-implementation-evidence.org=.
It closes the strict compact vocabulary requirement by rendering
=agent-coding-quality/<domain.signal>= in compact text while keeping raw
=signalId= values in JSON =agentQualitySignals= metadata, and it records the
Python root policy repair that removed the remaining module ownership blockers.

The final closure audit is recorded in
=docs/30-39-research/31.29-agent-quality-final-closure-evidence.org=.  It
records the schema-profile cleanup, the Rust full-test compile drift repair,
the Python full-policy clean run, and the final cross-provider gates for the v1
ready-provider rollout.

A signal may become a blocking candidate only when:

- an official/manual source supports the rule intent
- provider parser facts prove the local condition
- a fixture proves stable output
- a real-project run shows the finding improves localization, repair, or
  validation
- the provider has a narrow config/advice path rather than a broad allowlist

* Provider readiness audit

Current repository evidence shows uneven readiness.

** Python

Python is ready for Phase 2 control-flow mapping.  The provider exposes
agent-policy catalog and readability surfaces, including:

- =languages/python-lang-project-harness/src/python_lang_project_harness/_agent_policy_catalog.py=
- =languages/python-lang-project-harness/src/python_lang_project_harness/agent_readability/algorithm_shape.py=
- =languages/python-lang-project-harness/src/python_lang_parser/_control_flow.py=
- =languages/python-lang-project-harness/src/python_lang_parser/_native_collection_loops.py=

The existing =PY-AGENT-R009= and =PythonFunctionControlFlow= surfaces are the
seed for =control-flow.decision-stack=, =control-flow.traversal-knot=,
=control-flow.literal-dispatch-chain=, and =native-idiom.manual-transform-loop=.

** Rust

Rust is ready for Phase 2 control-flow mapping.  The provider exposes
agent-policy algorithm shape and native syntax control-flow surfaces:

- =languages/rust-lang-project-harness/src/rules/agent_policy/algorithm_shape.rs=
- =languages/rust-lang-project-harness/src/parser/native_syntax/control_flow.rs=

The first Rust mapping should prefer enum/match, iterator pipelines, typed
errors, domain newtypes, type-state, and explicit async boundaries.  RFC 004
continues to own structural/config policy; this RFC only owns shared signal
identity and cross-language wording.

** TypeScript

TypeScript is ready for the Phase 2 control-flow and native transform-loop
mapping.  The provider exposes agent-policy, function-shape, and
discriminated-union related surfaces:

- =languages/typescript-lang-project-harness/src/rules/agent_policy/pack.ts=
- =languages/typescript-lang-project-harness/src/rules/agent_policy/quality_signals.ts=
- =languages/typescript-lang-project-harness/src/parser/function-shape.ts=
- =languages/typescript-lang-project-harness/src/parser/native_syntax/control_flow.ts=
- =languages/typescript-lang-project-harness/src/cli/semantic-search/api.ts=

The first TypeScript mapping focuses on parser-owned control-flow narrowing
facts plus collection-transform loop facts: decision stacks, traversal knots,
literal dispatch chains, broad linear phases, and manual native transform
loops.  Error/resource signals remain partial until TypeScript exposes
provider-owned error/resource boundary facts; discriminated unions, =never=
exhaustiveness, =any= erasure, and Effect-specific resource facts should widen
the mapping only after those facts are provider-owned.

** Julia

Julia is planned, not ready for the same Phase 2 scope.  The provider has
registry and snapshot surfaces:

- =languages/JuliaLangProjectHarness.jl/src/agent_registry.jl=
- =languages/JuliaLangProjectHarness.jl/src/agent_snapshot.jl=

The first Julia work should add JuliaSyntax-derived facts for setup/kernel
separation, type-stable returns, variable type changes, mutation contracts
through =!= naming, macro-heavy public surfaces, generic indexing assumptions,
and global-state algorithm dependencies.  Until those parser facts exist, Julia
should advertise planned or partial mappings rather than emit shared signal ids.

* Severity policy

Default severity is advisory:

#+begin_example
official manual + parser fact + community practice = strong advisory
official manual + parser fact + paper risk = promotable advisory
official manual + community practice + paper risk + self-fixture =
  blocking candidate
#+end_example

Blocking is provider-local and opt-in until a real-project evidence pass proves
that the signal improves agent localization, repair, and validation without
creating broad allowlist pressure.

* Acceptance criteria

- =schemas/agent-quality-signal.v1.schema.json= validates as JSON Schema.
- RFC 013 names =agent-coding-quality.v1= and does not rename existing search,
  query, compact graph, or structural config contracts.
- RFC 013 defines at least the v1 control-flow signals listed above, and future
  catalog instances must validate against =agent-quality-signal.v1=.
- Provider mappings record readiness honestly: Python/Rust/TypeScript as ready
  or partial for v1 control-flow, Julia as planned until parser facts land.
- Compact finding vocabulary starts with
  =agent-coding-quality/<domain.signal>=.
- No provider emits a blocking finding solely from this RFC.

* Open questions

1. Should future provider finding packets embed =agentQualitySignals[]= inside
   =semantic-search-packet= / =semantic-query-packet=, or should policy checks
   keep a separate packet family?

2. Should severity promotion be controlled by provider config, root ASP
   config, or both?

3. Should Effect-specific TypeScript facts be a provider extension namespace or
   a general =error-resource= / =concurrency= mapping once enough projects use
   the same pattern?

4. Which real projects should be the first calibration targets for false
   positive and repair usefulness measurement?
