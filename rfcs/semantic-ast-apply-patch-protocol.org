#+TITLE: RFC: Semantic AST Apply Patch Protocol
#+AUTHOR: agent-semantic-protocols
#+DATE: 2026-06-02
#+RFC: 007
#+DEPENDS: rfcs/code-intelligence-lane.org (003), rfcs/cli-first-harness-ux.org (005), rfcs/semantic-query-projection-protocol.org (010)
#+SEE_ALSO: docs/30-39-research/31.13-ast-apply-patch-direction-report.org

* One-sentence summary

=agent-semantic-protocol= should expose an AST-backed patch verification
protocol for small structural edits.  Language providers own native AST
adapters, while the shared binary owns schema validation, receipts, hook
integration, and Codex-safe verification.

* Motivation

The current agent loop has a gap between understanding and editing:

#+begin_example
search/query finds the right owner and symbol
  -> compact code output omits source details
  -> agent guesses a text patch
  -> apply_patch may fail or produce syntax drift
#+end_example

Codex's text =apply_patch= is a strong fallback, but it is still text matching.
It cannot know whether a snippet is valid in a TypeScript argument list, a Rust
impl block, a Python class body, or a Julia module.  This project already has
language-native parsers.  The provider should own structural edits that can be
validated before any file content changes.

* Non-goals

- Do not replace =apply_patch= for arbitrary text edits.
- Do not make the LLM write full files.
- Do not expose raw AST dumps to the prompt.
- Do not require one shared AST implementation across languages.
- Do not let fuzzy search or compact outline output become mutation authority.

* Protocol roles

| Layer | Responsibility |
|-------+----------------|
| =search= | discover owners, symbols, tests, graph frontiers |
| =query= | return parser-owned compact outline, node locators, patch affordances |
| =asp ast-patch= | validate one structural edit intent and emit a receipt |
| provider AST adapter | resolve native AST nodes and parse operation snippets |
| =apply_patch= | text fallback, audit backend, or final diff application |
| =check= | verify project policy after the edit |

The critical split is:

#+begin_example
compact outline may be lossy
ast patch target must be lossless enough to validate
#+end_example

* Public CLI shape

The public agent-facing entrypoint is a shared binary, not one public AST patch
surface per language:

#+begin_src shell
asp ast-patch template --language <language> --owner <path> --read <path:start:end> --op <operation> [--snippet '<source>'] [--field <key=value> ...] > semantic-ast-patch.json
asp ast-patch verify --packet <ast-patch.json> .
asp ast-patch dry-run --packet <ast-patch.json> .
#+end_src

For the current shared Codex adapter, =asp ast-patch= is verification-only.  It
must not pretend to own every provider's mutation semantics.  A successful
shared dry-run proves that compact/query output is patchable and tells the agent
whether the packet is provider-native or only a controlled text fallback.
=template= is the agent-facing packet builder.  It exists so a hook denial can
tell the agent one command shape instead of causing another broad search for
=schemaId=, =target.read=, =expectedSnippet=, or other packet fields.

The normal source-edit flow is:

#+begin_src shell
asp <language> query --from-hook direct-source-read --selector <path:start:end> --code .
asp ast-patch template --language <language> --owner <path> --read <path:start:end> --op <operation> [--snippet '<exact source or inserted snippet>'] [--field <key=value> ...] > semantic-ast-patch.json
asp <language> ast-patch dry-run --packet semantic-ast-patch.json .
# If the receipt says mutationSource=provider-native:
asp <language> ast-patch apply --packet semantic-ast-patch.json .
# Codex apply_patch is only an explicit text fallback, never the default source-edit path.
asp <language> check --changed .
#+end_src

Trusted local provider runtimes expose:

#+begin_src shell
asp <language> ast-patch apply --packet <ast-patch.json> .
#+end_src

The first provider-native apply backend is intentionally narrower than the full
protocol surface: Rust may expose trusted local =replace_item= and
=split_owner_items= mutation after the provider parses selected syntax with
native Rust ASTs, verifies =target.read= against parser-owned item boundaries,
runs =rustfmt=, and reparses the formatted files.  =split_owner_items= reads the
selected owner items locally, emits the destination module, inserts the owner
module declaration, and carries =destinationPath= / =moduleName= as
operation-owned fields instead of asking the model to write a hunk.  Other
operations remain dry-run or verification-only until their parser-owned apply
semantics are implemented.

Language providers expose a narrow internal adapter surface behind the shared
binary.  This adapter may be a CLI subcommand, library API, or crate call, but
it is not the first thing agents learn:

#+begin_src shell
<lang-harness> ast-patch verify --packet <ast-patch.json> .
<lang-harness> ast-patch dry-run --packet <ast-patch.json> .
#+end_src

The shared binary normalizes provider discovery, schema validation, receipts,
failure kinds, and follow-up checks.

Provider-native dry-run adapters may emit receipts with
=capability=provider-ast-dry-run=.  They validate the request through the
language parser and report concrete changed ranges, but they must not mutate
files.  The Codex adapter still performs the final edit through =apply_patch=
after the receipt proves that compact/query evidence was precise enough.

* Real-project evidence artifacts

Provider implementation tests may use hermetic fake projects for stable CI, but
protocol claims about save-token compact output and native apply safety need
representative real-project evidence.  Those artifacts must be metadata-only:
they record repository identity, commit, query target, parser-owned
=patchSafety= target, compact/exact byte counts, semantic responsibilities, and
dry-run/temp-apply receipt events.  They must not store external project source
or full compact code payloads.

Rust provider real-project evidence uses
=rust-ast-patch-real-project-evidence.v1=.  It is intentionally Rust-specific
because the evidence includes Rust provider syntax labels such as
=save-token-rustfmt= and Rust =replace_item= apply receipts.  The shared
contract is still useful at the repository level: it makes evidence fixtures
schema-validated, keeps compact token-savings measurable, and distinguishes
non-mutating real-checkout dry-runs from temp-copy apply checks.

Real-checkout repro commands should take a crate root, a source path, a query
term, and optional match disambiguators:

#+begin_src shell
ASP_AST_PATCH_REAL_ROOT=/path/to/tokio/tokio \
ASP_AST_PATCH_REAL_PATH=src/runtime/builder.rs \
ASP_AST_PATCH_REAL_QUERY=Builder \
ASP_AST_PATCH_REAL_TARGET_KIND=impl \
ASP_AST_PATCH_REAL_APPLY_TEMP=1 \
cargo test --features cli,search \
  cli::ast_patch_scenarios::cli_ast_patch_real_checkout_query_target_dry_runs_from_env \
  -- --ignored
#+end_src

The implementation root should be a new Rust crate:

#+begin_example
crates/agent-semantic-protocol

├── command/ast_patch.rs
├── command/compact_verify.rs
├── protocol/
└── receipt/
#+end_example

=agent-semantic-hook= remains a first-class dependency of this crate.  It is

#+begin_src shell

#+end_src

This keeps the current Codex adapter stable while letting the public binary grow

=rust-lang-project-harness= policy/check surface the same way this repository

#+end_src

If the crate touches hook integration, also run the existing
=agent-semantic-hook= tests because the hook crate remains the implementation

* AST patch packet

The shared packet should be language-neutral at the envelope level and
provider-specific only where syntax requires it:

#+begin_src json
{
  "schemaId": "agent.semantic-protocols.semantic-ast-patch",
  "schemaVersion": "1",
  "languageId": "typescript",
  "projectRoot": ".",
  "target": {
    "ownerPath": "src/cli/semantic-search/render.ts",
    "locator": "src/cli/semantic-search/render.ts#fn:renderSemanticSearchSeedPacket#if:packet.searchSynthesis",
    "read": "src/cli/semantic-search/render.ts:120:180",
    "itemKind": "fn",
    "itemName": "renderSemanticSearchSeedPacket",
    "nodeKind": "if_statement",
    "role": "body"
  },
  "operation": {
    "op": "append_to_block",
    "snippet": "lines.push(renderSearchGraphLine(packet.searchSynthesis.graph));"
  },
  "precondition": {
    "targetHash": "sha256:...",
    "worktreeHash": "sha256:...",
    "parserVersion": "typescript:..."
  }
}
#+end_src

Line locations use the existing compact source-locator form =path:10:43=.
Do not introduce =startLine= / =endLine= fields in this protocol.  The
line-range locator is a read hint and audit coordinate, not mutation
authority.  The authoritative target is the parser-owned AST identity:
=ownerPath=, =locator=, =itemKind=, =itemName=, and provider-owned structural
roles.  If the line range is stale after an earlier patch or formatter pass,
the provider must reparse the current file and re-resolve the unique AST item
by identity before applying or dry-running the operation.

* Patch affordances

Compact query results should not merely expose a function outline.  They should
also expose supported structural operations for selected nodes:

#+begin_example
|patch target=src/cli/semantic-search/render.ts#fn:renderSemanticSearchSeedPacket read=src/cli/semantic-search/render.ts:120:180 ops=append_to_block,replace_call_arg,insert_before_statement patchable=true
#+end_example

When output is only a lossy outline, providers must say so:

#+begin_example
|patch target=... patchable=false reason=lossy-outline next=query --patch-affordances
#+end_example

The shared =semantic-query-packet.v1= contract exposes this as =patchSafety= at
packet and match scope.  Packet scope is the default safety level for the
query result; match scope overrides it for a concrete parser item.

| Level | Meaning | Agent behavior |
|-------+---------+----------------|
| =navigation-safe= | output only identifies owners, items, or follow-up searches | do not patch; issue a narrower query or exact read |
| =read-safe= | output is suitable for understanding and bounded source reads | do not patch from compact text; read the exact locator before editing |
| =patch-verify-safe= | compact output has a stable parser target and stale-check metadata | may build a =semantic-ast-patch= packet and run =ast-patch verify/dry-run= |
| =ast-patch-safe= | provider can validate and later apply a structural edit | may use native AST patch apply when the adapter supports mutation |

=patch-verify-safe= and =ast-patch-safe= entries must include:

- =target.ownerPath=
- =target.locator=
- =target.read= using the =path:10:43= source-locator grammar
- =target.location.path= and =target.location.lineRange=
- =target.itemKind= and =target.itemName=, or an equivalent provider-owned
  identity under =target.fields=
- =sourceFingerprint=
- =parserVersion=
- non-empty =allowedOperations=

They must not include legacy =startLine= or =endLine= fields.  When compact
projection is lossy, the provider should emit =navigation-safe= or =read-safe=
with a =reason= and =nextAction= instead of letting an agent infer braces,
commas, or statement anchors from outline text.

* Mutation source contract

=ast-patch= is not only an =apply_patch= preflight.  For supported structural
edits, the supported workflow must remove handwritten hunks from the prompt:
the agent selects parser-owned identities and operation arguments, while the
provider reads source locally, applies the AST transform, formats, reparses, and
writes files.  Receipts distinguish three mutation sources:

| =mutationSource= | Meaning | Prompt/code policy |
|------------------+---------+--------------------|
| =provider-native= | provider owns source read, AST transform, formatter, reparse, and file write | no source hunk in prompt; success may be quiet |
| =agent-snippet= | agent supplies a bounded snippet, provider parses and normalizes it before mutation | snippet is allowed, but raw hunk context is not |
| =codex-text-fallback= | unsupported structural edit or prose/text edit goes through Codex text patching | explicit fallback only, never the preferred AST path |

Hook denials for protected source edits should route to =provider-native= when
the provider advertises the operation.  They should not tell agents to compose
manual hunks for supported AST operations.  The normal hook-guided source edit
flow is:

#+begin_src shell
asp <language> query --from-hook direct-source-read --selector <path:start:end> --code .
asp ast-patch template --language <language> --owner <path> --read <path:start:end> --op <operation> --field <key=value> > semantic-ast-patch.json
asp <language> ast-patch dry-run --packet semantic-ast-patch.json .
asp <language> ast-patch apply --packet semantic-ast-patch.json .
asp <language> check --changed .
#+end_src

For =provider-native= operations, =snippetRequired=false=, =codeInPrompt=false=,
and receipt output should summarize changed paths/ranges instead of printing a
full source diff.  This is the token-saving contract: source bytes are consumed
locally by the provider, not round-tripped through the model context.

* Operation vocabulary

Version 1 should stay small:

| Operation | Required target | Snippet parse context |
|-----------+-----------------+-----------------------|
| =append_to_block= | block-like node | statement |
| =insert_before_statement= | statement node | statement |
| =insert_after_statement= | statement node | statement |
| =replace_statement= | statement node | statement |
| =replace_expression= | expression node | expression |
| =replace_call_arg= | call expression | expression |
| =insert_import= | module/file root | import declaration |
| =remove_import= | import declaration | none |
| =remove_statement= | statement node | none |
| =remove_item= | item/module member node | none |
| =replace_item= | item/module member node | item |
| =split_owner_items= | contiguous top-level owner items | none; =destinationPath= and =moduleName= fields |

Providers may expose additional operations under provider-owned fields, but the
shared vocabulary above must be enough for cross-language agent guidance.

* Large mechanical edits

=ast-patch= is especially valuable when a human or LLM would otherwise repeat a
fragile text edit across many structurally identical sites.  Examples:

- delete a bounded set of debug/logging statements selected by parser-owned
  query results
- replace a family of statement or item nodes after a symbol/API migration
- insert or remove imports across a known owner/test frontier
- remove generated compatibility shims once the parser can prove every target
  is the same syntactic role
- split an oversized owner by moving parser-selected top-level items into a
  provider-created submodule without asking the agent to write source hunks

Large mechanical edits must be opt-in.  A packet that wants large-change safety
sets:

#+begin_src json
{
  "operation": {
    "op": "remove_statement",
    "expectedSnippet": "console.log(value);",
    "mechanicalKind": "bounded-multi-node",
    "maxEdits": 20,
    "allowLargeMechanicalEdit": true
  }
}
#+end_src

The shared verifier only proves the packet is structurally bounded and
audit-ready.  Provider-native AST dry-run should then resolve exact affected
nodes and emit a receipt with =mechanicalEditPlan=.  When the operation is
advertised as =provider-native=, provider apply is the preferred mutation path.
Codex =apply_patch= remains only an explicit =codex-text-fallback= for
unsupported structural operations or non-code text edits.

Do not use large mechanical AST patching for prose edits, formatting-only
changes, uncertain targets, or regex-shaped rewrites.  Unknown operations must
fail with =unsupported operation= instead of silently becoming text search and
replace.

* Validation stages

Every AST patch attempt has the same stages:

1. Resolve project root from native anchors such as =Cargo.toml=,
   =package.json=, or =pyproject.toml=, bounded by the repository root.
2. Resolve the target locator to exactly one parser node.  Treat the packet
   line range as a hint.  If the range no longer parses to the expected item,
   reparse the current file and resolve by parser-owned AST identity.  When
   the identity is unique, continue with the current range and record
   =target-locator-stale= plus =target-re-resolved= in the receipt
   verification list.  When the identity is missing or ambiguous, fail with
   =failureKind=target-locator-stale= and report candidate current locators
   when available.
3. Verify the target hash/precondition if supplied.
4. Parse the snippet in the target operation's grammar context when the
   operation has =snippetRequired=true=.  Provider-native operations with
   =snippetRequired=false= must derive changed syntax from parser-owned source
   nodes and operation fields instead.
5. Apply the structural transform to an in-memory syntax tree or provider-owned
   source projection.
6. Print or splice changed files from provider-owned syntax facts without
   requiring the agent to author a source hunk.
7. Run the declared language formatter capability when available.
8. Re-parse the formatter-normalized changed file.
9. Verify the target owner/symbol still resolves, unless the operation
   intentionally removes it.
10. Emit a receipt containing changed ranges, formatter outcome, and a textual
    diff.

If any stage fails, the provider must not mutate files.

The formatter stage is part of the safety model, not a display preference.  A
native AST edit may generate awkward but parseable text; the formatter converts
that text into the language or workspace-owned style before the provider
re-parses it.  This lets providers compress code aggressively for agent reading
without making the compact form writable.  The write path remains: exact
locator, native AST transform, formatter, reparse, receipt.
Formatter-normalized text is a stable comparison substrate, not the source of
truth.  The source of truth is always the current parser AST identity.  A
formatter may move spans or change line counts; the provider must use the
formatted text only after the AST identity has been resolved and must reparse
the formatter output before any mutation is committed.

Formatter execution belongs to the =asp= runtime environment, not the user's
ambient shell.  When =asp= invokes a provider directly or through the client
backend, it sets =PRJ_CACHE_HOME= to the project cache root and prepends
=${PRJ_CACHE_HOME}/agent-semantic-protocol/runtime/bin= to =PATH=.  The same
path is exposed as =ASP_RUNTIME_BIN_DIR=.  Rust =rustfmt=, TypeScript/Python
formatter shims, and future provider-owned command shims must be resolved from
that runtime bin directory before falling back to the ambient =PATH=.  This
keeps formatter evidence reproducible under =.cache/agent-semantic-protocol=
and prevents installed or user-shell formatter drift from changing AST patch
receipts.

The goal is not to show the agent fully formatted source.  Formatters were
designed for human readers and often add indentation, blank lines, wrapping,
and alignment that cost prompt tokens.  The provider uses the formatter because
the resulting spelling matches the code distribution that LLMs understand
well, then returns compact AST projection or a receipt over exact ranges.  In
other words: use formatter-normalized syntax as a semantic safety substrate,
but do not expose human-oriented formatting as the default agent prompt
surface.

Formatter style is intentionally opaque to the shared protocol.  The user or
workspace may choose a different formatter, version, line width, plugin set, or
import style, and the provider may not know that upfront.  The provider should
discover and record the formatter profile, then verify native AST preservation.
The protocol cares that formatter output reparses and aligns structurally; it
does not require a single canonical token layout across projects.

Formatter/core-AST disagreement is a hard failure.  A provider must not treat
alignment failure between the original AST and the formatter-normalized AST as
a recoverable fallback.  The whole point of the formatter stage is to prove
that the workspace formatter preserves the native AST target.  If node
identity, symbol path, structural role, or structural fingerprint alignment
fails, the AST patch attempt fails before any mutation and before any compact
projection is trusted.  Token fingerprints may be recorded as diagnostics, but
they cannot be the cross-workspace alignment contract.  The receipt should
record =failureKind=formatter-alignment-failed= or an equivalent structured
failure.

Formatter ownership is language-specific.  Rust uses =rustfmt=.  Python uses
=ruff format= for pure formatting; =ruff check --fix= is a stronger lint-fix or
organize-imports capability and must be recorded separately.  TypeScript and
JavaScript should use the workspace-declared formatter from =package.json= or a
    provider registry descriptor, not a hard-coded global default.  Julia uses
    provider-declared =JuliaFormatter.jl= capability behind =asp julia=.

* Receipt

The dry-run and apply result should share one receipt shape:

#+begin_src json
{
  "schemaId": "agent.semantic-protocols.semantic-ast-patch-receipt",
  "schemaVersion": "1",
  "status": "applied",
  "languageId": "typescript",
  "target": {
    "ownerPath": "src/cli/semantic-search/render.ts",
    "locator": "src/cli/semantic-search/render.ts#fn:renderSemanticSearchSeedPacket#if:packet.searchSynthesis",
    "read": "src/cli/semantic-search/render.ts:120:180"
  },
  "operation": "append_to_block",
  "changed": [
    {
      "path": "src/cli/semantic-search/render.ts",
      "before": "src/cli/semantic-search/render.ts:140:145",
      "after": "src/cli/semantic-search/render.ts:140:146"
    }
  ],
  "diff": "...",
  "verification": [
    "target-resolved",
    "snippet-parsed",
    "formatter-ran",
    "file-reparsed",
    "symbol-resolved"
  ],
  "next": "ts-harness check --changed ."
}
#+end_src

Failure receipts should be equally structured:

#+begin_src json
{
  "status": "failed",
  "failureKind": "snippet-parse-error",
  "message": "expected statement",
  "repair": {
    "next": "query",
    "selector": "src/cli/semantic-search/render.ts:120:180"
  }
}
#+end_src

* Language provider boundaries

** TypeScript

Use the TypeScript Compiler API for target resolution and snippet parsing.
The first implementation should support function/block statement insertion,
call argument replacement, and import insertion/removal.  Formatting must use
the workspace-declared formatter capability, such as a =package.json= script,
=prettier=, =biome=, or =oxfmt=.  The provider must disclose which formatter
was selected.

** Rust

Use the Rust provider's native parser stack.  The first implementation should
support item insertion in modules, statement insertion in blocks, and expression
replacement.  Rust formatting should run through =rustfmt= after the provider
prints or splices the changed syntax tree and before the final reparse.

** Python

Python should prefer a lossless or formatting-preserving parser layer when
available.  The built-in =ast= module is useful for validation but not enough for
round-tripping comments and whitespace.  Provider output must disclose whether
an operation is lossless.  Pure formatting should use =ruff format=; lint-fix,
import organization, or semantic cleanups must be separate capabilities.

** Julia

Julia AST patch support should use JuliaSyntax-derived facts and the same
facade/provider contract as search and query.  The compiled =aslp-julia-harness=
owns provider identity; =julia-project-harness.jl= is only a source-provider
development fallback.

* Interaction with RFC 003 mutation boundary

RFC 003 states that code-intelligence lanes plan and validate, while the client
runtime owns workspace mutation.  This RFC does not remove that boundary.

The shared Codex-compatible adapter remains verification/dry-run only, while a
trusted provider may own native mutation for operations it explicitly advertises:

#+begin_example
provider validates AST edit
  -> provider-native apply writes files when mutationSource=provider-native
  -> or client text fallback applies only an explicit codex-text-fallback packet
  -> provider check validates result
#+end_example

Only trusted local runtimes should expose =ast-patch apply=.  Rust now uses that
trusted-local boundary for =replace_item= and =split_owner_items=;
Codex-compatible hook flows may still choose dry-run plus client-owned mutation
only when the receipt marks an explicit text fallback.

* Interaction with Codex apply_patch

Codex =apply_patch= remains a useful reference and fallback.  The parts to
mirror are:

- typed parse stage before application
- separate parse and apply failures
- preimage/context validation
- deterministic replacement ordering
- affected-path delta receipt

The part not to mirror as the primary strategy is line-hunk editing for
syntax-shaped changes.  Parser-owned providers should use AST node identity and
grammar-context snippet parsing instead of asking the LLM to infer missing
braces, commas, sibling anchors, indentation, or other source layout from
compact search output.  Compact projections are valid edit planning evidence,
but they are not indentation-preserving patch preimages.  Provider-native
operations should keep syntax construction inside the provider.  When the final
mutation falls back to Codex =apply_patch=, its context must come from an exact
source window for the selected locator, not from compact code.

** Experimental hook interception

The Codex adapter may block source-file =apply_patch= calls before mutation
when the patch text targets an activated language provider source path.  This
is not an automatic text-patch-to-AST conversion.  The first safe behavior is:

- detect Codex direct =apply_patch= payloads and shell heredocs that contain
  =*** Begin Patch= / =*** End Patch= hunks;
- extract =Add File=, =Delete File=, =Update File=, and =Move to= paths;
- if any extracted path matches an activated source selector, deny with
  =reasonKind=semantic-ast-patch-required=;
- return provider exact-read routes so the agent can validate the source
  preimage for the selected locator;
- instruct the agent to build a packet with =asp ast-patch template=, run
  provider =ast-patch dry-run=, and then run provider =ast-patch apply= when the
  receipt says =mutationSource=provider-native=;
- mention Codex =apply_patch= only for receipts that explicitly require a text
  fallback;
- allow non-source/prose patches to continue through ordinary Codex
  =apply_patch=.

This rule exists because compact AST projection intentionally removes layout
that text patch engines depend on.  A compact locator may authorize the next
semantic query, but it must not become a hunk preimage.  The hook therefore
guards the source-mutation boundary while RFC 007 evolves provider-native AST
patch dry-run and trusted AST patch apply.

* Acceptance gates

1. =asp hook install --client codex .= delegates to the
   existing =agent-semantic-hook= implementation.
2. =rs-harness check --changed crates/agent-semantic-protocol= passes after the
   new crate is introduced.
3. TypeScript dry-run can append one statement inside
   =renderSemanticSearchSeedPacket= from a compact query target without reading
   the full source file into the prompt.
4. A malformed TypeScript snippet returns =snippet-parse-error= and leaves the
   worktree unchanged.
5. A stale target hash returns =target-stale= and suggests a repair query.
6. =asp hook pre-tool --client codex= denies source-file Codex =apply_patch=
   payloads with =reasonKind=semantic-ast-patch-required=, prints an
   executable =asp ast-patch template= -> provider =ast-patch dry-run= command
   chain, and still allows non-source =README.md= patches.
7. Provider-native receipts can be applied with provider =ast-patch apply=;
   Codex =apply_patch= is present only for explicit text fallback receipts.
8. Rust and Python providers reject unsupported operations with structured
   =unsupported-operation= receipts rather than falling back to raw text edits.
