#+TITLE: RFC: CLI-First Harness UX — stable commands, local daemon, no MCP
#+AUTHOR: agent-semantic-protocols

#+RFC: 005
#+DEPENDS: rfcs/code-intelligence-lane.org (003), rfcs/structural-quality-config-policy.org (004)

* One-sentence summary

The harness CLI is the public protocol.  A local daemon is a private
acceleration cache.  MCP is explicitly excluded from the architecture.

* Design principle

#+begin_example
Agent client
  ↓ reads SKILL.org
hooks
  ↓ call fixed commands
ts-harness CLI
  ↓ optional acceleration
ts-harness local daemon / cache
  ↓
TypeScript parser facts / reasoning tree / compact diagnostics
#+end_example

Agent clients call stable CLI commands.  The server is a CLI implementation
detail — invisible to the agent, replaceable without contract changes.

* Why not MCP

MCP has three structural problems for harness:

1. /Protocol layer is too dynamic./  Tool discovery, schema negotiation, and
   runtime capability advertisement make the agent's dependency surface
   unpredictable.  Harness diagnostics should be deterministic — same
   project, same output.

2. /Failure modes are unstable./  When an MCP server crashes, restarts,
   version-mismatches, or times out, the agent receives transport-level
   errors that it cannot repair.  A CLI with stable exit codes has exactly 5
   failure states, all documented.

3. /It turns harness into a conversational-tool-ecosystem dependency./  The
   harness's value is /deterministic structural facts/ — parser-owned,
   project-scoped, policy-evaluated.  Embedding that behind a dynamic tool
   discovery layer adds indirection without adding precision.

The right direction is the opposite: shrink the surface, make it deterministic,
and keep parser-owned facts as the source of authority.

* What NOT to build: the anti-patterns

The CLI must not implement intent parsing, subagent planning, or task
routing.  These are LLM responsibilities.  Specifically, these commands
should never exist:

#+begin_src shell
# DO NOT BUILD:
rs-harness graph subagent-plan --intent "change dependency reqwest features"
rs-harness graph next --from synthesis
rs-harness explore nl "find ..."
rs-harness search ast --intent "find clone in loop"
#+end_src

Also: do NOT split exploration into =graph=, =cargo=, =explore= as separate
top-level commands.  To the agent client, everything is search — just
different views.

* CLI is the protocol: one search surface

Language-neutral examples use =<lang-harness>=.  The TypeScript implementation
ships this as =ts-harness=; Rust/Cargo-specific lanes use =rs-harness=.

The agent sees three top-level commands:

#+begin_src shell
<lang-harness> search <view> ...   # all exploration / location / evidence
<lang-harness> check ...           # policy / style / changed / full
<lang-harness> agent ...           # agent-client install and diagnostics
#+end_src

** =search= — the unified exploration surface

Every read-only exploration goes through =<lang-harness> search=.  Common views:

#+begin_src shell
<lang-harness> search prime

<lang-harness> search owner <path-or-owner>
<lang-harness> search dependency <crate-or-import-or-package>

<lang-harness> search symbol <name>
<lang-harness> search callsite <name>
<lang-harness> search import <root>
<lang-harness> search pattern <recipe>
<lang-harness> search tests <owner-or-path>
<lang-harness> search text <query>
#+end_src

=search owner= resolves semantic owners first, then parser-visible modules,
then existing project paths.  A parser-visible module that is not a reasoning
owner should return =source=parser-visible-module=, =parserOwner=false=, and
module facts such as role, layer, line count, validity, and diagnostics.  A path
outside the parser module set may return =source=path-only= and
=next=ingest:<path>= so agents can keep the path in the protocol instead of
falling back to raw shell output.

=search text= covers the provider's semantic text index, such as parser-visible
source text, owner paths, exports, symbols, or language-native docs when
implemented.  If docs, schema files, generated artifacts, or other non-parser
text are outside that index, the provider should return a =next=ingest:<query>=
action and the agent must route external candidates through =search ingest=.

Rust adds Cargo-native views under the same search namespace:

#+begin_src shell
rs-harness search workspace
rs-harness search targets
rs-harness search deps
rs-harness search deps <dep[/subpath][@version][::api]>
rs-harness search deps <dep[/subpath][@version][::api]> public-api
#+end_src

=search dependency= is a high-level composite view: native package/dependency
facts + source usage + owner graph in one compact output.  For Rust,
=search deps= is the practical dependency-version API usage view: manifest
version/feature/target facts and local API usage owners for the version range
declared by the current workspace.  If a query asks for =dep@version= that does
not match the workspace declaration, the packet must mark it as
=versionScope=external= and avoid presenting current owner usage as evidence
for that external version.  Current workspace versions are reported as
=currentWorkspaceVersion= when a lockfile or manifest exact version proves the
value.  For current workspace versions,
=::api= narrows owner hits to dependency owners that also mention the API token.
=search prime= is the low-token entry point with graph structure.

** Multi-pipe: deterministic packet composition

A provider can optionally chain multiple pipe stages in one invocation.  This
is an RFC target for providers that can keep the packet final-only and
deterministic; agents must still treat the registry's =methods= and
=methodDescriptors= as the callable truth:

#+begin_src shell
<lang-harness> search <source> <pipe1> <pipe2> ...
rg -n "<query>" <roots> | <lang-harness> search ingest <pipe1> <pipe2> ...
#+end_src

Source stages (produce the initial packet):

#+begin_src text
prime                 project reasoning-tree
dependency <dep>      dependency fact + source usage + owner graph
features <feature>    feature -> optional-dep -> target cfg chain
deps <dep>            native manifest dependency facts
symbol <symbol>       symbol definition + callsites
callsite <symbol>     all usage sites
pattern <recipe>      structural recipe search
ingest                stdin auto-sniff (rg -n / fd / git diff)
#+end_src

Candidate pipe stages (enrich the packet, all optional):

#+begin_src text
owners      resolve to owner graph nodes
items       resolve to intra-file items
tests       find owned tests/benches/examples
examples    find owned examples
benches     find owned benches
docs        attach API docs facts (kind, signature, feature, async, ...)
public-api  mark whether dependency types leak into public API
cfg         attach cfg / target facts
features    expand feature graph
dependents  who depends on this owner/dep
#+end_src

Each pipe is a deterministic transform.  The CLI renders only the final
packet.  The agent client selects the pipe combination based on the task; the CLI
never infers intent.

Examples:

#+begin_src shell
# Dependency change with docs and test coverage:
rs-harness search dependency ignore items docs tests --package file-search

# Feature/cfg change across owners and docs:
rs-harness search features io-uring cfg owners tests docs --package tokio

# Public API leakage check:
rs-harness search dependency bytes public-api docs tests --package tokio

# External candidate stream with grouping:
rg -n "require_git|git_ignore|parents" agent-rs/file-search | rs-harness search ingest items tests
#+end_src

When a provider advertises diagnostic tracing, it should return a compact
stage list rather than full intermediate packets:

#+begin_src text
[search-trace] source=dependency query=ignore pipes=items,docs,tests
|stage dependency cargo=1 owners=1
|stage items owners=1 items=4
|stage docs docs=1
|stage tests tests=1
#+end_src

Internal model:

#+begin_src rust
struct SearchPacket {
    header: SearchHeader,
    packages: Vec<PackageFact>,
    cargo: Vec<CargoFact>,
    owners: Vec<OwnerFact>,
    items: Vec<ItemFact>,
    docs: Vec<DocsFact>,
    tests: Vec<TestFact>,
    edges: Vec<EdgeFact>,
    next: Vec<NextAction>,
}

trait SearchPipe {
    fn apply(&self, ctx: &SearchContext, packet: SearchPacket) -> SearchPacket;
}
#+end_src

** Wire contract: semantic language registry + shared =semantic-search= JSON Schema

Every provider first advertises through the semantic language registry, then
maps native search packets to the shared =semantic-search= wire contract when
=--json= is requested.  The registry and packet schemas live at:

#+begin_src text
schemas/semantic-language-registry.v1.schema.json
schemas/semantic-search-packet.v1.schema.json
#+end_src

The protocol repository's =schemas/= directory is the source of truth.
Provider packages that run their own CI from independent checkouts may carry
package-local copies at the same relative paths.  Their tests should validate
against the package-local files, and monorepo/submodule checkouts should compare
the package-local copies with the repository source schemas to prevent drift.

Schema evolution rules:

#+begin_src text
Additive v1 change     add optional fields, enum members, or descriptors
Breaking change        new schema file and schemaVersion, e.g. *.v2.schema.json
Provider adoption      update package-local copies and sync tests together
Registry advertisement advertise every emitted schema id/version/path
Client behavior        accept known schema versions; reject unknown versions with a compact diagnostic
#+end_src

Never change the meaning of a v1 field in place.  If an agent could interpret
old and new packets differently, the change is breaking and needs a new schema
version plus registry advertisement.

Registry identity fields:

#+begin_src text
registryId      = agent.semantic-protocols.semantic-language-registry
registryVersion = 1
protocolId      = agent.semantic-protocols.semantic-language
protocolVersion = 1
languageId      = rust | typescript | julia | python | ...
providerId      = rs-harness | ts-harness | jl-harness | ...
binary          = rs-harness | ts-harness | jl-harness | ...
namespace       = agent.semantic-protocols.languages.<languageId>.<providerId>
methods         = search/prime | search/dependency | check/full | agent/doctor | ...
methodDescriptors = method + command + view + outputSchemaIds + output modes
#+end_src

=methods= is the callable truth.  The shared packet schema can contain a
cross-language =view= enum, but that enum does not advertise provider
capability.  Agents should only call methods present in the registry and use
=methodDescriptors= to discover the command grammar: whether a method is a
=search= view, which schema it emits, whether it requires one query positional,
whether it consumes stdin, whether =--package= is accepted, and whether
compact/JSON output is supported.  A =search/*= descriptor must carry its
=view=, emitted schema ids, and CLI input semantics; a =check/*= descriptor must
not pretend to be a search view; an =agent/*= descriptor can advertise
registry-shaped JSON output.

Registry invariants are deliberately strict:

#+begin_src text
languageId   stable lowercase source language id, not a file extension list
providerId   stable provider implementation id, e.g. ts-harness
binary       exact executable name the agent should invoke
namespace    agent.semantic-protocols.languages.<languageId>.<providerId>
methods      complete callable set
descriptors  exactly one descriptor for every method, no extras, no duplicates
#+end_src

Providers advertise exactly one semantic-language identity in the registry.
Extra local entry points, if a package has any, are outside the protocol and
must not appear in registry output, generated guides, schemas, or RFC examples.
For TypeScript, the only advertised provider and binary is =ts-harness=.

The descriptor command must match the method prefix.  For =search/<view>=,
=view= must be the method suffix, =outputSchemaIds= must include
=agent.semantic-protocols.semantic-search-packet=, and the descriptor owns CLI
input semantics such as =requiresQuery=, =acceptsStdin=, and
=supportsPackageScope=.  CLI parsers should consume this provider metadata
instead of maintaining a second hard-coded view registry.  Each provider's
tests should compare =methods= and =methodDescriptors= as sets and reject
duplicate descriptor methods.

Descriptor grammar is intentionally small in v1:

#+begin_src text
command              search | check | agent
view                 search method suffix only
requiresQuery         exactly one query positional before PROJECT_ROOT
acceptsStdin          stdin is part of the method input
supportsPackageScope  --package <package-id> is accepted
acceptedPipes         final-only pipe names accepted after the method query
capabilities          direct searchable surfaces for this method
ingestRequiredFor     surfaces that must route through search ingest
supportsJson          --json is accepted
supportsCompact       compact text is accepted/default
outputSchemaIds       JSON schemas emitted by the method
#+end_src

If a provider needs additional public controls, they must be added to the
descriptor schema first.  Provider-private flags may exist for local debugging,
but they are not semantic-language protocol until they are registry-described.
This keeps agent clients from learning accidental flags or inferring CLI shape
from the shared packet =view= enum.

Search capability descriptors answer the two questions agents need before they
spend tokens expanding context:

#+begin_src text
capabilities      what the provider can search directly
ingestRequiredFor what needs rg/fd/external expansion then search ingest
#+end_src

The common registry schema owns only the descriptor shape:
=languageId=, =namespace=, and =name=.  It must not maintain one global enum for
TypeScript, Rust, Python, Julia, and JavaScript capability names.  Each language
harness owns its language-specific capability schema under that language
repository's =schemas/= directory and can advertise that schema in its provider
registry.

The protocol repository may keep language-specific schema templates under root
=schemas/=, for example
=schemas/typescript-semantic-capabilities-template.v1.schema.json=.  Those
templates document the expected active schema shape and vocabulary, but the
provider's own repository still carries the active schema it advertises and
validates in CI.

For example, a TypeScript provider can advertise
=languageId=typescript namespace=typescript name=parser-visible-source-text-search=
for =search/text= while also advertising TypeScript-scoped
=ingestRequiredFor= records for docs text, schema JSON, generated artifacts,
and other non-parser text.  Agents should use that registry fact before reading
packet notes or running external search.

Minimal multi-pipe support is also registry-described.  A provider advertises
=acceptedPipes= on the source method, for example TypeScript =search/text=
can advertise =owner,tests= and accept:

#+begin_src shell
ts-harness search text OrderStatus owner tests .
#+end_src

The output remains one final packet; intermediate pipe packets are not rendered.

Search packet identity fields:

#+begin_src text
schemaId        = agent.semantic-protocols.semantic-search-packet
schemaVersion   = 1
protocolId      = agent.semantic-protocols.semantic-language
protocolVersion = 1
languageId      = rust | typescript | julia | python | ...
providerId      = rs-harness | ts-harness | jl-harness | ...
binary          = rs-harness | ts-harness | jl-harness | ...
namespace       = agent.semantic-protocols.languages.<languageId>.<providerId>
method          = search/workspace | search/prime | search/dependency | search/deps | ...
view            = workspace | prime | owner | dependency | deps | symbol |
                  callsite | import | cfg | patterns | pattern | docs |
                  api | public-external-types | tests | text | ingest
#+end_src

The schema owns the stable packet envelope: =header=, =packages=, =nodes=,
=edges=, =owners=, =items=, =hits=, =findings=, =nextActions=, =notes=, and
optional =inputDetection=.  Language-specific facts must go under typed
=fields= maps instead of changing the envelope.  Search views are consumers of
native parser facts, not substitutes for them: TypeScript package dependency
facts come from the TypeScript JSON AST for =package.json=, source usage comes
from TypeScript import-resolution facts, Rust dependency facts come from Cargo
metadata/TOML parsing, Julia facts come from JuliaSyntax and project metadata,
and Python facts come from =ast= plus project metadata.  This lets TypeScript,
Rust, Julia, Python, and later harnesses preserve compiler-native facts while
giving agent clients one JSON shape to validate and cache.

Compact text remains the default prompt surface.  JSON is the cross-language
artifact and test contract.

This mirrors the useful part of Language Server Protocol naming discipline:
=protocolId= identifies the semantic-language protocol, =namespace= identifies
the concrete provider method space, =languageId= identifies the source
language, =providerId= identifies the implementation, and =binary= names the
executable.  Agent clients should route by registry identity and method, not by
hard-coded platform-specific argument structs.

* The two-phase search model: deep prime → multi-pipe

Phase 1 starts with =<lang-harness> search workspace= only when the selected
root is a multi-package workspace.  That command is a router/index: it lists
packages, workspace-internal edges, external dependency axes, findings, and
next package candidates.  It must not deep-search source files.

Phase 1 then uses =<lang-harness> search prime= for the selected package or for
single-package projects.  Prime is still a purpose-built, deep but bounded graph
packet designed to give the LLM a package-level search map.  Multi-pipe is Phase
3+, used by subagents after the parent has selected package and axis.

#+begin_src text
Phase 1a: <lang-harness> search workspace   (workspace router, if applicable)
Phase 1b: <lang-harness> search prime       (package-level deep map)
Phase 2:  agent parent reasons about axes   (LLM task, not CLI)
Phase 3:  focused searches                  (dependency/symbol/owner/ingest)
Phase 4:  parent synthesis → edit plan      (LLM task, not CLI)
Phase 5:  <lang-harness> check --changed    (validation)
#+end_src

** What workspace covers (and what it does not)

Workspace is a package router, not a repository-wide search.  It outputs just
enough package routing facts for the agent to choose a package and then run
=search prime --package <package-id>=.  It never outputs raw hits, source
lines, docs prose, or full per-owner graphs:

#+begin_src text
workspace/package scale
workspace package list (ranked/capped)
workspace-internal package edges
external dependency axes across packages
finding groups by package
next hints (package/dependency axes, not hits)
#+end_src

Workspace routing invariants:

#+begin_src text
package id     root-relative package path; exact value accepted by --package
package name   native package name from package metadata, when present
package role   workspace-root | workspace-package | single-package
package config native config path, e.g. package.json, Cargo.toml, Project.toml
package edges  workspace-internal import/reference/member edges only
findings       package-scoped counts/groups, not full diagnostics
owners/items   empty in workspace packets
hits           empty in workspace packets
#+end_src

If a provider advertises =search/workspace= for a single-package project, it may
return one =single-package= row with =next=prime:.=.  Agent guides should still
skip workspace for known single-package roots and start with =search prime=.

Workspace output format:

#+begin_example
[search-workspace] mode=workspace-index package=@repo/root packages=12 shown=12 edge=18 external=7 find=3
|package . name=@repo/root role=workspace-root packages=12 patterns=packages/* dependencies=7 next=prime:.
|package packages/core name=@repo/core role=workspace-package pattern=packages/* type=module config=packages/core/tsconfig.json diagnostics=0 next=prime:packages/core
|package apps/web name=@repo/web role=workspace-package pattern=apps/* type=module config=apps/web/tsconfig.json diagnostics=0 next=prime:apps/web
|edge P:. -workspace-> P:packages/core importKind=type-import via=package-name package=@repo/core from=src/index.ts
|next prime:packages/core,prime:apps/web
#+end_example

** What prime covers (and what it does not)

Prime outputs package-level structure, signals, and next candidates.  It never outputs
raw hits, source lines, or docs prose:

#+begin_src text
native dependency / feature / cfg or package/config signals
owner graph (ranked, capped)
source_large owner warnings
public API owners
test/bench/example surfaces
finding groups
docs candidates
next hints (axes, not hits)
#+end_src

** Prime budget

Strict internal caps — not exposed as flags:

#+begin_src text
max_packages=8   max_owners=12     max_edges=16
max_features=8   max_findings=8    max_api_candidates=8
max_test_surfaces=8
#+end_src

Auto mode selection:

#+begin_src text
package within normal bounds   → mode=package (full graph)
single small package           → mode=package (full graph + extras)
#+end_src

** Owner ranking in prime

Owners are ranked, not listed alphabetically:

#+begin_src text
1. root / facade / interface
2. public API owners
3. owners with findings
4. owners with external dependency edges
5. source_large owners
6. owners with tests / benches
#+end_src

** What prime must NOT output

#+begin_src text
- no raw hits or callsite lists
- no source lines
- no docs prose
- no full dependency graph
- no per-file item listings (use source_large + next=items instead)
#+end_src

** How the agent client uses prime to build the subagent flow graph

Agent guide:

```md
## Phase 1: deep prime

Always start complex Rust tasks with:

`rs-harness search prime`

Read mode, package, feature, cfg, owner, large-owner, api-candidate,
test-surface, and next fields.

Do not spawn subagents before reading prime unless the user provides an
exact file and symbol.

## Phase 2: parent planning

Use prime output to choose one or more search axes:

- feature/cfg axis
- dependency/API docs axis
- owner/item axis
- public API/callsite axis
- tests/bench/examples axis
- harness finding axis

Spawn read-only subagents only after selecting axes.

Example: if prime shows `feature io-uring ... requires_cfg=tokio_unstable`:

feature subagent:
  `rs-harness search features io-uring cfg owners tests`
cfg subagent:
  `rs-harness search cfg tokio_unstable owners tests`
source subagent:
  `rg -n "io_uring|tokio_unstable" tokio/src tokio/tests | rs-harness search ingest items tests`
```

** =next= placement rule for prime

=next= on owner lines targets that owner implicitly.  Global =|next= only
for candidates without a clear owner context.  Never duplicate an
owner-specific next into the global =|next= line.

** Confidence modeling in prime

When facts cannot be confirmed from parser/Cargo data alone, prime uses
explicit confidence markers rather than guessing:

#+begin_src text
reason=public-api-carrier         confirmed from parser facts
reason=possible-public-api-carrier uncertain, needs public-api pipe
reason=likely-parser-owner        heuristic, needs owner pipe
public_external_allowed=true      confirmed from cargo metadata
source=workspace|path|crates-io|git  dependency origin
#+end_src

This prevents prime from misleading the agent client with false certainty on
cross-crate or macro-heavy codebases like rustc.  The =source= field on
=cargo= lines distinguishes workspace-internal from crates-io
dependencies, which affects search strategy and docs-source selection.

** Strong signal: =source_large=true=

When prime marks an owner as =source_large=true=, the agent client must NOT open the
file directly.  Instead:

```sh
rs-harness search owner <file> items
```

This rule prevents token explosion from files like =file-search/src/lib.rs=
(1192 lines, ~12 public items) or =tokio/src/runtime/mod.rs= (668 lines).

** =check= — policy evaluation

#+begin_src shell
<lang-harness> check --changed
<lang-harness> check --full
#+end_src

** =agent= — agent-client lifecycle

#+begin_src shell
<lang-harness> agent install
<lang-harness> agent doctor
<lang-harness> agent doctor --json
#+end_src

Installs/updates =SKILL.org=, hooks, and config into the configured agent
workspace.  =doctor= checks daemon health, protocol version, cache freshness,
and hook integrity.  =doctor --json= emits the semantic-language registry
contract for callable method discovery.

** Shared flags on all =search= views

#+begin_src text
                 default agent output (compact line protocol)
--json                    tool / IDE / flowhub output
#+end_src

If a daemon is running, the CLI internally routes through the daemon socket.
If no daemon is running, the CLI runs the native parser directly.  The agent
never knows which path was taken.

public contract:
  CLI args + stdout compact/json + exit code

private optimization:
  local daemon, TypeScript Program cache, file watcher, socket IPC

* Daemon failure matrix

| Daemon State          | CLI Behavior                          |
|-----------------------+---------------------------------------|
| running, cache fresh  | fast incremental result               |
| running, cache stale  | auto-rebuild cache, then respond      |
| not running           | direct parser run (same output)       |
| protocol mismatch     | deterministic upgrade message, exit 4 |
| crashed mid-request   | CLI retries once, then direct run     |

The agent never receives "server unavailable" as an error — only as a
=daemon=missing= diagnostic field when a provider chooses to expose daemon
status, which is informational.

* Compact format specification: line protocol + ASCII graph

=<lang-harness> search= output is a /line protocol/, not a Markdown report.
It inherits the existing Rust harness compact contract: stable tags, single
locator syntax, short action lines, minimal fact lines, no audit headers, no
prose.

Rules:

#+begin_src text
- no blank lines
- no Markdown headings
- no bullet indentation
- no source excerpts in default search packets
- no Help / Contract unless it is a check finding
- no prose paragraphs
- paths relative to project root
- every line independently readable
- every block default ≤ 12 lines
- no Unicode arrows, no box-drawing, no DOT/Mermaid/Graphviz
#+end_src

Line shape:

#+begin_src text
[search-kind] k=v k=v k=v          ← header line, always first
|node KIND:id k=v                   ← typed graph node
|edge KIND:id -kind[:label]-> KIND:id  ← ASCII edge, always " -kind-> "
|owner path k=v next=...             ← owner with inline next actions
|item name kind=... k=v next=...     ← intra-file item with inline next
|find RULE xN at=KIND:id            ← finding attached to node
|next kind:arg,kind:arg(scope=x)     ← global next actions (rare, only when
                                       no owner/item context exists)
|note k=v                           ← exceptional info (keep ≤ 1)
#+end_src

=|item= lines only appear when the request includes the =items= pipe or an
advertised provider method returns item facts by default.
=|next= is an inline field on owner/item/package/cargo lines, not a separate
line type.  A standalone =|next= line is only used when no owner/item
context exists to attach to.

Next actions use compact command fragments:

#+begin_src text
next=dependency:reqwest
next=callsite:create_order
next=pattern:public-error-boundary
next=text:timeout(scope=tests)
next=owner:src/api/orders.rs
next=tests
next=bench:parse
#+end_src

If the next action targets the current owner, the owner is implicit.  If it
targets a different owner, it is explicit: =next=owner:src/api/orders.rs=.

Node prefixes:

#+begin_src text
P:  package
O:  owner / module owner
C:  dependency / crate
T:  test / bench target
F:  finding group
S:  symbol
#+end_src

Edge kinds (all ASCII, no Unicode):

#+begin_src text
-mod->          module child edge
-path-mod->     #[path] mod edge
-include->      include! edge
-crate->        local owner dependency (same package)
-workspace->    workspace-internal dependency (different member)
-crates-io->    crates.io external dependency
-path-dep->     path-based external dependency
-ext->          external dependency (generic, when kind unresolved)
-test->         owned test / bench edge
-find->         finding at owner
-call->         symbol / callsite edge
-import->       import root edge
-reexport->     facade re-export edge
#+end_src

Distinguishing =workspace->= from =crates-io->= matters in monorepos like
rust-lang/rust and agent-client repos where cross-crate edges are workspace-internal
but still cross-package.

A provider-private diagnostic method, if advertised, also uses this line
protocol rather than prose:

[search-plan] view=dependency q=reqwest
|step deps dep=reqwest
|step native use_root=reqwest

* External candidate adapters: =search ingest=

When the agent client deliberately needs wider lexical or file-path candidates, it
calls =rg= or =fd= directly — but always pipes output back into
=rs-harness search ingest= for owner grouping, dedup, ranking, and compact
packet rendering.  Raw =rg= / =fd= output is never pasted into the chat.

There is exactly one ingest command — no sub-kinds, no =--query=:

#+begin_src shell

#+end_src

Shared controls (output / scope control only, no source declaration):

#+begin_src text
--json

#+end_src

** Auto-detection

#+begin_src text

first non-space char is "{"   → jsonl (rg --json)

line matches path:line:text   → rg-n / grep-n
line is existing project path → path-list

otherwise                     → unknown (compact error)

** Unified pipeline

#+begin_src text

** Recency-biased ranking

When a search view has multiple same-class candidates after semantic rank,
provider score, and deduplication, project-internal path recency is the next
tie-breaker.  Newer last-modified paths sort before older paths because agents
usually edit the files they are currently reasoning about.  This applies to
owner-grouped hits, ingest candidates, dependency usage owners, and equal-rank
prime owners.

Recency is only a ranking signal.  It must not replace semantic relevance,
public/API/root owner priority, exact-match scoring, or version-scope rules.
Providers only stat paths that resolve inside the selected project or package
root; missing or non-project mtimes fall back to deterministic path/name order.
Default compact output does not expose mtime fields.

** =search docs= / =search api=

Dependency API docs are first-class search objects — not prose blobs, but
native-parser API facts connected to local callsites, features, public API
leakage, and tests.  The Rust harness must treat its native parser as the
authority for local source shape; registry and rustdoc sources can enrich facts
only after parser-owned workspace/path facts are exhausted.

Baseline distinction: =search deps= owns local usage for the dependency version
range declared by the current workspace; =search docs= owns external/reference
API facts for an explicitly requested version.  A future composite docs+usage
packet should be added only after a provider can join those two fact sources
without hiding provenance; it is not part of the baseline schema or TypeScript
provider.

#+begin_src shell
rs-harness search docs tokio::runtime::Builder
rs-harness search docs bytes::Buf --package tokio
rs-harness search api tokio::sync::mpsc::Sender::send
rs-harness search public-external-types --package tokio
#+end_src

Distinction:

| Command                    | Returns                                          |
|----------------------------+--------------------------------------------------|
| =search docs=              | doc facts + API shape, minimal prose             |
| =search api=               | API shape only: kind, signature, async, unsafe,  |
|                            | feature, receiver, return, error, trait impls    |
| =search public-external-types=| external dependency types appearing in         |
|                            | public API signatures                            |

Data source priority:

#+begin_src text
1. workspace / path dependency source (native parser facts + rustdoc comments)
2. Cargo registry source (downloaded crate source)
3. local rustdoc JSON cache (target/doc/)
4. docs.rs rustdoc JSON (optional cache/download)
5. HTML docs (last resort, not default)
#+end_src

For Rust, MVP implementation starts with parser-native facts only:
=kind=, =signature=, =params=, =async=, =unsafe=, =receiver=, =return=, and
=error=.  The CLI renderer must not infer these from compact text, regex-only
matches, or prose docs.
This MVP includes public free functions, public inherent impl methods, and trait
impl methods, so receiver shape, impl type, and trait path are parser-owned.
The CLI renderer must project those facts without inferring association itself.

Implementation rule: every new Rust API fact lands first in the native parser
fact model with parser-level tests.  =rs-harness search= may only project those
facts into compact text or JSON; it is not the source of truth.

The header line declares the data source for trust calibration:

#+begin_src text
[search-docs] q=tokio::runtime::Builder crate=tokio source=native-parser
[search-docs] q=bytes@0.6.0::Buf crate=bytes requestedVersion=0.6.0 versionScope=external source=registry-source
[search-docs] q=ignore@0.4.23::WalkBuilder::require_git crate=ignore requestedVersion=0.4.23 versionScope=external source=docs-rs-json
#+end_src

If the external source is not available, the packet admits it instead of
presenting current workspace parser facts as external-version docs:

#+begin_src text
|note docsSource=registry-source missing=true
#+end_src

Dependency-prefixed docs/API queries such as =serde::Serialize= use
=source=registry-source= unless the provider has parser facts for a local
workspace/path dependency source.

Output format (API facts, not docs prose):

#+begin_example
[search-docs] q=fallible pkg=. docs=1 source=native-parser
|api src/lib.rs:11 kind=fn name=fallible next=owner:src/lib.rs source=native-parser docs=local-parser signature=fn(input:String)->anyhow::Result<Thing> params=input:String async=true unsafe=true receiver=- return=anyhow::Result<Thing> error=anyhow::Result impl=- trait=-
#+end_example

=search deps ignore@0.4.23::WalkBuilder::require_git= remains the local usage
entry point.  It may point to =search docs ignore::WalkBuilder::require_git=
when an external docs provider is available, but it must not require docs to
answer local usage.

=search docs= never outputs long prose by default.  Doc-comment excerpts are
not part of the baseline contract; providers can expose them only through an
advertised detail method or method descriptor field.

* Exit codes

Stable and documented:

| Code | Meaning                                          |
|------+--------------------------------------------------|
| 0    | clean / ok                                       |
| 1    | harness findings exist                           |
| 2    | invalid CLI usage                                |
| 3    | project / config / protocol error                |
| 4    | daemon required but unavailable                  |
|      | (developer-only diagnostic mode, not an agent path) |
| 5    | stale cache / retryable internal error           |

Daemon unavailable /does not/ cause failure by default.  Only explicit
developer diagnostics may make it an error, and those diagnostics are not
advertised in the semantic-language registry.

* Local daemon design

The daemon serves the CLI.  It is never exposed to the agent client.

** Transport

Priority order:

#+begin_src text
Unix:   .ts-harness/cache/server.sock
Win:    named pipe
Last resort: loopback HTTP 127.0.0.1 with random port file
#+end_src

No MCP, no dynamic tool discovery, no runtime schema negotiation.

** Protocol

Boring NDJSON.  Not an RPC framework.

Request:

#+begin_src json
{"v":1,"id":"1","method":"snapshot","params":{"root":".","mode":"compact"}}
#+end_src

Response:

#+begin_src json
{"v":1,"id":"1","ok":true,"result":{"compact":"Modules: source=184 owners=42\n..."}}
#+end_src

Error:

#+begin_src json
{"v":1,"id":"1","ok":false,"error":{"code":"STALE_CACHE","message":"cache invalidated; retry cold"}}
#+end_src

Rules:

#+begin_src text
- every request has a timeout
- every response carries protocol version
- result size has an upper bound
- search limit defaults to 20
- all paths are project-root-relative
- output ordering is deterministic
- daemon never returns unbounded JSON
#+end_src

** Responsibilities

The daemon:

#+begin_src text
- keeps TypeScript Program warm
- watches tsconfig / package.json / source changes
- maintains reasoning tree
- maintains owner graph
- maintains import/export/callsite index
- answers bounded queries
- invalidates cache deterministically
#+end_src

The daemon does NOT own policy authority.  That stays in the library/CLI:

#+begin_src text
daemon → gives facts fast
CLI    → validates facts version
       → renders compact text
       → owns exit code
       → owns direct parser run
#+end_src

A daemon bug cannot corrupt the agent contract.

** Lifecycle

The semantic-language protocol does not expose daemon lifecycle commands.
Providers may keep private developer controls for starting, stopping, or
inspecting a daemon, but those controls are not =methods=, are not advertised by
=agent doctor --json=, and are not used by agent guides or hooks.  The
agent-visible health surface is =<lang-harness> agent doctor=.

}

* Agent guide: teach recipes, not intent

The generated agent guide tells an agent client how to think about the
primitives.  It does not tell the client to call a planner command.  It
provides command-group recipes for common task types, subagent guidance, and a
synthesis format — all of which the agent client executes by composing
primitives.

````md
---
name: rs-harness
description: Use this guide in Rust repositories for reasoning-tree
  navigation, Cargo dependency facts, owner-aware search, structural policy
  checks, and changed-file repair loops.
---

# rs-harness Agent Client Guide

Use `rs-harness` before broad manual reading.  Prefer graph/search/deps
primitives over raw `rg`, `fd`, or hand-written `ast-grep`.

The CLI does not infer task intent.  The agent client must choose the
exploration recipe and decide whether to spawn subagents.

## Phase 1: deep prime — always first

Do not begin by reading files or requesting hits.  If the root is a workspace,
first get the workspace router:

```sh
rs-harness search workspace
```

Then select a package and get the package map:

```sh
rs-harness search prime
```

Read =mode=, =package=, =feature=, =cfg=, =owner=, =source_large=,
=api-candidate=, =test-surface=, =surface=, and =next= fields.

Critical rules from prime:
- =source_large=true= → never open the file; use =search owner <file> items=.
- =api-candidate= → a dependency API or workspace type may need docs.
- =feature ... requires_cfg= → the feature is guarded by a cfg; search both.
- =surface public_external= → external types are allowed in public API.

Do not spawn subagents before reading prime unless the user provides an
exact file and symbol.

## Phase 2: parent planning

Use prime output to choose search axes:

- feature/cfg axis
- dependency/API docs axis
- owner/item axis
- public API/callsite axis
- tests/bench/examples axis
- harness finding axis

Spawn read-only subagents only after selecting axes.

## Phase 3: focused multi-pipe or bounded single-view

Subagents use focused commands, not broad searches.  Prefer one multi-pipe
command per subagent when evidence surfaces are predictable:

```sh
# Feature/cfg subagent:
rs-harness search features <feature> cfg owners tests --package <pkg>

# Dependency/docs subagent:
rs-harness search dependency <dep> items docs tests --package <pkg>

# Source expansion subagent:
rg -n "<query>" <roots> | rs-harness search ingest items tests
```

Only request separate hit-level searches after selecting owners:
```sh
rs-harness search owner <owner> items
rs-harness search callsite <symbol> --view hits
rg -n "<query>" tests | rs-harness search ingest
```

Choose a focused recipe below.

## Subagent rule: graph slice, not full search

For broad tasks, spawn read-only explorer subagents yourself.

Parent gets graph; subagents get slices; hits only within slices:
- Parent: =search prime= or =search dependency <dep> --view graph=
- Dependency subagent: =search deps <dep[/subpath][@version][::api]>=
- Source subagent: =search dependency <dep> owners items --package <pkg>=
- Test subagent: =search tests <owner> --view graph=

Each subagent should:
1. run only the command group assigned by the parent agent;
2. avoid editing files;
3. return compact evidence as =[search-subagent]= line + =|= fact lines;
4. every =|= line must carry a =next= action if evidence is missing.

The parent agent merges subagent evidence into =[search-synthesis]= and
decides the next search phase.

## Output contract for subagents (line protocol)

```text
[search-subagent] role=deps dep=reqwest evidence=key:reqwest import:reqwest kind:normal feat:json,rustls-tls missing=- next=dependency:reqwest
[search-subagent] role=src dep=reqwest evidence=owner:src/http/client.rs hit_kind=use locations=3:5,8:1 pub=false missing=error-boundary next=pattern:public-error-boundary
[search-subagent] role=test dep=reqwest evidence=test:tests/unit/http/client.rs bench:benches/http_client.rs missing=timeout-test next=text:timeout(scope=tests)
```

Parent merges into:

```text
[search-synthesis] flow=dep dep=reqwest owners=src/http/client.rs,src/api/orders.rs tests=2 risk=public-error-boundary,timeout missing=2 next=pattern:public-error-boundary(owner=src/http),text:timeout(scope=tests)
```

If =missing= is non-empty, parent selects another focused =rs-harness search=
command using the =next= actions.

## Recipe: dependency change

Use when editing `Cargo.toml`, dependency features, dependency kind, target
cfg, or code using an external crate.

Parent first runs:

```sh
rs-harness search prime
rs-harness search dependency <dep>
```

Useful parallel subagents:

Dependency subagent:
```sh
rs-harness search deps <dep[/subpath][@version][::api]>
rs-harness search targets
```
=versionScope=external= starts an external docs/API research branch; do not use
current workspace owner hits as evidence for that requested version.

Source usage subagent:
```sh
rs-harness search dependency <dep> owners
rs-harness search import <dep-import-root>
```

Test/bench subagent:
```sh
rs-harness search dependency <dep> tests
rs-harness search tests <dep-or-owner>
```

Policy subagent:
```sh
rs-harness check --changed
rs-harness search pattern public-error-boundary
```

## Recipe: public API change

Use when changing public functions, exported structs/enums, error types,
request/config types, or public constructors.

Parent first runs:

```sh
rs-harness search symbol <symbol>
rs-harness search callsite <symbol>
```

If an owner is known:

```sh
rs-harness search owner <owner-or-path>
rs-harness search tests <owner-or-path>
```

Useful parallel subagents:

Owner boundary subagent:
```sh
rs-harness search owner <owner-or-path>
rs-harness search owner <owner-or-path> public-api
```

Callsite subagent:
```sh
rs-harness search callsite <symbol>
rs-harness search import <module-or-owner>
```

Test subagent:
```sh
rs-harness search tests <owner-or-path>
rg -n "<symbol>" tests | rs-harness search ingest
```

## Recipe: module move or namespace change

Use when moving files, renaming modules, splitting facades, moving
`mod.rs`, or changing imports.

Parent first runs:

```sh
rs-harness search owner <path-or-owner>
rs-harness search import <old-module-root>
```

Useful parallel subagents:

Owner graph subagent:
```sh
rs-harness search owner <path-or-owner>
rs-harness search prime
```

Import subagent:
```sh
rs-harness search import <old-module-root>
rs-harness search import <new-module-root>
```

Test mount subagent:
```sh
rs-harness search tests <owner-or-path>
rg -n "#[path" tests | rs-harness search ingest
```

## Flow: dependency API docs and usage

Use when the task asks how a dependency API is used, whether a feature
enables an API, how docs define an API, or whether a dependency type
leaks into public API.

Parent:
```sh
rs-harness search deps <dep[/subpath][@version][::api]>
rs-harness search docs <dep>::<item>
```

Subagents:
docs=`rs-harness search docs <dep>::<item>`
usage=`rs-harness search deps <dep[/subpath][@version][::item]>`
public=`rs-harness search dependency <dep> public-api`
tests=`rg -n "<item-name>" tests | rs-harness search ingest`

Parent synthesis fields:
dependency, version, features, api_shape, local_usage, public_leakage,
tests, risk, next

## Recipe: compact finding repair

Use when task starts from a harness finding.

Parent first runs:

```sh
rs-harness check --changed
rs-harness search owner <finding-path>
```

If the finding is import or dependency related:
```sh
rs-harness search import <import-root>
rs-harness search dependency <dep>
```

If the finding is API related:
```sh
rs-harness search symbol <symbol>
rs-harness search callsite <symbol>
```

If the finding is structural:
```sh
rs-harness search owner <path>
rs-harness search tests <owner-or-path>
```

## Recipe: unknown scope

Use when the user request names a concept but no owner/file is known.

Start with:

```sh
rs-harness search workspace
rs-harness search prime
rs-harness search symbol <concept>
```

If symbol search is too sparse, expand with lexical/file candidates
(always pipe back to ingest — never paste raw output):

```sh
rg -n "<concept>" src tests | rs-harness search ingest
fd -e rs "<concept>" src tests | rs-harness search ingest
```

If many matches appear, pick the top owner and narrow:

```sh
rs-harness search owner <top-match-path>
rs-harness search tests <top-match-path>
```

Then continue with a more specific recipe.

## External candidate expansion rule

When built-in =search text= or =search symbol= returns too few results,
use =rg= or =fd= — but always pipe back to =search ingest=:

```sh
rg -n "<query>" src tests | rs-harness search ingest
fd -e rs "<name>" src tests | rs-harness search ingest
git diff --name-only | rs-harness search ingest
```

Never paste raw =rg= / =fd= output.  The ingest command groups results by
owner or path, annotates the input source, and keeps downstream navigation in
the shared packet schema.  This is required for docs, schema files, generated
artifacts, and any test files or other text that are outside the provider's
=search text= index.

## Structural pattern rule

Use =search pattern <recipe>= for code shape queries.  List available
recipes with:

```sh
rs-harness search patterns
```

Do not write raw =ast-grep= patterns.  Select a recipe by name.  If no
recipe matches, fall back to =search text= or =rg | ingest=.

## After edits

Always run:

```sh
rs-harness check --changed
```

If Cargo.toml changed:

```sh
rs-harness search deps
rs-harness check --changed
```

Before closeout, choose the proper cargo lane:

```sh
cargo check --all-targets
cargo test
```

## Parent synthesis format (line protocol)

After merging subagent evidence, the parent agent produces a single
compact synthesis line (not a CLI input):

```text
[search-synthesis] flow=dep dep=reqwest owners=src/http/client.rs,src/api/orders.rs tests=2 risk=public-error-boundary,timeout missing=2 next=pattern:public-error-boundary(owner=src/http),text:timeout(scope=tests)
```

This synthesis is an agent-client reasoning artifact — the CLI never reads it.  If
=missing= is non-empty, the agent client selects additional primitives from the
relevant recipe using the =next= actions.
````

* Agent search flow: structured, not chaotic

The agent should follow this order:

#+begin_example
1. search workspace → route multi-package workspaces before package search
2. search prime     → see bounded package owner/dependency/finding map
3. search dependency → inspect external dependency source usage when relevant
4. search owner     → understand current file's responsibility
5. search symbol    → find exported symbol definitions
6. search callsite  → find owner-level import/reexport sites
7. search import    → inspect import/reexport owner edges
8. search tests     → locate tests importing the edited owner
9. search text      → find broader symbol/path candidates
10. search ingest   → normalize external candidate streams when needed
11. edit            → apply change
12. check           → verify change against policy
13. check --full    → closeout when needed
#+end_example

NOT:

#+begin_example
rg keyword
open many files
guess public API
run big test suite
#+end_example

* Failure mode reference

| Scenario                | CLI Behavior                                |
|-------------------------+---------------------------------------------|

| daemon stale            | auto-rebuild cache, then respond            |

* MVP order: M1–M7

The real MVP is not "many search subcommands."  It's four capabilities that
cover Tokio, rust-lang/rust, and large agent-client class projects:

#+begin_src text
1. search workspace + search prime --package
2. search features/cfg/deps dependency-version usage
3. search owner <path> items
4. search ingest auto-sniff + owner/item grouping
#+end_src

| Phase | Deliverable                                               |
|-------+-----------------------------------------------------------|
| M1    | Multi-pipe grammar + final-only renderer.  =search <source> |
|       | <pipe1> <pipe2> ...= with deterministic stages.  Trace      |
|       | output is descriptor-advertised.  Full field names,         |
|       | =locations=/                                                |
|       | =calls=/=uses= fold hits, next= inline on owner/item lines. |
| M2    | Cargo feature/cfg pipes.  =search features <f> cfg     |
|       | owners tests docs=.  Feature→optional-dep→target cfg chain. |
|       | Tokio =io-uring= / =loom= fixtures.                         |
| M3    | Items pipe.  =search owner <path> items=.  Nearest-item for |
|       | ingest.  =source-large= role.  agent-client file-search fixture.   |
| M4    | Docs/API facts.  Workspace/registry-source API facts: kind, |
|       | signature, async, unsafe, feature, receiver, return, error. |
|       | =search docs <crate>::<item>= plus versioned                 |
|       | =search deps <crate[/subpath][@version][::item]>=.          |
|       | bytes::Buf /                                                 |
|       | ignore::WalkBuilder fixtures.                               |
| M5    | Public-api pipe.  =search dependency <dep> public-api=.     |
|       | Tokio allowed_external_types + rustc_middle smallvec.       |
| M6    | Ingest auto-sniff + 8 pattern recipes.  Single command,     |
|       | auto-detect stdin.  Always owner/item-grouped output.       |
| M7    | SKILL.org multi-pipe flows.  8 flows with multi-pipe         |
|       | templates: dependency/feature/public-api/external-candidate/ |
|       | namespace/finding-repair/unknown/docs-api.                  |

Drill-verified implementation order — deep PM first, pipes after:

#+begin_src text
Step 1: Deep PM minimum viable loop
    rs-harness search workspace / prime / prime --package <pkg>
    mode=workspace-index for workspace, mode=package for prime
    |package, |dep, |feature, |cfg, |owner, |edge, |finding,
    |api-candidate, |test-surface, |surface, |next
    source=workspace|path|crates-io|git on cargo lines

Step 2: source_large + item slice
    rs-harness search owner <path> items
    source_large=true signal (guide: never open the file directly)
    agent-client file-search/src/lib.rs fixture

Step 3: Cargo feature/cfg
    rs-harness search features <feature>
    rs-harness search cfg <cfg>
    rs-harness search cfg <cfg>
    feature→optional-dep→target-cfg→docs.rs-metadata chain
    Tokio io-uring / tokio_unstable fixtures

Step 4: Multi-pipe final-only renderer
    rs-harness search dependency <dep> items tests
    rs-harness search features <feature> cfg owners tests
    descriptor-advertised trace for compact stage list
    single enriched packet output

Step 5: Docs/API facts
    rs-harness search docs <crate>::<item>
    rs-harness search deps <crate[/subpath][@version][::item]>
    rs-harness search api <crate>::<item>
    rs-harness search public-external-types --package <pkg>
    native parser API facts only (kind, signature, params, async, unsafe, receiver, return, error, impl, trait)
    public-external-types joins parser-native public API type facts with Cargo dependency facts
    bytes::Buf / ignore::WalkBuilder fixtures

Step 6: SKILL.org two-phase guide
    Phase 1: deep prime always first (mode/source_large/api-candidate rules)
    Phase 2: parent axis selection
    Phase 3: subagent multi-pipe/ingest
    Phase 4: parent synthesis
    Phase 5: rs-harness check --changed
#+end_src

The dominant finding: deep PM is the single most valuable capability
because it determines whether the LLM can correctly organize the rest of
the subagent flow graph without full-repo traversal.  Multi-pipe is a
Phase 3 acceleration, not a replacement for the Phase 1 map.

* Existing TypeScript harness readiness

The TypeScript harness already has:

- A CLI binary: =ts-harness= → =./dist/src/cli/main.js=
- =package.json= scripts: =build=, =check=, =format=, =lint=, =test=, =harness=
- Stable protocol commands: =search=, =check=, and =agent doctor=
- A provider registry emitted by =ts-harness agent doctor --json=
- A =SKILL.org= guide that teaches protocol commands, not direct renderer flags
- Public library surface: =renderTypeScriptProjectHarnessAgentCompactText=,
  =renderTypeScriptProjectHarnessAgentSnapshot=,
  =runTypeScriptProjectHarnessAgentSnapshot=, =runTypeScriptProjectHarness=,
  verification/render/profile/task index APIs

The TypeScript slice is the reference implementation for the semantic-language
registry contract.  The CLI advertises =ts-harness= only, rejects old direct
renderer flags as protocol commands, and validates search JSON against the
shared schema.  Its API search slice is intentionally parser-owned:
=ts-harness search api <query>= emits current-workspace exported/public API
facts only, while dependency-prefixed queries such as
=react@18.0.0::jsx= return an explicit missing-provider note until an external
docs/API source exists.  For equal relevance scores, TypeScript search ranks
recently modified project files before older files and then falls back to path
order, so active repair context rises without adding a new protocol field.

* The correct division of labor

#+begin_example
SKILL.org:
  writes exploration protocol, subagent usage, command-group recipes,
  synthesis format

Agent parent:
  judges task type
  selects recipe
  spawns subagents
  merges evidence
  decides next search phase
  decides edit plan

Agent subagents:
  read-only
  run assigned rs-harness command group
  return compact evidence

rs-harness CLI:
  search / check / agent primitives
  registry-described methods
  deterministic output
  no natural-language intent
  no subagent planning
  no task router

hooks:
  inject reminders
  prevent obviously bad searches
  changed check after edits
  closeout guard before stop
#+end_example

One sentence:

/Put "intent → plan → subagents" in the agent client.  Put native facts, owner
slices, search packets, compact checks, and registry-described methods in the
harness.  That is the correct harness-client boundary./
