* One-sentence summary

The harness CLI is the public protocol.  A local daemon is a private
acceleration cache.  MCP is explicitly excluded from the architecture.

* Design principle

#+begin_example
Agent client
  ↓ reads SKILL.org
hooks
  ↓ call fixed commands
ts-harness CLI
  ↓ optional acceleration
ts-harness local daemon / cache
  ↓
TypeScript parser facts / reasoning tree / compact diagnostics
#+end_example

Agent clients call stable CLI commands.  The server is a CLI implementation
detail — invisible to the agent, replaceable without contract changes.

* Why not MCP

MCP has three structural problems for harness:

1. /Protocol layer is too dynamic./  Tool discovery, schema negotiation, and
   runtime capability advertisement make the agent's dependency surface
   unpredictable.  Harness diagnostics should be deterministic — same
   project, same output.

2. /Failure modes are unstable./  When an MCP server crashes, restarts,
   version-mismatches, or times out, the agent receives transport-level
   errors that it cannot repair.  A CLI with stable exit codes has exactly 5
   failure states, all documented.

3. /It turns harness into a conversational-tool-ecosystem dependency./  The
   harness's value is /deterministic structural facts/ — parser-owned,
   project-scoped, policy-evaluated.  Embedding that behind a dynamic tool
   discovery layer adds indirection without adding precision.

The right direction is the opposite: shrink the surface, make it deterministic,
and keep parser-owned facts as the source of authority.

* What NOT to build: the anti-patterns

The CLI must not implement intent parsing, subagent planning, or task
routing.  These are LLM responsibilities.  Specifically, these commands
should never exist:

#+begin_src shell
# DO NOT BUILD:
rs-harness graph subagent-plan --intent "change dependency reqwest features"
rs-harness graph next --from synthesis
rs-harness explore nl "find ..."
rs-harness search ast --intent "find clone in loop"
#+end_src

Also: do NOT split exploration into =graph=, =cargo=, =explore= as separate
top-level commands.  To the agent client, read-only discovery is =search= with
different views.  Parser-native extraction, owner-local item lookup, hook
reroutes, and compact code projection are =query=.

* CLI is the protocol: search plus query surface

Language-neutral examples use =<lang-harness>=.  The TypeScript implementation
ships this as =ts-harness=; Rust/Cargo-specific lanes use =rs-harness=.
Julia is intentionally not a bin parity target: LLVM/JIT warm-up makes a
short-lived installed/global Julia provider binary a poor agent protocol
surface. Julia alignment should use the workspace-managed command
=julia --project=languages/JuliaLangProjectHarness.jl
languages/JuliaLangProjectHarness.jl/bin/julia-project-harness.jl= and should
not design a standalone =julia-project-harness= bin as part of provider
alignment.  RFC 008 may let =agent-semantic-client= cache validated
workspace-managed Julia provider exports and serve repeated searches from the
Rust client cache, but that cache is an execution optimization, not a new Julia
bin-parity requirement or a Rust-owned Julia parser.

The agent sees four language-provider top-level commands:

#+begin_src shell
<lang-harness> search <view> ...   # all exploration / location / evidence
<lang-harness> query ...           # parser-owned compact code/query reroutes
<lang-harness> check ...           # policy / style / changed / full
<lang-harness> agent ...           # agent-client install and diagnostics
#+end_src

RFC 007 adds the shared =asp ast-patch= verification path.
Language providers still own AST adapters, but agents should use the shared
binary for schema validation, receipts, and Codex-safe dry-run behavior.

** =search= — discovery and candidate acquisition

Every read-only discovery step goes through =<lang-harness> search=.  Search
returns compact evidence, owner/test seeds, dependency facts, and candidate
frontiers.  It should not be the primary path for parser-owned code extraction.
Common views:

#+begin_src shell
<lang-harness> search prime

<lang-harness> search owner <path-or-owner>
<lang-harness> search dependency <crate-or-import-or-package>

<lang-harness> search symbol <name>
<lang-harness> search callsite <name>
<lang-harness> search import <root>
<lang-harness> search pattern <recipe>
<lang-harness> search tests <owner-or-path>
<lang-harness> search fzf <query>
#+end_src

=search fzf= is the preferred agent-facing lexical candidate command.  The
name is intentional: LLMs already know =fzf= means fuzzy filtering/ranking over
a candidate stream, so the command communicates the search behavior better than
a provider-invented =text= abstraction.  The provider still owns the pipeline:
=rg= or an equivalent scanner builds bounded candidates, =fzf= or an
equivalent scorer ranks them headlessly, then the provider groups by owner,
deduplicates, caps, and renders compact seeds.

Do not expose the old text-named compatibility view.  Exact lexical matching,
fixed-string matching, and fuzzy matching are modes of =search fzf= or provider
finder pipelines, not separate public views.  =search fzf= is not a language
query: code-shaped questions, known owners, symbols, declarations, parser
items, and hook read reroutes should use top-level =query= first.

Agent-facing =search fzf= grammar keeps search composition and fzf-native
options separate:

#+begin_src text
<lang-harness> search fzf <query> [owner|tests|items|...] [--scope <path-or-kind>] \
  [--pipe <surface>] [--view seeds|hits|graph|both] [PROJECT_ROOT] \
  [--fzf-arg <fzf-option> ...]
<lang-harness> search fzf <query> [search-options...] [PROJECT_ROOT] --fzf <fzf-options...>
#+end_src

=--scope=, =--pipe=, =--view=, =--query-set=, and the optional project root
belong to the harness search grammar.  =--fzf-arg <value>= passes one
provider-approved fzf-native option without changing grammar ownership.
=--fzf= is a boundary marker: it must be the last harness option and all
following tokens are fzf-native options, so agents should place =PROJECT_ROOT=
before =--fzf= or omit it and let the provider discover the project root.
Providers must reject interactive or shell-shaped fzf options on agent paths
and record accepted/rejected options in the search packet =finder= field.

Path scopes passed to =search fzf= are search scopes, not extra project roots.
Providers must normalize them against the discovered project/package root and
its ancestors.  For example, when the package root is =crates/demo= inside a
workspace, =search fzf run owner crates/demo/tests .= must scope to package-local
=tests= instead of resolving =crates/demo/crates/demo/tests=.

In fuzzy mode, a plain query containing multiple whitespace-separated tokens may
be promoted to the same homogeneous query-set machinery as explicit
=--query-set= terms.  The provider must mark the output with =querySet=N= and a
query-set selector so agents can see that the single fzf phrase was decomposed
into bounded same-axis lexical terms.  Exact mode keeps the query as one literal
fzf pattern.

=search owner= resolves semantic owners first, then parser-visible modules,
then existing project paths.  A parser-visible module that is not a reasoning
owner should return =source=parser-visible-module=, =parserOwner=false=, and
module facts such as role, layer, line count, validity, and diagnostics.  A path
outside the parser module set may return =source=path-only= and
=next=ingest:<path>= so agents can keep the path in the protocol instead of
falling back to raw shell output.

** =query= — first-class parser-native extraction and reroutes

Provider-native item query is the default code extraction path for agents:

#+begin_src shell
<lang-harness> query <project-path> --query <symbol-or-a|b|c>
<lang-harness> query <project-path> --term <symbol-a> --term <symbol-b> --names-only
<lang-harness> query --from-hook direct-source-read --selector <exact-source-path[:line-range]> --code
<lang-harness> query --from-hook direct-source-read --selector <glob> --term <intent-term>
#+end_src

The language provider owns parser/AST lookup, exact item matching, multi-term
query expressions, hook reroute normalization, and compact code rendering.
Even when Julia stays workspace-managed rather than bin-parity, its owner-local
item query must use this same shared =semantic-query-packet= contract for JSON
output:

#+begin_src shell
julia --project=languages/JuliaLangProjectHarness.jl \
  languages/JuliaLangProjectHarness.jl/bin/julia-project-harness.jl \
  query src/CliExample.jl --term run --json languages/JuliaLangProjectHarness.jl
#+end_src

For an exact =direct-source-read= selector with =--code=, the provider should
return a bounded compact local window for that path/range, not owner metadata.
The compact output is parser-owned code text only; the selector already carries
the path/range, so =--code= must not repeat =line= or =endLine= metadata or return
owner seeds.  Rust, TypeScript, Python, and Julia all use their native parser
facts for this surface; Julia uses the workspace-managed provider command
rather than a global short-lived binary.
If the selector is too wide for bounded code transport, the provider returns a
=read-plan= frontier, not a denial-only outline.  The active contract is
=semantic-read-packet=: =readPlan.mode= is =range-frontier= or =search-repair=,
=readPlan.frontier= names executable next actions, and =readPlan.avoid= names forbidden next actions such as =repeat-wide-read=,
=manual-window-scan=, and =raw-read=.  A range-only plan must use =windows= and
=frontier= entries; it must not synthesize a fake =symbol= such as
=local-window=.  Parser-owned symbols are preferred when available, and
mechanical range splits are only the fallback.  Every frontier =code= action
must map back to:

#+begin_src shell
<lang-harness> query --from-hook direct-source-read --selector <path[:line-range]> --code .
#+end_src

For exact selectors without =--code=, the provider may normalize to
=search owner <path> items= rather than owner seeds.  That gives the agent
parser-owned item names, read locators carrying line ranges, and test
reachability without dumping raw source.  For exact selectors with a query term,
the provider should keep the normal item query/code projection path.  Wildcard
or broad selectors still require concrete terms and should remain bounded
candidate/seed searches.

Hook route construction must preserve exact line-range selectors when the agent
already supplied one.  The hook may strip =:line[-end]= for language profile
matching, but the provider-facing route must keep the bounded selector:

#+begin_example
rtk read src/lib.rs:10:20
=> <lang-harness> query --from-hook direct-source-read --selector src/lib.rs:10:20 --code .
#+end_example

Dropping the range turns a precise local-read intent into a larger owner query
and encourages follow-up raw reads.  That is a protocol bug, not a user
preference.

The language provider must make that bounded selector useful.  A query such as
=<lang-harness> query src/lib.rs:10:20 --code .= should return a bounded
parser-owned local window for that exact range, not whole-owner item names.
This is required for test/assertion edits, import/header inspection, and hook
recovery after a denied =sed= / =rtk read= / tool-native read.  The root hook
only preserves the selector; native parser providers own the local-window
projection.

Providers should exact-match item names first; if no exact item exists, they may
fall back to contains matching.  A hit emits both parser-owned item metadata and
compact code:

#+begin_example
[search-owner] q=src/search/owner_view.rs pkg=. own=1 item=1 itemQuery=render_search_owner
|query itemQuery=render_search_owner status=hit match=exact item=1 reason=parser-item-exact next=code
|item render_search_owner kind=fn next=symbol:render_search_owner read=src/search/owner_view.rs:22:40
|code path=src/search/owner_view.rs lineRange=22:40 reason=item-query truncated=false nodes=n0:fn:declaration:0:22:40,n1:if:control-flow:1:28:32 text="pub(super) fn render_search_owner(\nproject_root: &Path,"
#+end_example

After an agent has already selected a concrete owner and item query, it may ask
for pure compact code:

#+begin_src shell
<lang-harness> search owner <path> items --query 'fun1|fun2|fun3' --code
#+end_src

The =--code= projection emits only compact parser-owned code projection text.
It omits the =|owner= / =|item= / =|code= line protocol metadata and is mutually
exclusive with =--json=.  Providers should keep the default =--query= output
metadata-bearing because =status=hit=, =match=exact=, and visible fallback
markers are the accuracy contract for exploration and hook recovery.  =--code=
is therefore a post-selection token optimization, not a replacement for search
evidence.

For item query results, the =|code text="..."= field must carry the same
compact parser-owned projection as =--code=.  The only difference is transport:
line protocol JSON-escapes the compact text into a field, while =--code= prints
the compact text directly.  Providers must not apply a second whitespace-only
compaction layer at line-render time; that hides AST structure, makes
=text="..."= diverge from =--code=, and pushes agents back into repeated local
reads.

Provider compact code must also preserve literal payload semantics.  Whitespace
inside Rust raw strings, Python triple-quoted strings, JavaScript template
literals, Julia string macros, and equivalent language-owned literal regions is
not syntax trivia.  A provider may compact whitespace only outside such
payloads, or it must summarize/omit the payload and route the agent to an exact
read locator.  JSON query packets expose this decision as
=projection.compactSafety=; line protocol =--code= users see the same compact
text and must not infer original indentation from it unless
=compactSafety.whitespacePolicy= is =exact-source=.

Line-protocol =|code= rows may include a parser-owned =nodes= field before
=text=.  The field is a comma-separated compact node list:
=id:kind:role:depth:start:end=.  The JSON query renderer must prefer this
parser-owned node list for =projection.nodes[]= and =expandActions[]=, using
compact =text= classification only as a fallback when a legacy provider does
not emit =nodes=.  This keeps visible line protocol compact while preventing
JSON projections from being reconstructed by renderer heuristics.

Compact code is not raw source minification.  It is an agent-facing AST
projection produced by the language-native parser.  The projection may omit
trivia such as blank lines, comments, and indentation, but it must preserve the
semantic structure an agent needs to reason about ownership and edits:

- item boundary and exact read locator
- block nesting and branch ownership
- return / throw / panic / await / unsafe / mutation markers where the parser
  can expose them
- helper calls and hot-block locators when a full item is too large

Rust and TypeScript providers may use token/braces compact syntax because block
ownership survives in ={ ... }=.  Python, Julia, and indentation-sensitive
languages must not flatten raw source indentation into ambiguous one-line code.
They should emit a parser-owned block projection such as =def f(x) { if x > 0
{ return x } return 0 }= or a semantic outline with hot-block locators.  This
projection is for agent understanding, not an editable source replacement.
Exact source remains available through the read locator.

The query response is a progressive reasoning-tree traversal, not a set of
human display modes.  Providers should select the smallest layer that preserves
the next decision:

#+begin_example
owner tree -> item names -> item outline -> compact hot blocks -> exact read for edit
#+end_example

Compact projection is bidirectional.  The forward direction is native parser
facts to agent-facing projection: declarations, branches, calls, exits,
mutation/effect markers, and hot blocks.  The reverse direction is projection
back to exact provider actions: every important node should carry a stable
node id plus a read locator or expansion action so the agent can ask for the
exact block instead of restarting broad search.  The compact text may be lossy
for token savings, but the projection metadata must explain what was omitted
and how to expand it.

#+begin_example
parser AST -> projection.nodes[] -> compact/outline text -> expandActions[] -> exact read
#+end_example

Formatter-assisted compaction is allowed only after the native parser has
proved the structure.  The provider may use the language formatter to choose a
workspace/style-owned source dialect before projection or after an edit
dry-run, because formatter-normalized text lets the compact renderer remove
whitespace, blank lines, and local trivia without guessing semantics.  The
formatter is not the source of AST facts.  It is a safety gate for printable
text that still follows the native AST.

This intentionally separates formatter language from formatted source.  The
original purpose of formatters is human readability: indentation, vertical
spacing, alignment, and style conventions make code comfortable for people to
scan.  Those same choices waste tokens for an agent.  However, LLMs are trained
mostly on formatted code, so the formatter's canonical spelling is still a
useful prior for code understanding.  Compact AST should therefore reuse the
formatter dialect, not the full formatted layout.  Providers may normalize
syntax with the formatter, then project the parser facts into a shorter
agent-facing form that drops human-only layout while keeping grammar,
ownership, and edit locators.

The protocol must not assume one fixed formatter style.  Different Rust,
Python, TypeScript, JavaScript, and future provider workspaces may choose
different formatter tools, versions, line widths, import styles, or plugin
settings.  Those choices are user/workspace policy.  The provider records the
formatter profile it discovered and treats the exact style as opaque; the
shared invariant is that the formatted output still parses to the same
language-native structure.  The shared query projection schema must not expose
formatter names or styles as a cross-language enum; formatter profile belongs
to provider registry/capability metadata or provider-local diagnostics.

Formatter alignment is fail-closed.  If a provider declares that compact output
uses formatter-normalized syntax, the original parser facts and the formatted
parser facts must align through native node identity, symbol path, structural
role, and structural fingerprints.  Token fingerprints may help diagnose
alignment, but they are not the primary contract because formatter styles are
not stable across workspaces.  If AST alignment fails, the provider must not
silently return parser-only compact output as a fallback.  It should emit a
structured failure or diagnostic receipt and stop the formatter-assisted path.
Parser-only compact projection is valid only as an explicitly declared
capability mode before formatter normalization is attempted; it is not a
recovery path after formatter/core AST disagreement.

The formatter stage is language-owned:

- Rust uses =rustfmt=.
- Python uses =ruff format= for formatting; =ruff check --fix= is a separate
  lint-fix or organize-imports capability, not formatting.
- TypeScript and JavaScript use the workspace-declared formatter from
  =package.json= or provider registry facts, such as =prettier=, =biome=, or
  =oxfmt=.
- Julia remains skipped for bin parity.  If the Julia lane reopens, it should
  use =JuliaFormatter.jl= as a workspace-managed capability.

Compact AST output must not break the normal agent edit loop.  The agent reads
compact projection to understand and choose a target, then expands an exact
read locator before writing code.  The compact projection is not an editable
source replacement and must never be the only path back to source.  It is also
not a text patch context: compact renderers may remove indentation, punctuation
lines, comments, or formatter layout.  =apply_patch= preimages and other
text-hunk contexts must come from an exact source window selected by the
projection's =read=/=exactRead= locator.  A provider that relies on
formatter-normalized text for maximum compression must still return stable node
ids, locators, omitted facts, and expansion actions.

Standalone layout punctuation is never a useful compact line.  Lines such as
=}=, =})=, =});=, =];=, or =,= are formatter/source-layout residue, not
agent-facing facts.  A provider may keep punctuation inside a semantic line
when it helps expression understanding, such as =.map((value) => ({=, but it
must not emit punctuation-only lines as compact code.  If block structure
matters, it belongs in parser nodes, depth, edges, or exact expand actions.

Compact rendering must also be node-unique: one native AST node id produces at
most one rendered row in a given projection.  Nested owners, local classes,
closures, or visitor methods may be summarized under their parent or exposed as
expandable descendants, but the renderer must not emit the same descendant both
inline and as a sibling row.

**** AST projection identity and uniqueness

AST projection uniqueness is a protocol invariant, not a renderer preference.
Within one query/search packet, every projected native AST node must have one
stable projection identity and at most one primary rendered row.

The provider owns the concrete native identity, but the shared projection
identity must be derived from parser facts, not text formatting.  A projection
node identity should include enough provider-owned material to distinguish the
node within the source fingerprint:

- owner path and owner/item identity
- native node kind
- structural role
- parent projection id or structural parent path
- source range or parser-owned stable span
- same-kind ordinal when the language parser requires it
- optional structural fingerprint for diagnostics

In the packet contract this is not optional metadata.  Every compact projection
node must carry =nativeId=, =role=, =depth=, =read=, and
=structuralFingerprint= so the agent can distinguish parser-native identity,
rendering hierarchy, exact source expansion, and stale/ambiguous projections
without guessing from formatted text.

Rendered compact rows are views over projection nodes:

- one rendered row has one primary =nodeId=
- one native AST node may not be rendered as two primary rows
- child nodes may be referenced inline only as labels, edges, or summaries
- repeated semantic references such as callee, type, binding, or symbol use are
  references, not duplicate projection nodes
- =omitted[]= and =expandActions[]= must reference existing projection node ids
  or exact read locators instead of re-rendering hidden code
- owner-local name discovery, fallback candidates, and routing actions such as
  =owner-names= belong to query/candidate output, not to compact AST
  =projection.expandActions=
- every projection action must be executable without guessing: =exact-read=
  and =hot-block= actions carry a =read= locator; =node-query= actions carry
  provider =argv= and must target an existing projection node rather than a
  read locator
- if an =exact-read= or =hot-block= action also emits provider =argv=, its
  =--selector= argument must be the same canonical =read= locator; it must not
  widen back to an owner path or glob

Nested owners need an explicit mode.  A provider may either summarize a nested
function/class/closure under its parent or expose it as a child projection node
with =parentId= and an expansion action.  It must not do both in the same
projection.  If a provider cannot assign unique parser-owned projection ids for
the requested item, it must not claim compact-AST output for that item.  It may
return a weaker, explicitly declared surface such as owner names or outline,
but not a duplicate or ambiguous compact AST projection.

This is the core parser loop.  Native parsers should improve the compact
algorithm and the expansion metadata together: if a provider learns to omit
large match cases, helper bodies, closures, decorators, or type clauses, it
must also emit the omitted fact and the next exact read/query action.  A
projection without reverse navigation is only a display optimization; a
projection with nodes and expansion actions is an agent reasoning interface.
Therefore =mode=compact= requires =sourceFingerprint=, =losslessStructure=,
=exactRead=, =nodes[]=, and =renderedNodeIds[]=.  A provider that cannot
populate those fields must use a weaker mode such as =outline= or =names=.
For compact mode, =exactRead= is the same canonical locator as the match-level
=read= field, and =sourceFingerprint= must bind that locator so stale compact
projections can be diagnosed without comparing arbitrary formatted text.

An exact small item hit should produce compact AST code.  A broad, mixed, or
missed query should produce parser-owned names/candidates.  An exact but large
item should produce semantic outline plus hot-block locators before full compact
code.  The agent should not have to choose among these layers blindly; provider
certainty, item size, and token budget decide the projection.  =--code= is the
explicit override after the agent has selected the item to read.

*** Parser compact capability plan

The Rust, TypeScript, and Python compact lanes must be advanced as native parser
implementations, not as renderer tricks.  The renderer may choose line protocol,
JSON, or pure =--code= transport, but it must serialize parser projection nodes
that already exist in the language provider.  A provider that can only trim
whitespace has not implemented compact code.  Julia is skipped for this
bin-parity and compact-search lane; it remains workspace-managed only and must
not be used to justify a standalone =julia-project-harness= binary.

The shared capability ladder is:

1. /Declaration boundary./  Item name, visibility, signature, type-only status,
   and exact read locator.
2. /Control flow./  =if=/=else=, =match=/=case=, loops, and branch-local
   ownership.
3. /Terminal and effects./  =return=, =break=, =continue=, =throw=/=raise=,
   =panic=, =await=, =try/?=, =unsafe=, and equivalent language-owned exits.
4. /Mutation and binding./  Assignment targets, low-value ignored bindings,
   summarized initializer shape, and member/field mutation when the parser can
   expose it.
5. /Calls and macros./  Stable callee names plus bounded argument evidence for
   high-signal calls without dumping whole expressions.
6. /Expansion metadata./  Stable node ids, parent ids, exact read locators,
   omitted facts, and next query/read actions for hot blocks.

The durable plan is:

1. /Snapshot the current parser contract./  Each provider keeps a small fixture
   project and a stable compact-query snapshot.  The snapshot records the
   agent-facing =code= projection, =projection.nodes[]=, =omitted[]=, and
   =expandActions[]=.  It excludes temp roots, timings, and other runtime-only
   fields.  A compact algorithm change must update this fixture intentionally.
   The root repository mirrors these fixtures under
   =tests/fixtures/parser-compact/= as feature-level cases with one provider
   variant per language, for example =cases/control-flow-basic/rust.json=.
   Each variant has three shared artifacts: compact line output,
   =semantic-query-packet.v1= JSON, and =parser-compact-token-cost.v1=
   measurement.  The =parser-compact-snapshots= runner only wraps the provider
   commands from the manifest and compares artifacts so Rust, TypeScript,
   Python, and Julia can compare compact algorithms against the same schema and
   token-cost contract.

2. /Strengthen native parser facts before renderer rules./  New compact
   behavior should come from native parser nodes: Rust =syn=, the TypeScript
   compiler API, Python =ast=, or JuliaSyntax.  Parser facts own branch
   ownership, call labels, return/raise/throw exits, await/effect markers,
   mutation targets, decorators, class/member boundaries, and type/signature
   facts where the language exposes them.  Regex post-processing may normalize
   display text but must not invent structure.

3. /Keep projection bidirectional./  Every hot node that affects editing must
   carry a stable =id=, =parentId= when nested, an exact =read= locator, and an
   executable =expandAction= whose =--selector= equals that exact locator.
   Direct-source-read selectors with line ranges must return the bounded parser
   item or source window, not owner top-items.

4. /Add capability tests by language feature./  After the baseline snapshot,
   add one snapshot per parser feature class: nested branches, loops, terminal
   statements, mutation, call/macro summaries, async calls/effects,
   class/member bodies, decorators or modifiers, large item outline truncation,
   and selector-range expansion.  These snapshots are the parser compact
   acceptance tests.  Rust should lock both the pure =--code= projection and the
   line-protocol =|code text="..."= transport so renderer changes cannot drift
   away from parser facts.

5. /Measure search-flow impact after snapshots are stable./  Sandtable and
   real-project runs should compare command count, repeated owner/test search,
   direct-read reroutes, and query packet size before and after compact parser
   improvements.  Performance optimizations are accepted only when the snapshot
   contract stays stable or changes intentionally.

Root hooks must not maintain a parallel read/query engine for the normal flow.
The =|query= line is the stop/accuracy contract for agents: =status=hit= with
=match=exact= means the parser selected the requested item; =fallback-contains=
means the provider had to broaden after no exact item existed, including
language-owned function-body matching when the requested evidence lives inside
the code rather than in the symbol name; =status=miss= means the query should
be corrected rather than followed by raw file search.

Owner-local item discovery has a names-only phase.  Agents should use it when a
selected owner is known but the exact item names are still uncertain:

#+begin_example
[search-owner] q=src/command/raw_search.rs pkg=. own=1 item=6 itemQuery=parse_
|query itemQuery=parse_ status=hit match=fallback-contains item=6 reason=parser-item-fallback output=names next=select-item
|item parse_ripgrep_like kind=fn read=src/command/raw_search.rs:97:201
|item parse_fd kind=fn read=src/command/raw_search.rs:311:358
#+end_example

In names-only mode, providers must not emit =|code= lines.  They should emit
read locators and a visible =output=names= field so the agent can choose the
exact follow-up query before requesting compact code.  Names-only hit or mixed
results should use =next=select-item=.  Misses or weak fallback top-item
inventories should use =next=revise-query= unless the provider has a
high-confidence parser-owned replacement item.

When an owner-local item query misses, the provider should emit bounded
parser-owned candidates if they can be inferred from the selected owner's item
index:

#+begin_example
|query itemQuery=parse_ripgrep_scope status=miss match=none item=0 reason=parser-item-miss output=names candidates=parse_ripgrep_like next=revise-query
#+end_example

Candidate names must come from parser-owned item names in the requested owner.
They are query repair evidence, not textual search hits.  A shell-safe repeated
=--term= form is preferred over unquoted pipe expressions because
=--query RawSearchScope|terms_for_raw_search= is interpreted by shells as a
pipeline unless the whole expression is quoted.

When provider-native =query --json= is requested for an owner item query, it
emits =semantic-query-packet.v1= rather than =semantic-search-packet.v1=.  The
JSON packet uses =queryCoverage= and =candidateItems= for the same repair
evidence that the line protocol exposes through =|query ... candidates=...=.
Parser compact snapshots must use this =query --json= form; using
=search ... --json= records the wrong packet class and hides compact-query
cost behind search-packet fields.

Workspace-managed Julia search still uses the shared search packet for search
JSON:

#+begin_src shell
julia --project=languages/JuliaLangProjectHarness.jl \
  languages/JuliaLangProjectHarness.jl/bin/julia-project-harness.jl \
  search fzf run owner tests --view seeds --json languages/JuliaLangProjectHarness.jl
#+end_src

The hook wildcard form uses the same packet class when a client needs a cached
search frontier rather than compact text:

#+begin_src shell
julia --project=languages/JuliaLangProjectHarness.jl \
  languages/JuliaLangProjectHarness.jl/bin/julia-project-harness.jl \
  query --from-hook direct-source-read \
    --selector '**/*.jl' \
    --term render_julia_search_packet \
    --surface owners,tests \
    --view seeds \
    --json languages/JuliaLangProjectHarness.jl
#+end_src

That packet is for validators, IDEs, and Rust-client cache reuse.  It should
mirror the compact search line protocol with =owners=, =hits=, =nextActions=,
=querySet= and =queryCoverage= when terms were provided, =searchSynthesis=,
optional =nativeSyntaxFacts=, and optional =semanticHandles=.  It does not
replace the agent-facing compact default, and it must not invent a
Julia-private search payload.

Providers may additionally expose a =read-packet= query output mode for exact
selected source windows:

#+begin_src shell
<lang-harness> query --from-hook direct-source-read --selector <project-path> --view read-packet .
#+end_src

That mode emits =semantic-read-packet= under the normal
=agent.semantic-protocols.semantic-language= protocol.  It is provider-owned
parser transport, not a root hook command.  The packet should only carry
bounded exact windows that the provider can justify with parser-owned owner,
item, or syntax facts.  Broad selectors must remain seed/candidate queries.

When an owner-scoped item query has no exact or contains match, the provider may
return a bounded owner-local fallback instead of an empty packet.  The fallback
must stay parser-owned, scoped to the requested owner, and visibly marked in the
header, for example =fallback=owner-top-items=.  This is intended for weak
path-derived hook queries such as =types|Types= or =packet-base|PacketBase=:
the hook supplies a cheap route, and the language provider chooses a small set
of top-level declarations that are most useful for compact code inspection.
Providers that implement this behavior should advertise it on the =search/owner=
registry descriptor:

#+begin_src json
{
  "fallbacks": [
    {
      "name": "owner-top-items",
      "trigger": "item-query-miss",
      "appliesToPipes": ["items"],
      "maxItems": 4
    }
  ]
}
#+end_src

Top-level =query= is also the provider-native syntax intent surface.  It is
for code-shaped input where the language parser can prove intent before text
search: =pub use rules=, =fn format_field=, =struct PacketCollections=,
=#[cfg(test)]=, =import {Foo}=, =def run=, and equivalent language-native
forms.  Future syntax-intent variants must live under =query= rather than
=search fzf=.  Providers must route these inputs through a Native Syntax Fact
Index instead of treating them as plain text.  The compact packet should expose
the normalized intent and facts:

#+begin_example
[search-query] q=pub use rules pkg=. intent=syntax.import own=1 fact=1
|query intent=syntax.import term=rules status=hit routed=native-syntax selected=1
|fact id=rust:src/lib.rs:4:reexport:rules kind=reexport source=native-parser owner=src/lib.rs line=4 visibility=public name=rules languageKind=use
[search-graph] mode=query root=. alg=native-syntax-query
legend: ID=kind:role(value)!next; edge SRC>{DST:rel}; frontier ID.next
alias: graph:{G=search,L=owner,F=fact}
L=owner:path(src/lib.rs)!owner
F=fact:symbol(rust:src/lib.rs:4:reexport:rules)
L>{F:contains}
rank=L,F frontier=L.owner
#+end_example

The root protocol owns the shared =semantic-native-syntax-fact-index.v1=
schema.  It defines language-neutral fact identity, kind, owner/location,
visibility, query keys, relations, and provider extension fields.  Rust,
TypeScript, Python, Julia, and future providers each own their concrete fact
builders and language-specific fields under the common schema.  The shared
schema is intentionally not a lowest-common-denominator AST; it is the
agent-facing index of facts that can answer search/query flows without raw text
scans.

The shared fact kind vocabulary must be broad enough for providers to expose
useful native facts without leaking their AST wholesale.  It therefore includes
portable items such as =owner=, =module=, =function=, =method=, =struct=,
=type=, =import=, =export=, =call=, =test=, =doc=, =include=, =field=,
=binding=, =constant=, =argument=, and =macro=.  Language-specific categories
stay in =languageKind=, while provider detail, parser text, tags, and
language-only fields stay under =fields=.

=search fzf= covers fuzzy lexical candidates: source text, owner paths, string
literals, comments, docs, schema files, generated artifacts, config keys,
error messages, and other surfaces where parser-native facts are unavailable or
insufficient.  The default pipeline is =rg+fzf=: =rg= or an equivalent scanner
builds candidates, then =fzf= or an equivalent headless scorer filters/ranks
them.  The provider, not the backend, owns project scope, path filtering, owner
grouping, per-term fairness, deduplication, ranking integration, and compact
packet rendering.

If the query is code-shaped, =search fzf= must not execute the native syntax
API on behalf of the caller.  It should return a visible =next=query= action so
the agent retries through the explicit parser-native API.  If text lies outside
the provider's local index, the provider may return =next=ingest:<query>= and
the agent must route external candidates through =search ingest=.

=search fzf= must remain lexical/fuzzy search.  It may support match modes
such as =literal=, =fixed=, =regex=, =word=, =case-folded=, and =fuzzy=, and
backends/pipelines such as =rg=, =rg+fzf=, =ugrep=, or a provider-maintained
trigram/indexed lexical engine.  Backend selection should be visible in packets
through runtime cost/provenance fields, but it should not change the semantic
contract: parser-owned extraction is still =query=, and lexical candidate
expansion is still =search fzf= or =search ingest=.

=ast-grep= is not the fuzzy text backend for this lane.  It is a structural
search/rewrite engine: useful behind =search pattern= recipes or provider-owned
native-fact builders, but not a replacement for =rg= plus approximate lexical
matching.  Agents should never be asked to author raw =ast-grep= DSL as a
general search fallback.

*** Roadmap: agent-facing fzf lexical search

The lexical-search roadmap is intentionally separate from =query=:

1. =search fzf= becomes the preferred public lexical/fuzzy search view for
   agents.  The command name matches the mental model LLMs already have:
   fuzzy filtering over a candidate list.
2. The first =search fzf= backend should be =rg+fzf=.  =rg= owns fast source
   scanning, ignore/glob semantics, and structured hit provenance; =fzf= owns
   fuzzy ranking over candidate lines, paths, or owner labels.  Agent-facing
   runs must use non-interactive filtering/scoring, not a terminal picker.
   Human-facing diagnostics may optionally expose an interactive preview later.
3. =ugrep= or a provider-owned lexical index remains a later alternative when
   fuzzy matching, Boolean combinations, case folding, or cold large-workspace
   pruning cannot be expressed efficiently as =rg+fzf=.
4. Shared schemas should add backend provenance only after the RFC is stable:
   =textBackend=, =matchMode=, =sourceSearchPasses=, =candidateFileBasis=,
   =fuzzyFilter=, =queryTerms=, and bounded runtime cost.  These fields explain
   how text candidates were found; they do not make text results parser-owned
   facts.
5. Provider guides should teach the decision ladder:
   =query= for parser-owned extraction, =search owner/tests/deps= for
   structured discovery, =search fzf= for broad lexical/fuzzy candidates, and
   =search ingest= for externally generated candidate streams.

This keeps the competitive advantage clear: native syntax parsing plus
first-class =query= gives precise code slices and read locators with fewer
tokens and less semantic drift.  =search fzf= is the widening tool, not the
authority layer.

*** Shared finder tools schema

The protocol repository owns =semantic-finder-tools.v1= as the common contract
for provider-approved lexical and structural finder pipelines.  It is not a
new command surface and it does not expose raw shell argv to agents.  It
describes:

- the approved tool catalog, such as =rg=, =fzf=, =fd=, =ast-grep=,
  =ugrep=, or a provider index
- tool roles, such as source scan, fuzzy filter, path list, structural recipe,
  candidate normalization, or scoring
- approved pipelines, such as =fzf-rg-lines=, =text-rg-json=, or
  =pattern-ast-grep-recipe=
- provenance fields that search packets can later embed, such as
  =finderPipelineId=, =matchMode=, =candidateBasis=, =sourceSearchPasses=,
  =fuzzyFilter=, candidate counts, and runtime cost

The first target pipeline is:

#+begin_src text
search fzf <query>
  -> provider-owned rg candidate scan
  -> provider-owned fzf headless filter/rank
  -> provider normalize
  -> owner/test seeds
#+end_src

=rg= owns fast lexical candidate generation.  =fzf= owns fuzzy filtering and
ranking over candidate lines, paths, or owner labels.  The language provider
owns the semantic boundary after that: path normalization, owner resolution,
nearest-item resolution, test frontier selection, deduplication, caps, and
packet rendering.

Agents do not choose the =rg= command, stage order, or a shell pipe shape.
Agents may request provider-approved fzf match options only through
=--fzf-arg= or the terminal =--fzf= boundary.  The provider owns validation,
normalization, defaulting, and provenance; unsupported options are rejected
with a compact error or recorded as rejected finder arguments in JSON.

=search fzf= may reuse the same finder schema with
=surface=search-fzf=, but the preferred agent-facing surface is
=surface=search-fzf= and view =fzf=.

Agent-facing finder runs must be headless and provider-owned.  Human-facing
diagnostics may later expose interactive =fzf= preview, but that is outside
the agent protocol path.

*** ast-grep boundary

=ast-grep= has a useful place, but it is not a substitute for provider-native
=query=.  The provider's native parser and query implementation is stronger for
that language because it owns project configuration, package/workspace scope,
compiler or parser facts, item identity, test reachability, provider-specific
handles, and compact code projection.  For Rust, =rs-harness query/search= is
the authority for Rust semantic extraction; =ast-grep= must not replace it as a
parallel Rust query engine.

The correct =ast-grep= boundary is structural evidence where native provider
facts are absent, incomplete, or intentionally recipe-scoped:

- =search pattern <recipe>= can use provider-approved =ast-grep= recipes for
  structural patterns without asking the agent to write raw =ast-grep= DSL
- a Rust provider may combine =rs-harness= owner/query evidence with
  =ast-grep= structural matches when the structural recipe adds non-native
  evidence, but the final packet must mark =ast-grep= as structural provenance,
  not parser-owned truth
- for languages without a mature provider, =ast-grep= can supply bounded
  structural candidates that are then normalized by the generic finder pipeline
  and clearly marked as non-native structural evidence
- cross-language investigations may combine one language provider's native
  query results with =ast-grep= structural candidates from unsupported
  languages, as long as the synthesis preserves source/provenance boundaries

In short: native harness =query= is the precision layer; =ast-grep= is a
structural adapter and recipe runner; =rg+fzf= is the lexical fuzzy candidate
pipeline.

** Source owner context is provider query transport, not root read transport

The durable discovery path is parser-native =search= output.  Providers should
return compact parser-owned evidence: owners, items, tests, dependency edges,
findings, and =searchSynthesis=.  Once the hook has one exact selected source
path, the root runtime must route back to the owning provider's parser query
surface.  Root-level read recovery is removed and must not be advertised as the
recovery route.  If
=semantic-read-packet= is useful, it is emitted by the language provider as an
optional =query/*= output schema after that route has been followed.  The
read-plan frontier shape is part of the active provider-owned read-packet
contract; providers must not advertise a separate legacy recovery route.

Direct source reads and exact wrapper reads should be denied and rerouted to a
compact code window:

#+begin_src shell
asp <language> query --from-hook direct-source-read --selector <project-path[:line-range]> --code .
#+end_src

The hook decision packet must label that recovery route as =kind=query=.
Provider output may still contain owner, item, test, and read-locator facts, but
the executable route is the provider query surface, not a =search owner= route.
This keeps exact source paths, wrapper line locators such as =rtk read
src/lib.rs:10:20=, and cross-language direct reads on the same semantic
transport.

Broad selector reads should instead route to provider =query --from-hook=,
=query --term ...=, =search fzf ... owner tests=, or =search ingest ...=
depending on the evidence already available.  The provider owns the parser
facts, frontier selection, owner item projection, and broad discovery; the hook
only classifies and routes.

The hook-facing =query --from-hook= form is for broad selectors where no single
owner is known, for example =Read *.ts= or =less **/*.tsx= paired with concrete
terms from the agent's intent:

#+begin_src shell
asp <language> query --from-hook direct-source-read \
  --selector '**/*.{ts,tsx,js}' \
  --intent inspect-cli-search \
  --term parseSearchArgs \
  --term querySets \
  --surface owners,tests \
  --view seeds .
#+end_src

It is a compact route into the provider's parser-owned query layer.  It should
emit bounded query evidence: query coverage, owner candidates, item candidates,
test seeds when requested, and any visible runtime prefilter cost.  It must not
expand the matched files into source text.  =query --from-hook= is the only
agent-facing wildcard direct-read recovery route.  Providers must not advertise
or accept =search query --from-hook= as a compatibility alias; keeping both
forms creates route drift and breaks guide-quality replay.

When the selector is one exact project-relative source path and the agent has no
reliable item query yet, top-level =query --from-hook direct-source-read
--selector <path>= without =--code= may return owner =items= context directly.  This is still
parser-owned compact context, not source rendering: it lists item names, line
ranges, read locators, and owner/test frontiers so the next step can be a precise
item query or focused edit.

Rust adds Cargo-native views under the same search namespace:

#+begin_src shell
rs-harness search workspace
rs-harness search targets
rs-harness search deps
rs-harness search deps <dep[/subpath][@version][::api]>
rs-harness search deps <dep[/subpath][@version][::api]> public-api
#+end_src

=search dependency= is a high-level composite view: native package/dependency
facts + source usage + owner graph in one compact output.  For Rust,
=search deps= is the practical dependency-version API usage view: manifest
version/feature/target facts and local API usage owners for the version range
declared by the current workspace.  If a query asks for =dep@version= that does
not match the workspace declaration, the packet must mark it as
=versionScope=external= and avoid presenting current owner usage as evidence
for that external version.  Current workspace versions are reported as
=currentWorkspaceVersion= when a lockfile or manifest exact version proves the
value.  For current workspace versions,
=::api= narrows owner hits to dependency owners that also mention the API token.
=search prime= is the low-token entry point with graph structure.

** Multi-pipe: deterministic packet composition

A provider can optionally chain multiple pipe stages in one invocation.  This
is an RFC target for providers that can keep the packet final-only and
deterministic; agents must still treat the registry's =methods= and
=methodDescriptors= as the callable truth:

#+begin_src shell
<lang-harness> search <source> <pipe1> <pipe2> ...
rg -n "<query>" <roots> | <lang-harness> search ingest <pipe1> <pipe2> ...
#+end_src

Source stages (produce the initial packet):

#+begin_src text
prime                 project reasoning-tree
dependency <dep>      dependency fact + source usage + owner graph
features <feature>    feature -> optional-dep -> target cfg chain
deps <dep>            native manifest dependency facts
symbol <symbol>       symbol definition + callsites
callsite <symbol>     all usage sites
pattern <recipe>      structural recipe search
ingest                stdin auto-sniff (rg -n / fd / git diff)
#+end_src

Candidate pipe stages (enrich the packet, all optional):

#+begin_src text
owners      resolve to owner graph nodes
items       resolve to intra-file items
tests       find owned tests/benches/examples
examples    find owned examples
benches     find owned benches
docs        attach API docs facts (kind, signature, feature, async, ...)
public-api  mark whether dependency types leak into public API
cfg         attach cfg / target facts
features    expand feature graph
dependents  who depends on this owner/dep
#+end_src

Each pipe is a deterministic transform.  The CLI renders only the final
packet.  The agent client selects the pipe combination based on the task; the CLI
never infers intent.

Examples:

#+begin_src shell
# Dependency change with docs and test coverage:
rs-harness search dependency ignore items docs tests --package file-search

# Feature/cfg change across owners and docs:
rs-harness search features io-uring cfg owners tests docs --package tokio

# Public API leakage check:
rs-harness search dependency bytes public-api docs tests --package tokio

# External candidate stream with grouping:
rg -n "require_git|git_ignore|parents" agent-rs/file-search | rs-harness search ingest items tests
#+end_src

When a provider advertises diagnostic tracing, it should return a compact
stage list rather than full intermediate packets:

#+begin_src text
[search-trace] source=dependency query=ignore pipes=items,docs,tests
|stage dependency cargo=1 owners=1
|stage items owners=1 items=4
|stage docs docs=1
|stage tests tests=1
#+end_src

*** Dev-mode command log

Providers may record every =*-harness= CLI invocation when development mode is
explicitly enabled:

#+begin_src shell
SEMANTIC_PROTOCOL_DEV_MODE=1
#+end_src

This is a provider debugging surface, not an agent prompt surface.  Its purpose
is to measure real command count, latency, repeated search loops, view/pipeline
selection, and hook reroute behavior without asking the agent to expand source
or request JSON during normal exploration.

The canonical cache location is:

#+begin_src text
$PRJ_CACHE_HOME/semantic_protocol/<language>/<provider>/commands/YYYY-MM-DDTHH-MM-SSZ-000001-<event-id>.jsonl
#+end_src

If =SEMANTIC_PROTOCOL_TRACE_DIR= is set, providers should write under that
directory instead of deriving =PRJ_CACHE_HOME=.  If neither location can be
resolved, the log write is silently disabled.  Command execution must not fail
because a dev-mode log directory cannot be created or appended.

Each JSONL row must validate against
=semantic-dev-command-log.v1.schema.json=.  Providers record argv, cwd,
project root, language/provider identity, normalized command method,
=startedAtUtc=, =finishedAtUtc=, elapsed time, exit code, and stdout/stderr
byte counts.  Providers must not record full stdout/stderr content by default;
those streams can contain source text, private paths, or model-visible prompt
material.  Obvious secret-bearing arguments such as =--token=, =--api-key=,
=--password=, and =--secret= must be redacted in the recorded argv.

The file name carries a UTC second timestamp for human directory inspection,
but analysis tools should sort by row fields rather than file names.  Within a
session, providers should allocate a best-effort monotonically increasing
=sessionOrdinal= before command execution starts.  The ordinal is provider
local; fan-out analyzers that combine Rust, TypeScript, Python, and other
providers should sort by =sessionId=, =startedAtUtc=, =sessionOrdinal=, and
=eventId= rather than assuming a single global counter.  When a client can pass
context directly, it should set =SEMANTIC_PROTOCOL_SESSION_ID=,
=SEMANTIC_PROTOCOL_PARENT_EVENT_ID=, and =SEMANTIC_PROTOCOL_HOOK_RUN_ID=.
Because real agents often remember a hook warning and then invoke =*-harness=
directly without any hook-provided environment, hook runtimes may also write a
short-lived active-context marker:

#+begin_src text
$PRJ_CACHE_HOME/semantic_protocol/dev-context/<project-root-hash>.json
#+end_src

Providers should read that marker only as a best-effort development trace aid,
with a short TTL.  The marker must validate against
=semantic-dev-active-context.v1.schema.json=.  Providers should mark inherited
command events with =fields.contextSource= =active-context=.  When no
environment or active marker exists, the provider may fall back to a
project-scoped session id.  This makes the trace useful for reconstructing
which model step fanned out into which harness commands, even when commands
overlap, finish out of order, or run after the original hook event has already
returned.

The command log is implemented at each provider CLI entrypoint.  The root hook
may pass hook context such as a hook run id through environment variables, but
it is not the primary log owner: direct CLI usage, sandtable replay, and
hook-rerouted commands should all land in the same provider-owned log stream.

Internal model:

** Wire contract: semantic language registry + shared =semantic-search= JSON Schema

Every provider first advertises through the semantic language registry, then
maps native search packets to the shared =semantic-search= wire contract when
=--json= is requested.  The registry and packet schemas live at:

#+begin_src text
schemas/semantic-language-registry.v1.schema.json
schemas/semantic-search-packet.v1.schema.json
#+end_src

The protocol repository's =schemas/= directory is the source of truth.
Provider packages that run their own CI from independent checkouts may carry
package-local copies at the same relative paths.  Their tests should validate
against the package-local files, and monorepo/submodule checkouts should compare
the package-local copies with the repository source schemas to prevent drift.

Schema evolution rules:

#+begin_src text
Additive v1 change     add optional fields, enum members, or descriptors
Breaking change        new schema file and schemaVersion, e.g. *.v2.schema.json
Provider adoption      update package-local copies and sync tests together
Registry advertisement advertise every emitted schema id/version/path
Client behavior        accept known schema versions; reject unknown versions with a compact diagnostic
#+end_src

Never change the meaning of a v1 field in place.  If an agent could interpret
old and new packets differently, the change is breaking and needs a new schema
version plus registry advertisement.

Registry identity fields:

#+begin_src text
registryId      = agent.semantic-protocols.semantic-language-registry
registryVersion = 1
protocolId      = agent.semantic-protocols.semantic-language
protocolVersion = 1
languageId      = rust | typescript | julia | python | ...
providerId      = rs-harness | ts-harness | jl-harness | ...
binary          = rs-harness | ts-harness | jl-harness | ...
namespace       = agent.semantic-protocols.languages.<languageId>.<providerId>
methods         = search/prime | search/dependency | check/full | agent/doctor | ...
methodDescriptors = method + command + view/input + output modes + schemas
#+end_src

=methods= is the callable truth.  The shared packet schema can contain a
cross-language =view= enum, but that enum does not advertise provider
capability.  Agents should only call methods present in the registry and use
=methodDescriptors= to discover the command grammar: whether a method is a
=search= view, which schema it emits, whether it requires one query positional,
whether it consumes stdin, whether =--package= is accepted, and whether
compact/JSON output is supported.  A =search/*= descriptor must carry its
=view=, emitted schema ids, and CLI input semantics; a =check/*= descriptor must
not pretend to be a search view; an =agent/*= descriptor only requires
=outputSchemaIds= when =supportsJson=true=.  Text-only helper methods such as
=agent/guide= can be registry-described without inventing a JSON schema they do
not emit.

A =query/*= descriptor must use =command="query"=, must not carry a search
=view=, and must advertise the schema it emits.  For owner-local item lookup,
providers should expose a descriptor such as =query/owner-items= with
=input="owner-path"=, =requiredOptions=["--term"]=,
=outputSchemaIds=["agent.semantic-protocols.semantic-query-packet"]=,
=querySetScopes=["owner"]=, and =outputModes= such as =compact=, =json=,
=code=, and =names=.  That makes query repair discoverable through
=agent doctor --json= rather than hidden in hook prose.

When a search view embeds a shared sub-schema inside the search packet, the
descriptor must advertise both the packet schema and that sub-schema.  For
example, =search/public-external-types= emits the stable
=semantic-search-packet= envelope and may populate =typeSurfaces= entries owned
by =semantic-type-surface=, so providers list both schema ids in
=outputSchemaIds=.  This keeps client validation and cache keys contract-visible
without turning provider-local type systems into shared protocol fields.

Registry invariants are deliberately strict:

#+begin_src text
languageId   stable lowercase source language id, not a file extension list
providerId   stable provider implementation id, e.g. ts-harness
binary       exact executable name the agent should invoke
namespace    agent.semantic-protocols.languages.<languageId>.<providerId>
methods      complete callable set
descriptors  exactly one descriptor for every method, no extras, no duplicates
#+end_src

Providers advertise exactly one semantic-language identity in the registry.
Extra local entry points, if a package has any, are outside the protocol and
must not appear in registry output, generated guides, schemas, or RFC examples.
For TypeScript, the only advertised provider and binary is =ts-harness=.

The descriptor command must match the method prefix.  For =search/<view>=,
=view= must be the method suffix, =outputSchemaIds= must include
=agent.semantic-protocols.semantic-search-packet=, and the descriptor owns CLI
input semantics such as =requiresQuery=, =acceptsStdin=, and
=supportsPackageScope=.  CLI parsers should consume this provider metadata
instead of maintaining a second hard-coded view registry.  Each provider's
tests should compare =methods= and =methodDescriptors= as sets and reject
duplicate descriptor methods.

Descriptor grammar is intentionally small in v1:

#+begin_src text
command              search | check | agent
view                 search method suffix only
requiresQuery         exactly one query positional before PROJECT_ROOT
acceptsStdin          stdin is part of the method input
supportsPackageScope  --package <package-id> is accepted
supportsQuerySet      --query-set <term> may replace the single query positional
acceptedQuerySetSelectors exact-set | prefix-set | stdin-path-set
querySetScopes        project | package | owner contexts accepted by --query-set
acceptedPipes         final-only pipe names accepted after the method query
capabilities          direct searchable surfaces for this method
ingestRequiredFor     surfaces that must route through search ingest
fallbacks             bounded provider fallbacks for weak or missing query hits
clients               agent clients or hooks with specialized command shapes
requiredOptions       options required by the specialized input shape
input                 compact human-readable grammar for non-default input
supportsJson          --json is accepted
supportsCompact       compact text is accepted/default
outputModes           descriptor-advertised output modes, e.g. compact, json, code, names, outline
outputSchemaIds       JSON schemas emitted when supportsJson=true
#+end_src

If a provider needs additional public controls, they must be added to the
descriptor schema first.  Provider-private flags may exist for local debugging,
but they are not semantic-language protocol until they are registry-described.
This keeps agent clients from learning accidental flags or inferring CLI shape
from the shared packet =view= enum.

Search capability descriptors answer the two questions agents need before they
spend tokens expanding context:

#+begin_src text
capabilities      what the provider can search directly
ingestRequiredFor what needs rg/fd/external expansion then search ingest
fallbacks         what bounded provider-owned recovery can run after a miss
#+end_src

The common registry schema owns only the descriptor shape:
=languageId=, =namespace=, and =name=.  It must not maintain one global enum for
TypeScript, Rust, Python, Julia, and JavaScript capability names.  Each language
harness owns its language-specific capability schema under that language
repository's =schemas/= directory and can advertise that schema in its provider
registry.

The protocol repository may keep language-specific schema templates under root
=schemas/=, for example
=schemas/typescript-semantic-capabilities-template.v1.schema.json= and
=schemas/python-semantic-capabilities-template.v1.schema.json=.  Those templates
document the expected active schema shape and vocabulary, but the provider's own
repository still carries the active schema it advertises and validates in CI.

For example, a TypeScript provider can advertise
=languageId=typescript namespace=typescript name=parser-visible-source-text-search=
for =search/fzf= while also advertising TypeScript-scoped
=ingestRequiredFor= records for docs text, schema JSON, generated artifacts,
and other non-parser text.  Agents should use that registry fact before reading
packet notes or running external search.

Minimal multi-pipe support is also registry-described.  A provider advertises
=acceptedPipes= on the source method, for example TypeScript =search/fzf=
can advertise =owner,tests= and accept:

#+begin_src shell
ts-harness search fzf OrderStatus owner tests .
#+end_src

The output remains one final packet; intermediate pipe packets are not rendered.

Search packet identity fields:

#+begin_src text
schemaId        = agent.semantic-protocols.semantic-search-packet
schemaVersion   = 1
protocolId      = agent.semantic-protocols.semantic-language
protocolVersion = 1
languageId      = rust | typescript | julia | python | ...
providerId      = rs-harness | ts-harness | jl-harness | ...
binary          = rs-harness | ts-harness | jl-harness | ...
namespace       = agent.semantic-protocols.languages.<languageId>.<providerId>
method          = search/workspace | search/prime | search/dependency | search/deps | ...
view            = workspace | prime | owner | dependency | deps | symbol |
                  callsite | import | cfg | patterns | pattern | docs |
                  api | public-external-types | tests | text | ingest
#+end_src

The schema owns the stable packet envelope: =header=, =packages=, =nodes=,
=edges=, =owners=, =items=, =hits=, =findings=, =nextActions=, =notes=, and
optional =inputDetection=.  Optional =typeSurfaces= carries shared public API
and dependency type surface facts when a view has parser-owned type evidence.
Language-specific facts must go under typed
=fields= maps instead of changing the envelope.  Search views are consumers of
native parser facts, not substitutes for them: TypeScript package dependency
facts come from the TypeScript JSON AST for =package.json=, source usage comes
from TypeScript import-resolution facts, Rust dependency facts come from Cargo
metadata/TOML parsing, Julia facts come from JuliaSyntax and project metadata,
and Python facts come from =ast= plus project metadata.  This lets TypeScript,
Rust, Julia, Python, and later harnesses preserve compiler-native facts while
giving agent clients one JSON shape to validate and cache.

Compact text remains the default prompt surface.  JSON is the cross-language
artifact and test contract.

Semantic handles are the shared layer for non-code facts that agents still need
to query precisely.  They cover stable things that are not necessarily parser
items: policy rule ids, schema fixtures, test cases, config keys, CLI command
surfaces, dependency API handles, and provider capability names.  The
=schemas/semantic-handle.v1.schema.json= contract owns the cross-language
fields: stable id, kind, source, title, aliases, labels, owner path,
implementation owner, test paths, locations, relations, and provider-owned
=fields=.  Search packets and query packets can embed these facts through
optional =semanticHandles=.

This is deliberately not a Python-only policy shortcut.  Rust harness policy
ids, TypeScript schema fixture names, Python rule ids, Julia package checks, and
future provider-specific facts should all use the same handle shape when the
fact is stable and useful as an agent query target.  Provider implementations
remain language-native; the shared protocol only makes the handle observable.

Policy-rule lookup is the motivating example:

#+begin_src shell
<lang-harness> search policy <rule-id-or-alias> owner tests --view seeds .
#+end_src

Expected compact facts:

#+begin_example
|handle kind=policy-rule id=PY-PROJ-R001 title="src layout" aliases=src-layout,packaged-project-layout owner=src/python_lang_project_harness/_project_policy_catalog.py implementation=src/python_lang_project_harness/_project_policy_layout.py tests=tests/unit/harness/project_policy/test_layout.py
|next owner:src/python_lang_project_harness/_project_policy_layout.py,tests:tests/unit/harness/project_policy/test_layout.py
#+end_example

Owner item query should not pretend these ids are function names.  If a selected
owner is a catalog, the provider may expose a handle-scoped pipe or query mode:

#+begin_src shell
<lang-harness> search owner <catalog-path> handles --query <rule-id-or-alias> .
#+end_src

That output should return =semanticHandles= first and code only when the agent
asks for the implementation owner or exact item.  For broad owner-local
discovery such as =test=, =minimal_packet=, or =parse_=, providers should return
=outputMode=names= or =outline= before code windows.

Type surface facts are the first shared sub-schema extracted from the search
envelope.  =schemas/semantic-type-surface.v1.schema.json= owns only the
language-neutral facts that an agent can use across providers: type name, kind,
role, owner path, visibility, member shape, external origin, and version scope.
It intentionally does not model full Rust, TypeScript, Python, Julia, or future
provider type systems.  Language-specific type-checker details remain
provider-owned =fields= facts or provider-local schemas.  Search packets can
embed these shared facts through optional =typeSurfaces=, primarily for
=search api=, =search public-external-types=, and parser-native query outputs
that need contract-visible type surfaces instead of stringly =fields= only.

This mirrors the useful part of Language Server Protocol naming discipline:
=protocolId= identifies the semantic-language protocol, =namespace= identifies
the concrete provider method space, =languageId= identifies the source
language, =providerId= identifies the implementation, and =binary= names the
executable.  Agent clients should route by registry identity and method, not by
hard-coded platform-specific argument structs.

* The two-phase search model: deep prime → multi-pipe

Phase 1 starts with =<lang-harness> search workspace= only when the selected
root is a multi-package workspace.  That command is a router/index: it lists
packages, workspace-internal edges, external dependency axes, findings, and
next package candidates.  It must not deep-search source files.

Phase 1 then uses =<lang-harness> search prime= for the selected package or for
single-package projects.  Prime is still a purpose-built, deep but bounded graph
packet designed to give the LLM a package-level search map.  Multi-pipe is Phase
3+, used by subagents after the parent has selected package and axis.

#+begin_src text
Phase 1a: <lang-harness> search workspace   (workspace router, if applicable)
Phase 1b: <lang-harness> search prime       (package-level deep map)
Phase 2:  agent parent reasons about axes   (LLM task, not CLI)
Phase 3:  focused searches                  (dependency/symbol/owner/ingest)
Phase 4:  parent synthesis → edit plan      (LLM task, not CLI)
Phase 5:  <lang-harness> check --changed    (validation)
#+end_src

Public TypeScript sandbox runs calibrate this model against large real
repositories.  =vitejs/vite= requires =search workspace= to ingest
=pnpm-workspace.yaml= before prime, otherwise the parent agent starts from an
incorrect single-package map.  When a workspace pattern also includes fixture
packages such as =packages/**/__tests__/**=, the workspace router must rank
source packages before fixture packages so the capped package list still gives
the parent useful =prime:<package>= choices.  =microsoft/TypeScript= requires
prime to degrade through parser-version and deep-AST pressure while preserving
per-file import resolution, otherwise the next =owner= / =import= / =tests=
pipes lose graph edges after fallback.  Text-flow evidence such as
=surface=source|test=, =ownerRole=, and bounded =text= snippets exists so
follow-up filters do not re-search the original query token.

** What workspace covers (and what it does not)

Workspace is a package router, not a repository-wide search.  It outputs just
enough package routing facts for the agent to choose a package and then run
=search prime --package <package-id>=.  It never outputs raw hits, source
lines, docs prose, or full per-owner graphs:

#+begin_src text
workspace/package scale
workspace package list (ranked/capped)
workspace-internal package edges
external dependency axes across packages
finding groups by package
next hints (package/dependency axes, not hits)
#+end_src

Workspace routing invariants:

#+begin_src text
package id     root-relative package path; exact value accepted by --package
package name   native package name from package metadata, when present
package role   workspace-root | workspace-package | single-package
package config native config path, e.g. package.json, Cargo.toml, Project.toml
package edges  workspace-internal import/reference/member edges only
findings       package-scoped counts/groups, not full diagnostics
owners/items   empty in workspace packets
hits           empty in workspace packets
#+end_src

If a provider advertises =search/workspace= for a single-package project, it may
return one =single-package= row with =next=prime:.=.  Agent guides should still
skip workspace for known single-package roots and start with =search prime=.

Workspace output format:

#+begin_example
[search-workspace] mode=workspace-index package=@repo/root packages=12 shown=12 edge=18 external=7 find=3
|package . name=@repo/root role=workspace-root packages=12 patterns=packages/* dependencies=7 next=prime:.
|package packages/core name=@repo/core role=workspace-package pattern=packages/* type=module config=packages/core/tsconfig.json diagnostics=0 next=prime:packages/core
|package apps/web name=@repo/web role=workspace-package pattern=apps/* type=module config=apps/web/tsconfig.json diagnostics=0 next=prime:apps/web
|edge P:. -workspace-> P:packages/core importKind=type-import via=package-name package=@repo/core from=src/index.ts
|next prime:packages/core,prime:apps/web
#+end_example

** What prime covers (and what it does not)

Prime outputs package-level structure, signals, and next candidates.  It never outputs
raw hits, source lines, or docs prose:

#+begin_src text
native dependency / feature / cfg or package/config signals
owner graph (ranked, capped)
source_large owner warnings
public API owners
test/bench/example surfaces
finding groups
docs candidates
next hints (axes, not hits)

Owners are ranked, not listed alphabetically:

#+begin_src text

** Graph-derived search synthesis

Search providers may run bounded graph algorithms over their native reasoning
tree to make LLM planning cheaper.  The graph is an internal computation input;
the prompt-facing packet gets only compact graph evidence and synthesis inside
the existing =search= result.  The graph vocabulary is owned by
=schemas/semantic-graph.v1.schema.json= and embedded into search packets through
compatible =nodes=, =edges=, and =searchSynthesis= fields.  Providers must not
add a separate =search graph= lane for the agent, and must not hide graph
semantics in language-private =fields= maps:

#+begin_src text
algorithm        named algorithm, e.g. owner-rank-frontier or bounded-reachability-depth1
scope            prime | owner | dependency | query-set
highImpactOwners ranked owner paths that explain the selected axis
frontierOwners   owner paths one graph step beyond the current packet
editFrontier     source owners the next edit/read window should consider
testFrontier     test owners tied to the selected source frontier
windowSet        typed owner/test/read window targets selected before source windows
findingOwners    owner paths with policy findings in the current graph slice
seeds            next actions derived from graph frontier, not text coincidence
#+end_src

Graph synthesis must never replace parser-owned facts.  It ranks, trims,
explains, and proposes next actions from owner/dependency/test edges already
owned by the language harness.  If the provider cannot prove an edge from
parser, package metadata, schema registry, or test ownership facts, the graph
algorithm must not present it as an impact path.
RFC 006 defines how prompt-facing =seed= and =synthesis= semantics are folded
back into the graph feature: seeds are selected frontier actions, and synthesis
is rendered through =alg=, =rank=, =frontier=, and =avoid= rather than a
separate prompt-facing protocol.

For =search prime --view seeds=, providers should still use their native parser
and reasoning tree.  A seed packet built from directory/file discovery alone is
too close to a path list and causes the LLM to fan out across unrelated files.
The compact seed packet should run the same owner graph ranking as full prime
and derive its frontier actions from =searchSynthesis=:

#+begin_src text
highImpactOwners  ranked owners shown in the packet
frontierOwners    adjacent owners one parser-owned graph step outside the packet
findingOwners     owners carrying grouped findings
seeds             owner/test/dependency frontier actions derived from the graph
#+end_src

For =search owner <path> --view seeds=, providers should emit
=bounded-reachability-depth1=: incoming owner count, outgoing owner count,
frontier owners, finding owners, and owner/test seeds.  This makes the next
agent step a graph decision instead of a new round of text fragments.

RFC 006 owns the compact rendering.  The old single =|synthesis= line is
superseded by graph-integrated rank and frontier facts:

#+begin_src text
[search-graph] mode=prime root=. alg=owner-rank-frontier
legend: ID=kind:role(value)!next; edge SRC>{DST:rel}; frontier ID.next
alias: graph:{G=search,L=owner,T=owner}
L=owner:path(src/lib.rs)!owner
T=owner:path(src/tests.rs)!owner
L>{T:frontier}
rank=L,T frontier=L.owner,T.owner
#+end_src

Agent-facing graph feature semantics are specified separately in RFC 006,
=rfcs/agent-compact-graph-feature.org=.  RFC 005 owns the CLI surface and
search/query command lifecycle; RFC 006 owns the LLM-oriented graph facts,
rank, compact projection, and next-action semantics shared by all =search=
surfaces.

** How the agent client uses prime to build the subagent flow graph

Agent guide:

```md
## Phase 1: deep prime

Always start complex Rust tasks with:

`rs-harness search prime`

Read mode, package, feature, cfg, owner, large-owner, api-candidate,
test-surface, and next fields.

Do not spawn subagents before reading prime unless the user provides an
exact file and symbol.

## Phase 2: parent planning

Use prime output to choose one or more search axes:

- feature/cfg axis
- dependency/API docs axis
- owner/item axis
- public API/callsite axis
- tests/bench/examples axis
- harness finding axis

Spawn read-only subagents only after selecting axes.

Example: if prime shows `feature io-uring ... requires_cfg=tokio_unstable`:

feature subagent:
  `rs-harness search features io-uring cfg owners tests`
cfg subagent:
  `rs-harness search cfg tokio_unstable owners tests`
source subagent:
  `rg -n "io_uring|tokio_unstable" tokio/src tokio/tests | rs-harness search ingest items tests`
```

** =next= placement rule for prime

=next= on owner lines targets that owner implicitly.  Global =|next= only
for candidates without a clear owner context.  Never duplicate an
owner-specific next into the global =|next= line.

** Confidence modeling in prime

When facts cannot be confirmed from parser/Cargo data alone, prime uses
explicit confidence markers rather than guessing:

#+begin_src text
reason=public-api-carrier         confirmed from parser facts
reason=possible-public-api-carrier uncertain, needs public-api pipe
reason=likely-parser-owner        heuristic, needs owner pipe
public_external_allowed=true      confirmed from cargo metadata
source=workspace|path|crates-io|git  dependency origin
#+end_src

This prevents prime from misleading the agent client with false certainty on
cross-crate or macro-heavy codebases like rustc.  The =source= field on
=cargo= lines distinguishes workspace-internal from crates-io
dependencies, which affects search strategy and docs-source selection.

** Strong signal: =source_large=true=

When prime marks an owner as =source_large=true=, the agent client must NOT open the
file directly.  Instead:

```sh
rs-harness search owner <file> items
```

This rule prevents token explosion from files like =file-search/src/lib.rs=
(1192 lines, ~12 public items) or =tokio/src/runtime/mod.rs= (668 lines).

** Shared flags on all =search= views

#+begin_src text
                 default agent output (compact line protocol)
--json                    tool / IDE / flowhub output
#+end_src

Agent clients must use compact output for interactive exploration.  =--json=
is only for schema-contract tests, sandtable receipts/replay, IDE/Flowhub
consumers, or a local validator that consumes the JSON directly.  It is not a
prompt-facing discovery mode: repeated commands such as
=search fzf <query> owner tests --json .= waste tokens, hide the intended line
protocol guidance, and should be rejected by agent hooks with a safe compact
route.

When a provider does emit =--json= for a machine consumer, the renderer should
prefer minified JSON.  Pretty JSON is easier for humans but produces the same
facts with extra prompt-visible whitespace; schema-contract sandtables and
receipts should validate the structured payload instead of relying on
line-oriented JSON formatting.

When an agent asks for JSON during exploration, the hook denial should preserve
the semantic route, explain that agent search commands must not use =--json=,
and remove only the output-mode error.  For example:

#+begin_src text
denied: asp typescript search fzf location.path owner tests --json .
guide:  asp typescript search fzf location.path owner tests --view seeds .
#+end_src

If the rejected JSON-output command already carries a concrete owner path, the
hook should not fall back to a generic guide.  It should route to the
provider's owner query command and derive the compact query from the path or
provider-owned query fields, using this shape:

#+begin_src text
asp <language> search owner <path> items --query <derived-query> .
#+end_src

For =search fzf=, the first required query positional is owned by the query,
even when the literal text begins with a dash.  For example,
=ts-harness search fzf --json --view seeds .= searches for the literal
=--json= token.  A structured output request must put the output flag after the
query, e.g. =ts-harness search fzf OrderStatus --json .=.  This prevents a
flag-like source token from silently becoming a prompt-facing JSON packet.

If a daemon is running, the CLI internally routes through the daemon socket.
If no daemon is running, the CLI runs the native parser directly.  The agent
never knows which path was taken.

#+begin_example
public contract:
  CLI args + stdout compact/json + exit code

private optimization:
  local daemon, TypeScript Program cache, file watcher, socket IPC
#+end_example

* Daemon failure matrix

| Daemon State          | CLI Behavior                          |
|-----------------------+---------------------------------------|
| running, cache fresh  | fast incremental result               |
| running, cache stale  | auto-rebuild cache, then respond      |
| not running           | direct parser run (same output)       |
| protocol mismatch     | deterministic upgrade message, exit 4 |
| crashed mid-request   | CLI retries once, then direct run     |

The agent never receives "server unavailable" as an error — only as a
=daemon=missing= diagnostic field when a provider chooses to expose daemon
status, which is informational.

* Compact format specification: line protocol + ASCII graph

=<lang-harness> search= output is a /line protocol/, not a Markdown report.
It inherits the existing Rust harness compact contract: stable tags, single
locator syntax, short action lines, minimal fact lines, no audit headers, no
prose.

Rules:

#+begin_src text
- no blank lines
- no Markdown headings
- no bullet indentation
- no source excerpts in default search packets
- no Help / Contract unless it is a check finding
- no prose paragraphs
- paths relative to project root
- every line independently readable
- every block default ≤ 12 lines
- no Unicode arrows, no box-drawing, no DOT/Mermaid/Graphviz
#+end_src

Path values are canonical project-root-relative paths, not display locators.
They must not include rank prefixes, node slots, URI schemes, line ranges,
column numbers, or command prefixes.  In particular, values such as
=0:src/components/WorkflowExecution.tsx=, =owner:src/lib.rs=, and
=src/lib.rs:42:90= are not paths.  Put rank/score in separate fields
(for example =rank=0= or =score=2), put line/column data in =locations=, and
use command fragments only inside =next= values.

This rule also applies when the provider internally works with package-local
paths.  Agent-facing owner paths, hit paths, query selectors, and next-action
arguments remain project-root-relative.  If a provider needs to expose a package
name, package root, package-local path, rank, graph slot, or display locator, it
must use a separate field such as =package=, =packageRoot=, =packagePath=,
=rank=, or =node=.  It must not shorten =packages/typescript/src/foo.ts= to
=src/foo.ts= in command-ready fields, because agents will copy that value into a
read/search/edit command against the workspace root.

Edge endpoints are graph node ids, not path values.  Owner node ids use the
typed locator =O:<projectPath>=; they must not use numeric prefixes such as
=0:<projectPath>=.  If the provider wants to render ordering, it should add
=rank=N= instead of changing the owner/path identity.

Line shape:

#+begin_src text
[search-kind] k=v k=v k=v          ← header line, always first
|node KIND:id k=v                   ← typed graph node
|edge KIND:id -kind[:label]-> KIND:id  ← ASCII edge, always " -kind-> "
|owner path k=v next=...             ← owner with inline next actions
|item name kind=... k=v next=...     ← intra-file item with inline next
|hit path=<path> line=N column=N ... ← hit location uses fields, not path:line
|find RULE xN path=<path> node=KIND:id ← finding path first, graph node second
|next kind:arg,kind:arg(scope=x)     ← global next actions (rare, only when
                                       no owner/item context exists)
|note k=v                           ← exceptional info (keep ≤ 1)
#+end_src

=|item= lines only appear when the request includes the =items= pipe or an
advertised provider method returns item facts by default.
=|item= lines must not repeat fields already encoded by their read locator:
=read=path:start-end= carries the owner path and line range.  Providers should
also suppress normal boolean defaults such as =public=false=, =doc=false=,
=exported=false=, and =typeOnly=false=; emit them only when they carry positive
or exceptional information.
=|owner= lines should suppress healthy/default validation fields such as
=public=false=, =parserOwner=false=, =valid=true=, =syntaxDiagnostics=0=, and
=semanticDiagnostics=0=.  Emit diagnostics, invalid states, parser ownership,
or public/export facts only when they change the agent's next action.
=|edge= lines should suppress normal boolean defaults such as =typeOnly=false=
and =test=false=.  Emit =typeOnly=true=, =test=true=, or exceptional resolution
facts when they affect the edit/test boundary.
=|hit= lines must render the path as =path=<projectPath>= and line/column as
separate fields.  They must not use =src/lib.rs:42:7= or
=src/lib.rs:42:90= as the positional value because agents will copy that token
as a file path.
=|hit= lines should suppress =owner=<path>= when =owner= equals =path=, and
should suppress true/default provenance fields such as =realOwner=true= unless
the value is exceptional.
=|next= is an inline field on owner/item/package/cargo lines, not a separate
line type.  A standalone =|next= line is only used when no owner/item
context exists to attach to.

#+begin_src text

* Output mode: compact is the default

=rs-harness= is agent-first.  All commands render compact text by default.
There is no =--compact= flag.  =--json= is for structured consumers.  Provider
descriptors may also advertise narrower text projections such as =--code= for
already-selected parser item queries.

#+begin_src text
rs-harness search prime            → compact packet (default)
rs-harness search prime --json     → structured JSON
rs-harness search owner src/lib.rs items --query 'run' --code → compact code only
rs-harness check --changed         → compact finding text (default)
rs-harness check --changed --json  → structured JSON report
#+end_src

For agent exploration, =--json= is a hook-deniable output-mode error, not a
better search.  Agents should request compact =--view seeds= when they need the
smallest prompt-facing packet and reserve JSON for tests, receipts, or
machine-to-machine validation.

* Graph-first view controls

#+begin_src text
--view graph|hits|both|seeds
--package <pkg>
#+end_src

=--view= selects the final render mode for a search packet.  =--package=
selects a workspace package after =search workspace=.  Depth, edge, hit, owner,
line, feature, target, and seed caps are provider-owned deterministic limits
unless a future =methodDescriptor= explicitly advertises them for that method.
They are not shared semantic-language flags.

There is no shared =--backend= flag today.  Execution-route selection is fully
internal unless a method descriptor later advertises bounded text match modes.
Even then, the agent client chooses intent-level match semantics, not raw tool
execution.  RFC 008 extends this boundary to the cross-language =asp= client:
local native provider CLI, Cloud Flight, and hybrid route selection are client
configuration and diagnostics state, not search/query flags agents must pick
during normal exploration.

* External candidate adapters: =search ingest=

When the agent client deliberately needs wider lexical or file-path candidates, it
calls =rg= or =fd= directly — but always pipes output back into
=rs-harness search ingest= for owner grouping, dedup, ranking, and compact
packet rendering.  Raw =rg= / =fd= output is never pasted into the chat.

There is exactly one ingest command — no sub-kinds, no =--query=:

#+begin_src shell

#+end_src

Shared controls (output / scope control only, no source declaration):

#+begin_src text
--json
--package <pkg>
--view graph|hits|both|seeds
#+end_src

No =--query=, no =ingest rg= / =ingest paths= / =ingest fd= sub-kind.
Ingest auto-detects the format from the first few KB of stdin.
Owner, scope, hit, item, line, and per-owner caps are provider-owned
deterministic limits unless a method descriptor explicitly advertises them.

** Auto-detection

#+begin_src text

#+begin_src text

** Recency-biased ranking

When a search view has multiple same-class candidates after semantic rank,
provider score, and deduplication, project-internal path recency is the next
tie-breaker.  Newer last-modified paths sort before older paths because agents
usually edit the files they are currently reasoning about.  This applies to
owner-grouped hits, ingest candidates, dependency usage owners, and equal-rank
prime owners.

Recency is only a ranking signal.  It must not replace semantic relevance,
public/API/root owner priority, exact-match scoring, or version-scope rules.
Providers only stat paths that resolve inside the selected project or package
root; missing or non-project mtimes fall back to deterministic path/name order.
Default compact output does not expose mtime fields.

contract."

#+begin_example

|owner src/index.rs hit_kind=text locations=88:1,94:1 next=owner:src/index.rs
#+end_example

No raw =ast-grep= syntax, no =backend= line.  The agent client selects the recipe; the
CLI executes it.

** =search docs= / =search api=

Dependency API docs are first-class search objects — not prose blobs, but
native-parser API facts connected to local callsites, features, public API
leakage, and tests.  The Rust harness must treat its native parser as the
authority for local source shape; registry and rustdoc sources can enrich facts
only after parser-owned workspace/path facts are exhausted.

Baseline distinction: =search deps= owns local usage for the dependency version
range declared by the current workspace; =search docs= owns external/reference
API facts for an explicitly requested version.  A future composite docs+usage
packet should be added only after a provider can join those two fact sources
without hiding provenance; it is not part of the baseline schema or TypeScript
provider.

#+begin_src shell
rs-harness search docs tokio::runtime::Builder
rs-harness search docs bytes::Buf --package tokio
rs-harness search api tokio::sync::mpsc::Sender::send
rs-harness search public-external-types --package tokio
#+end_src

Distinction:

| Command                    | Returns                                          |
|----------------------------+--------------------------------------------------|
| =search docs=              | doc facts + API shape, minimal prose             |
| =search api=               | API shape only: kind, signature, async, unsafe,  |
|                            | feature, receiver, return, error, trait impls    |
| =search public-external-types=| external dependency types appearing in         |
|                            | public API signatures                            |

Data source priority:

#+begin_src text
1. workspace / path dependency source (native parser facts + rustdoc comments)
2. Cargo registry source (downloaded crate source)
3. local rustdoc JSON cache (target/doc/)
4. docs.rs rustdoc JSON (optional cache/download)
5. HTML docs (last resort, not default)
#+end_src

For Rust, MVP implementation starts with parser-native facts only:
=kind=, =signature=, =params=, =async=, =unsafe=, =receiver=, =return=, and
=error=.  The CLI renderer must not infer these from compact text, regex-only
matches, or prose docs.
This MVP includes public free functions, public inherent impl methods, and trait
impl methods, so receiver shape, impl type, and trait path are parser-owned.
The CLI renderer must project those facts without inferring association itself.

Implementation rule: every new Rust API fact lands first in the native parser
fact model with parser-level tests.  =rs-harness search= may only project those
facts into compact text or JSON; it is not the source of truth.

The header line declares the data source for trust calibration:

#+begin_src text
[search-docs] q=tokio::runtime::Builder crate=tokio source=native-parser
[search-docs] q=bytes@0.6.0::Buf crate=bytes requestedVersion=0.6.0 versionScope=external source=registry-source
[search-docs] q=ignore@0.4.23::WalkBuilder::require_git crate=ignore requestedVersion=0.4.23 versionScope=external source=docs-rs-json
#+end_src

If the external source is not available, the packet admits it instead of
presenting current workspace parser facts as external-version docs:

#+begin_src text
|note docsSource=registry-source missing=true
#+end_src

Dependency-prefixed docs/API queries such as =serde::Serialize= use
=source=registry-source= unless the provider has parser facts for a local
workspace/path dependency source.

Output format (API facts, not docs prose):

#+begin_example
[search-docs] q=fallible pkg=. docs=1 source=native-parser
|api path=src/lib.rs line=11 kind=fn name=fallible next=owner:src/lib.rs source=native-parser docs=local-parser signature=fn(input:String)->anyhow::Result<Thing> params=input:String async=true unsafe=true receiver=- return=anyhow::Result<Thing> error=anyhow::Result impl=- trait=-
#+end_example

=search deps ignore@0.4.23::WalkBuilder::require_git= remains the local usage
entry point.  It may point to =search docs ignore::WalkBuilder::require_git=
when an external docs provider is available, but it must not require docs to
answer local usage.

=search docs= never outputs long prose by default.  Doc-comment excerpts are
not part of the baseline contract; providers can expose them only through an
advertised detail method or method descriptor field.

* Exit codes

Stable and documented:

| Code | Meaning                                          |
|------+--------------------------------------------------|
| 0    | clean / ok                                       |
| 1    | harness findings exist                           |
| 2    | invalid CLI usage                                |
| 3    | project / config / protocol error                |

* Local daemon design

The daemon serves the CLI.  It is never exposed to the agent client.

** Transport

Priority order:

#+begin_src text
Unix:   .ts-harness/cache/server.sock
Win:    named pipe
Last resort: loopback HTTP 127.0.0.1 with random port file
#+end_src

No MCP, no dynamic tool discovery, no runtime schema negotiation.

** Protocol

Boring NDJSON.  Not an RPC framework.

Request:

#+begin_src json
{"v":1,"id":"1","method":"snapshot","params":{"root":".","mode":"compact"}}
#+end_src

Response:

#+begin_src json
{"v":1,"id":"1","ok":true,"result":{"compact":"Modules: source=184 owners=42\n..."}}
#+end_src

Error:

#+begin_src json
{"v":1,"id":"1","ok":false,"error":{"code":"STALE_CACHE","message":"cache invalidated; retry cold"}}
#+end_src

Rules:

#+begin_src text
- every request has a timeout
- every response carries protocol version
- result size has an upper bound
- search limit defaults to 20
- all paths are project-root-relative
- output ordering is deterministic
- daemon never returns unbounded JSON
#+end_src

** Responsibilities

The daemon:

#+begin_src text
- keeps TypeScript Program warm
- watches tsconfig / package.json / source changes
- maintains reasoning tree
- maintains owner graph
- maintains import/export/callsite index
- answers bounded queries
- invalidates cache deterministically
#+end_src

The daemon does NOT own policy authority.  That stays in the library/CLI:

#+begin_src text
daemon → gives facts fast
CLI    → validates facts version
       → renders compact text
       → owns exit code
       → owns direct parser run
#+end_src

A daemon bug cannot corrupt the agent contract.

** Lifecycle

The semantic-language protocol does not expose daemon lifecycle commands.
Providers may keep private developer controls for starting, stopping, or
inspecting a daemon, but those controls are not =methods=, are not advertised by
=agent doctor --json=, and are not used by agent guides or hooks.  The
agent-visible health surface is =<lang-harness> agent doctor=.

* Agent guide: teach recipes, not intent

The generated agent guide tells an agent client how to think about the
primitives.  It does not tell the client to call a planner command.  It
provides command-group recipes for common task types, subagent guidance, and a
synthesis format — all of which the agent client executes by composing
primitives.

````md
---
name: rs-harness
description: Use this guide in Rust repositories for reasoning-tree
  navigation, Cargo dependency facts, owner-aware search, structural policy
  checks, and changed-file repair loops.
---

# rs-harness Agent Client Guide

Use `rs-harness` before broad manual reading.  Prefer graph/search/deps
primitives over raw `rg`, `fd`, or hand-written `ast-grep`.

The CLI does not infer task intent.  The agent client must choose the
exploration recipe and decide whether to spawn subagents.

## Phase 1: deep prime — always first

Do not begin by reading files or requesting hits.  If the root is a workspace,
first get the workspace router:

```sh
rs-harness search workspace
```

Then select a package and get the package map:

```sh
rs-harness search prime
```

Read =mode=, =package=, =feature=, =cfg=, =owner=, =source_large=,
=api-candidate=, =test-surface=, =surface=, and =next= fields.

Critical rules from prime:
- =source_large=true= → never open the file; use =search owner <file> items=.
- =api-candidate= → a dependency API or workspace type may need docs.
- =feature ... requires_cfg= → the feature is guarded by a cfg; search both.
- =surface public_external= → external types are allowed in public API.

Do not spawn subagents before reading prime unless the user provides an
exact file and symbol.

## Phase 2: parent planning

Use prime output to choose search axes:

- feature/cfg axis
- dependency/API docs axis
- owner/item axis
- public API/callsite axis
- tests/bench/examples axis
- harness finding axis

Spawn read-only subagents only after selecting axes.

## Phase 3: focused multi-pipe or bounded single-view

Subagents use focused commands, not broad searches.  Prefer one multi-pipe
command per subagent when evidence surfaces are predictable:

```sh
# Feature/cfg subagent:
rs-harness search features <feature> cfg owners tests --package <pkg>

# Dependency/docs subagent:
rs-harness search dependency <dep> items docs tests --package <pkg>

# Source expansion subagent:
rg -n "<query>" <roots> | rs-harness search ingest items tests
```

Only request separate hit-level searches after selecting owners:
```sh
rs-harness search owner <owner> items
rs-harness search callsite <symbol> --view hits
rg -n "<query>" tests | rs-harness search ingest
```

Choose a focused recipe below.

## Query-set composition

When the parent already has several queries for the same semantic axis, the
provider should accept one bounded query-set packet instead of forcing the
agent to issue one command per query.  This is /homogeneous composition/:
one search view, one package/scope context, one set of same-kind query terms.
It is not a general batch language and must not concatenate unrelated commands.

Supported first-hop forms:

```sh
rs-harness search dependency <dep1>,<dep2>,<dep3> items tests --package <pkg>
rs-harness search owner <path1>,<path2>,<path3> items --package <pkg>
ts-harness search fzf --query-set <q1> --query-set <q2> owner tests --owner <path> .
```

Providers may add query-set support to other views only when each query term
uses the same parser-owned source and the merged packet can de-duplicate owners,
items, tests, next actions, and notes without hiding which query caused which
fact.  Query-set selectors are explicit:

- =exact-set= means every term is a literal query value supplied by the agent.
- =stdin-path-set= means a path list was normalized by =search ingest= before
  becoming owner/path evidence.
- =prefix-set= may be used only when the provider derives matches from
  parser-owned package/dependency facts, not from raw substring scanning.

The line protocol keeps the normal view header and adds composition metadata:

```text
[search-dependency] q=dep1,dep2 querySet=2 selector=exact-set pkg=<pkg> dep=2 own=4 api=0 item=8 tests=2
[search-owner] q=src/a.rs,src/b.rs querySet=2 selector=exact-set pkg=<pkg> own=2 item=12
```

JSON packets expose the normalized query terms through =querySet= and the merge
policy through =queryComposition=.  Agents should prefer a query-set packet when
the alternative is a repeated series of same-view searches, but should keep
separate packets for distinct axes such as dependency, owner, callsite, and
tests.

Owner-scoped text query-sets are the TypeScript stress case.  A real trace
around =src/cli/semantic-search/render.ts= searched the same owner with
=location.path=, =location.column=, =location.line=, =renderLocation=,
=function renderFields=, =if (location=, and string-literal fragments after
the owner was already known.  That pattern is not a user intent change; it is
one text axis with several exact parser-visible terms and one owner scope.
Providers must not rely on comma-separated text such as
=location.path,location.column= because that is a valid literal search query.

The registry must advertise this explicitly:

#+begin_src text
method=search/fzf
supportsQuerySet=true
acceptedQuerySetSelectors=exact-set
acceptedPipes=owner,tests
querySetScopes=project,owner
#+end_src

The packet should then carry =querySet= terms and a =queryComposition.scope=
with the owner path:

#+begin_src json
{
  "method": "search/fzf",
  "view": "text",
  "querySet": [
    {"value": "location.path", "kind": "text", "selector": "exact"},
    {"value": "location.column", "kind": "text", "selector": "exact"}
  ],
  "queryComposition": {
    "mode": "query-set",
    "view": "text",
    "selector": "exact-set",
    "scope": {"ownerPath": "src/cli/semantic-search/render.ts"},
    "merge": ["owners", "hits", "items", "nextActions", "notes"]
  }
}
#+end_src

This is still homogeneous composition.  It does not batch =owner=, =text=,
=tests=, and shell commands together.  It collapses repeated text searches
after the owner/scope is known, lets the provider de-duplicate hits and tests,
and gives the agent one next action instead of encouraging token-fragment
probing.

For compact =--view seeds= output, a query-set packet should expose the graph
frontier when the provider can derive a change frontier from parser-visible
owners and tests.  This is the root fix for traces that alternate between text,
owner, tests, and raw reads: the first same-axis query-set packet must identify
the source edit frontier, the test frontier, and typed follow-up actions
without teaching the agent a second prompt protocol.

The durable fix is not simply to concatenate those terms.  A mechanical
=search fzf --query-set= union can still let broad terms such as =metadata= or
=relative_to= consume the bounded hit budget, hiding the layout-policy owner
that caused the trace.  Python =search fzf --query-set= must therefore:

- record =queryCoverage= for every term, including misses and truncated terms
- merge hits fairly across terms before applying the compact hit limit
- classify selected owners with =ownerResolution=
- emit =searchSynthesis= seeds that point to owner/tests follow-ups instead of
  encouraging more text fragments
- keep the cold-start cost visible in sandtable receipts so provider indexing
  work is not confused with prompt-surface compression

This keeps query-set as homogeneous composition while making the provider own
the explanation that otherwise becomes repeated agent-side guessing.

## Prime-driven fan-out/fan-in

=search prime= is the parent planning packet.  When it exposes several
independent semantic axes, the agent should fan out the bounded follow-up work
before editing and then fan in the compact evidence for one decision.  The
fan-out unit is not an arbitrary shell command.  It is one semantic axis, one
scope, and one command group selected from the prime packet:

#+begin_src text
parent: search prime
deps worker:  search deps <dep[/subpath][@version][::api]>
source worker: search owner <owner> items
tests worker:  search tests <owner>
policy worker: check --changed or check --full when the edit boundary requires it
parent: merge receipts into [search-synthesis]
#+end_src

Clients with multi-agent support can run these workers as read-only subagents.
Clients with tool parallelism can run independent search commands in parallel
inside the parent turn.  Clients without either capability must run the same
command groups sequentially.  The protocol does not depend on the client
feature flag.  The portable contract is the prime packet, the bounded search
packets, the =[search-subagent]= receipts, and the parent
=[search-synthesis]= line.

This differs from query-set composition.  A query-set merges repeated queries
inside one view and one scope.  Fan-out runs distinct views or scopes in
parallel after prime has selected them.  Providers should keep both visible:
query-set metadata explains why one packet contains several same-axis terms;
subagent receipts explain why several independent packets were explored before
the parent chose the edit boundary.

## Real-trigger sandtable loop

Hand-written search-flow drills are necessary but incomplete: they prove a
known command sequence, not whether a real agent chooses that sequence from a
task intent.  Sandtable calibration therefore includes a real-trigger loop:
start an agent on a representative large project with a feature-level intent,
record the search phase, stop before edits, and replay the harness-facing
command flow as a normal sandtable scenario.

The first real-trigger target is a Rust/Tokio package.  The intent should be
broad enough to require =search prime=, parent-axis selection, focused
follow-ups, optional subagent fan-out, and at least one hook denial.  The
recorded artifact is not a full terminal transcript.  It is a compact receipt:

#+begin_src text
project identity and workdir
user intent
accepted <lang-harness> search commands before editing
denied raw reads/searches and their safe guide commands
subagent search requests and compact receipts
elapsed time, stdout/stderr bytes, line counts
missing facts, confusing next actions, repeated searches
query-set or multi-pipe merge opportunities
#+end_src

The edit-stop boundary is strict.  Once the agent is ready to patch a source
file, the real-trigger recording ends.  Implementation belongs to the language
provider repository and its normal tests; the protocol repository keeps the
search behavior evidence.

The replay scenario runs only deterministic harness-facing commands:

#+begin_src text
prime -> focused search -> owner/items -> tests -> ingest -> hook deny guide
#+end_src

It does not launch Codex, simulate LLM reasoning, or embed source excerpts.
If the real agent used =rg= or =fd= to widen candidates, the replay must pipe
that output through =search ingest= so the provider owns grouping, ranking,
deduplication, tests, and next actions.

Keep the trigger layer distinct from the target library.  Codex CLI, Claude, or
another client supplies hook payloads such as =PreToolUse= and should be tested
through hook replay at that boundary.  Libraries such as Tokio, Vite, Vue, or
Playwright are exploration targets that exercise provider search quality.  A
complete real-trigger sandtable can therefore have both:

#+begin_src text
client trigger: Codex PreToolUse -> hook deny/allow guide
target replay:  <lang-harness> search prime/fzf/owner/tests/ingest
#+end_src

This prevents a large target-library replay from being mistaken for the Codex
CLI trigger itself, while still keeping the provider-facing commands
deterministic and replayable.

Real-trigger evidence feeds two optimization loops:

1. /Prime quality/: did =search prime= expose the owners, features, cfgs,
   dependencies, tests, and next actions needed to choose focused axes?
2. /Flow compression/: which repeated same-view searches should become a
   query-set packet, and which predictable follow-ups should become a
   multi-pipe command?

Hook denials in these scenarios test guide quality, not only enforcement.  A
useful denial must include the reason category, language/provider route, next
safe command, pipe shape for =search ingest= when needed, subagent receipt
requirements when applicable, and no leaked source text.  The route argv is a
separate assertion surface from the human-readable message: sandtables may
require canonical command fragments such as =query --from-hook= and reject stale
fragments such as =search query --from-hook= or non-canonical surface names.

Sandtable summaries should make the improvement visible:

#+begin_src text
[sandtable-flow] scenario=rust.tokio-real-trigger commands=7 stdoutBytes=...
|prime quality=pass missing=...
|merge view=owner queries=3 saveCommands=2
|guide raw-search=pass direct-read=pass
|token delta=lower repeatedSearches=...
#+end_src

Warnings are acceptable while a real-trigger pattern is being calibrated.  Once
the flow is stable, command count, packet size, latency, repeated-search, and
guide-quality budgets can become regression gates through =--fail-on-warn=.

## Large-library intent fixture matrix

Real-trigger calibration must not collapse to one favorite project or one task
shape.  Each language provider should keep a portable large-library fixture
matrix with at least three representative libraries and the same three agent
intent classes:

#+begin_src text
feature-implementation      implement or modify a named feature
api-usage                   find how a public API is used or exposed
implementation-principle    explain how a behavior is implemented internally
#+end_src

The CLI still does not parse natural-language intent.  The scenario records
the human task in =evidence.intentCases=, then exercises deterministic harness
commands such as =search prime=, =search fzf=, =search dependency=,
=search owner=, =search tests=, or =search ingest=.  The agent client owns the
intent-to-command choice; the provider owns only parser/schema/RFC facts and
ranked compact packets.

Each large-library scenario carries typed metadata:

#+begin_src text
evidence.fixtureTier=large-library
evidence.targetLibrary={language,name,package,workdirKind}
evidence.intentCases=[{intentKind,intent,stepIds...,queryTerms...}]
#+end_src

The workdir must be optional through an environment variable or local checkout
candidate.  Missing large-library checkouts skip the scenario instead of
failing CI.  Schema and coverage tests still validate that Rust, TypeScript,
Python, and future providers publish the same matrix shape.

The first baseline matrix is:

| Language   | Libraries                         |
|------------+-----------------------------------|
| Rust       | tokio, bytes, ignore              |
| TypeScript | typescript, vite, playwright      |
| Python     | fastapi, rich, pandas             |

Every library row must include all three intent classes.  A scenario may group
the three intent cases for one library, but it must name the step ids and query
terms that exercise each case so later receipts can compare command count,
packet size, frontier quality, and repeated-search compression by intent kind.
When same-view text probes are homogeneous, prefer one =search fzf
--query-set= step over three separate text steps; keep the per-intent
=queryTerms= metadata as the audit join key.

Language providers may use filesystem tools such as =fd= and =rg= to prune
large-library text/query-set candidate files before parsing, but those tools
only own candidate ordering and caps.  Owners, symbols, callsites, imports,
test edges, and packet facts still come from the language parser.  Any such
prefilter must be visible in the packet through =runtimeCost= and/or notes,
and query coverage remains =partial= when the provider has intentionally
pruned a broad text frontier.

When a text prefilter handles =--query-set=, it should treat the query-set as
one same-axis search frontier.  Running one multi-pattern =rg= pass and then
fairly capping per-term parser inputs is preferred over N full source scans for
N query terms.  The packet should keep this auditable through
=runtimeCost.fields=, for example =queryTerms= and =sourceSearchPasses=.
When that first source scan already proves a large enough candidate frontier
and every query-set term has enough source matches to satisfy the per-term cap,
the provider may skip a separate full file-list pass.  Path-only or weak-source
terms must still fall back to a file-list/path-match pass.  The packet should
make this visible with fields such as =fileListPasses= and
=candidateFileBasis= so audits can distinguish exact all-file counts from
source-matched lower bounds.

For large TypeScript workspaces, =search fzf --query-set ... owner tests= must
rank parser-visible implementation owners above broad declaration, generated,
or fixture-heavy owners when the implementation owner has owner-local exported
symbol/type matches for multiple query terms.  A query-set such as
=APIRequestContext,BrowserContextAPIRequestContext,fetch= should keep the owner
that declares the matching request context classes in the first frontier even
when generated =.d.ts= files or top-level tests have more lexical occurrences.
This is still a fuzzy search result: the provider may use =rg+fzf= or another
prefilter to find candidates, but the final rank must be parser-owned and must
prefer owner-local exported item evidence over weak path/text frequency.

Coverage audit reports this matrix explicitly:

#+begin_src text
|intent-matrix language=rust libraries=3/3 missing=0
|intent-library language=rust package=tokio intents=api-usage,feature-implementation,implementation-principle scenarios=rust.tokio-intent-matrix
#+end_src

When =--fail-on-missing= is enabled, missing large-library rows or missing
intent classes are coverage failures, just like missing protocol surfaces.

## Subagent rule: graph slice, not full search

For broad tasks, spawn read-only explorer subagents yourself.

Parent gets graph; subagents get slices; hits only within slices:
- Parent: =search prime= or =search dependency <dep> --view graph=
- Dependency subagent: =search deps <dep[/subpath][@version][::api]>=
- Source subagent: =search dependency <dep> owners items --package <pkg>=
- Test subagent: =search tests <owner> --view graph=

Each subagent should:
1. run only the command group assigned by the parent agent;
2. avoid editing files;
3. return compact evidence as =[search-subagent]= line + =|= fact lines;
4. every =|= line must carry a =next= action if evidence is missing.

The parent agent merges subagent evidence into =[search-synthesis]= and
decides the next search phase.

## Output contract for subagents (line protocol)

```text
[search-subagent] role=deps dep=reqwest evidence=key:reqwest import:reqwest kind:normal feat:json,rustls-tls missing=- next=dependency:reqwest
[search-subagent] role=src dep=reqwest evidence=owner:src/http/client.rs hit_kind=use locations=3:5,8:1 pub=false missing=error-boundary next=pattern:public-error-boundary
[search-subagent] role=test dep=reqwest evidence=test:tests/unit/http/client.rs bench:benches/http_client.rs missing=timeout-test next=text:timeout(scope=tests)
```

Parent merges into:

```text
[search-synthesis] flow=dep dep=reqwest owners=src/http/client.rs,src/api/orders.rs tests=2 risk=public-error-boundary,timeout missing=2 next=pattern:public-error-boundary(owner=src/http),text:timeout(scope=tests)
```

If =missing= is non-empty, parent selects another focused =rs-harness search=
command using the =next= actions.

## External candidate expansion rule

When built-in =search fzf= or =search symbol= returns too few results,
use =rg= or =fd= — but always pipe back to =search ingest=:

```sh
rg -n "<query>" src tests | rs-harness search ingest
fd -e rs "<name>" src tests | rs-harness search ingest
git diff --name-only | rs-harness search ingest
```

Never paste raw =rg= / =fd= output.  The ingest command groups results by
owner or path, annotates the input source, and keeps downstream navigation in
the shared packet schema.  This is required for docs, schema files, generated
artifacts, and any test files or other text that are outside the provider's
=search fzf= index.

## Structural pattern rule

Use =search pattern <recipe>= for code shape queries.  List available
recipes with:

```sh
rs-harness search patterns
```

Do not write raw =ast-grep= patterns.  Select a recipe by name.  If no
recipe matches, fall back to =search fzf= or =rg | ingest=.
````

* Hooks: inject reminders, don't plan

Hooks are shell scripts.  They do not connect to the daemon, do not generate
subagent plans, and do not consume natural-language intent.

* Agent search flow: structured, not chaotic

The agent should follow this order:

#+begin_example
1. search workspace → route multi-package workspaces before package search
2. search prime     → see bounded package owner/dependency/finding map
3. search dependency → inspect external dependency source usage when relevant
4. search owner     → understand current file's responsibility
5. search symbol    → find exported symbol definitions
6. search callsite  → find owner-level import/reexport sites
7. search import    → inspect import/reexport owner edges
8. search tests     → locate tests importing the edited owner
9. search fzf      → find broader symbol/path candidates
10. search ingest   → normalize external candidate streams when needed
11. edit            → apply change
12. check           → verify change against policy
13. check --full    → closeout when needed
#+end_example

NOT:

#+begin_example
rg keyword
open many files
guess public API
run big test suite
#+end_example

* MVP order: M1–M7

The real MVP is not "many search subcommands."  It's four capabilities that
cover Tokio, rust-lang/rust, and large agent-client class projects:

#+begin_src text
1. search workspace + search prime --package
2. search features/cfg/deps dependency-version usage
3. search owner <path> items
4. search ingest auto-sniff + owner/item grouping
#+end_src

| Phase | Deliverable                                               |
|-------+-----------------------------------------------------------|
| M1    | Multi-pipe grammar + final-only renderer.  =search <source> |
|       | <pipe1> <pipe2> ...= with deterministic stages.  Trace      |
|       | output is descriptor-advertised.  Full field names,         |
|       | =locations=/                                                |
|       | =calls=/=uses= fold hits, next= inline on owner/item lines. |
| M2    | Cargo feature/cfg pipes.  =search features <f> cfg     |
|       | owners tests docs=.  Feature→optional-dep→target cfg chain. |
|       | Tokio =io-uring= / =loom= fixtures.                         |
| M3    | Items pipe.  =search owner <path> items=.  Nearest-item for |
|       | ingest.  =source-large= role.  agent-client file-search fixture.   |
| M4    | Docs/API facts.  Workspace/registry-source API facts: kind, |
|       | signature, async, unsafe, feature, receiver, return, error. |
|       | =search docs <crate>::<item>= plus versioned                 |
|       | =search deps <crate[/subpath][@version][::item]>=.          |
|       | bytes::Buf /                                                 |
|       | ignore::WalkBuilder fixtures.                               |
| M5    | Public-api pipe.  =search dependency <dep> public-api=.     |
|       | Tokio allowed_external_types + rustc_middle smallvec.       |
| M6    | Ingest auto-sniff + 8 pattern recipes.  Single command,     |
|       | auto-detect stdin.  Always owner/item-grouped output.       |
| M7    | SKILL.org multi-pipe flows.  8 flows with multi-pipe         |
|       | templates: dependency/feature/public-api/external-candidate/ |
|       | namespace/finding-repair/unknown/docs-api.                  |

Drill-verified implementation order — deep PM first, pipes after:

#+begin_src text
Step 1: Deep PM minimum viable loop
    rs-harness search workspace / prime / prime --package <pkg>
    mode=workspace-index for workspace, mode=package for prime
    |package, |dep, |feature, |cfg, |owner, |edge, |finding,
    |api-candidate, |test-surface, |surface, |next
    source=workspace|path|crates-io|git on cargo lines

Step 2: source_large + item slice
    rs-harness search owner <path> items
    source_large=true signal (guide: never open the file directly)
    agent-client file-search/src/lib.rs fixture

Step 3: Cargo feature/cfg
    rs-harness search features <feature>
    rs-harness search cfg <cfg>
    rs-harness search cfg <cfg>
    feature→optional-dep→target-cfg→docs.rs-metadata chain
    Tokio io-uring / tokio_unstable fixtures

Step 4: Multi-pipe final-only renderer
    rs-harness search dependency <dep> items tests
    rs-harness search features <feature> cfg owners tests
    descriptor-advertised trace for compact stage list
    single enriched packet output

Step 5: Docs/API facts
    rs-harness search docs <crate>::<item>
    rs-harness search deps <crate[/subpath][@version][::item]>
    rs-harness search api <crate>::<item>
    rs-harness search public-external-types --package <pkg>
    native parser API facts only (kind, signature, params, async, unsafe, receiver, return, error, impl, trait)
    public-external-types joins parser-native public API type facts with Cargo dependency facts
    bytes::Buf / ignore::WalkBuilder fixtures

Step 6: SKILL.org two-phase guide
    Phase 1: deep prime always first (mode/source_large/api-candidate rules)
    Phase 2: parent axis selection
    Phase 3: subagent multi-pipe/ingest
    Phase 4: parent synthesis
    Phase 5: rs-harness check --changed
#+end_src

The dominant finding: deep PM is the single most valuable capability
because it determines whether the LLM can correctly organize the rest of
the subagent flow graph without full-repo traversal.  Multi-pipe is a
Phase 3 acceleration, not a replacement for the Phase 1 map.

* Existing TypeScript harness readiness

The TypeScript harness already has:

- A CLI binary: =ts-harness= → =./dist/src/cli/main.js=
- =package.json= scripts: =build=, =check=, =format=, =lint=, =test=, =harness=
- Stable protocol commands: =search=, =check=, and =agent doctor=
- A provider registry emitted by =ts-harness agent doctor --json=
- A =SKILL.org= guide that teaches protocol commands, not direct renderer flags
- Public library surface: =renderTypeScriptProjectHarnessAgentCompactText=,
  =renderTypeScriptProjectHarnessAgentSnapshot=,
  =runTypeScriptProjectHarnessAgentSnapshot=, =runTypeScriptProjectHarness=,
  verification/render/profile/task index APIs

The TypeScript slice is the reference implementation for the semantic-language
registry contract.  The CLI advertises =ts-harness= only, rejects old direct
renderer flags as protocol commands, and validates search JSON against the
shared schema.  Its API search slice is intentionally parser-owned:
=ts-harness search api <query>= emits current-workspace exported/public API
facts only, while dependency-prefixed queries such as
=react@18.0.0::jsx= return an explicit missing-provider note until an external
docs/API source exists.  It also implements
=ts-harness search public-external-types <pkg>= as a parser-owned dependency
leakage slice: direct =import("pkg").Type= public type text is confirmed, while
named import text is marked possible until the parser exposes import binding
attribution.  For equal relevance scores, TypeScript search ranks
recently modified project files before older files and then falls back to path
order, so active repair context rises without adding a new protocol field.

* Existing Python harness readiness

The Python harness now exposes the same CLI-first protocol lane:

- A CLI binary: =py-harness= → =python_lang_project_harness:run_cli_from_env=
- Stable protocol commands: =search=, =check=, and =agent doctor=
- Owner item query: =py-harness search owner <path> items --query '<a|b>' .=
  returns =|query=, =|item=, and indentation-stripped =|code= lines from Python
  parser facts, including fallback matching against function bodies.
- Transitional Codex hooks deny direct Python source reads, including shell
  content readers such as =sed ... file.py= and tool-native =Read= calls, with
  =reasonKind=direct-source-read= plus a concrete public hook route
  =asp python query --from-hook direct-source-read --selector <path> --code .=.
- A provider registry emitted by =py-harness agent doctor --json=
- Python-local capability schema:
  =schemas/python-semantic-capabilities.v1.schema.json=
- Package-local common schema copies:
  =schemas/semantic-language-registry.v1.schema.json= and
  =schemas/semantic-search-packet.v1.schema.json=

The first Python search slice is intentionally smaller than the Rust and
TypeScript surfaces but follows the same contract.  =py-harness search
workspace= emits package-root routing facts and =next= prime targets.
=py-harness search prime= emits parser-owned reasoning-tree owners, import
edges, dependency nodes, findings, owner-scoped text seeds, and de-duplicated
deps next actions.
=py-harness search callsite <query>= emits parser-owned =PythonCall= facts so
agents can move from a symbol or API owner to the modules that invoke it without
re-running broad text search.
=py-harness search public-external-types <package>= emits parser-owned public
function, class, decorator, base-class, and annotated-assignment surfaces that
expose dependency types; direct package/alias references are confirmed and
named import matches are marked possible.
=py-harness search deps <package[@version][::api]>= now emits the same flow
shape as the TypeScript lane: dependency, public-external-types, api, text, and
tests next actions when the requested dependency version is current.
=py-harness search fzf <query> owner tests= searches parser-visible owner
paths, namespaces, public exports, symbols, and source text, then keeps matched
owners plus importing tests in one packet.  =py-harness search ingest= detects
=rg -n=, vimgrep, plain path-list, and NUL path-list stdin and maps hits back to
parser-visible owners when possible.  Dependency queries use =pyproject.toml=
metadata and local import usage, with =versionScope= rendered as =current=,
=external=, or =unknown=.

* The correct division of labor

#+begin_example
SKILL.org:
  writes exploration protocol, subagent usage, command-group recipes,
  synthesis format

Agent parent:
  judges task type
  selects recipe
  spawns subagents
  merges evidence
  decides next search phase
  decides edit plan

Agent subagents:
  read-only
  run assigned rs-harness command group
  return compact evidence

rs-harness CLI:
  search / check / agent primitives
  registry-described methods
  deterministic output
  no natural-language intent
  no subagent planning
  no task router

hooks:
  inject reminders
  prevent obviously bad searches
  changed check after edits
  closeout guard before stop
#+end_example

One sentence:

/Put "intent → plan → subagents" in the agent client.  Put native facts, owner
slices, search packets, compact checks, and registry-described methods in the
harness.  That is the correct harness-client boundary./

  teaches agent clients how to use harness

CLI:
