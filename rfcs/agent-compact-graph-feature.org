* One-sentence summary

Every agent-facing =search= response carries a typed decision graph feature:
schema-owned semantic graph facts are projected into role-typed aliases,
grouped relation edges, a legend-declared search root, algorithm-owned rank,
inline next actions, exact locators, and explicit omission/avoidance facts.

* Problem

The current compact search output started as line protocol:

#+begin_src text
|owner src/tool_action.rs role=source ...
|item command_source_paths kind=fn read=src/tool_action.rs:221:241
|code path=src/tool_action.rs lineRange=221:241 nodes=... text="..."
|synthesis algorithm=bounded-reachability-depth1 ...
#+end_src

That shape is better than raw source, but it still repeats paths, field names,
and owner context.  More importantly, it makes the agent reconstruct graph
relations from flat lines.  The hidden graph is already present in
=semantic-graph.v1= and =searchSynthesis=, but the prompt-facing rendering does
not expose the graph as the agent's attention surface.

Human formatting is the wrong target.  An LLM does not need a pretty diagram,
a Markdown report, or a source-like outline.  It needs a minimal structure that
preserves the next reasoning decision.  That structure must be part of the
whole =search= interface, not a special =owner= view.

* LLM reasoning model

For code work, the useful prompt-facing unit is not a line of source text.  It
is a decision node connected to other nodes by typed relations:

- owner, package, dependency, item, block, test, finding, and read window nodes
- relation labels such as =contains=, =imports=, =covers=, =calls=, =guards=,
  =returns=, =writes=, and =frontier=
- graph-derived attention order
- executable follow-up actions
- exact locators for expansion or edit preparation
- explicit omission and anti-action facts

This is a code reasoning graph, not a visual graph.  The provider should
compress repeated syntax, but it must not compress away relation labels,
rank order, next actions, locators, or omission reasons.

* Research line and invariants

This RFC is the protocol surface for the research line:

#+begin_src text
Agent Compact Code Graph: typed decision graph for LLM code navigation
#+end_src

The claim is not that code should be made to look like a graph.  The claim is
that agent-facing code graph serialization is a typed decision interface.  Its
purpose is to expose the smallest graph slice that preserves:

1. what entities exist
2. why they were selected
3. how they relate
4. what order the agent should attend to them
5. what executable follow-up actions are safe
6. what source/code facts were intentionally omitted

The durable pipeline is:

#+begin_src text
parser/compiler/static-analysis facts
  -> typed code/repo graph
  -> ranked frontier
  -> compact prompt serialization
  -> executable next action
#+end_src

LLMs may infer structure, but the protocol must not depend on inference for
grammar.  Compact graph output therefore follows six invariants:

- Typed Node: every alias declares a node kind.
- Typed Target: every alias target declares a target role such as =path=,
  =term=, =pkg=, =symbol=, or =range=.
- Typed Edge: every relation has an explicit relation label.
- Algorithmic Rank: =rank= comes from =searchSynthesis.algorithm=, not renderer
  sorting.
- Executable Frontier: every =frontier= entry maps to a provider follow-up
  action.
- Bounded Locator: every read/edit/query frontier has an exact locator or a
  target that the provider can recover into one.

* Agent-driven search flow boundary

The compact graph renderer is part of an agent-driven search flow.  It does not
move natural-language intent parsing into the harness.

The intended flow is:

#+begin_src text
user intent
  -> agent/LLM interprets the task
  -> agent reads search guide and provider capabilities
  -> agent runs search prime for the initial graph frontier
  -> agent chooses a typed reasoning profile and selectors
  -> provider returns schema-backed graph facts/evidence
  -> agent chooses query/read/edit/test actions
#+end_src

=search guide= tells the agent which deterministic flows and evidence profiles
exist.  =search prime= returns an initial graph frontier without user intent.
=search reasoning <profile>= consumes typed selectors chosen by the agent and
returns schema-backed facts.  None of these commands accepts natural-language
=--goal=, =--intent=, or task prose, and none of them performs agent planning.

Valid reasoning commands are shaped by guide-declared profiles:

#+begin_src shell
<lang-harness> search reasoning owner-query \
  --owner <project-path> \
  --query <term> \
  [--dependency <pkg>] \
  [--view facts|seeds|both] \
  .

<lang-harness> search reasoning query-deps \
  --query <term> \
  --dependency <pkg> \
  [--view facts|seeds|both] \
  .

<lang-harness> search reasoning owner-tests \
  --owner <project-path> \
  [--view facts|seeds|both] \
  .

<lang-harness> search reasoning finding-frontier \
  --query <rule-id-or-diagnostic-id> \
  [--owner <project-path>] \
  [--view facts|seeds|both] \
  .
#+end_src

The graph feature is the prompt-facing projection for guide, prime, and
reasoning evidence.  It compresses the returned facts and frontier into a
low-token graph, but the agent still owns intent-to-flow selection.

* Non-goals

This RFC does not introduce a new top-level =graph= command.  Discovery remains
=search= with views such as =seeds=, =hits=, =graph=, and =both=.
=asp graph render= is a machine adapter for rendering a
validated packet; it is not an agent-facing discovery command and must not
replace =search= in agent workflows.

This RFC does not make graph projection an =owner= special case.  Prime, owner,
dependency, query-set, ingest, tests, policy, and item-oriented search/query
surfaces should all expose graph facts when they can produce parser-owned nodes,
edges, rank, or next actions.

This RFC does not make Mermaid, DOT, Graphviz, or Markdown diagrams a prompt
protocol.  Those formats are trained priors, but they are diagram-first and
too verbose for the harness's agent path.

This RFC does not replace JSON schema packets.  =semantic-graph.v1= and
=semantic-search-packet.v1= remain the canonical machine contract.

* Canonical data boundary

The graph feature is a renderer over schema-owned facts:

#+begin_src text
provider parser facts
  -> semantic-search-packet nodes/edges/searchSynthesis
  -> asp graph render
  -> agent compact graph feature
#+end_src

Provider-private maps must not determine prompt-visible rank, relation labels,
or next actions.  If a relation or action changes the agent's decision, it must
be representable in the JSON packet before it is rendered in compact graph
output.

Field names are canonical, not provider aliases.  For example, query-set count
is derived from root-level =querySet=, and source read locators on graph
frontier actions use =searchSynthesis.seeds[].read=.  The shared renderer must
not accept alternate names such as =locator= or =readLocator= because aliases
let providers drift while producing superficially similar prompt text.

The prompt-facing graph renderer is shared protocol logic.  Native language
providers own parser facts and schema-valid packets; they do not own separate
long-term compact graph render algorithms.  The renderer lives in
=agent-semantic-protocol= so Rust, TypeScript, Python, Julia cache paths, local
ASP cache hits, and Cloud Flight results project through the same alias,
edge, rank, frontier, profile, omit, and avoid rules.

Language providers under =languages/= invoke the renderer through a process
boundary:

#+begin_src shell
asp graph render --packet - --view seeds
#+end_src

That shell-out is the provider integration boundary, not a temporary escape
hatch.  Provider crates and packages must not link or vendor the renderer
library, including =languages/rust-lang-project-harness=.  The target hot path
described by RFC 008 is ASP's Rust-side cache/query/render path over validated
provider packet artifacts or native index rows.

* Projection grammar

The projection has one view-native header line and a small set of compact fact
lines:

#+begin_src text
[search-<view>] (q=<query>|owner=<path>|root=<path-or-.>) [querySet=<n>|terms=<n>] [selector=<selector>] [scope=<scope>] [view=<view>] alg=<searchSynthesis.algorithm>
legend: ID=kind:role(value)!next; edge SRC>{DST:rel}; frontier ID.next
alias: graph:{G=search,...}
ID=<nodeKind>:<targetRole>(<target>)[!<action>]
G>{ID:relation,ID:relation}
SRC>{DST:relation,DST:relation}
(SRC,SRC)>DST:relation
rank=ID,ID,ID frontier=ID.action,ID.action
entries=profile(ID,ID=>return+return),profile(ID=>return)
omit=<omitted-semantic>[,<omitted-semantic>]
avoid=<anti-action>[,<anti-action>]
#+end_src

This graph projection is schema-derived and prompt-facing.  The default
agent-facing =--view seeds= output renders the minimal graph frontier as
view-native =search= output, aliases, and the combined =rank= / =frontier=
line.  It must not also render independent =|seed= or =|synthesis= rows as a
second action protocol.

The same shape applies to code-level graphs:

#+begin_src text
[search-query] q=<query> selector=<path> alg=<algorithm>
ID=<nodeKind>:<targetRole>(<target>)[@<path-or-range>][!<action>]
SRC>{DST:relation,DST:relation}
rank=ID,ID,ID frontier=ID.action,ID.action
entries=profile(ID,ID=>return+return),profile(ID=>return)
omit=<omitted-semantic>[,<omitted-semantic>]
#+end_src

Alias ids are packet-local.  They are stable only within one rendered packet
and must not be copied into source paths, JSON paths, or edit commands.
=targetRole(target)= tells the agent and parser how to interpret the target
value.  =!action= declares an action available from a node.  =frontier=
selects the seed actions for this packet from those node actions.  The
=entries= line is optional and names guide-declared reasoning profiles whose
typed selectors match aliases rendered in this packet; each selector inside
=profile(...)= is a packet-local alias id, and the =...=>...= suffix names the
return entries exposed by that profile.  The
micro-legend line is the minimal grammar contract for agents that have not
recently read this RFC.
Every selector alias used by an =entries= row must resolve to the current
packet's =alias: graph:{...}= declaration, and known shared profile selectors
must resolve to the profile's node-kind catalog.  For example,
=owner-query(O,Q=>...)= requires =O= to resolve to an =owner= alias and =Q= to
resolve to a =query= alias; =query-deps(O,T=>...)= is invalid even if =O= and
=T= are otherwise valid graph aliases.  This is a parser-checkable contract, not
a compatibility hint for the model to infer.  The structured catalog lives in
=semantic-compact-graph-render.v1.schema.json= as =reasoningProfileContracts=,
and =semantic-search-packet.v1.schema.json= validates implemented profile
packets against the same selector order and return-entry contract.

Profile names are shared protocol catalog entries, not compatibility aliases or
provider-private labels.  The initial catalog is =owner-query=, =query-deps=,
=owner-tests=, =finding-frontier=, and =feature-cfg=.  Adding another
prompt-facing profile name requires a schema/RFC update first so every provider
can either emit the same meaning or deliberately omit it.
Provider guides should print this catalog explicitly, for example
=|catalog reasoningProfiles=owner-query,query-deps,owner-tests,finding-frontier,feature-cfg entries=owner-query,query-deps,owner-tests,finding-frontier,feature-cfg routes=read-frontier=.
Direct-read or repair flows that are not reasoning profiles must be rendered as
routes or commands rather than as =entry= rows.
Provider-specific guide lines may list fewer =entries= than the shared catalog
until that provider implements the matching reasoning command.  Rust carries
=finding-frontier= and =feature-cfg= through the existing =--query= selector
slot; adding =--finding= or =--feature= would be a new public command surface and
needs its own RFC/schema pass.
When a provider implements one of these profiles, the returned packet must carry
the selector as a typed graph action, for example =nextActions=[{kind:"finding",
target:<finding>}]= or =nextActions=[{kind:"feature", target:<feature>}]=, so
the shared renderer can emit =F=finding= / =F=feature= aliases and concrete
=entries= rows without relying on model inference.

The alias legend is strict in compact mode: every alias id that appears in
alias lines, edges, =rank=, or =frontier= must be declared with its node kind,
for example =alias: graph:{G=search,O=owner,S=symbol,S2=symbol}=.  A prefix
legend such as =S*=symbol= may be introduced later as a dense-mode extension,
but it is not the default compact grammar because it requires parser and LLM
inheritance rules that are not otherwise needed.

=G= is the packet-local search root.  It is declared by the legend entry
=G=search= and by appearing as an edge source such as =G>{...}=.  Renderers
must not emit a standalone root alias line such as =G=search:result!query=:
that line repeats the legend, looks like an actionable node, and increases the
chance that an agent treats the root as a follow-up target.  =G= gives
synthesis-derived edges a real source node without making the =rank= /
=frontier= line act as a graph node.  =G= is not a recommended follow-up action
and must not appear in =rank= or =frontier=.  When a provider has no
parser-owned =imports=, =calls=, or =covers= edges for a surface, it may still
render honest synthesis-derived relations from =G= such as =selects=,
=matches=, =covers=, =uses=, =imports=, =contains=, =explains=, =flags=, and
=gates=.

* Line protocol atomics

Compact graph output is line protocol.  Each rendered graph fact is a complete
line atom.  The micro-legend and alias legend lines are each valid only as one
complete line, for example:

#+begin_src text
legend: ID=kind:role(value)!next; edge SRC>{DST:rel}; frontier ID.next
alias: graph:{G=search,O=owner,T=test}
#+end_src

Renderers, schema examples, JSON fixtures, and sandtable assertions must not
split this line into structural fragments such as:

#+begin_src json
["alias: graph:{", "G=search:result!query"]
#+end_src

That shape is invalid even though =alias: graph:{...}= itself is valid.  It
turns an opening delimiter into a standalone assertion and makes the next line
look like a continuation or a root alias.  The standalone root alias
=G=search:result!query= is also forbidden by this RFC.

When tests need resilient substring matching rather than exact output matching,
they may assert a stable prefix that includes the first alias assignment, such
as =alias: graph:{G=search=.  They must not assert only =alias: graph:{=.
Exact-output fixtures should store the full legend line.  JSON arrays that
model =stdoutContains= are lists of independent line substrings, not a way to
continue a graph grammar token across elements.

* Owner search example

#+begin_src text
[search-owner] q=src/classifier.rs scope=tests/unit/classifier/routes alg=bounded-reachability-depth1
legend: ID=kind:role(value)!next; edge SRC>{DST:rel}; frontier ID.next
alias: graph:{G=search,C=owner,M=owner,P=owner,W=test,R=test}
C=owner:self(root)!items
M=owner:path(src/command/mod.rs)!owner
P=owner:path(src/protocol.rs)
W=test:path(wrappers.rs)!tests
R=test:path(raw_search.rs)!tests
G>{C:selects,W:covers,R:covers,M:selects,P:selects}
C>{M:imports,P:decision}
(W,R)>C:covers
rank=C,W,R,M,P frontier=C.items,M.owner,W.tests,R.tests
avoid=raw-read
#+end_src

This says:

- =C= is the selected owner and its next action is =items=
- =M= and =P= are frontier owners
- =W= and =R= are tests that cover =C=
- =G= tells the agent these nodes were selected by this search result, not by
  source-order adjacency
- the attention order is =C,W,R,M,P=
- the selected seed actions are =C.items,M.owner,W.tests,R.tests=
- raw source reads are explicitly not the next action

The paths appear once.  The owner/test roles appear once.  Same-relation test
edges are grouped.  The agent does not have to infer rank from output order.

* Query item example

#+begin_src text
[search-query] q=command_source_paths selector=src/tool_action.rs terms=1 view=seeds alg=exact-item
legend: ID=kind:role(value)!next; edge SRC>{DST:rel}; frontier ID.next
alias: graph:{G=search,I=item,A=item,L=item,B=item,P=item}
I=item:symbol(command_source_paths)@221:241!code
A=guard:range(range_paths.nonempty@224:226)!read
L=loop:range(path_like_tokens@228:239)!read
B=branch:range(embedded.empty@230:233)
P=branch:range(embedded.present@234:238)
G>{I:selects,A:contains,L:contains}
I>{A:return,L:collect}
L>{B:push_path,P:push_embedded}
rank=I,A,L,B,P frontier=I.code,A.read,L.read
omit=trivia,comments,blank-lines
#+end_src

For same-file code graphs, =@start:end= inherits =root=.  Cross-file nodes
must include a project-root-relative path before the range, for example:

#+begin_src text
T=test:range(tests/tool_action.rs@12:30)!tests
#+end_src

The compact graph is not an editable source replacement.  It is a reasoning
surface that tells the agent which item or block to expand through provider
query/read actions.

Owner-local item search uses the same rule, but the selected owner is context
and the matched items are the immediate frontier.  The header must preserve the
fact that this is an owner-local item query rather than an ordinary owner
frontier:

#+begin_src text
[search-owner] owner=src/cli/search_output/graph.rs selector=items terms=4 view=seeds alg=seed-frontier
legend: ID=kind:role(value)!next; edge SRC>{DST:rel}; frontier ID.next
alias: graph:{G=search,O=owner,S=symbol,S2=symbol,S3=symbol,S4=symbol}
O=owner:path(src/cli/search_output/graph.rs)!owner
S=symbol:symbol(render_search_graph_packet)@60:93!code
S2=symbol:symbol(graph_header)@521:548!code
S3=symbol:symbol(header_mode)@550:555!code
S4=symbol:symbol(packet_view)@570:582!code
G>{O:selects,S:matches,S2:matches,S3:matches,S4:matches}
O>{S:contains,S2:contains,S3:contains,S4:contains}
rank=S,S2,S3,S4,O frontier=S.code,S2.code,S3.code,S4.code
omit=code,comments,blank-lines,nonmatching-items
avoid=repeat-owner,raw-read,full-json
#+end_src

This avoids a common search loop: after =search owner <path> items --query ...=,
the selected owner has already been expanded, so =O.owner= is not the next
frontier action.  The owner remains in =rank= as context, but matched symbols
rank first and their bounded =code= actions are the recommended follow-up.
Edges keep the two meanings separate: =G>{S:matches}= says the search matched
the symbol; =O>{S:contains}= says the owner contains it.  Because the header
already declares the owner path, same-owner code locators use =@start:end=.
Cross-owner item locators must use =@path:start:end= so the provider can route
the next =code= action without a raw file read.

* Prime search example

#+begin_src text
[search-prime] root=. alg=owner-rank-frontier
legend: ID=kind:role(value)!next; edge SRC>{DST:rel}; frontier ID.next
alias: graph:{G=search,L=owner,C=owner,T=owner,U=test}
L=owner:path(src/lib.rs)!owner
C=owner:path(src/classifier.rs)!owner
T=owner:path(src/tool_action.rs)!owner
U=test:path(tests/unit_test.rs)!tests
G>{L:selects,C:selects,T:selects,U:covers}
L>{C:exports,T:exports}
U>{C:covers,T:covers}
rank=L,C,T,U frontier=L.owner,C.owner,T.owner,U.tests
entries=owner-tests(L=>covering-tests+test-entrypoints+fixtures)
#+end_src

Prime output still avoids full dependency or callsite expansion.  The graph
feature names the selected owner map and next frontier so the agent can choose a
focused owner/test action instead of fanning out through paths.

* Compression rules

Compression may remove repeated syntax.  It must not remove decision
semantics.

Allowed compression:

- move shared path context into =root= or =base=
- use =self= for the root owner
- replace repeated paths with packet-local aliases
- group same-relation edges
- attach next action with =!action=
- select seed actions with =frontier=
- omit default/healthy facts
- pack large packets with semicolons in dense mode

Forbidden compression:

- remove relation labels such as =imports=, =covers=, =guards=, or =returns=
- remove =rank=
- hide next or seed actions inside prose
- render seed actions from a path list that is not derived from the graph
- drop exact read locators for hot nodes
- use alias ids as path values
- invent relation labels that are not backed by parser/schema facts
- shorten semantic words without a declared legend and enough packet size to
  amortize that legend

* Compact and dense modes

Compact mode is the default for agent exploration.  It keeps one alias line per
high-impact node and one edge line per source or grouped relation.

Dense mode may pack aliases and edges with semicolons:

#+begin_src text
legend: ID=kind:role(value)!next; edge SRC>{DST:rel}; frontier ID.next
C=owner:self(root)!items;M=owner:path(src/command/mod.rs)!owner;P=owner:path(src/protocol.rs);W=test:path(wrappers.rs)!tests;R=test:path(raw_search.rs)!tests
alias: graph:{G=search,C=owner,M=owner,P=owner,W=test,R=test}
G>{C:selects,W:covers,R:covers,M:selects,P:selects};C>{M:imports,P:decision};(W,R)>C:covers
rank=C,W,R,M,P frontier=C.items,M.owner,W.tests,R.tests
#+end_src

Dense mode is valid only when it preserves the same relation labels, rank,
actions, frontier actions, locators, =omit=, and =avoid= facts as compact mode.
Providers should prefer compact mode for small packets because line boundaries
are useful LLM anchors.  Dense mode is for high-volume graph output where path
and field-name deduplication saves more tokens than the lost line anchors cost.

* Search interface mapping

Graph is a feature of the whole search interface.  The provider should compute
schema-backed graph nodes, edges, rank, and next actions whenever the selected
search surface can justify them.

=--view seeds= should render the minimal graph feature needed for action
selection: header, aliases for selected/frontier nodes, enough edges to justify
the frontier, =rank=, =frontier=, and =avoid= when relevant.  It is not a
separate seed protocol.

=--view graph= should render the fuller compact graph projection: aliases,
grouped edges, =rank=, =frontier=, =omit=, =avoid=, and any graph facts needed
to explain the selected search surface.

=--view both= may render the graph feature followed by bounded hit, item,
finding, or code facts that are not yet compactly represented by aliases and
edges.  Those extra facts must reference the same graph nodes or locators; they
must not introduce an independent ranking or seed set.

* Seed and synthesis integration

=seed= is not an independent line family in this RFC.  A seed is a selected
frontier action over a graph node.  Compact output renders it inside the
combined =rank=... frontier=...= line, where each =ID.action= must correspond
to a node alias that declares =!action= or to an action represented in the JSON
packet.

=synthesis= is not an independent prose summary in this RFC.  Synthesis is the
graph algorithm evidence that selects the rank, frontier, omit/avoid facts, and
next actions:

#+begin_src text
searchSynthesis.algorithm -> header alg=
searchSynthesis selected/order/frontier -> rank= and frontier=
searchSynthesis edit/test/window/finding frontier -> typed aliases and edges
searchSynthesis.seeds -> frontier=
avoidNextActions -> avoid=
#+end_src

RFC 006 does not define independent =|seed= or =|synthesis= prompt-facing
rows.  Providers that still emit them are pre-RFC-006 implementations and must
converge on the graph-integrated form.  Renderer tests should assert =rank=,
=frontier=, aliases, grouped edges, and =avoid= instead of accepting duplicate
seed/synthesis rows as a second contract.

* Rank semantics

=rank= is not cosmetic output order.  It is the agent attention order selected
by =searchSynthesis.algorithm=.

Renderers must not sort aliases alphabetically, by path length, by file system
order, or by original source order when the graph algorithm provides a frontier
or attention order.  If several nodes have equal algorithmic rank, the provider
may keep parser/source order inside that tie.

* Schema implications

Before implementing this renderer broadly, verify that the shared schemas can
represent every prompt-visible fact:

- node alias source: =nodes[]=
- target roles: =nodes[].targetRole=, =nextActions[].targetRole=, or
  schema-owned render action specs
- typed relations: =edges[]=
- rank algorithm and order: =searchSynthesis=
- frontier seed actions: =searchSynthesis.seeds= or graph/search synthesis
  action fields
- node-local next actions: =nextActions= or graph/search synthesis action
  fields
- anti-actions: =avoidNextActions=
- omitted facts and expansion locators: query/read projection metadata

If a field is missing, extend the schema before provider-specific rendering.
Do not hide projection-only decisions in language-private =fields= maps.

The prompt-facing text template is also schema-owned.  Providers must align to
=schemas/semantic-compact-graph-render.v1.schema.json= for:

- view-native header grammar
- micro-legend grammar
- alias line grammar
- dense alias separator
- combined =rank=... frontier=...= line
- optional =entries=...=>returns= return-entry line
- source-kind to node/target-role/action vocabulary

Rust, TypeScript, Python, and future language providers must not implement
independent prompt-facing compact graph renderers.  They construct
schema-valid packets and invoke the shared renderer through
=asp graph render=.  Provider-local code may contain only a
thin process adapter and tests that prove the adapter passes canonical packets
to the shared renderer without fallback.

* Implementation order

1. Keep this RFC as the language-neutral contract.
2. Audit =semantic-graph.v1= and =semantic-search-packet.v1= for alias,
   action, rank, frontier seed, synthesis, omission, and avoid coverage.
3. Maintain =semantic-compact-graph-render.v1= as the shared prompt-facing
   text template.
4. Add schema tests for any missing fields or template vocabulary drift.
5. Implement shared renderer tests that prove prime, owner, query-set, ingest,
   tests, and item query outputs derive their compact lines from the graph
   feature.
6. Replace renderer assertions that target independent =|seed= or
   =|synthesis= rows with assertions over =rank=, =frontier=, =avoid=, aliases,
   and grouped edges.
7. Align Rust, TypeScript, and Python provider integration tests to the same

* Acceptance criteria

- The compact graph output can be parsed by an agent through the schema-owned
  micro-legend plus =alias: graph:{...}= legend and does not require prose to
  explain aliases.
- The output has fewer repeated paths and field names than legacy line protocol.
- Target roles, relation labels, rank order, frontier seed actions, node-local
  next actions, exact locators, and omission facts remain visible.
- The JSON packet remains the source of truth for graph data.
- =seed= and =synthesis= are represented through graph aliases, =rank=,
  =frontier=, and =avoid= rather than as independent prompt-facing protocols.
- Providers produce the same graph meaning across Rust, TypeScript, and Python.
- Julia remains out of bin-parity requirements for this lane unless a separate
  Julia-specific workspace-managed effort is opened.
