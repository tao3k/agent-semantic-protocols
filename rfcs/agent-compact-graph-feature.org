* One-sentence summary

Every agent-facing =search= response carries a typed decision graph feature:
schema-owned semantic graph facts are projected into role-typed aliases,
grouped relation edges, a legend-declared search root, algorithm-owned rank,
inline next actions, exact locators, and explicit omission/avoidance facts.

* Problem

The current compact search output started as line protocol:

#+begin_src text
|owner src/tool_action.rs role=source ...
|item command_source_paths kind=fn read=src/tool_action.rs:221:241
|code path=src/tool_action.rs lineRange=221:241 nodes=... text="..."
|synthesis algorithm=bounded-reachability-depth1 ...
#+end_src

That shape is better than raw source, but it still repeats paths, field names,
and owner context.  More importantly, it makes the agent reconstruct graph
relations from flat lines.  The hidden graph is already present in
=semantic-graph.v1= and =searchSynthesis=, but the prompt-facing rendering does
not expose the graph as the agent's attention surface.

Human formatting is the wrong target.  An LLM does not need a pretty diagram,
a Markdown report, or a source-like outline.  It needs a minimal structure that
preserves the next reasoning decision.  That structure must be part of the
whole =search= interface, not a special =owner= view.

* LLM reasoning model

For code work, the useful prompt-facing unit is not a line of source text.  It
is a decision node connected to other nodes by typed relations:

- owner, package, dependency, item, block, test, finding, and read window nodes
- relation labels such as =contains=, =imports=, =covers=, =calls=, =guards=,
  =returns=, =writes=, and =frontier=
- graph-derived attention order
- executable follow-up actions
- exact locators for expansion or edit preparation
- explicit omission and anti-action facts

This is a code reasoning graph, not a visual graph.  The provider should
compress repeated syntax, but it must not compress away relation labels,
rank order, next actions, locators, or omission reasons.

* Research line and invariants

This RFC is the protocol surface for the research line:

#+begin_src text
Agent Compact Code Graph: typed decision graph for LLM code navigation
#+end_src

The claim is not that code should be made to look like a graph.  The claim is
that agent-facing code graph serialization is a typed decision interface.  Its
purpose is to expose the smallest graph slice that preserves:

1. what entities exist
2. why they were selected
3. how they relate
4. what order the agent should attend to them
5. what executable follow-up actions are safe
6. what source/code facts were intentionally omitted

The durable pipeline is:

#+begin_src text
parser/compiler/static-analysis facts
  -> typed code/repo graph
  -> ranked frontier
  -> compact prompt serialization
  -> executable next action
#+end_src

Terminology is deliberately narrow:

- =frontier= means locator/next-action candidates selected from typed facts.
- =compact graph render= means the prompt-facing serialization format for those
  facts; it is not a second kind of frontier.
- =code compact= means parser/format-aware code projection used by query/code
  surfaces.  Search/frontier output must not pretend to be code compact.

LLMs may infer structure, but the protocol must not depend on inference for
grammar.  Compact graph output therefore follows six invariants:

- Typed Node: every alias declares a node kind.
- Typed Target: every alias target declares a target role such as =path=,
  =term=, =pkg=, =symbol=, or =range=.
- Typed Edge: every relation has an explicit relation label.
- Algorithmic Rank: =rank= comes from =searchSynthesis.algorithm=, not renderer
  sorting.
- Executable Frontier: every =frontier= entry maps to a provider follow-up
  action.
- Bounded Locator: every read/edit/query frontier has an exact locator or a
  target that the provider can recover into one.

* Agent-driven search flow boundary

The compact graph renderer is part of an agent-driven search flow.  It does not
move natural-language intent parsing into the harness.

The intended flow is:

#+begin_src text
user intent
  -> agent/LLM interprets the task
  -> agent reads search guide and provider capabilities
  -> agent runs search prime for the initial graph frontier
  -> agent chooses a typed reasoning profile and selectors
  -> provider returns schema-backed graph facts/evidence
  -> agent chooses query/read/edit/test actions
#+end_src

=search guide= tells the agent which deterministic flows and evidence profiles
exist.  =search prime= returns an initial graph frontier without user intent.
=search reasoning <profile>= consumes typed selectors chosen by the agent and
returns schema-backed facts.  None of these commands accepts natural-language
=--goal=, =--intent=, or task prose, and none of them performs agent planning.

Valid reasoning commands are shaped by guide-declared profiles:

#+begin_src shell
<lang-harness> search reasoning owner-query \
  --owner <project-path> \
  --query <term> \
  [--dependency <pkg>] \
  [--view facts|seeds|both] \
  .

<lang-harness> search reasoning query-deps \
  --query <term> \
  --dependency <pkg> \
  [--view facts|seeds|both] \
  .

<lang-harness> search reasoning owner-tests \
  --owner <project-path> \
  [--view facts|seeds|both] \
  .

<lang-harness> search reasoning finding-frontier \
  --query <rule-id-or-diagnostic-id> \
  [--owner <project-path>] \
  [--view facts|seeds|both] \
  .
#+end_src

The graph feature is the prompt-facing projection for guide, prime, and
reasoning evidence.  It compresses the returned facts and frontier into a
low-token graph, but the agent still owns intent-to-flow selection.

* Non-goals

This RFC does not introduce a new top-level =graph= command.  Discovery remains
=search= with views such as =seeds=, =hits=, =graph=, and =both=.
=asp graph render= is a machine adapter for rendering a
validated packet; it is not an agent-facing discovery command and must not
replace =search= in agent workflows.

This RFC does not make graph projection an =owner= special case.  Prime, owner,
dependency, query-set, ingest, tests, policy, and item-oriented search/query
surfaces should all expose graph facts when they can produce parser-owned nodes,
edges, rank, or next actions.

This RFC does not make Mermaid, DOT, Graphviz, or Markdown diagrams a prompt
protocol.  Those formats are trained priors, but they are diagram-first and
too verbose for the harness's agent path.

This RFC does not replace JSON schema packets.  =semantic-graph.v1= and
=semantic-search-packet.v1= remain the canonical machine contract.

* Canonical data boundary

The graph feature is a renderer over schema-owned facts:

#+begin_src text
provider parser facts
  -> semantic-search-packet nodes/edges/searchSynthesis
  -> asp graph render
  -> agent compact graph feature
#+end_src

Provider-private maps must not determine prompt-visible rank, relation labels,
or next actions.  If a relation or action changes the agent's decision, it must
be representable in the JSON packet before it is rendered in compact graph
output.

Field names are canonical, not provider aliases.  For example, query-set count
is derived from root-level =querySet=, and source read locators on graph
frontier actions use =searchSynthesis.seeds[].read=.  The shared renderer must
not accept alternate names such as =locator= or =readLocator= because aliases
let providers drift while producing superficially similar prompt text.

The prompt-facing graph renderer is shared protocol logic.  Native language
providers own parser facts and schema-valid packets; they do not own separate
long-term compact graph render algorithms.  The renderer lives in
=agent-semantic-protocol= so Rust, TypeScript, Python, Julia cache paths, local
ASP cache hits, and Cloud Flight results project through the same alias,
edge, rank, frontier, profile, omit, and avoid rules.

Language providers under =languages/= invoke the renderer through a process
boundary:

#+begin_src shell
asp graph render --packet - --view seeds
#+end_src

That shell-out is the provider integration boundary, not a temporary escape
hatch.  Provider crates and packages must not link or vendor the renderer
library, including =languages/rust-lang-project-harness=.  The target hot path
described by RFC 008 is ASP's Rust-side cache/query/render path over validated
provider packet artifacts or native index rows.

* Projection grammar

The projection has one view-native header line and a small set of compact fact
lines:

#+begin_src text
[search-<view>] (q=<query>|owner=<path>|root=<path-or-.>) [querySet=<n>|terms=<n>] [selector=<selector>] [scope=<scope>] [view=<view>] alg=<searchSynthesis.algorithm>
legend: aliases ID:kind; node ID=kind:role(value)!next; edge SRC>{DST:rel}; frontier ID.next
aliases=G:search,...
ID=<nodeKind>:<targetRole>(<target>)[!<action>]
G>{ID:relation,ID:relation}
SRC>{DST:relation,DST:relation}
(SRC,SRC)>DST:relation
rank=ID,ID,ID frontier=ID.action,ID.action
entries=profile(ID,ID=>return+return),profile(ID=>return)
omit=<omitted-semantic>[,<omitted-semantic>]
avoid=<anti-action>[,<anti-action>]
#+end_src

This graph projection is schema-derived and prompt-facing.  The default
agent-facing =--view seeds= output renders the minimal graph frontier as
view-native =search= output, aliases, and the combined =rank= / =frontier=
line.  It must not also render independent =|seed= or =|synthesis= rows as a
second action protocol.

The same shape applies to code-level graphs:

#+begin_src text
[search-query] q=<query> selector=<path> alg=<algorithm>
ID=<nodeKind>:<targetRole>(<target>)[@<path-or-range>][!<action>]
SRC>{DST:relation,DST:relation}
rank=ID,ID,ID frontier=ID.action,ID.action
entries=profile(ID,ID=>return+return),profile(ID=>return)
omit=<omitted-semantic>[,<omitted-semantic>]
#+end_src

Alias ids are packet-local.  They are stable only within one rendered packet
and must not be copied into source paths, JSON paths, or edit commands.
=targetRole(target)= tells the agent and parser how to interpret the target
value.  =!action= declares an action available from a node.  =frontier=
selects the seed actions for this packet from those node actions.  The
=entries= line is optional and names guide-declared reasoning profiles whose
typed selectors match aliases rendered in this packet; each selector inside
=profile(...)= is a packet-local alias id, and the =...=>...= suffix names the
return entries exposed by that profile.  The
micro-legend line is the minimal grammar contract for agents that have not
recently read this RFC.
Every selector alias used by an =entries= row must resolve to the current
packet's =aliases=...= declaration, and known shared profile selectors
must resolve to the profile's node-kind catalog.  For example,
=owner-query(O,Q=>...)= requires =O= to resolve to an =owner= alias and =Q= to
resolve to a =query= alias; =query-deps(O,T=>...)= is invalid even if =O= and
=T= are otherwise valid graph aliases.  This is a parser-checkable contract, not
a compatibility hint for the model to infer.  The structured catalog lives in
=semantic-compact-graph-render.v1.schema.json= as =reasoningProfileContracts=,
and =semantic-search-packet.v1.schema.json= validates implemented profile
packets against the same selector order and return-entry contract.

Profile names are shared protocol catalog entries, not compatibility aliases or
provider-private labels.  The initial catalog is =owner-query=, =query-deps=,
=owner-tests=, =finding-frontier=, and =feature-cfg=.  Adding another
prompt-facing profile name requires a schema/RFC update first so every provider
can either emit the same meaning or deliberately omit it.
Provider guides should print this catalog explicitly, for example
=|catalog reasoningProfiles=owner-query,query-deps,owner-tests,finding-frontier,feature-cfg entries=owner-query,query-deps,owner-tests,finding-frontier,feature-cfg routes=read-frontier=.
Direct-read or repair flows that are not reasoning profiles must be rendered as
routes or commands rather than as =entry= rows.
Provider-specific guide lines may list fewer =entries= than the shared catalog
until that provider implements the matching reasoning command.  Rust carries
=finding-frontier= and =feature-cfg= through the existing =--query= selector
slot; adding =--finding= or =--feature= would be a new public command surface and
needs its own RFC/schema pass.
When a provider implements one of these profiles, the returned packet must carry
the selector as a typed graph action, for example =nextActions=[{kind:"finding",
target:<finding>}]= or =nextActions=[{kind:"feature", target:<feature>}]=, so
the shared renderer can emit =F:finding= / =F:feature= aliases and concrete
=entries= rows without relying on model inference.

The aliases declaration is strict in compact mode: every alias id that appears
in node lines, edges, =rank=, or =frontier= must be declared with its node kind,
for example =aliases=G:search,O:owner,S:symbol,S2:symbol=.  A prefix
legend such as =S*=symbol= may be introduced later as a dense-mode extension,
but it is not the default compact grammar because it requires parser and LLM
inheritance rules that are not otherwise needed.

=G= is the packet-local search root.  It is declared by the legend entry
=G:search= and by appearing as an edge source such as =G>{...}=.  Renderers
must not emit a standalone root alias line such as =G=search:result!query=:
that line repeats the legend, looks like an actionable node, and increases the
chance that an agent treats the root as a follow-up target.  =G= gives
synthesis-derived edges a real source node without making the =rank= /
=frontier= line act as a graph node.  =G= is not a recommended follow-up action
and must not appear in =rank= or =frontier=.  When a provider has no
parser-owned =imports=, =calls=, or =covers= edges for a surface, it may still
render honest synthesis-derived relations from =G= such as =selects=,
=matches=, =covers=, =uses=, =imports=, =contains=, =explains=, =flags=, and
=gates=.

* Line protocol atomics

Compact graph output is line protocol.  Each rendered graph fact is a complete
line atom.  The micro-legend and aliases declaration lines are each valid only
as one complete line, for example:

#+begin_src text
legend: aliases ID:kind; node ID=kind:role(value)!next; edge SRC>{DST:rel}; frontier ID.next
aliases=G:search,O:owner,T:test
#+end_src

Renderers, schema examples, JSON fixtures, and sandtable assertions must not
split this line into structural fragments such as:

#+begin_src json
["aliases=", "G:search,O:owner"]
#+end_src

That shape is invalid even though =aliases=...= itself is valid.  It
turns a declaration key into a standalone assertion and makes the next line
look like a continuation.  The standalone root alias =G=search:result!query=
is also forbidden by this RFC.

When tests need resilient substring matching rather than exact output matching,
they may assert a stable prefix that includes the first alias assignment, such
as =aliases=G:search=.  They must not assert only the bare =aliases= key.
Exact-output fixtures should store the full aliases declaration line.  JSON
arrays that model =stdoutContains= are lists of independent line substrings,
not a way to continue a graph grammar token across elements.

* Owner search example

#+begin_src text
[search-owner] q=src/classifier.rs scope=tests/unit/classifier/routes alg=bounded-reachability-depth1
legend: aliases ID:kind; node ID=kind:role(value)!next; edge SRC>{DST:rel}; frontier ID.next
aliases=G:search,C:owner,M:owner,P:owner,W:test,R:test
C=owner:self(root)!items
M=owner:path(src/command/mod.rs)!owner
P=owner:path(src/protocol.rs)
W=test:path(wrappers.rs)!tests
R=test:path(raw_search.rs)!tests
G>{C:selects,W:covers,R:covers,M:selects,P:selects}
C>{M:imports,P:decision}
(W,R)>C:covers
rank=C,W,R,M,P frontier=C.items,M.owner,W.tests,R.tests
avoid=raw-read
#+end_src

This says:

- =C= is the selected owner and its next action is =items=
- =M= and =P= are frontier owners
- =W= and =R= are tests that cover =C=
- =G= tells the agent these nodes were selected by this search result, not by
  source-order adjacency
- the attention order is =C,W,R,M,P=
- the selected seed actions are =C.items,M.owner,W.tests,R.tests=
- raw source reads are explicitly not the next action

The paths appear once.  The owner/test roles appear once.  Same-relation test
edges are grouped.  The agent does not have to infer rank from output order.

* Query item example

#+begin_src text
[search-query] q=command_source_paths selector=src/tool_action.rs terms=1 view=seeds alg=exact-item
legend: aliases ID:kind; node ID=kind:role(value)!next; edge SRC>{DST:rel}; frontier ID.next
aliases=G:search,I:item,A:item,L:item,B:item,P:item
I=item:symbol(command_source_paths)@221:241!code
A=guard:range(range_paths.nonempty@224:226)!read
L=loop:range(path_like_tokens@228:239)!read
B=branch:range(embedded.empty@230:233)
P=branch:range(embedded.present@234:238)
G>{I:selects,A:contains,L:contains}
I>{A:return,L:collect}
L>{B:push_path,P:push_embedded}
rank=I,A,L,B,P frontier=I.code,A.read,L.read
omit=trivia,comments,blank-lines
#+end_src

For same-file code graphs, =@start:end= inherits =root=.  Cross-file nodes
must include a project-root-relative path before the range, for example:

#+begin_src text
T=test:range(tests/tool_action.rs@12:30)!tests
#+end_src

The compact graph is not an editable source replacement.  It is a reasoning
surface that tells the agent which item or block to expand through provider
query/read actions.

Owner-local item search uses the same rule, but the selected owner is context
and the matched items are the immediate frontier.  The header must preserve the
fact that this is an owner-local item query rather than an ordinary owner
frontier:

#+begin_src text
[search-owner] q=crates/agent-semantic-hook/src/tool_action.rs pkg=crates/agent-semantic-hook selector=items querySet=6 alg=item-frontier
legend: ID=kind:role(value)!next; edge SRC>{DST:rel}; frontier ID.next
aliases: graph:{G=search,O=owner,Q=query,I=item,H=hot}
O=owner:path(crates/agent-semantic-hook/src/tool_action.rs)!owner
Q=query:term(tool_action|structured|payload|command_intent|from_payload|from_action)!query
I=item:symbol(payload_string)@crates/agent-semantic-hook/src/tool_action.rs:212:214!code
I2=item:symbol(collect_tool_actions)@crates/agent-semantic-hook/src/tool_action.rs:216:419!outline
I3=item:symbol(nested_tool_actions)@crates/agent-semantic-hook/src/tool_action.rs:555:571!code
H=hot:symbol(command_source_paths)@crates/agent-semantic-hook/src/tool_action.rs:397:401!code
H2=hot:symbol(nested_action_from_tool_use)@crates/agent-semantic-hook/src/tool_action.rs:567:568!code

G>{O:selects,Q:matches}
O>{I:contains,I2:contains,I3:contains,H:contains,H2:contains}
Q>{I:matches,I2:matches,I3:matches,H:revise,H2:revise}

rank=H,H2,I,I2,I3,O
frontier=H.code,H2.code,I.code,I2.outline,I3.code
revise=command_intent->command_source_paths,from_action->nested_action_from_tool_use
omit=code,projection-nodes,large-item-text
avoid=inline-code-in-search,raw-read,repeat-owner
#+end_src

This avoids a common search loop: after =search owner <path> items --query ...=,
the selected owner has already been expanded, so =O.owner= is not the next
frontier action.  The owner remains in =rank= as context, but hot revision
targets and matched items rank first and their bounded =code= or =outline=
actions are the recommended follow-up.  Edges keep the meanings separate:
=G>{Q:matches}= says the search matched the owner-local query, =Q>{I:matches}=
says the item matched it, =Q>{H:revise}= says a query term should be revised to
a parser-owned candidate, and =O>{I:contains,H:contains}= records ownership.
Owner-item search output must not inline source code.  It gives exact
locators/frontier only; a follow-up =query --code= or direct-read route is the
pure code extraction boundary.

* Prime search example

#+begin_src text
[search-prime] root=. alg=owner-rank-frontier
legend: aliases ID:kind; node ID=kind:role(value)!next; edge SRC>{DST:rel}; frontier ID.next
aliases=G:search,L:owner,C:owner,T:owner,U:test
L=owner:path(src/lib.rs)!owner
C=owner:path(src/classifier.rs)!owner
T=owner:path(src/tool_action.rs)!owner
U=test:path(tests/unit_test.rs)!tests
G>{L:selects,C:selects,T:selects,U:covers}
L>{C:exports,T:exports}
U>{C:covers,T:covers}
rank=L,C,T,U frontier=L.owner,C.owner,T.owner,U.tests
entries=owner-tests(L=>covering-tests+test-entrypoints+fixtures)
#+end_src

Prime output still avoids full dependency or callsite expansion.  The graph
feature names the selected owner map and next frontier so the agent can choose a
focused owner/test action instead of fanning out through paths.

* Compression rules

Compression may remove repeated syntax.  It must not remove decision
semantics.

Allowed compression:

- move shared path context into =root= or =base=
- use =self= for the root owner
- replace repeated paths with packet-local aliases
- group same-relation edges
- attach next action with =!action=
- select seed actions with =frontier=
- omit default/healthy facts
- pack large packets with semicolons in dense mode
- omit graph-debug rows such as =scores=, =paths=, =trace=, =explain=,
  =cache=, and =metrics= from prompt-visible =search pipe --view seeds= output
  when the same response still preserves parser-owned facts, typed relations,
  =queryCoverage=, exact =S1=, and =nextCommand=

Forbidden compression:

- remove relation labels such as =imports=, =covers=, =guards=, or =returns=
- remove =rank=
- hide next or seed actions inside prose
- render seed actions from a path list that is not derived from the graph
- drop exact read locators for hot nodes
- use alias ids as path values
- invent relation labels that are not backed by parser/schema facts
- shorten semantic words without a declared legend and enough packet size to
  amortize that legend

* Read-loop and failure-loop memory

The graph-turbo request may carry =readMemory.seenSelectors=.  These selectors
are not advisory prose.  They are an algorithm input: ranked/frontier nodes
whose exact selector is already present in memory must be suppressed before the
agent-facing frontier is rendered.  The result packet must expose
=algorithmMetrics.readMemorySuppressedCount= and add =seen-selector= to
=avoid= when suppression occurs, so tests and sandtables can prove the loop was
prevented rather than merely observed after the fact.

ASP's =search pipe= runtime reads this memory from
=$PRJ_CACHE_HOME/agent-semantic-protocol/read-loop-memory.json= before building
the graph-turbo request.  The runtime accepts both =seenSelectors=[...]= and
=entries=[{selector:...}]= so sandtable-produced memory can be replayed without
lossy translation.  If the memory packet carries =projectRoot= and it does not
resolve to the current project root, the runtime must ignore it.

Sandtable observations also project command-output memory:

- =readLoopMemory= records exact selector reads, owners, line ranges, command
  fingerprints, output bytes, result fingerprints, read counts, and avoid
  reasons such as =duplicate-read=, =manual-window-scan=, and =repeat-owner=.
- =failureLoopMemory= records executable failure-frontier actions selected from
  =asp <lang> search failure --from-last-check --view seeds .= output.
- =failureFrontierOutputPrecision= records whether the failure frontier kept
  failure facts, assertions, hot blocks, query profiles, omission facts, avoid
  facts, and executable frontier actions.

These memory packets are part of the loop contract.  A future provider or
client optimization may reduce tokens only if it preserves these facts and the
corresponding precision gates.

* Compact and dense modes

Compact mode is the default for agent exploration.  It keeps one alias line per
high-impact node and one edge line per source or grouped relation.

Dense mode may pack aliases and edges with semicolons:

#+begin_src text
legend: aliases ID:kind; node ID=kind:role(value)!next; edge SRC>{DST:rel}; frontier ID.next
C=owner:self(root)!items;M=owner:path(src/command/mod.rs)!owner;P=owner:path(src/protocol.rs);W=test:path(wrappers.rs)!tests;R=test:path(raw_search.rs)!tests
aliases=G:search,C:owner,M:owner,P:owner,W:test,R:test
G>{C:selects,W:covers,R:covers,M:selects,P:selects};C>{M:imports,P:decision};(W,R)>C:covers
rank=C,W,R,M,P frontier=C.items,M.owner,W.tests,R.tests
#+end_src

Dense mode is valid only when it preserves the same relation labels, rank,
actions, frontier actions, locators, =omit=, and =avoid= facts as compact mode.
Providers should prefer compact mode for small packets because line boundaries
are useful LLM anchors.  Dense mode is for high-volume graph output where path
and field-name deduplication saves more tokens than the lost line anchors cost.

* Search interface mapping

Graph is a feature of the whole search interface.  The provider should compute
schema-backed graph nodes, edges, rank, and next actions whenever the selected
search surface can justify them.

=--view seeds= should render the minimal graph feature needed for action
selection: header, aliases for selected/frontier nodes, enough edges to justify
the frontier, =rank=, =frontier=, and =avoid= when relevant.  It is not a
separate seed protocol.

=--view graph= should render the fuller compact graph projection: aliases,
grouped edges, =rank=, =frontier=, =omit=, =avoid=, and any graph facts needed
to explain the selected search surface.

=--view both= may render the graph feature followed by bounded hit, item,
finding, or code facts that are not yet compactly represented by aliases and
edges.  Those extra facts must reference the same graph nodes or locators; they
must not introduce an independent ranking or seed set.

* Seed and synthesis integration

=seed= is not an independent line family in this RFC.  A seed is a selected
frontier action over a graph node.  Compact output renders it inside the
combined =rank=... frontier=...= line, where each =ID.action= must correspond
to a node alias that declares =!action= or to an action represented in the JSON
packet.

=synthesis= is not an independent prose summary in this RFC.  Synthesis is the
graph algorithm evidence that selects the rank, frontier, omit/avoid facts, and
next actions:

#+begin_src text
searchSynthesis.algorithm -> header alg=
searchSynthesis selected/order/frontier -> rank= and frontier=
searchSynthesis edit/test/window/finding frontier -> typed aliases and edges
searchSynthesis.seeds -> frontier=
avoidNextActions -> avoid=
#+end_src

RFC 006 does not define independent =|seed= or =|synthesis= prompt-facing
rows.  Providers that still emit them are pre-RFC-006 implementations and must
converge on the graph-integrated form.  Renderer tests should assert =rank=,
=frontier=, aliases, grouped edges, and =avoid= instead of accepting duplicate
seed/synthesis rows as a second contract.

* Rank semantics

=rank= is not cosmetic output order.  It is the agent attention order selected
by =searchSynthesis.algorithm=.

Renderers must not sort aliases alphabetically, by path length, by file system
order, or by original source order when the graph algorithm provides a frontier
or attention order.  If several nodes have equal algorithmic rank, the provider
may keep parser/source order inside that tie.

* Schema implications

Before implementing this renderer broadly, verify that the shared schemas can
represent every prompt-visible fact:

- node alias source: =nodes[]=
- target roles: =nodes[].targetRole=, =nextActions[].targetRole=, or
  schema-owned render action specs
- typed relations: =edges[]=
- rank algorithm and order: =searchSynthesis=
- frontier seed actions: =searchSynthesis.seeds= or graph/search synthesis
  action fields
- node-local next actions: =nextActions= or graph/search synthesis action
  fields
- anti-actions: =avoidNextActions=
- omitted facts and expansion locators: query/read projection metadata

If a field is missing, extend the schema before provider-specific rendering.
Do not hide projection-only decisions in language-private =fields= maps.

The prompt-facing text template is also schema-owned.  Providers must align to
=schemas/semantic-compact-graph-render.v1.schema.json= for:

- view-native header grammar
- micro-legend grammar
- alias line grammar
- dense alias separator
- combined =rank=... frontier=...= line
- optional =entries=...=>returns= return-entry line
- source-kind to node/target-role/action vocabulary

Rust, TypeScript, Python, and future language providers must not implement
independent prompt-facing compact graph renderers.  They construct
schema-valid packets and invoke the shared renderer through
=asp graph render=.  Provider-local code may contain only a
thin process adapter and tests that prove the adapter passes canonical packets
to the shared renderer without fallback.

** Graph turbo packet boundary

=asp-graph-turbo= is a typed ranking engine for parser-owned graph facts, not a
second prompt renderer.  Its durable output is
=semantic-graph-turbo-result.v1=, which records the selected profile, seed
nodes, ranked node ids, ranking budget, frontier actions, relation edges,
scores, typed paths, cache trace, algorithm trace, rank explanations, and
algorithm metrics.  When graph turbo emits compact text, it must use the same
compact graph grammar owned by this RFC: header, legend, =aliases=, node
aliases, edge lines, =rank=, =frontier=, profile-specific projection lines,
=scores=, =paths=, =cache=, =trace=, =explain=, =metrics=, =profiles=,
=omit=, and =avoid=.  ASP-owned fast-search adapters may append query-pipeline
projection lines after the graph frontier, such as =pipePlan=, =pipeExpr=,
=pipeProjections=, =pipeStages=, =frontierActions=, =pipeCommands=, and
=pipeChoice=, and =stop=.  Those lines describe the ASP pipe expression and
selected agent-facing projections; they must not be treated as graph-turbo ranking facts.  The
default search frontier header is
=[graph-frontier]=.  A profile-specific surface such as =failure-frontier= may
use a narrower header such as =[search-failure]= only when the rest of the
grammar and =semantic-graph-turbo-result.v1= evidence remain graph-turbo
owned.  It must not invent a second =[graph-turbo]= prompt format.  Ordinary
non-turbo =semantic-search-packet= fallback output may continue to render
through the shared =semantic-compact-graph-render.v1= template, but
ranking/profile/path/score/trace projection for graph-turbo requests is owned
by =asp-graph-turbo=.

Graph turbo must not infer missing relation labels, node roles, or frontier
actions from prose.  Any prompt-visible rank or action produced by the turbo
engine must be visible in the result packet before another renderer serializes
it for an agent.

SciPy-first typed graph intelligence engine.

=asp-graph-turbo= remains a SciPy-first typed sparse ranking engine for the
agent hot path.  The default runtime path should continue to use lightweight
SciPy sparse matrices, typed relation policy, graph cache fingerprints, and
schema-owned request/result packets.  PyG, GNN message passing, and
=HeteroData= export are optional lab/offline research surfaces only: they must
not become default package dependencies, default CLI execution paths, hook
runtime requirements, or fallback conditions for search/query/read-plan.

The next graph-turbo investments should strengthen ASP's own typed facts and
deterministic sparse scoring before any learned reranker is considered.  The
priority order is typed fact graph quality, profile-specific sparse scoring,
failure/read-loop feedback, and command-inflation metrics.  Current result
metrics expose read-loop guard counts for direct code actions, duplicate
selectors, adjacent range windows, and same-owner scans.  Future fact quality
signals such as provenance, confidence, freshness, and profile transition
evidence should remain packet-visible so rank explanations, =frontierActions=,
=omit=, and =avoid= can be audited outside the Python implementation.  The
artifact timeline report also projects cached direct-code read-loop risk through
=readLoopRisk= JSON and =[graph-turbo-read-loop]= text evidence so historical
agent runs can be evaluated before another live replay.

The =asp <language> search failure= surface is a graph-turbo adapter, not a
Rust-owned renderer.  The facade may parse the last check transcript, collect
candidate facts, and construct a =semantic-graph-turbo-request.v1= packet with
=profile=failure-frontier=.  It must not rank failure facts, decide the
frontier order, or synthesize =frontierActions= outside the graph-turbo result.
The =failure-frontier= profile owns failure, assert, owner, hot, key, evidence,
and test node transitions; the compact =[search-failure]= text is only a
projection of that profile result.
The default =[search-failure]= projection is an agent prompt surface, so it
must keep failure facts, candidate owners, hot blocks, query profiles,
frontier actions, =omit=, and =avoid=, while omitting diagnostic rows such as
=scores=, =paths=, =cache=, =trace=, =explain=, and =metrics=.  Those
diagnostics remain available through graph-turbo packet/result debug surfaces;
removing them from seeds is valid only when the selected hot-block frontier and
failure facts are unchanged.

The check projection route follows the same boundary.  =asp <language> check
changed --view seeds .= and =asp <language> check --changed --view seeds .=
execute the provider check without forwarding the ASP =--view seeds= flag,
persist the failing transcript as the last-check artifact, and then render
=search failure --from-last-check --view seeds= through the graph-turbo
=failure-frontier= profile.  =search failure --view graph-turbo-request= is a
debug/schema surface for auditing that request packet; it is not the default
agent prompt surface.

Graph turbo requests are schema-owned too.  A caller that wants the Python
ranking engine to run =typed-ppr-diverse= should send
=semantic-graph-turbo-request.v1= with =profile=, =algorithm=, =seedIds=,
=budget=, typed =graph.nodes[]=/=graph.edges[]= facts, optional
=kindBudgets=, and optional =windowMerge= controls.  Default fast-search
candidate packets should include owner/item/test facts plus hot range nodes
when a candidate locator is known, connected through typed edges such as
=owner->item contains=, =item->hot contains=, and =owner->test covers= so path
ranking can select direct code follow-ups without reparsing compact text.
When the bounded candidate set exposes data-shape intent, such as field, type,
collection, map, set, or list terms, the request should also carry typed
semantic nodes for =field=, =type=, and =collection=.  High-concurrency,
resource-lifecycle, and API-usage terms such as =concurrency=, =queue=,
=stream=, =scope=, or =fiber= may also request semantic facts as candidate
context: the provider should expand only from stdin-selected candidate owners,
not from an unbounded whole-repository structural scan.  This keeps real API
queries able to see parser-owned data-shape evidence without sacrificing the
lexical/API relevance of the ranked frontier.  These are schema-visible graph
facts rather than compact-renderer heuristics: =field= records the owner, field
name, type text, collection kind, locator, and matched text; =type= records the
field type; and =collection= records the collection family.  The default
owner-query adapter connects them through typed edges such as =query->field
matches=, =owner->field contains=, =field->type has_type=,
=field->collection collection_of=, =type->collection collection_of=, and
=field->hot contains=.  This lets graph turbo project field-level selectors for
questions such as =Vec collection fields= or =Effect concurrency Stream Scope=
without requiring the agent to run a second owner search, broad fzf, or
tree-sitter guide probe after the ranked frontier has selected the evidence
boundary.
For Rust, these data-shape facts are provider-owned parser facts produced by
the harness's =syn= project scan.  Search-pipe lexical candidates may seed
priority, but they must not be the authority or the only files parsed for
=field=/=type=/=collection= facts; otherwise a lexical =Vec= hit outside the
struct owner prevents the graph from ever seeing the field evidence.
Provider alignment uses the command shape =search semantic-facts <query>
--json= with candidate lines on stdin.  The JSON response is provider-owned
=nodes=/=edges= graph fact payload that ASP appends to the graph-turbo request.
Rust owns these facts with =syn=, Python owns them with =ast=, TypeScript owns
them with the TypeScript compiler API through the provider parser layer, and
Julia owns them with JuliaSyntax-derived field facts in the Julia harness.
Unsupported providers may return a non-zero command result; ASP treats that as
no extra facts so rollout can be incremental without breaking existing search.
Field nodes should preserve the exact field declaration locator for display
and metadata, and may also expose a parser-owned =contextLocator= for code
projection.  When =contextLocator= is present, branch =query-selector= commands
should use that bounded field context selector rather than a single declaration
line, so the next read contains enough surrounding structure without forcing a
manual widening search.
When provider-owned =field= nodes are present in an =owner-query= result, the
bounded fanout projection should select branches from those field nodes first
and must not fill the remaining branch budget with same-owner lexical =item= or
=hot= matches merely to reach the branch cap.
When a projected branch already has an exact selector, prompt-visible compact
output must make =query-selector= the first next action for that branch.
=search-reasoning= is still available for branch expansion, but it is a
follow-up after reading the exact selector or a fallback when no exact selector
exists.  Rendered =frontierActions=, =recommendedNext=, =nextCommand=, and
=pipeCommands= should preserve that selector-first ordering so the agent does
not spend turns on owner reasoning before it has consumed the ranked evidence
boundary.  Sandtable and hook diagnostics should therefore treat
=prime -> pipe -> query-selector= as a complete complex-pipe success when the
pipe result exposes an exact selector; =search-reasoning= is measured as
optional branch expansion in that case, not as a required stage.
The prompt-visible top-three fanout is a conditional frontier, not the default
read plan.  The ranked graph may retain =S1=/=S2=/=S3= internally for audit and
debug views, but =search pipe --view seeds= should expose only the primary
=S1= exact selector in =frontierActions= and =pipeCommands= when =S1= is exact.
Secondary selectors and reasoning branches belong in =conditionalActions= as
metadata-only branch previews, without runnable =asp ...= commands or exact
secondary selector ranges.  A commands/advisory surface may materialize those
branch commands only after an explicit request or after the primary selector
evidence is known insufficient.
The seeds view is a decision packet, not a graph-debug dump.  When an exact
primary selector is projected, prompt-visible seeds output should keep the
primary graph facts, =frontierActions=S1=, =nextCommand=, compact policy,
cache/metric summaries, and avoid/stop guidance, while omitting score vectors,
typed path lists, full algorithm traces, and rank explanations.  Debug callers
can still request the graph-turbo packet/result surfaces for those diagnostics;
agents deciding the next command should not pay that token cost by default.
Output compaction is precision-preserving only.  It may remove diagnostic
volume such as score vectors, typed path lists, full traces, and rank
explanations, but it must not remove parser-owned semantic evidence that
changes the agent's decision.  For field/type/collection questions, seeds
output must still expose the primary =field=, =type=, =collection= facts,
the =has_type= / =collection_of= relationships, query coverage, the exact
=S1= selector, and the same =nextCommand=.  If a proposed compaction hides
those facts or changes the selected frontier, it is a correctness regression,
not a token optimization.
Language =guide= output is a pre-search tool map.  Once a pipe result has
projected an exact primary selector, guide should not be run as a post-selector
confirmation step; diagnostics may count =guideCommands= so this drift is
visible in live sandtable reports.
Command-line arguments may override request profile, seeds, or budget for
local debugging, but provider integration should prefer the request packet so
the algorithm input and response can be audited together.  The response is
=semantic-graph-turbo-result.v1= and must echo the effective profile,
algorithm, seed ids, budget, per-kind budget, merged windows, profile
compatibility, typed paths, graph cache, algorithm trace, rank explanations,
algorithm metrics, plus prompt-visible =profiles=, =omit=, and =avoid= facts
used by compact graph-frontier rendering.  Graph cache state is owned by the
Python ranking engine; persistent backend entries live under =$PRJ_CACHE_HOME=
or the git toplevel =.cache=, and the packet fingerprint must include graph
nodes, edges, profile, seeds, budgets, path, and window controls so changed
source facts do not replay stale rankings.

Language implementations may use their ecosystem's normal graph algorithm
library, such as Python =NetworkX= or Julia =Graphs.jl=, for traversal,
centrality, and reachability.  These libraries are implementation dependencies:
the durable protocol remains the schema-owned graph turbo result packet and
does not expose provider-local library names as ranking algorithms.

For Python MVP 11 P1, =typed-ppr-diverse= is implemented over a SciPy sparse
CSR backend.  The algorithm constructs a profile-filtered typed adjacency
matrix, runs personalized PageRank from =seedIds=, filters by profile depth,
applies diversity and per-kind budgets, merges overlapping/nearby source
windows, and emits profile compatibility evidence.  NetworkX can be useful for
experimentation, but the production Python turbo path should use SciPy sparse
linear algebra because P2 flow/path ranking and P3 trace/cache evidence also
depend on stable matrix-oriented graph state.

Python MVP 11 P2 adds typed path evidence to the same response packet:
=sourceSinkFrontier= records the effective sources and sinks,
=typedPaths= records constrained shortest and k-shortest typed paths, and
=flowLite= records the local path ranking order.  The path evidence is
schema-owned response data, not another compact prompt protocol.

Python MVP 11 P3 adds auditability around the algorithm run:
=packetFingerprint= is the canonical request/graph hash, =graphCache= reports
the sparse backend cache status, =algorithmTrace= lists the stable algorithm
steps, =rankExplanations= explains each ranked node, and =algorithmMetrics=
records sandtable-friendly counts such as path count, ranked count, merged
window count, and cache status.

Python MVP 12 makes the ranking policy explicit rather than allowing the
algorithm to consume unweighted or all-graph edges.  Graph turbo must normalize
missing edge weights through a shared relation policy before constructing the
CSR matrix: =selects= 1.2, =matches= 1.5, =repairs= 1.8, =contains= 1.0,
=calls= 1.2, =covers= 1.4, =imports= 0.8, =uses= 0.9, =flows-to= 1.6, and
=split= 0.7.  Provider-compatible aliases such as =covered_by= and
=depends_on= may map to the same initial weights as their canonical relation
families, but unknown unweighted relations must not silently become neutral
rank edges.

Each profile owns a node-kind bonus map and an explicit transition mask.  The
initial bonuses are =owner-query= field +0.4, hot +0.35, collection +0.3, type
+0.28, item +0.25, test +0.15, owner +0.1; =prime= owner +0.3, query +0.2,
dependency +0.15, test +0.15; and =read-frontier= window +0.4, symbol +0.3,
range -0.2.  The initial typed transitions are =owner-query= query->item,
query->field, query->type, query->collection, owner->item, owner->field,
item->hot, field->hot, field->type, field->collection, type->collection,
owner->test; =query-deps= query->owner, owner->dependency, dependency->owner,
owner->test; and =read-frontier= range->symbol, range->window, symbol->window.  The result
packet's =profileCompatibility= evidence must expose allowed relations,
allowed transitions, and kind bonuses so another renderer or agent does not
have to infer the policy from rank output.

Diversity is a score adjustment layer over typed PPR, not a prompt heuristic.
The initial penalties are same owner 0.25, same symbol name 0.4, and repeated
same kind beyond the soft budget 0.3; contiguous windows that can merge receive
a 0.5 bonus.  Explicit per-kind budgets remain hard caps.  This policy is meant
to suppress repeated outputs such as =semantic_string_type= x8 while still
preserving parser-owned rank evidence for distinct owners, symbols, tests, and
windows.

* Implementation order

1. Keep this RFC as the language-neutral contract.
2. Audit =semantic-graph.v1= and =semantic-search-packet.v1= for alias,
   action, rank, frontier seed, synthesis, omission, and avoid coverage.
3. Maintain =semantic-compact-graph-render.v1= as the shared prompt-facing
   text template.
4. Add schema tests for any missing fields or template vocabulary drift.
5. Implement shared renderer tests that prove prime, owner, query-set, ingest,
   tests, and item query outputs derive their compact lines from the graph
   feature.
6. Replace renderer assertions that target independent =|seed= or
   =|synthesis= rows with assertions over =rank=, =frontier=, =avoid=, aliases,
   and grouped edges.
7. Align Rust, TypeScript, and Python provider integration tests to the same
   examples and surface classes by asserting packet construction plus shell-out
   rendering, not provider-local renderer implementations.
8. Add sandtable scenarios that measure command count, token size, repeated
   search loops, and hook-recovery clarity.
9. Use real-project traces before widening graph rendering beyond the proven
   surfaces; require every =--view seeds= follow-up action to remain
   graph-derived from the start.
