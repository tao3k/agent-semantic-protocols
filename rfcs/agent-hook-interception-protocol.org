#+AUTHOR: agent-semantic-protocols
#+DATE: 2026-05-31
#+STATUS: draft
#+DEPENDS: rfcs/cli-first-harness-ux.org (005)

* One-sentence summary

Agent hooks should be a thin, platform-adapted enforcement layer over the
semantic language CLI registry.  The root protocol owns the normalized hook
decision shape and classification algorithm; language harnesses own their
language profile descriptors and provider commands.

* Problem

The current hook implementations are drifting:

- Rust has the strongest stateful gate: direct source-read blocks, raw broad
  search blocks, edit-before-prime checks, subagent receipt checks, and stop
  checks.
- TypeScript has a standalone Codex hook implementation, but it reimplements
  platform parsing and language routing in TypeScript-specific code.
- Python and Julia do not yet have the same enforcement surface.
- Root =.codex/config.toml= can install multiple language hook commands, but
  the algorithm is not standardized.

This drift is risky because hook correctness depends on platform runtime
details, not only on language facts.  A hook sees a structured tool payload for
=Read=, a shell command string for =Bash=/=exec_command=, patch text for
=apply_patch=, and event-specific stop/subagent fields.  A language-specific
regex gate cannot be trusted to classify those consistently across platforms.

* Source Model

The hook protocol has three layers:

#+begin_example
agent platform event
  -> platform adapter
  -> normalized hook invocation
  -> root classification algorithm
  -> language provider routes
  -> platform-specific response
#+end_example

The root protocol standardizes the middle two layers.  Provider repositories
still own the language-native pieces:

- source extensions and config files;
- project roots and ignored directories;
- provider binary and method descriptors;
- check command;
- search guide commands.

* Platform Facts

Claude Code hook documentation describes command hooks as JSON on stdin plus
stdout/stderr/exit-code or structured JSON decisions.  =PreToolUse= receives
=tool_name= and =tool_input=; a deny decision can be returned in
=hookSpecificOutput.permissionDecision=.  Bash permission matching is applied
to subcommands, and complex commands should be treated conservatively.

Codex has a Claude-style hook configuration surface, but its current coverage
is not yet equivalent across every built-in tool.  Public Codex issue threads
also document that =updatedInput= rewrite should not be treated as portable
and that tool coverage has changed over recent releases.

Therefore v1 must be conservative:

- do not rely on input rewriting;
- do not rely on =PreToolUse= context injection;
- use deny-with-guide for unsafe actions;
- use =SessionStart=, =UserPromptSubmit=, =SubagentStart=, =PostToolUse=, and
  =Stop= for context/state when the platform supports them;
- use =stop_hook_active= / equivalent loop guards when present.

The adapter is intentionally dumb.  It extracts strings and paths, but it does
not decide whether a path is Rust, TypeScript, Python, or Julia.  It also does
not try to infer agent intent.

* Language Profile Descriptor

Each language provider supplies a profile descriptor derived from the semantic
language registry:

#+begin_src text

The descriptor is not a global enum in the common schema.  Language-specific
repositories own language-specific capability names, while the root protocol
only requires that provider routes are explicit and machine-readable.

* Classifier

The classifier is deterministic and evidence-first.  It evaluates one
normalized invocation against the enabled language profiles and current hook
state.

** Tool intent

Classify the tool call into one or more intents:

#+begin_src text
direct-read       structured Read/read_file/MCP read of a source path
content-dump      Bash content reader: cat, sed, nl, bat, head, tail, awk, less
raw-search        rg, grep, ag, fd, find, git grep, git ls-files, ast-grep
semantic-search   <provider-binary> search ...
changed-check     <provider-binary> check --changed
edit              apply_patch, Edit, Write, or shell patch/write command
subagent-receipt  SubagentStop final message
stop              Stop event
#+end_src

Shell commands must be tokenized as commands, not matched as arbitrary
substrings.  The tokenizer does not need to evaluate shell expansion, but it
must detect pipelines, command separators, and content-reader stages.  If a
compound command cannot be parsed safely, treat it as broad when it touches a
known source root, source glob, or project root.

** Profile routing

Paths route to profiles by descriptor:

- source extension or config file -> matching language profile;
- source root plus source glob -> matching profile;
- ignored path prefixes such as =target/=, =node_modules/=, =dist/=,
  =coverage/=, and generated cache roots -> no profile;
- root-scoped search such as =rg ... .= or =fd ... .= -> every enabled profile
  unless narrowed by globs.

Raw shell search is allowed only when one of these is true:

- it is an exact-file search over one known owner file;
- it is piped into the matching provider's =search ingest= command;
- it is outside enabled language profiles;
- the policy explicitly marks it advisory.

This is the key token-saving rule: root-scoped =rg= may collect candidates, but
it must not dump them into the model.  It should normalize through
=search ingest= so the provider can group by owner, tests, dependency, and
next actions.

** Decisions

The decision order is:

1. block direct source reads unless the provider route is already a semantic
   search command;
2. block broad content dumps and broad raw searches unless they pipe to ingest;
3. block broad edits to language-profile files before prime/search evidence;
4. allow exact-file edits before prime only when policy enables the exception;
5. record semantic-search evidence in post-tool state;
6. record changed checks in post-tool state;
7. require compact subagent receipt shape at subagent stop;
8. require changed checks at stop when dirty profile files exist.

Blocking decisions must include the next safe command, not only a denial.

* Decision Packet

The common JSON contract is:

#+begin_src text
schemas/semantic-agent-hook-decision.v1.schema.json
#+end_src

The packet is not the raw platform response.  Each platform adapter maps it to
the platform's expected output:

- Claude =PreToolUse=: structured =hookSpecificOutput.permissionDecision= or
  exit-code 2 with stderr.
- Codex =PreToolUse=: structured deny payload when supported.
- =SessionStart= / =UserPromptSubmit= / =SubagentStart=: context output when
  supported.
- =Stop= / =SubagentStop=: block decision with loop guard.

* Downstream Plan

1. Root protocol:
   - keep the shared decision schema;
   - document platform adapter rules;
   - keep sandtable drills for hook payload fixtures.

2. Rust provider:
   - keep its current state machine as the reference implementation;
   - replace Rust/TS-specific branches with descriptor-driven profile routes;
   - emit the shared decision packet before rendering Codex output.

3. TypeScript provider:
   - remove hardcoded Codex-only naming from the classifier;
   - share the normalized invocation and decision concepts;
   - keep TypeScript-specific profile descriptors local.

4. Python and Julia providers:
   - implement the same =agent hook --client <platform> <event>= CLI;
   - start with direct-read, raw-search-to-ingest, edit-before-prime, and
     changed-check gates;
   - add subagent/stop state once the core gates are stable.

* Open Questions

- Whether root should eventually ship a small standalone hook router binary
  that dispatches to installed language providers.  The v1 design does not
  require it because provider binaries already exist.
- Whether Codex will add portable =updatedInput= support.  Until then,
  provider hooks should not depend on rewriting tool inputs.
- Whether shell parsing should use a common library in each implementation or
  a generated conformance fixture suite.  The v1 requirement is behavioral:
  parse command boundaries conservatively and route broad output through
  ingest.
