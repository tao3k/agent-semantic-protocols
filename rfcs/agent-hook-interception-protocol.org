#+TITLE: RFC: Agent Hook Interception Protocol
#+AUTHOR: agent-semantic-protocols
#+DATE: 2026-05-31
#+STATUS: implemented-single-bin-configurable-client-hook
#+DEPENDS: rfcs/cli-first-harness-ux.org (005)

* One-sentence summary

Agent hooks should be a thin, platform-adapted enforcement layer over the
semantic language CLI registry.  The root protocol owns the normalized hook
decision shape, activation algorithm, and classification algorithm; language
harnesses own provider manifests and semantic commands.

Status note: the single =agent-semantic-protocol= binary and configurable
client-hook lane is implemented and closed by the gates in [[*RFC Closure Gates]].
The downstream provider-convergence plan remains tracked separately below.

* Problem

The current hook implementations are drifting:

- Rust has the strongest stateful gate: direct source-read blocks, raw broad
  search blocks, edit-before-prime checks, subagent receipt checks, and stop
  checks.
- TypeScript has a standalone Codex hook implementation, but it reimplements
  platform parsing and language routing in TypeScript-specific code.
- Python now has a provider hook for direct-source-read denial; Julia does not
  yet have the same enforcement surface.
- Root =.codex/config.toml= can install multiple language hook commands, but
  the algorithm is not standardized.

This drift is risky because hook correctness depends on platform runtime
details, not only on language facts.  A hook sees a structured tool payload for
=Read=, a shell command string for =Bash=/=exec_command=, patch text for
=apply_patch=, and event-specific stop/subagent fields.  A language-specific
regex gate cannot be trusted to classify those consistently across platforms.

* Source Model

The hook protocol has three layers:

#+begin_example
agent platform event
  -> root Rust agent-semantic-hook platform adapter
  -> normalized hook invocation
  -> root Rust classification algorithm
  -> language provider routes
  -> platform-specific response
#+end_example

The root repository owns the platform adapter and classifier as a single Rust
runtime named =agent-semantic-hook=.  Provider repositories still own the
language-native pieces:

- source extensions and config files;
- source root detector defaults and ignored directories;
- provider binary and route descriptors;
- check command;
- search guide commands.

Provider-level =agent hook= commands must not keep independent Codex/Claude
payload parsing, shell classification, stop/subagent state machines, or
deny-guide rendering.  Those belong to the root runtime.  Providers expose
static manifests; the project installer activates those manifests for one
workspace.

* Root Rust Runtime

The v1 implementation target is the shared protocol binary in this repository.
The hook runtime itself stays in the =agent-semantic-hook= library crate:

#+begin_src text
asp hook pre-tool --client codex --activation .cache/agent-semantic-protocol/hooks/activation.json
asp hook pre-tool --client claude --activation .cache/agent-semantic-protocol/hooks/activation.json
asp hook pre-tool --client codex --activation .cache/agent-semantic-protocol/hooks/activation.json --emit decision
asp hook doctor --client codex --activation .cache/agent-semantic-protocol/hooks/activation.json .
asp hook install --client codex .
#+end_src

The runtime is intentionally language-neutral:

- parse platform hook JSON from stdin;
- normalize Codex/Claude event names and tool payloads;
- tokenize shell commands into command stages rather than substring matching;
- route paths and root-scoped broad searches through the project activation
  and provider manifests;
- emit =schemas/semantic-agent-hook-decision.v1.schema.json= before rendering
  the platform-specific response;
- default to platform hook payload output for =hook=, with =--emit decision=
  reserved for schema tests, receipts, and machine validation;
- own =.codex/config.toml= hook blocks that invoke
  =asp hook= directly;
- generate one project activation file that records enabled providers and
  workspace coverage, without copying provider route templates;
- maintain only shared hook state: prime/search evidence, dirty activation paths,
  changed-check evidence, and subagent receipt evidence.

The runtime must not call compiler parsers or inspect language syntax.  It
constructs safe next commands from provider manifests and lets the provider
binary own search/check semantics.

Provider crates may be linked only in build/dev contexts to generate manifest
fixtures and black-box runtime tests.  They must not become normal
=agent-semantic-hook= runtime dependencies; production classification consumes
the project activation plus static manifests, not provider APIs.

* Platform Facts

Claude Code hook documentation describes command hooks as JSON on stdin plus
stdout/stderr/exit-code or structured JSON decisions.  =PreToolUse= receives
=tool_name= and =tool_input=; a deny decision can be returned in
=hookSpecificOutput.permissionDecision=.  Bash permission matching is applied
to subcommands, and complex commands should be treated conservatively.

Codex has a Claude-style hook configuration surface, but its current coverage
is not yet equivalent across every built-in tool.  A local audit of
=openai/codex= found that =PreToolUse= serializes a canonical =tool_name= and
keeps matcher aliases as client-side compatibility inputs.  For example,
=apply_patch= may match legacy =Write=/=Edit= aliases, and =spawn_agent= may
match legacy =Agent= aliases, while dynamic/deferred tools can be introduced
through =tool_search=.  The durable boundary is therefore not a long
tool-name regex in =config.toml=; the config should dispatch broadly and let
=agent-semantic-hook= classify the normalized payload against project activation
and provider manifests.

The installed Codex hook command should also invoke =agent-semantic-protocol=
through =PATH=.  Copying a per-repository binary into
=.codex/agent-semantic-hook/bin= makes the runtime stale and easy to bypass:
the activation file and config block are repository-owned, but the classifier
binary is a global toolchain component.

=.codex/agent-semantic-protocol/hooks= is durable project hook policy.  The
versioned client policy lives at =config.toml= there.  Generated activation,
provider profile registries, and hook event logs must live under
=${PRJ_CACHE_HOME}/agent-semantic-protocol/hooks= when =PRJ_CACHE_HOME= is set,
otherwise under the git toplevel =.cache/agent-semantic-protocol/hooks=
directory.

Provider execution profiles are a separate runtime cache contract.  The root
runtime writes =profiles.json= under
=${PRJ_CACHE_HOME}/agent-semantic-protocol/runtime=, or otherwise under the git
toplevel =.cache/agent-semantic-protocol/runtime= directory.  Activation
continues to record which provider manifests and coverage roots are active;
runtime profiles record the resolved provider argv and health for the current
project.  Agent-facing provider execution must prefer a healthy profile argv
over shell =PATH= lookup.  This prevents an agent from accidentally selecting a
different provider binary when it forgets =direnv exec= or inherits the wrong
=PATH=, and it makes stale/missing runtime tools visible through doctor output
instead of hiding them behind generated bin symlinks.

Public Codex issue threads also document that =updatedInput= rewrite should
not be treated as portable and that tool coverage has changed over recent
releases.

Therefore v1 must be conservative:

- do not rely on input rewriting;
- do not rely on =PreToolUse= context injection;
- keep Codex hook matchers broad and perform language/policy classification in
  the =agent-semantic-hook= library behind =asp hook=
  instead of encoding tool-name allowlists in TOML;
- use deny-with-guide for unsafe actions;
- use =SessionStart=, =UserPromptSubmit=, =SubagentStart=, =PostToolUse=, and
  =Stop= for context/state when the platform supports them;
- use =stop_hook_active= / equivalent loop guards when present.

** Codex source audit: tool surface is not filesystem enforcement

A local source audit of =.data/codex/codex-rs= found that Codex =PreToolUse=
is a tool-registry hook, not a universal file-system interception layer:

- =core/src/tools/registry.rs= only calls =run_pre_tool_use_hooks= when a tool
  handler returns =pre_tool_use_payload=.
- =core/src/hook_runtime.rs= blocks only when the hook result has
  =should_block=true= and a non-empty block reason.  A malformed deny, missing
  reason, invalid JSON output, or unsupported decision shape degrades to
  continue/fail-open behavior at the Codex boundary.
- =core/src/tools/handlers/unified_exec/exec_command.rs= maps shell execution to
  the canonical =Bash= hook payload with a raw command string.
- =core/src/tools/handlers/unified_exec/write_stdin.rs= intentionally returns
  no =PreToolUse= payload.  A long-lived shell therefore has a second input
  surface that the hook cannot reclassify unless the original session launch was
  already constrained.
- =app-server-protocol= and =exec-server= expose =fs/readFile=,
  =fs/readDirectory=, =fuzzyFileSearch/*=, =command/exec=, and
  =process/spawn= RPC surfaces outside the ordinary model tool registry.  These
  are app/exec-server APIs, not guaranteed =PreToolUse= events.

The root classifier must therefore describe hook coverage explicitly instead of
pretending every source read is enforceable through the same event.

#+begin_src text
surface                        hook coverage     normalized operation
Bash / exec_command             pre-tool          shell-command
apply_patch                     pre-tool          apply-patch
mcp__*__read*                   pre-tool          direct-read
Read / read_file                pre-tool if tool  direct-read
multi_tool_use.parallel         pre-tool wrapper  nested-tools
write_stdin                     no pre-tool       stdin-continuation
fs/readFile                     app RPC           direct-read
fs/readDirectory                app RPC           directory-read
fuzzyFileSearch/*               app RPC           file-search
command/exec                    app RPC           shell-command
process/spawn                   app RPC           shell-command
#+end_src

** Codex Source Access Layer without a daemon

The Codex-internal route should be specified as a source-access layer, not as a
larger hook matcher.  In the no-daemon lane, the goal is to prevent
unbrokered source bytes from entering the model-visible transcript and to route
the agent back to provider-owned compact search/query commands.  It is not a
kernel monitor and it does not claim to intercept arbitrary subprocess
=open=/=read= calls.

MCP surfaces are out of scope for this v1 source-access layer.  MCP tools,
resources, and server-owned reads may still be classified opportunistically by
the hook runtime when their payloads are visible, but they are not part of the
Codex-internal enforcement guarantee.  This keeps the first contract focused on
Codex-owned APIs and avoids treating external MCP servers as if they were
controlled by the Codex process.

The enforcement contract is:

#+begin_src text
boundary                  operation              enforcement        transcript guarantee
codex-fs-api              read-file              hard               no source bytes returned
codex-fs-api              read-directory         hard               no source listing returned
codex-tool-action         read-file              hard               no source bytes returned
codex-shell-preflight     spawn-process          preflight          deny obvious raw reads before spawn
codex-shell-preflight     process-stdin          advisory           guide when stdin is visible
codex-shell-egress        tool-output            egress             suppress source dumps before model
subprocess open/read      arbitrary              not-enforced       no no-daemon guarantee
#+end_src

The source-access decision packet is separate from the hook decision packet:

#+begin_src text
schemas/semantic-source-access-decision.v1.schema.json
#+end_src

The packet records the Codex boundary, normalized operation, enforcement mode,
whether source bytes were returned by the local action, whether any bytes became
model-visible, and the provider route that should be used instead.  For shell
commands, =sourceBytesReturned=true= with =modelVisibleBytesReturned=false= is
a valid egress suppression result: the subprocess may have produced a source
dump, but Codex did not place it into the agent transcript.  For hard FS API
denials, both values must be false.

The root CLI exposes an internal no-daemon probe so Codex integration tests can
exercise this packet without installing app-server patches first:

#+begin_src text
asp source-access read-file --activation <activation.json> --rpc-method fs/readFile <path>
asp source-access shell-egress --activation <activation.json> --command <command> --output-digest <digest> <path>
#+end_src

This command is a black-box policy probe for Codex-owned integration points.  It
is not an agent exploration surface and it does not replace provider
=asp <language> query/search= commands.

Provider reads are authorized only through an explicit provider/broker
capability, normally surfaced as =asp <language> query/search=.  Raw Codex reads
and provider-owned compact reads therefore have different policy identities:

#+begin_src text
Codex Read src/lib.rs                  => deny, direct-source-read guide
shell sed -n 1,120p src/lib.rs         => deny preflight or suppress egress
asp rust query --from-hook ...         => allow provider-capability
#+end_src

The public status/doctor surface should report enforcement honestly:

Doctor reports two different checks.  =classifierProbe= is the fast in-process
hook classifier self-test: it feeds a representative =functions.exec_command=
source-dump payload through the same decision code and should return
=deny/bulk-source-dump= when client config is present.  =enforcementProbe= is
the live Codex runtime probe: it only proves the real tool channel when the
Codex exec probe is explicitly run.  If that runtime probe is disabled, doctor
must say =enforcement=unproven= and =enforcementProbe=codex-exec-disabled=,
not imply live filesystem/tool interception.

#+begin_src text
fsApi=enforced
toolAction=enforced
shellPreflight=best-effort
shellOutput=egress-enforced
subprocessOpen=not-enforced
mcp=out-of-scope
daemon=not-used
#+end_src

The shared decision packet should carry normalized =fields.toolSurface= and
=fields.operationIntent= for action-derived decisions.  These fields make
receipts explain the client/tool surface that triggered the route, and they let
black-box tests distinguish "blocked by hook" from "known app RPC surface that
needs a lower enforcement layer".

The classifier must operate primarily at the action layer:

- structured tool payloads are normalized by extracting action fields such as
  =path=, =file=, =files=, =uri=, =resource=, =document=, and nested tool calls;
- an unknown tool with a source selector path is treated as source access and
  routed through the same direct-source-read policy as Codex's built-in =Read=;
- known tool names are hints for platform surface reporting, not the policy
  authority;
- shell commands are parsed with the Bash grammar through =tree-sitter-bash=
  before direct-read, content-dump, raw-search, and provider-output-pipe
  decisions are made; the classifier should preserve parser-owned command
  stages and argument roles instead of flattening first and recovering meaning
  through wrapper-name heuristics;
- raw-search source selector matching remains provider-owned through activated
  =sourceExtensions= and =globset=: =tree-sitter-bash= owns selector roles,
  the shared matcher extracts the selector's extension pattern, and the
  pattern must match the provider-declared language extensions before the raw
  search is treated as source-content access.

Client-side hook customization is a typed config file, not a server process:

- default Codex path: =.codex/agent-semantic-protocol/hooks/config.toml=;
- install behavior: =asp hook install --client codex=
  creates this file when missing, with schema metadata plus commented examples,
  and preserves an existing valid project config;
- schema: =schemas/semantic-agent-hook-client-config.v1.schema.json=;
- load timing: every =asp hook= invocation;
- loader: the Rust runtime uses typed =serde= extraction through =figment=
  =Toml::file=; no live-reload, watcher, daemon, or server affordance is
  enabled in this client lane;
- language filtering: rule =languageIds= match action paths through activated
  provider coverage for the project;
- rule ordering: higher =priority= wins, and equal priorities keep config file
  order so the tie-break is deterministic and reviewable;
- rule id contract: rule ids are unique across the loaded config because they
  become machine receipts;
- rule receipts: config-derived decisions set =fields.configRuleId= to the
  matching rule id;
- experimental switches: generated Codex client configs include
  =[experimental.semanticAstPatch] enabled = false=.  The built-in source
  =apply_patch= to semantic AST patch route is experimental; hook runtimes may
  preserve legacy behavior when the switch is absent, but generated
  =.codex/agent-semantic-protocol/hooks/config.toml= must keep it disabled
  until the provider-native AST patch path is stable;
- runtime schema mirror: the loader rejects schema-shape errors before
  classification, including invalid identifiers, empty min-length strings,
  unsupported events/platforms, duplicate =languageIds=, empty route =argv=,
  and invalid route binary names;
- doctor: =asp hook doctor --client codex= reports the
  client config path and =clientConfigStatus= (=missing= or =ok=), and fails
  when the config is invalid;
- failure mode: invalid config emits a block decision so policy cannot silently
  degrade;
- no watcher, daemon, or background server behavior belongs to the client hook
  runtime.

** Codex runtime constraints

The root installer must treat Codex hook activation as a runtime contract, not
as a repository-local TOML snippet:

- =features.hooks=true= is required, or Codex will not run hook handlers.
- Project config is not a self-trust root.  Codex only runs project hooks after
  the project config layer is loaded and the hook's current hash is trusted from
  user/session/managed hook state.  Writing =hooks.state= only inside the same
  project =.codex/config.toml= has no enforcement value.
- Untrusted unmanaged hooks can appear in =hooks/list=, but they are discovery
  metadata; they do not become runnable handlers.
- =PreToolUse= matcher selection uses the canonical tool name plus compatibility
  aliases, but the hook stdin payload contains the canonical =tool_name= and
  =tool_input=.  Therefore match broadly in TOML and classify the normalized
  payload in the =agent-semantic-hook= library behind
  =asp hook=.
- Codex Desktop API entries such as =functions.exec_command= and nested
  =multi_tool_use.parallel= calls must be replayed as first-class hook payloads,
  then smoke-tested through the real runtime.  Wrapper paths such as
  =path.rs:10:20= and =path.rs:10:4= must be normalized before both policy
  matching and provider route construction.
- Codex Desktop and deferred-tool read payloads are not stable enough to trust
  only one field spelling.  The platform adapter must extract paths from the
  language-neutral field family =path=, =file=, =file_name=, =fileName=,
  =file_path=, =filePath=, =absolute_path=, =absolutePath=,
  =relative_path=, =relativePath=, =uri=, =paths=, =files=, =resource=,
  =resources=, =uris=, =document=, =documents=, =text_document=, and
  =textDocument=, including arrays and nested objects.  =file://= URIs are
  normalized to project paths before source selector matching.
- Codex hook output is schema-strict.  Unknown top-level fields can make the
  output fail to parse and convert an intended denial into no blocking action.
  The shared decision packet must either be emitted with =--emit decision= for
  tests/receipts, or embedded inside an accepted platform field such as
  =hookSpecificOutput.additionalContext=.
- The installed command must resolve =asp= through =PATH= so upgrades affect
  every repository.  The repository owns activation config and provider
  manifests; the classifier binary is a toolchain component.

This is the concrete root-cause model for the low escape cost observed during
the Codex CLI audit: a narrow matcher, a stale repo-local binary, missing hook
feature enablement, project-local self-trust, schema-invalid output, or a
payload path field outside the adapter's field family can each make the hook
non-enforcing before language policy is even evaluated.

Every platform adapter should produce the same internal record:

       post-tool | subagent-start | subagent-stop | stop | unknown

The adapter is intentionally dumb.  It extracts strings and paths, but it does
not decide whether a path is Rust, TypeScript, Python, or Julia.  It also does
not try to infer agent intent.

* Provider Manifest And Project Activation

The hook runtime does not use a generated command registry as its source of
truth.  Provider truth and project activation are separate contracts.

Provider manifests are static descriptors owned by the provider package or by
the root runtime's built-in provider registry.  The shared JSON shape is:

#+begin_src text
schemas/semantic-agent-hook-provider-manifest.v1.schema.json
#+end_src

A provider manifest records only stable provider facts:

#+begin_src text
manifestId
manifestVersion
languageId
providerId
namespace
binary

source.defaultConfigFiles[]
source.defaultSourceRoots[]
source.defaultIgnoredPathPrefixes[]
policy.directSourceRead = block | allow | advisory
policy.bulkSourceDump = block | allow | advisory
policy.rawSourceSearch = block | allow | advisory
policy.agentSearchJson = block | allow | advisory

routes.owner(argv)

routes.ingest(argv, stdinMode)

#+end_src

The manifest is not a workspace cache.  It must not be regenerated under
=.codex=.  Command display text is derived from =argv= at render time and is
not stored as an independent field.

Project activation is the repository-local generated file.  The shared JSON
shape is:

#+begin_src text
schemas/semantic-agent-hook-activation.v1.schema.json
#+end_src

Activation records only which provider manifests are enabled in this workspace
and what coverage the installer resolved:

#+begin_src text

providers[].coverage.configFiles[]
providers[].coverage.sourceExtensions[]
providers[].coverage.ignoredPathPrefixes[]
providers[].manifestDigest
#+end_src

Activation does not repeat command templates, policies, or guide text.  The
hook hot path loads activation, resolves each =manifestId= against static
manifests, then builds an in-memory dispatch index:

#+begin_example
extension/config/glob/source-root -> activated provider manifest
hook intent -> provider route kind -> argv rendered from manifest + activation
#+end_example

If activation is missing, malformed, or references an unknown manifest, the
hook event must still fail closed for source-like direct reads and source
dumps that can be classified from the raw platform payload.  The denial route
may degrade to a repair-oriented guide when no provider route can be trusted,
but activation errors must not turn obvious source reads into allow decisions.
=doctor= and =install= are strict: they rebuild activation, verify manifest
digests, and fail if no provider can be activated.  They must not run provider
=guide= or any other provider command during discovery; PATH executable checks
and static workspace detector facts are enough.

* Shared Hook State

Generated hook state is diagnostic and routing evidence, not a source of
provider truth.  It lives under
=${PRJ_CACHE_HOME}/agent-semantic-protocol/hooks= when =PRJ_CACHE_HOME= is set,
otherwise under the git toplevel =.cache/agent-semantic-protocol/hooks=.

The append-only event log is =events.jsonl=.  Each non-empty line must be one
complete JSON object with:

#+begin_src text
schemaId = agent.semantic-protocols.hook.event
schemaVersion = 1
protocolId = agent.semantic-protocols.hook
protocolVersion = 1
platform
event
decision
reasonKind
languageIds[]
subject
routeKinds[]
recordedAtUnixMs
#+end_src

Hook invocations may overlap because agent clients can execute tools in
parallel.  Event writes therefore must be serialized with an operating-system
file lock or an equivalent process-safe append protocol.  A JSONL write is not
valid unless the reader can parse every line independently; interleaved partial
writes are a protocol violation, not only a logging defect.

=asp hook install= and =asp hook doctor= must treat incompatible or corrupted
event logs as migratable cache, not durable policy.  If an existing
=events.jsonl= contains a non-empty line with a legacy =schemaId=, legacy
=protocolId=, or invalid JSON, install may remove that generated file and
start a new current-protocol log.  The versioned project policy remains
=.codex/agent-semantic-protocol/hooks/config.toml=.

* Classifier

The classifier is deterministic and evidence-first.  It evaluates one
normalized invocation against the activated provider dispatch index and current
hook state.

** Tool intent

Classify the tool call into one or more intents:

#+begin_src text
direct-read       structured Read/read_file/namespaced read_file/MCP read of a source path
directory-read    structured listFiles/list_files directory enumeration action
content-dump      Bash content reader: cat, sed, nl, bat, head, tail, awk, less
raw-search        rg, grep, ag, fd, find, git grep, git ls-files, ast-grep
semantic-search   <provider-binary> search ...
search-json       <provider-binary> search ... --json during agent exploration
changed-check     <provider-binary> check --changed
edit              apply_patch, Edit, Write, or shell patch/write command
ast-patch-gate    source-file apply_patch requires semantic AST patch validation
subagent-receipt  SubagentStop final message
stop              Stop event
#+end_src

Shell commands must be tokenized as commands, not matched as arbitrary
substrings.  The tokenizer does not need to evaluate shell expansion, but it
must detect pipelines, command separators, and content-reader stages.  If a
compound command cannot be parsed safely, treat it as broad when it touches a
known source root, source glob, or project root.

Structured direct-read tool names must be normalized before classification.
Bare names such as =Read= and =read_file=, namespaced deferred tools such as
=functions.read_file=, and MCP read aliases are the same hook intent when they
carry a source path.

Codex app-server =CommandAction::{Read,ListFiles,Search}= and core
=ParsedCommand::{Read,ListFiles,Search}= are parser-owned Explore evidence, not
raw shell text.  The hook must classify structured =Read/read= as
=direct-read=, structured =ListFiles/list_files= as =directory-read=, and
structured =Search/search= as =raw-search= before considering the rendered
command string.  A =ListFiles/list_files= action over an activated source root
or source-owned path is denied with
=reasonKind=source-directory-enumeration= and the owning provider's existing
=ingest= route.  Directory listings outside activated source scope remain
allowed.

Source-file =apply_patch= is a special edit intent.  If a Codex direct
=apply_patch= payload or shell heredoc contains patch hunks for activated
provider source paths, the classifier should deny with
=reasonKind=semantic-ast-patch-required= and return the same exact-read route
used by direct source reads.  This does not validate or rewrite patch hunks.
It only prevents compact AST projection output from being used as text patch
context.  Non-source patches remain ordinary edit tool calls.

** Activation routing

Paths route to activated providers through manifest and activation facts:

- source extension or config file -> matching activated provider;
- source root plus source glob -> matching activated provider through a
  compiled glob selector, not ad hoc suffix string matching;
- ignored path prefixes such as =target/=, =node_modules/=, =dist/=,
  =coverage/=, and generated cache roots -> no provider;
- root-scoped search such as =rg ... .= or =fd ... .= -> every activated
  provider unless narrowed by globs.

The classifier should maintain one shared source-selector matcher.  Structured
read tools, wrappers such as =rtk read=, content dumps, and raw search scopes
all consult the same activated =sourceExtensions=, =sourceRoots=, and
=configFiles=.  Adding another language should normally require adding a
provider manifest and workspace detector only; the classifier algorithm should
not gain a language-specific branch.

For performance, a selector is normalized and parsed once per classifier pass:
the matcher owns glob detection, extension-pattern extraction, config filename
matching for exact direct reads, and compiled globsets.  Classifier actions
consume that parsed selector rather than compiling glob patterns inside every
provider-extension pair.

Glob-like selectors such as =*.rs=, =src/**/*.tsx=,
=packages/*/src/**/*.py=, =crates/**/lib.rs=, =*.{ts,py}=, and =*.[jt]s=
must be matched with a glob engine such as Rust =globset=.  Command parsing
still owns argument roles: =rg -e runner.py docs= treats =runner.py= as the
search pattern, not as a Python source selector, while =rg -g '*.py' docs= or
=git ls-files '*.py'= are source selectors.

Selectors without a source suffix remain valid scope limiters, but for
=rg=/=fd=-style raw search they are not sufficient source evidence by
themselves.  The necessary raw-search source signal is a parser-owned
language selector that matches the activated provider's required
=sourceExtensions= through the shared globset matcher, such as =*.rs=,
=**/*.{ts,tsx}=, or =src/model.py=.
Roots such as =src=, =crates=, or =.= can narrow the
search after that signal exists, but they do not turn =env | rg PATH=,
=rg Foo src=, or =fd Foo crates= into source search.

Raw shell search is allowed only when one of these is true:

- it is piped into the matching provider's =search ingest= command;
- it is outside activated language providers;
- the policy explicitly marks it advisory.

This is the key token-saving rule: root-scoped =rg= may collect candidates, but
it must not dump them into the model.  It should normalize through
=search ingest= so the provider can group by owner, tests, dependency, and
next actions.
Exact source-file raw search such as =rg -n Foo src/model.ts= is still raw
source exploration.  It should be denied with the same provider-owned guide
instead of being treated as a safe exception; use =search owner= or a scoped
=search fzf= packet when the owner is already known.

Semantic search JSON is not the agent-facing exploration surface.  A
provider-binary =search ... --json= command is allowed only when the caller is a
schema-contract test, sandtable receipt/replay, IDE/Flowhub consumer, or local
validator that consumes the JSON as a machine artifact.  During interactive
agent exploration, hooks should deny it with =reasonKind=agent-search-json= and
return the equivalent compact command, usually preserving the search route and
adding =--view seeds= when the provider supports it.

** Decisions

The decision order is:

1. block direct source reads unless the provider route is already a semantic
   search command;
2. block prompt-facing =search ... --json= exploration unless the invocation is
   a registered machine-consumer context;
3. block broad content dumps and broad raw searches unless they pipe to ingest
   or route to a provider-declared hook query command;
4. block broad edits to activated provider files before prime/search evidence;
5. allow exact-file edits before prime only when policy enables the exception;
6. record semantic-search evidence in post-tool state;
7. record changed checks in post-tool state;
8. require compact subagent receipt shape at subagent stop;
9. require changed checks at stop when dirty activated provider files exist.

The raw command classifier is action-aware first and selector-aware second.
Directory names such as =docs/**= or =crates/**= are not language evidence on
their own; only selectors whose basename or glob extension pattern matches
provider-declared =sourceExtensions= are raw-search source evidence.

| Action family | Source evidence | Term evidence | Decision |
| Direct read/edit wrapper | exact source file | exact selector | deny with provider =query --from-hook direct-source-read --selector <path> .= route |
| Direct read/edit wrapper | broad source suffix selector | none | deny with provider prime route |
| Content dump (=sed=, =less=, =head=, =tail=, =nl=, =bat=) | exact source file | exact selector | deny bulk-source-dump with provider =query --from-hook direct-source-read --selector <path> .= route |
| Content dump (=sed=, =less=, =head=, =tail=, =nl=, =bat=) | source glob | not required | deny bulk-source-dump with provider prime route |
| Content diff (=git diff=, =git diff --no-index /dev/null <path>=) | exact source file | exact selector | deny bulk-source-dump with provider =query --from-hook direct-source-read --selector <path> .= route |
| Metadata diff (=git diff --stat=, =git diff --name-only=, =git diff --name-status=) | source or non-source scope | metadata only | allow |
| Raw search (=rg=, =grep=, =fd=, =find=, =git grep=) | parser-owned selector extension matches activated =sourceExtensions= | concrete query term | deny with provider query route when declared |
| Raw search (=rg=, =grep=, =fd=, =find=, =git grep=) | parser-owned selector extension matches activated =sourceExtensions= | no concrete term | deny with provider ingest route |
| Structured directory/listFiles action | activated source root/path | none | deny source-directory-enumeration with provider ingest route |
| Raw search | non-source scope only, such as markdown/docs | any term | allow |
| Raw search piped to provider ingest | source scope | candidates on stdin | allow |
| Prompt-facing provider =--json= search | registered interactive shell | any | deny with compact text/owner route |

Untracked source files do not justify raw full-file or no-index diff reads.
If an agent needs context for a new source file that is absent from the index,
the hook should route the exact path to the owning provider query/owner command.
The provider must treat parser-visible untracked files inside declared source
roots as normal query candidates; the only Git command that remains safe at
this stage is metadata discovery such as =git status --short= or
=git diff --name-only=.

Blocking decisions must include the next safe route, not only a denial.  The
=routes= array is the machine-readable next-action source of truth.  For one
exact project-relative source path, the hook should surface the owning
provider's exact =query --selector ... --workspace ... --code= route instead of
falling back to a generic guide pointer, source rendering, or
=--from-hook direct-source-read=.  For TypeScript, source-read recovery
therefore routes through the public =asp= language facade while preserving
=providerId=ts-harness=:

#+begin_example
direct-source-read denied; route: asp typescript query --from-hook direct-source-read --selector src/components/WorkflowExecution.tsx --code .
#+end_example

The hook only materializes the exact selector route; the language provider
owns parser matching, owner ranking, exact item selection, and compact owner
context rendering.

For broad or pattern-based denials that do not carry one exact source path, the
=message= field remains an agent-facing guide pointer.  If the matched provider
manifest declares a query route and the raw search contains concrete terms, the
route should use that route with ={selector}= and ={termArgs}= expanded from
activated suffixes and shell-tokenized search terms.  Otherwise, it should wrap
the provider-owned guide command in a compact form such as:

#+begin_example
direct-source-read denied; provider guide: rust => asp rust guide .; python => asp python guide .
raw-broad-search denied; provider guide: typescript => asp typescript guide .
#+end_example

For Codex and Claude prompt surfaces, the rendered reason text is Markdown, not
a pipe-delimited template and not a replacement for the JSON decision packet.
The packet remains the machine-readable contract; the prompt string teaches the
next safe agent move.  A deny message should contain these sections in order:

#+begin_example
# ASP Hook Recovery

## Stop
Do not retry raw source reads, shell source dumps, or broad raw search on the
matched source path.

## Run Next
asp <language> query --from-hook direct-source-read ...

## Detected Binaries
- language=<language> provider=<providerId> command=<runtime-profile argv> facade=asp <language>

## Agent Flow
1. Start from asp <language> guide . when the tool map is needed.
2. Use search prime for project mapping.
3. Use query guide treesitter only for syntax pattern work.
4. Use query --code only after exact selector or unique match.
5. Patch with apply_patch unless the loaded hook config enables semantic ast-patch.
#+end_example

The =Detected Binaries= section is diagnostic, not the executable public route.
It must use the generated runtime profile's resolved =argv= when available so
PATH, direnv, symlink, and stale binary drift are visible.  The public route in
=routes[].argv= still uses the =asp <language>= facade and keeps =providerId=
as identity.  If =experimental.semanticAstPatch.enabled = false= in the loaded
client hook config, the rendered flow must explicitly tell agents to use
=apply_patch= and not route ordinary edits through provider =ast-patch=.

The root hook must not maintain a parser/query engine.  Exact selected source
paths route back to the owning provider's normal =query --selector
<path-or-range> --workspace <workspace-root> --code= surface through =asp
<language>=, while =providerId= records the native provider identity such as
=rs-harness=, =ts-harness=, or =py-harness=.  Hook routes must not expose
provider implementation binaries as the executable command surface.
=--from-hook direct-source-read= is reserved for audited low-priority exact
windows after parser-owned search/frontier evidence, and must not be advertised
as the normal direct-read recovery route for broad or unbounded source access.

If a provider wants to return bounded source windows after that route, it may
emit =semantic-read-packet.v1= from its =query/*= method with
=protocolId=agent.semantic-protocols.semantic-language=.  The root hook still
emits only a hook decision packet and a =kind=query= route; it must not expose
or emulate a root read command.

* RFC Closure Gates

This RFC is closed for the single-protocol-bin configurable client-hook lane
when the following implementation gates pass:

1. =agent-semantic-hook= is no longer an install-facing binary target.  Cargo
   metadata must show only =lib=, =test=, and =custom-build= targets for that
   crate, while =agent-semantic-protocol= owns the only hook-facing bin target.
2. Generated Codex config uses
   =exec asp hook <event> --client codex= for every
   installed event: =session-start=, =user-prompt=, =pre-tool=,
   =permission-request=, =post-tool=, =subagent-start=, =subagent-stop=, and
   =stop=.
3. Generated config, skill text, docs, schemas, and fixtures do not instruct
   users to run retired =agent-semantic-hook= command forms.  Automated scans
   should keep exact legacy executable strings out of human-facing artifacts.
4. Missing client config preserves the built-in classifier behavior; invalid
   config fails closed with a structured block decision; doctor reports
   =clientConfig= and =clientConfigStatus= and rejects invalid config.
5. Config-derived decisions emit =fields.configRuleId=, duplicate rule ids are
   rejected before classification, higher priority wins, and equal-priority
   rules keep config order.
6. Runtime config validation mirrors the shared schema shape for identifiers,
   min-length strings, event/platform enums, language id uniqueness, route
   =argv=, and route binary names.
7. Config changes are observed on the next =asp hook=
   process invocation without any watcher, daemon, or server loop.
8. Validation gates pass:
   - =cargo test -p agent-semantic-hook=
   - =cargo test -p agent-semantic-protocol=
   - schema tests for =agent-semantic-hook-decision= and
     =agent-semantic-hook-client-config=
   - =rs-harness check --changed crates/agent-semantic-hook=
   - =rs-harness check --changed crates/agent-semantic-protocol=
9. Direct-read payload field-family coverage is tested through the classifier
   and through the real =asp hook pre-tool= runtime for at least these Codex
   shapes: =path=, =file=, =absolutePath=, nested =resource.path=,
   =resources[].filePath=, and =file://= URI lists.  Each source match denies
   with =reasonKind=direct-source-read= and a route shaped as
   =asp <language> query --selector <path> --workspace <workspace-root> --code=.
10. Concurrent hook invocations cannot corrupt =events.jsonl=.  Unit tests must
    exercise overlapping appends, and a real-runtime smoke must prove that
    every event log line parses as JSON with
    =schemaId=agent.semantic-protocols.hook.event= and
    =protocolId=agent.semantic-protocols.hook=.
11. Prompt-facing deny output is a Markdown recovery prompt, not a pipe-template
    or raw JSON replacement.  It must include =# ASP Hook Recovery=, =## Stop=,
    =## Run Next=, =## Detected Binaries=, =## Agent Flow=, and =## Rules=.
12. =Detected Binaries= must render the current runtime profile command argv
    when available, while =routes[].argv= remains the public =asp <language>=
    facade route.  A real hook smoke must prove the displayed command path can
    expose stale PATH/direnv drift without teaching agents to bypass the facade.
13. Hook recovery output must branch on the loaded client config: when
    =experimental.semanticAstPatch.enabled = false=, the prompt says to use
    =apply_patch= and not provider =ast-patch= for ordinary edits; when enabled,
    it may mention provider =ast-patch= only after dry-run receipt validation.
14. Generated skills and hook prompts must not reintroduce retired
    agent-prefixed guide spelling or the old search-wrapper hook query
    spelling.  The accepted guide entrypoint is =asp <language> guide .=, its
    help surface is =asp <language> guide --help .=, and the accepted hook
    recovery route is top-level =query --selector <path-or-range> --workspace <workspace-root> --code=.

* Downstream Plan

1. Root protocol:
   - keep the shared decision schema, provider manifest schema, and activation
     schema;
   - ship the Rust =agent-semantic-hook= library runtime behind the
     =agent-semantic-protocol= binary as the only classifier;
   - keep sandtable drills for hook payload fixtures and shell-tokenizer
     conformance.

2. Rust provider:
   - export a Rust provider manifest;
   - move current state-machine behavior into root-runtime tests;
   - remove provider-private Codex payload parsing once the root runtime covers
     direct reads, raw search, edit-before-prime, stop, and subagent receipts.

3. TypeScript provider:
   - export a TypeScript provider manifest;
   - delete the standalone Codex classifier or replace it with a thin wrapper
     that invokes =asp hook=;
   - keep TypeScript-specific source/config ownership local.

4. Python provider:
   - export a Python provider manifest;
   - replace any transitional provider-owned hook wrapper with =asp hook=, or
     keep only a thin compatibility wrapper that invokes =asp hook=;
   - deny direct Python source reads with =reasonKind=direct-source-read= and a
     concrete =asp python query --selector <path> --workspace <workspace-root>
  --code=
     route when the root runtime is installed;
   - continue moving shared classification into the root hook runtime behind
     =asp hook= for raw-search-to-ingest, edit-before-prime,
     changed-check, subagent, and stop gates.

5. Julia provider:
   - export a Julia provider manifest before adding provider-owned hook code;
   - rely on =asp hook= for direct-read,
     raw-search-to-ingest, edit-before-prime, changed-check, subagent, and stop
     gates.

* Open Questions

- Whether Codex will add portable =updatedInput= support.  Until then,
  provider hooks should not depend on rewriting tool inputs.
- Which Rust shell-tokenizer crate is acceptable for the production runtime.
  The conformance requirement is fixed now: parse command boundaries
  conservatively and route broad output through ingest.
