** Search Flow
Search is not a mandatory pipeline. Infer the current =evidenceState= first, choose the narrowest route whose preconditions are already satisfied, then follow =recommendedNext= or =nextCommand= when the packet proves it. Do not run =search prime= when an owner path, exact structural selector, dependency anchor, failure/test frontier, changed file, or user-provided file/symbol anchor already identifies a narrower route.

Before any ASP search command, record the route justification mentally or in the receipt: known anchors, chosen route, rejected broader routes, expected projection, code policy, and fallback reason when a fallback is used. =displayLineRange= and =sourceLocatorHint= are display or transport hints, not executable semantic identity. =query-code= requires exact parser-owned identity.

After each ASP command, verify the trajectory: =.syntax= actions must stay non-code unless upgraded through an exact selector, fallback reads must be bounded and justified, repeated prime/fzf/rg after a frontier is a route smell, feedback risks should link to =route-feedback:<id>= entries through =feedback.dataset-linked=, and completion claims require packet or test evidence.
| Evidence state | Preferred route | What the agent learns |
| capability unknown | =asp <language> guide .= | Supported surfaces, axes, recipes, and dependency profiles. |
| no precise owner, selector, dependency, failure/test, changed-file, or user anchor | =asp <language> search prime --workspace . --view seeds= | Workspace owners, provider roots, package/member hints, and first action frontier. |
| question terms but no owner | =asp <language> search pipe '<question>' --workspace . --view seeds= | Ranked evidence, query terms, missing facts, risk, and next command. |
| known owner path | =asp <language> search owner <owner> items --query '<term>' --workspace . --view seeds= | Owner-local symbols, tests, imports, docs-use, and exact selectors. |
| uncertain path or owner term | =asp fd -query '<path-or-owner-term>' --workspace <scope>= | Candidate owners when path or selector confidence is low. |
| packet asks for scoped text recovery | =asp rg -query '<term>' --workspace <scope>= | Candidate owners from indexed text when a search packet asks for scoped recovery. |
| exact parser-owned selector | =asp <language> query --selector <selector> --workspace . --code= | Source text for a precise selector already emitted by search. |

** Execution Fan-out
Subagents are optional client execution. They consume search-flow axes; they do
not define new query semantics.

| Trigger | Action |
| session search worker id | Open at most one ASP search worker per main session, record its id in parent state, and reuse that id for later search fan-out. |
| independent =actionFrontier= or =targetActions= rows | Assign each bounded command group to a read-only worker. |
| worker result | Return compact evidence, missing facts, risk, and next action to the parent. |
| no subagent support | Run the same command groups sequentially in the parent. |

** Dependency Search Topology
Dependency questions stay inside provider search until the provider reports a
missing dependency frontier.

| Command | Purpose |
| =asp <language> search deps <dependency[@version][::api]> --workspace . --view seeds= | Return current manifest owner, dependency edge, import owners, docs/public API use, source/runtime hints, tests, and next command. Use one selector token such as =tokio::spawn=; do not pass an extra positional API token. |
| =asp <language> search pipe '<question>' --workspace . --view seeds= | Graph-turbo may surface =search-deps= as an action affordance when dependency topology nodes prove a package frontier. |
| =asp <language> search owner <manifest-owner> items --query '<dependency>' --workspace . --view seeds= | Inspect dependency evidence in the manifest or package owner. |

** Document Search
path-context resolution is part of the document-provider contract: pass =--workspace= explicitly and recover content through exact parser-owned selectors.
Document providers expose AST facts rather than policy harnesses.
Use asp org capture --contract CONTRACT_ID for non-mutating Org entry plans backed by explicit contracts such as agent.task.v1, agent.plan.v1, agent.sdd.v1, agent.adr.v1, agent.bdd.v1, agent.prd.v1, or agent.execplan.v1.
The contract id must be explicit; do not use positional aliases such as task, plan, or sdd.

| Command | Purpose |
| =asp org guide --workspace .= | Show Org element kinds, fields, capture, and contract trace commands. |
| =asp org query --kind task --field todo=TODO --workspace . --view metadata= | Find TODO headlines as task facts. |
| =asp org query --kind checklistItem --field checked=false --workspace . --view metadata= | Find unchecked checklist items without collapsing them into tasks. |
| =asp org query --selector <structural-selector> --workspace . --content= | Print matched element content after a structural selector is known. |
| =asp org capture --contract agent.task.v1 --kind task --title <TITLE> --target-file <file.org>= | Render a non-mutating Org entry plan for review before writing. |
| =asp md guide --workspace .= | Show Markdown document query/search surfaces. |

** Provider Contract Rules
- Provider =guide= output is the command contract; do not infer unsupported axes.
- Dependency topology must include manifest, dependency edge, import owner,
  public API/docs use, test witness, and next action when the provider can
  prove them.
- Search output should be compact evidence, not source dumps.
- Treat =missing=, =evidenceGrade=unknown=, and =witness pending= as real
  frontier gaps.
- Keep language project parsing in the language provider. ASP routes and
  indexes; it does not reimplement package membership or source/test scope.
- Do not edit languages/org/templates/ASP_ORG_SKILL.org directly or patch the
  materialized project-state copy by hand. Change the governing contract,
  template, or Rust materialization source and rerun =asp sync= or the relevant
  =asp install ...= command so the skill is regenerated.
